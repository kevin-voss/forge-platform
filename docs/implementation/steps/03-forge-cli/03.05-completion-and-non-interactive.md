# Step 03.05: Shell completion + non-interactive mode

## Status

Not started

## Epic

`03-forge-cli` → [`../../epics/03-forge-cli.md`](../../epics/03-forge-cli.md)

## Goal

Add shell completion (bash/zsh/fish) via `forge completion <shell>` and harden non-interactive/CI usage: no TTY prompts, all input via flags/env/stdin, deterministic output, and a `--no-input` guarantee so the CLI is safe to script.

## Why this step exists

The spec lists shell completion and non-interactive mode as required features. The demo (03.06) and later automated demos run the CLI headless, so it must never block on prompts and completion must aid interactive use.

## Dependencies

* Prior steps in this epic: `03.01`–`03.04` (commands, config, output)
* Cross-epic prerequisites: none

## Scope

* `forge completion bash|zsh|fish` generating completion scripts
* Dynamic completion for known values where cheap (e.g. profile names from config)
* Non-interactive guarantees: no interactive prompts; missing required input → error, not a prompt
* `--no-input` flag (and CI detection) to fail fast rather than prompt
* Documented install instructions for completion

## Explicitly out of scope

* Server-side dynamic completion that requires network calls for every tab (keep to local/config-based completion; API-backed completion optional)
* Auth prompts / interactive login (epic 09)
* New resource commands

## Architecture

```text
forge completion zsh > _forge        # generated script
forge <tab>                          # framework completion + custom value providers

Non-interactive:
  detect: --no-input flag OR stdin not a TTY OR CI env
  behavior: never prompt; missing input → exit 2 with message
```

## Components

| Component | Purpose |
|---|---|
| `cmd/completion.go` | Emit completion scripts per shell |
| Completion value providers | Profiles from config; static enums (`--output`) |
| Non-interactive guard | Central check used before any would-be prompt |

## Contracts

No Control contract. CLI behavioral contract: in non-interactive mode the CLI never reads from a TTY; all required input comes from flags/env/stdin, and absence is an error.

## Data model

None.

## Configuration

```text
--no-input                 # force non-interactive; error instead of prompting
FORGE_NO_INPUT=1           # env equivalent
CI                         # if set, default to non-interactive
```

## Security considerations

* Completion scripts contain no secrets.
* Non-interactive mode avoids accidental credential prompts in logs/CI.

## Observability

* `completion` writes only the script to stdout (nothing else), so it can be redirected to a file.

## Failure handling

* Unknown shell arg to `completion` → exit `2` with supported list.
* Non-interactive + missing required input → exit `2` with the missing flag named.
