## Goal

Implement `forge deployment create` (record a desired deployment for a service) and `forge deployment status` (read a deployment's state) over Control's `02.05` endpoints, completing the CLI's coverage of the control-plane hierarchy.

## Why this step exists

Deployments are the developer's intent to run something. This command is the CLI's link to later runtime behavior: in epic 04 the same command produces a deployment that Runtime materializes. Delivering it now completes the CLI happy path against Control.

## Dependencies

* Prior steps in this epic: `03.02` (service commands, client, types), `03.01`
* Cross-epic prerequisites: Control `02.05` (deployments desired-state API)

## Scope

* `forge deployment create --service <id> --image <ref> [--replicas N] --env <environmentId>`
* `forge deployment status <deploymentId>` (single) and `forge deployment list --service <id>`
* Client methods for deployment create/get/list
* Idempotency: send an `Idempotency-Key` so retries do not duplicate (leveraging Control 02.06)

## Explicitly out of scope

* Actually running the container (epic 04)
* Watching/streaming status changes (Runtime 04.x / reconciler 07.x)
* Rollback / rolling update commands (epic 07)
* Log streaming (`forge logs`) — epics 04/12

## Architecture

```text
forge deployment create --service S --image localhost:5000/demo-go:latest --env E
   │ generate Idempotency-Key (uuid) unless provided
   ▼
control.Client.CreateDeployment(...) → POST /v1/services/{S}/deployments
   │
   ▼
print deployment id + status (pending)

forge deployment status <id> → GET /v1/deployments/{id} → render
```

## Contracts

```text
POST /v1/services/{serviceId}/deployments
  headers: Idempotency-Key: <uuid>
  body: {"image","desiredReplicas","environmentId"}
GET  /v1/deployments/{deploymentId}
GET  /v1/services/{serviceId}/deployments
```

## Data model

No local persistence. Deployment type mirrors Control DTO (`id`, `serviceId`, `environmentId`, `image`, `desiredReplicas`, `status`, timestamps).

## Configuration

Uses 03.01 config/flags. Optional `--idempotency-key` flag; auto-generated when absent.

## Security considerations

* No credentials on the deployment (local registry, dev mode).
* Auto-generated idempotency keys use a UUID; documented so scripts can supply their own for safe retries.

## Observability

* `--verbose` prints the request id and idempotency key used.
* `deployment status` prints `status` prominently for scripting (`--output json` exposes the raw field).

## Failure handling

* Unknown service/environment → Control `404` → readable error + non-zero exit.
* Invalid replicas/image → `400` → exit `2`.
* Retried create with same key returns the original deployment (no duplicate) and exit `0`.
