# Step 03.01: CLI skeleton, profiles, endpoint config, global flags

## Status

Not started

## Epic

`03-forge-cli` → [`../../epics/03-forge-cli.md`](../../epics/03-forge-cli.md)

## Goal

Create the `forge` Go CLI skeleton: a command tree with a root command, `version`, and a `config` group for managing named profiles and the Control endpoint, plus global flags (`--endpoint`, `--profile`, `--output`, `--timeout`, `--verbose`) and a config loader with clear precedence.

## Why this step exists

Every resource command (project/app/service/deployment) needs a consistent way to resolve which Control endpoint to call, which output format to use, and how to fail. Establishing config/profile loading and global flags first keeps later commands thin.

## Dependencies

* Prior steps in this epic: none (first step of epic 03)
* Cross-epic prerequisites: epic `00` tool template; a running Control (epic 02) is only needed for later steps, not to build the skeleton

## Scope

* Go module + `forge` binary entrypoint under `tools/forge-cli/`
* Command tree with `forge`, `forge version`, `forge config set|get|list|use`
* Named profiles stored in `~/.config/forge/config.yaml`
* Global persistent flags: `--endpoint`, `--profile`, `--output {table,json}`, `--timeout`, `--verbose`
* Config precedence: flag > env (`FORGE_ENDPOINT`/`FORGE_PROFILE`) > profile file > built-in default
* A thin HTTP client factory (base URL, timeout, request-id capture) shared by later steps
* Makefile targets (`build`, `test`, `install`)

## Explicitly out of scope

* Resource commands (`project`/`app`/`service`/`deployment`) — 03.02/03.03
* Output formatting rules and exit-code taxonomy details — 03.04 (basic wiring only here)
* Shell completion + non-interactive hardening — 03.05
* Auth/login — epic 09

## Architecture

```text
forge <group> <command> [flags]
   │
   ├── root: parse global flags, load config
   ├── config resolver: flag > env > profile file > default
   └── client factory: http.Client{ base=endpoint, timeout } + request-id capture

~/.config/forge/config.yaml
  current_profile: local
  profiles:
    local:   { endpoint: http://127.0.0.1:4001 }
    staging: { endpoint: https://control.staging.example }
```

## Components

| Component | Purpose |
|---|---|
| `main.go` | Entry; build root command; execute |
| `cmd/root.go` | Root command + persistent flags |
| `cmd/version.go` | Print version/build info |
| `cmd/config.go` | `config set/get/list/use` |
| `internal/config` | Load/save profiles; precedence resolver |
| `internal/client` | HTTP client factory + request-id extraction |

## Contracts

Consumes nothing yet beyond the Control base URL. Establishes the internal client contract: given a resolved endpoint + timeout, produce an `http.Client` and helpers that capture `X-Request-Id` from responses for later surfacing.

## Data model

CLI config file only:

```yaml
current_profile: local
profiles:
  local:
    endpoint: http://127.0.0.1:4001
```

No platform persistence.

## Configuration

```text
FORGE_ENDPOINT     # overrides profile endpoint
FORGE_PROFILE      # selects a named profile
FORGE_OUTPUT       # table|json (flag --output overrides)
FORGE_TIMEOUT      # e.g. 30s
Config file: $XDG_CONFIG_HOME/forge/config.yaml (default ~/.config/forge/config.yaml)
```

## Security considerations

* Config file created with `0600` permissions (it will later hold tokens).
* No secrets logged; `--verbose` must not print credentials.
* Endpoint URLs validated (scheme http/https) to avoid accidental misuse.

## Observability

* `--verbose` prints resolved endpoint/profile and request/response summaries (method, path, status, `requestId`) to stderr.
* Normal output goes to stdout; diagnostics to stderr (so `--output json` stays clean).

## Failure handling

* Missing profile / unknown profile → exit `2` with a clear message.
* Unwritable config path → exit `1` with the OS error.
* Invalid endpoint URL → exit `2`.
* Network errors are surfaced by later commands; the skeleton client returns typed errors.
