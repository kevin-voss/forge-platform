# Step 03.06: Demo `03-cli-control` + gate

## Status

Not started

## Epic

`03-forge-cli` → [`../../epics/03-forge-cli.md`](../../epics/03-forge-cli.md)

## Goal

Deliver `demos/03-cli-control`, which builds the `forge` CLI, points it at a running Control, and recreates the entire control-plane hierarchy — project, environment, application, service, deployment — using only CLI commands, verifying stable `--output json`, correct exit codes, and read-back, serving as the epic acceptance gate.

## Why this step exists

The spec defines a CLI demo that mirrors the Control demo using only CLI commands, and acceptance criteria (all Step 02 operations available via CLI, stable JSON output, useful errors, no DB access). This step ties every prior CLI step into one falsifiable end-to-end check.

## Dependencies

* Prior steps in this epic: `03.01`–`03.05` (full CLI)
* Cross-epic prerequisites: Control (epic 02, at least through `02.05`; `02.06` for idempotency); `make demo` runner

## Scope

* `demos/03-cli-control/` with `README.md` and `run.sh`
* Script: build CLI, start Control + Postgres, configure a profile, create the full hierarchy via CLI, read it back, assert JSON stability + exit codes
* `make demo DEMO=03` wiring
* Epic acceptance: assert all Step 03 criteria

## Explicitly out of scope

* Container execution of the deployment (epic 04)
* Auth/login flows (epic 09)
* Any new CLI commands

## Architecture

```text
make demo DEMO=03
   │
   ▼
demos/03-cli-control/run.sh
   ├── build forge (tools/forge-cli)
   ├── docker compose up postgres forge-control
   ├── forge config set endpoint … / use profile
   ├── forge project/app/service/deployment create (capture ids from --output json)
   ├── forge ... list/status → assert hierarchy
   ├── negative: forge project get <bogus> → assert exit code 3
   └── print success
```

## Contracts

Consumes the Control OpenAPI indirectly via the CLI. Asserts the CLI's stable JSON output contract and exit-code taxonomy from 03.04.

## Data model

None local. May target an isolated Control schema/DB for repeatable runs.

## Configuration

```text
FORGE_ENDPOINT=http://127.0.0.1:4001
FORGE_PROFILE=demo
CI=1                       # ensure non-interactive
```

## Security considerations

* Documents the dev auth bypass as temporary (Identity epic 09).
* No secrets; uses local Control and default profile.

## Observability

* `run.sh` echoes each command and a final pass/fail summary; dumps CLI stderr + Control logs on failure.

## Failure handling

* Any assertion failure → non-zero exit with the offending command/output.
* Control not ready in time → fail with logs.
* Cleanup on exit (trap): `docker compose down` / schema reset.

## Implementation tasks

1. Create `demos/03-cli-control/README.md`.
2. Write `run.sh`: build CLI, bring up Control + Postgres, wait for readiness.
3. Configure a CLI profile via `forge config`.
4. Create project→env→app→service→deployment via CLI, capturing ids from JSON output.
5. List/status to read the hierarchy back and assert shapes.
6. Add a negative case asserting exit code `3` on unknown id.
7. Wire `make demo DEMO=03`.
