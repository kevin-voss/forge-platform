# Step 03.04: Table/JSON output, exit codes, timeouts, request IDs

## Status

Not started

## Epic

`03-forge-cli` → [`../../epics/03-forge-cli.md`](../../epics/03-forge-cli.md)

## Goal

Harden the CLI's cross-cutting behavior: a consistent table renderer and stable `--output json`, a documented exit-code taxonomy, per-request timeouts with cancellation, and consistent surfacing of Control `requestId`s for debuggability.

## Why this step exists

Commands built in 03.02/03.03 need uniform, scriptable output and predictable failure semantics before the demo (03.06) asserts them. The spec explicitly requires stable machine-readable output, useful exit codes, timeouts, and request IDs.

## Dependencies

* Prior steps in this epic: `03.02`, `03.03` (commands exist), `03.01` (client/config)
* Cross-epic prerequisites: Control error envelope with `requestId` (02.06)

## Scope

* Shared `render` package: table (aligned columns) and JSON (stable field order/shape)
* Exit-code taxonomy applied across all commands
* Global `--timeout` honored via `context.Context` with cancellation on all HTTP calls
* `requestId` surfaced on every error (stderr) and on success with `--verbose`
* `--output json` emits raw Control resource JSON for single reads; a stable array for lists

## Explicitly out of scope

* New commands
* Completion / non-interactive (03.05)
* Retry/backoff policies (documented as future work)

## Architecture

```text
command → control.Client (ctx with --timeout)
   │ success → render(result, --output)
   │ error   → classify → exit code + message(+requestId)
   ▼
render:
  table  → tabwriter columns
  json   → stable marshaling (resource passthrough / array wrap)
```

Exit-code taxonomy:

```text
0  success
1  generic/unexpected error
2  usage / validation (bad flags, missing args)
3  not found (Control 404)
4  conflict (Control 409)
5  timeout / network
```

## Components

| Component | Purpose |
|---|---|
| `internal/render` | Table + JSON renderers |
| `internal/errmap` | Map Control error/HTTP → exit code + message |
| `internal/client` (extend) | Context/timeout + request-id extraction |
| Root command wiring | Apply timeout + output mode globally |

## Contracts

Reads Control's error envelope (`code`, `message`, `requestId`) to classify exit codes. `--output json` for a single resource passes through the Control resource body unchanged; lists are wrapped as a JSON array of resources.

## Data model

None.

## Configuration

```text
--timeout / FORGE_TIMEOUT   default 30s
--output  / FORGE_OUTPUT    table|json (default table)
--verbose                    print requestId on success too
```

## Security considerations

* JSON output goes to stdout only; diagnostics/verbose to stderr, so piped JSON stays clean.
* Never emit credentials in verbose logs.

## Observability

* On error, always print `requestId` (when present) to stderr to correlate with Control logs/traces.
* `--verbose` prints timing and `requestId` for successful calls.

## Failure handling

* Timeout/cancellation → exit `5` with a clear "request timed out" message.
* Control error envelope → mapped exit code per taxonomy.
* Unmarshalable/unexpected response → exit `1` with the raw status.

## Implementation tasks

1. Implement `internal/render` with table + JSON renderers and apply to all commands.
2. Define and implement the exit-code taxonomy in `internal/errmap`.
3. Thread `context.Context` with `--timeout` through all client calls; ensure cancellation.
4. Extract and surface `requestId` on errors (always) and successes (`--verbose`).
5. Standardize `--output json` shape (single vs list).
6. Document exit codes and output shapes in the CLI README.

## Manual verification

```bash
./forge --endpoint http://127.0.0.1:4001 project list --output json | python3 -c 'import sys,json;json.load(sys.stdin);print("json ok")'
./forge --endpoint http://127.0.0.1:4001 project get 00000000-0000-0000-0000-000000000000; echo "exit=$?"   # expect 3 (not found)
./forge --endpoint http://127.0.0.1:4001 --timeout 1ms project list; echo "exit=$?"                          # expect 5 (timeout)
```

## Acceptance criteria

* `--output json` produces stable, machine-parseable output for single resources and lists.
* Table output is aligned and human-readable.
* Exit codes follow the documented taxonomy (`0/1/2/3/4/5`).
* `--timeout` cancels in-flight requests and returns exit `5`.
* `requestId` is surfaced on errors and on `--verbose` successes.
