## Goal

Implement the resource commands `forge project create|list`, `forge app create|list`, and `forge service create|list` (plus `get` where useful) as thin clients over Control's `02.03`/`02.04` endpoints, using the shared config, client, and output plumbing from 03.01.

## Why this step exists

These commands cover the bulk of the developer happy path and produce the parent resources deployments need. They also exercise and validate the client/output conventions before the deployment commands (03.03).

## Dependencies

* Prior steps in this epic: `03.01` (skeleton, config, client)
* Cross-epic prerequisites: Control `02.03` (projects/environments) and `02.04` (applications/services) available

## Scope

* `forge project create --name [--slug]`, `forge project list`, `forge project get <id>`
* `forge env create --project <id> --name` (environments), `forge env list --project <id>`
* `forge app create --project <id> --name`, `forge app list --project <id>`
* `forge service create --app <id> --name --port`, `forge service list --app <id>`
* Typed Control API client methods for these resources
* Map Control error envelope → CLI error message + exit code

## Explicitly out of scope

* Deployment commands (03.03)
* Final output-format polish and exit-code taxonomy (03.04 — basic table/json here)
* Completion (03.05)
* Update/delete commands unless the underlying Control endpoint exists

## Architecture

```text
forge project create --name acme
   │ resolve endpoint (03.01)
   ▼
internal/control.Client.CreateProject(...)  → POST /v1/projects
   │
   ▼
render(result, --output)  → table or json
```

Commands accept parent ids by flag (`--project`, `--app`) and print created resource ids for scripting.

## Contracts

Consumes Control endpoints (see `contracts/openapi/forge-control.openapi.yaml`):

```text
POST /v1/projects                                   ← project create
GET  /v1/projects                                   ← project list
GET  /v1/projects/{id}                              ← project get
POST /v1/projects/{id}/environments                 ← env create
GET  /v1/projects/{id}/environments                 ← env list
POST /v1/projects/{id}/applications                 ← app create
GET  /v1/projects/{id}/applications                 ← app list
POST /v1/applications/{id}/services                 ← service create
GET  /v1/applications/{id}/services                 ← service list
```

## Data model

No local persistence. Types mirror Control DTOs (project, environment, application, service).

## Configuration

Uses 03.01 config/flags. No new configuration.

## Security considerations

* Sends the dev auth context expected by `FORGE_AUTH_MODE=dev` (no real credentials yet).
* Never prints request bodies containing anything sensitive (none in this epic).

## Observability

* `--verbose` logs each HTTP call (method, path, status, `requestId`) to stderr.

## Failure handling

* Control `404`/`409`/`400` → mapped to a readable stderr message + non-zero exit; `requestId` included.
* Network/timeout → distinct error message + non-zero exit.
* Missing required flags → usage error, exit `2`.
