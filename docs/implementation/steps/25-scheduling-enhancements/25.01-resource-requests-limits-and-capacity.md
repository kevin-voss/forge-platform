## Goal

Replace the abstract "slots" resource model with real CPU (millicores), memory (MB), and disk (MB) requests and limits, while keeping slots working — permanently — as a derived compatibility view. Nodes report real allocatable capacity from Runtime host introspection; an overcommit ratio is configurable per resource; limits are enforced on the container at creation time; and every rejected placement carries a structured, machine-readable unschedulable reason that epic 24's autoscaler consumes.

## Why this step exists

Epic 08 deliberately modeled capacity as "slots plus optional cpu/mem fields" and explicitly left "resource enforcement/cgroup limits inside Runtime" out of scope. The normative product manifest (`spec.resources: { cpu: 1000m, memory: 1024Mi }`) has been schedulable-looking since epic 06/07 but the scheduler never actually enforced it — a workload requesting 4 CPUs and one requesting 100m were indistinguishable to the epic-08 scheduler as long as both fit in "1 slot." This step makes `spec.resources` real: nodes stop over-committing on true resource usage, and every later step in this epic (labels, affinity, priority, GPU) filters and scores against real capacity instead of a placeholder unit.

## Scope

* `PlacementRequest.requirements` gains optional `requests: {cpu_millis, memory_mb, disk_mb}` and `limits: {cpu_millis, memory_mb, disk_mb}`, alongside the existing `slots` field
* Conversion rules so `slots`-only requests and real-unit requests are always mutually derivable (never both required, never double-counted)
* `nodes.capacity_json` gains real host capacity (`cpu_millis`, `mem_mb`, `disk_mb`) reported by Runtime introspection, replacing the demo-only `FORGE_NODE_SLOTS` as the source of truth for real deployments (kept as an override for deterministic demos/CI)
* `nodes.allocatable_json`: capacity × overcommit ratio − system-reserved headroom
* Capacity filtering in the scheduler generalizes the epic-08.03 `free.slots >= requirements.slots` check to also cover cpu/mem/disk when present
* Limit enforcement: Runtime applies CPU/memory/disk constraints at container creation when `limits` are present
* `placements.trace_json`: the structured, appendable explainability record every later step in this epic writes into — this step establishes its schema with the `capacity` filter
* Structured unschedulable reasons (`InsufficientCPU`, `InsufficientMemory`, `InsufficientDisk`, `InsufficientSlots`) surfaced on pending placements for epic 24 to consume

## Explicitly out of scope

* Labels, selectors, taints, architecture/OS constraints (`25.02`)
* Workload affinity and topology spread (`25.03`)
* Priority, preemption, disruption budgets (`25.04`)
* GPU requests, reservations, stateful pinning (`25.05`)

## Architecture

```text
PlacementRequest.requirements:
  { slots?: int, requests?: {cpu_millis,memory_mb,disk_mb}, limits?: {...} }

resolve(requirements):
  if requests present  -> use as-is; slots (if also present) is informational only
  if requests absent    -> derive requests from slots * FORGE_SLOT_{CPU_MILLIS,MEMORY_MB}
  # exactly one of the two is authoritative; never summed

CapacityAccounting (per node, on heartbeat):
  allocatable.cpu_millis    = capacity.cpu_millis * FORGE_OVERCOMMIT_CPU_RATIO    - reserved.cpu_millis
  allocatable.memory_mb     = capacity.mem_mb     * FORGE_OVERCOMMIT_MEMORY_RATIO - reserved.mem_mb
  allocatable.disk_mb       = capacity.disk_mb                                    - reserved.disk_mb   # never overcommitted

Scheduler.place(req) [extends 08.03]:
  candidates = fleet.online().filter(free.cpu_millis  >= resolved.requests.cpu_millis
                                   && free.memory_mb   >= resolved.requests.memory_mb
                                   && free.disk_mb     >= resolved.requests.disk_mb)
  trace.filters += { name: "capacity", eliminated: [...] }
  if candidates empty -> pending with reason=Insufficient<Resource>
  else -> 08.03 strategy picks among candidates (unchanged)

Runtime.createContainer(placement):
  if limits present -> apply Docker resource constraints (NanoCPUs, Memory, disk quota where supported)
  else              -> unconstrained (identical to pre-25.01 behavior)
```

## Components

| Component | Responsibility |
|---|---|
| `ResourceQuantity` | Parse/format `"1000m"` cpu and `"1024Mi"` memory strings to/from millis/MB |
| `RequirementsResolver` | Derive requests from slots or slots from requests; single source of truth per request |
| `CapacityAccounting` | Compute `allocatable` from `capacity`, overcommit ratios, and reserved headroom |
| `UnschedulableReason` | Enum + trace-entry builder for filter rejections |
| `PlacementTrace` | Appendable `{strategy, filters[], scores[]}` record persisted per placement |
| Runtime capacity reporter | Host introspection (CPU count, memory, disk) fed into heartbeat |
| Runtime limit enforcer | Applies `limits` to the Docker container spec at creation |

## Contracts

* `POST /v1/placements` request (extends `08.01`):

```json
{ "deployment_id": "dpl_1", "replica_index": 0,
  "requirements": {
    "requests": { "cpu_millis": 500, "memory_mb": 512, "disk_mb": 1024 },
    "limits":   { "cpu_millis": 1000, "memory_mb": 1024 }

```

* Pending response (extends `08.04`):

```json
{ "status": "pending",
  "reason": "InsufficientMemory: requested 512Mi, best node free 256Mi",
  "unschedulable_reasons": [
    { "node_id": "node-a", "reason": "InsufficientMemory", "detail": "requested 512, free 256" },
    { "node_id": "node-b", "reason": "InsufficientCPU", "detail": "requested 500, free 100" }
  ]
}
```

* New: `GET /v1/placements/{id}` → full record including `requests`, `limits`, and `trace` (additive; list endpoint from `08.01` unchanged)
* `GET /v1/nodes` response gains `allocatable` alongside existing `capacity`/`allocation`

## Data model

```sql
ALTER TABLE nodes
  ADD COLUMN allocatable_json JSONB NOT NULL DEFAULT '{}',
  ADD COLUMN reserved_json    JSONB NOT NULL DEFAULT '{}';
-- capacity_json already carries cpu_millis/mem_mb (08.02); disk_mb added as a new key, same column

ALTER TABLE placements
  ADD COLUMN requests_json JSONB,
  ADD COLUMN limits_json   JSONB,
  ADD COLUMN trace_json    JSONB;
```

All new columns nullable or defaulted; a placement row created under epic 08 (no `requests_json`) is read back exactly as before wherever new columns are not consulted.

## Configuration

```text
FORGE_SLOT_CPU_MILLIS               # units a slot is worth when only slots given (default 1000)
FORGE_SLOT_MEMORY_MB                # (default 1024)
FORGE_OVERCOMMIT_CPU_RATIO          # default 1.0 (no overcommit)
FORGE_OVERCOMMIT_MEMORY_RATIO       # default 1.0
FORGE_SYSTEM_RESERVED_CPU_MILLIS    # headroom subtracted from allocatable (default 100)
FORGE_SYSTEM_RESERVED_MEMORY_MB     # default 256
FORGE_ENFORCE_LIMITS                # default true; applies limits at container creation when present
FORGE_NODE_CAPACITY_SOURCE          # host | slots-only (default host; slots-only for deterministic demos/CI)
```

## Security considerations

* Overcommit ratios are platform-operator config, never settable from a product manifest (consistent with the manifest never containing machine-level facts)
* Limit enforcement uses Docker Engine's own resource-constraint primitives; no new privileged capability is requested from the Docker socket beyond what `04.01`/`04.03` already use
* Capacity/allocatable numbers carry no tenant-identifying data; safe to log and export as metrics

## Observability

* Log per placement: `{deployment_id, replica_index, requested, chosen_node|pending, reason}`
* Metrics: `forge_node_allocatable_cpu_millis{node}`, `forge_node_allocatable_memory_mb{node}`, `forge_placement_unschedulable_total{reason}`, `forge_limits_enforced_total{resource}`
* Trace span `scheduler.place` gains attributes `requested_cpu_millis`, `requested_memory_mb`; span `runtime.enforce_limits` around container creation

## Failure handling

* Runtime cannot introspect host resources (e.g. constrained sandbox) → falls back to `FORGE_NODE_SLOTS`-derived capacity with a logged warning, never crashes registration
* `limits` narrower than `requests` (a caller error) → rejected at `POST /v1/placements` with a typed validation error, not silently clamped
* Docker rejects a resource constraint (e.g. cgroup v1 without memory controller) → container creation fails explicitly; placement is marked `lost` and rescheduled via the `08.05` path, never left in a half-created state
* Overcommit ratio changed while nodes are running → recomputed on next heartbeat; never retroactively evicts existing placements
