## Goal

Generalize epic 08's "spread replicas of the same service across distinct nodes" into workload affinity/anti-affinity against arbitrary selectors, scoped by a topology key (`node`, `zone`, `region`, `provider`), with hard (`required`) and soft (`preferred`) variants, plus a `minimumDistinctNodes` spread constraint with deterministic scoring. This is what makes the epic's headline HA flow true: 3 replicas land on 3 distinct nodes across 2 zones, and losing one node loses exactly one replica.

## Why this step exists

`08.04`'s anti-affinity is hard-coded to "same service, node granularity only." Real placement needs to express "keep the cache near the API" (affinity) and "never put two database replicas in the same zone" (anti-affinity at a coarser topology level than node), and to reason about spread quantitatively (a score), not just as a binary same-node check. Without topology-aware spread, "multi-node" and "highly available" are not the same claim — two nodes in the same rack or zone can fail together. This step is the one that makes them the same claim.

## Scope

* `nodes.zone`, `nodes.region`, `nodes.provider` columns — topology facts, set the same way labels are (`NodePool` or `FORGE_NODE_ZONE`/`FORGE_NODE_REGION`/`FORGE_NODE_PROVIDER`), all defaulting to `'default'`/`'docker'` so a single-domain fleet behaves exactly as under epic 08
* `placement.affinity.workload`: `requiredDuringScheduling` / `preferredDuringScheduling`, each an array of `{selector: {service|labels}, topologyKey}`
* `placement.topologySpreadConstraints`: array of `{topologyKey, minimumDistinctNodes, maxSkew, whenUnsatisfiable: DoNotSchedule|ScheduleAnyway}`
* Generalizes `08.04`'s `anti_affinity: soft|hard` into a `topologyKey=node` anti-affinity constraint expressed the same way as any other topology key — the old field keeps working as sugar for it
* Spread scoring: a score component added to `08.03`'s least-allocated pass rewarding candidates in under-represented topology domains
* `placements.affinity_json`, `placements.topology_spread_json` — persisted for audit/explainability

## Explicitly out of scope

* Priority-aware spread (e.g. "high-priority workloads may violate spread to place") — spread and priority interact only through preemption in `25.04`, never by silently relaxing spread
* Rebalancing already-placed replicas to improve spread after the fact (placement-time only, same restriction as epic 08)
* Cross-region latency-aware scoring (epic `39-multi-region`)

## Architecture

```text
placement.affinity.workload.requiredDuringScheduling:
  [ { selector: {service: "cache"}, topologyKey: "zone" } ]   # must land in the same zone as a "cache" replica

placement.topologySpreadConstraints:
  [ { topologyKey: "node", minimumDistinctNodes: 3, whenUnsatisfiable: "DoNotSchedule" },
    { topologyKey: "zone", minimumDistinctNodes: 2, whenUnsatisfiable: "DoNotSchedule" } ]

Scheduler.place(req) [extends 25.02 trace]:
  candidates = (25.01 capacity) ∩ (25.02 selector/platform/taint) filters
  candidates = candidates.filter(satisfiesRequiredAffinity(req.affinity.required))
  trace.filters += { name: "workload_affinity", eliminated: [...] }
  candidates = candidates.filter(satisfiesTopologySpread(req.topologySpreadConstraints, hard-only))
  trace.filters += { name: "topology_spread", eliminated: [...] }
  if candidates empty -> pending (whenUnsatisfiable=DoNotSchedule) | ignore constraint (ScheduleAnyway)
  scores = candidates.map(node -> 08.03Score(node) + spreadScore(node, existingPlacements) + softAffinityScore(node))
  trace.scores = scores
  pick = scores.maxBy(score, tie -> lowest node id)   # deterministic

HA flow (topologyKey=node minDistinctNodes=3, topologyKey=zone minDistinctNodes=2, replicas=3):
  fleet: node-a,node-b (zone-a), node-c,node-d (zone-b)
  placement 1 -> node-a (zone-a)          # no existing replicas, any candidate scores equally by spread
  placement 2 -> node-c (zone-b)          # spread score favors the empty zone over node-b (same zone as #1)
  placement 3 -> node-b or node-d          # both node-level-distinct and zone-balanced; tie -> lowest id
  → 3 distinct nodes, both zones represented
  node hosting one replica dies -> 08.05 reschedule requests a fresh placement
  → candidates exclude the 2 nodes already running a replica (node-level hard constraint) and the dead node
  → only the 4th, previously-unused node qualifies -> exactly one replica ever left the fleet
```

## Components

| Component | Responsibility |
|---|---|
| `WorkloadAffinityFilter` | Eliminate nodes failing `requiredDuringScheduling` affinity/anti-affinity |
| `TopologySpreadFilter` | Eliminate nodes violating a hard (`DoNotSchedule`) spread constraint |
| `SpreadScorer` | Score component rewarding under-represented topology domains |
| `SoftAffinityScorer` | Score component from `preferredDuringScheduling` matches |
| `TopologyResolver` | Read a node's value for a given `topologyKey` (`node`→id, `zone`, `region`, `provider`) |

## Contracts

* `POST /v1/placements` request gains:

```json
{ "placement": {
    "affinity": {
      "workload": {
        "requiredDuringScheduling": [ { "selector": { "service": "svc_cache" }, "topologyKey": "zone" } ],
        "preferredDuringScheduling": [ { "weight": 50, "selector": { "service": "svc_api" }, "topologyKey": "node" } ]
```

* `GET /v1/nodes` response gains `zone`, `region`, `provider`

## Data model

```sql
ALTER TABLE nodes
  ADD COLUMN zone     TEXT NOT NULL DEFAULT 'default',
  ADD COLUMN region    TEXT NOT NULL DEFAULT 'default',
  ADD COLUMN provider  TEXT NOT NULL DEFAULT 'docker';

ALTER TABLE placements
  ADD COLUMN affinity_json         JSONB,
  ADD COLUMN topology_spread_json  JSONB;
```

Every existing node defaults to the single implicit domain `zone='default'`/`region='default'` — a fleet that never sets these collapses spread scoring back to plain least-allocated, matching epic 08 exactly.

## Security considerations

* Affinity selectors only match within the requesting workload's own project/environment scope — cross-project affinity is not possible (respects the isolation axis from the cross-cutting requirements)
* Topology values are operator-set facts, not attacker-controlled input; no injection surface beyond standard label validation from `25.02`

## Observability

* Log per placement: `{affinity_required_matched, spread_constraints_checked, chosen_topology}`
* Metrics: `forge_topology_spread_violations_prevented_total{topology_key}`, `forge_affinity_fallback_total` (soft affinity that couldn't be satisfied), `forge_spread_score{node,topology_key}` (gauge, last computed)
* Trace span `scheduler.place` gains `topology_keys_evaluated`, `distinct_nodes_used`, `distinct_zones_used` attributes

## Failure handling

* Hard `requiredDuringScheduling` unsatisfiable → `pending` with reason `AffinityUnsatisfiable` (never falls back to soft silently)
* Hard `topologySpreadConstraints` unsatisfiable with `whenUnsatisfiable=DoNotSchedule` → `pending`, reused `08.04` queue drains it once a qualifying node appears (new node registered, or a competing placement stops)
* `whenUnsatisfiable=ScheduleAnyway` → constraint ignored for scoring purposes only, recorded in trace as `spread_relaxed: true` (visible, never silent)
* Selector references a service with zero existing replicas → treated as "no matches," not an error (affinity simply contributes nothing to filtering/scoring yet)
