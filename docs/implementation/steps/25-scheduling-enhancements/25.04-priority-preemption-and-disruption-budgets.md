## Goal

Add a `PriorityClass` resource so workloads can outrank each other for scarce capacity, preemption so a high-priority placement can evict a lower-priority victim (gracefully, with a full audit trail) instead of only queueing, and disruption budgets so voluntary disruptions — drains, rolling updates, and preemption itself — never remove more of a deployment's replicas than the operator allows.

## Why this step exists

Epic 08's pending queue (`08.04`) is fair but priority-blind: a critical workload waits in the same FIFO line as a batch job when the cluster is full. Real capacity-constrained fleets need a way to say "this matters more" and have the scheduler act on it — by making room, not just by waiting longer. And once eviction is possible at all (preemption here, drains in epic 24, rollouts in epic 07/27), there needs to be a floor under how much of a deployment can be disrupted at once, or a burst of voluntary actions could take a service fully down even though every individual action looked safe.

## Dependencies

* Step `25.01` — real resource accounting (preemption needs to know exactly how much capacity a victim frees)
* Step `25.03` — topology spread (a rescheduled, preempting replica must still satisfy the same spread constraints as any other placement)
* Epic `08-multi-node-scheduler`, step `08.04` — pending queue, which preemption now tries *before* falling back to
* Epic `08-multi-node-scheduler`, step `08.05` — reschedule-on-node-loss, whose *involuntary* path this step explicitly does not touch

## Scope

* `PriorityClass` cluster-scoped resource: `name`, `value` (int, higher outranks lower), `preemptionPolicy: PreemptLowerPriority|Never`, `description`; a built-in `default` class (`value=0`, `Never`) that every pre-existing placement implicitly uses
* `spec.priorityClassName` on a workload (optional; omission = `default`, zero behavior change)
* Preemption: when no node satisfies a `PreemptLowerPriority` request's filters/capacity, the scheduler looks for the minimal set of strictly-lower-priority victims on a topologically-eligible node, gracefully evicts them, and places the preempting replica — tried *before* enqueueing to the pending queue
* Graceful eviction reuses Runtime's existing stop-with-grace lifecycle; evicted victims are themselves re-submitted as fresh placement requests (reschedule-or-pending, same as `08.05`)
* Disruption budgets: `min_available` or `max_unavailable` per deployment, checked by every *voluntary* removal path (drain, rolling update, preemption-as-victim); never checked on the *involuntary* node-loss path (`08.05`)
* Preemption audit trail: durable, queryable record of every eviction — victim, preemptor, priorities, node, reason
* Anti-starvation: pending items age; after a configurable duration a stuck low-priority item's effective queue position is boosted so a constant stream of higher-priority arrivals cannot starve it forever

## Explicitly out of scope

* Preempting equal-priority workloads (never happens — avoids order-dependent flapping)
* GPU-aware preemption (`25.05` adds GPU as a resource dimension; this step's victim selection is priority + capacity only, dimension-agnostic)
* Disruption budgets applying to a direct, explicit operator delete of a specific replica (an explicit action is not the kind of accidental cascading disruption a budget protects against)
* Multi-level preemption chains (preempting a victim that would itself need to preempt something else) — a single pass only; if one pass doesn't free enough capacity, fall back to pending

## Architecture

```text
priority_classes: default(0,Never) is seeded; operators add more (e.g. high=100 PreemptLowerPriority, low=-10 Never)

Scheduler.place(req) [extends 25.03]:
  candidates = (25.01..25.03 filters)
  if candidates nonempty -> 08.03 strategy + 25.03 scoring picks as before (priority irrelevant when capacity exists)
  else if req.priorityClass.preemptionPolicy == PreemptLowerPriority:
      victims = findMinimalVictims(fleet, req)   # strictly lower priority.value, topologically eligible node,
                                                   # smallest set whose freed capacity satisfies req
      if victims found AND all(v -> disruptionBudgetAllows(v.deployment)):
          evict(victims, grace=FORGE_PREEMPTION_GRACE_S)   # graceful stop; victims re-submitted as new placements
          record preemption_events row per victim
          place req on the freed node
      else -> enqueue (08.04 pending path, unchanged)
  else -> enqueue (08.04 pending path, unchanged)

Voluntary removal (drain [epic 24], rolling update [epic 07/27], preemption victim):
  disruptionBudgetAllows(deployment) = (running - 1) >= min_available  OR  (unavailable + 1) <= max_unavailable
  if false -> action deferred/rejected, never forced through

Pending queue aging (extends 08.04 QueueProcessor):
  each tick: pending.age > FORGE_PENDING_STARVATION_S -> effective_priority += 1 tier (bounded, logged)
```

## Data model

```sql
CREATE TABLE priority_classes (
  name              TEXT PRIMARY KEY,
  value             INT  NOT NULL,
  preemption_policy TEXT NOT NULL DEFAULT 'Never',   -- Never | PreemptLowerPriority
  description       TEXT,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
INSERT INTO priority_classes(name, value, preemption_policy, description)
  VALUES ('default', 0, 'Never', 'Implicit class for placements created before epic 25');

CREATE TABLE disruption_budgets (
  deployment_id   TEXT PRIMARY KEY REFERENCES deployments(id),
  min_available   INT,
  max_unavailable INT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE placements
  ADD COLUMN priority_class TEXT NOT NULL DEFAULT 'default' REFERENCES priority_classes(name),
  ADD COLUMN preempted_by_placement_id TEXT;

CREATE TABLE preemption_events (
  id                     TEXT PRIMARY KEY,
  victim_placement_id    TEXT NOT NULL,
  preemptor_placement_id TEXT NOT NULL,
  victim_priority        INT  NOT NULL,
  preemptor_priority     INT  NOT NULL,
  node_id                TEXT NOT NULL,
  reason                 TEXT NOT NULL,
  created_at             TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Every pre-existing placement defaults to `priority_class='default'` with `preemption_policy='Never'` — it can neither trigger nor suffer preemption until an operator opts a workload into a different class.

## Configuration

```text
FORGE_PREEMPTION_ENABLED       # default true (global kill switch)
FORGE_PREEMPTION_GRACE_S       # graceful stop window for a victim (default 10)
FORGE_PENDING_STARVATION_S     # age at which a pending item's effective priority is boosted (default 300)
FORGE_DEFAULT_PRIORITY_CLASS   # default 'default'
```

## Security considerations

* Only a platform operator can create/modify `PriorityClass` (cluster-scoped, same trust tier as `Node`/`NodePool`); a project cannot grant itself a higher-priority class than the operator allows
* Preemption never crosses project/environment isolation — a victim search is scoped to the same fleet-wide candidate set the requesting placement would itself be filtered against, but priority comparison and eviction only ever act on placements the platform already tracks; no cross-tenant data is exposed by the audit trail beyond ids already visible via the placements API
* Disruption budget bypass is not possible via any voluntary path; it is explicitly not applied to involuntary node loss (documented, not a loophole — a dead node cannot honor a budget)

## Observability

* Log every preemption: `{event=preemption, victim_placement, preemptor_placement, victim_priority, preemptor_priority, node}`
* Log every budget check: `{event=disruption_budget_check, deployment, allowed, min_available, running}`
* Metrics: `forge_preemptions_total{victim_priority,preemptor_priority}`, `forge_disruption_budget_blocked_total{deployment}`, `forge_pending_aged_total`
* Trace span `scheduler.preempt` with attributes `victims_count`, `node`, `freed_cpu_millis`, `freed_memory_mb`

## Failure handling

* No victim set frees enough capacity → falls back to the `08.04` pending path, never partially evicts
* Disruption budget would be violated by the only viable victim → preemption declined for that node; scheduler tries the next candidate node or falls back to pending
* Evicted victim can't be rescheduled (no capacity anywhere) → itself goes `pending`, visible and audited, not silently dropped
* Control restart mid-preemption → `preemption_events` + placement `status` are the durable record; recovery re-derives from persisted state, same discipline as `08.05`
* A priority class is deleted while placements still reference it → rejected (referential integrity via FK); classes are deactivated, never hard-deleted while in use

## Unit tests

* `PreemptionSelector`: given one high-priority request and multiple lower-priority candidates, picks the smallest victim set that frees enough capacity; never selects an equal-or-higher-priority victim
* `DisruptionBudgetGuard`: `min_available=2` on a 2-replica-running deployment blocks a third voluntary removal; `max_unavailable=1` allows exactly one concurrent removal
* Preemption respects topology spread: a victim search excludes nodes that would violate the preemptor's own `minimumDistinctNodes` constraint
* `PendingAgingPolicy`: an item older than `FORGE_PENDING_STARVATION_S` is boosted exactly once per threshold crossing, deterministically

## Integration tests

* Full node with a low-priority replica; deploy a `PreemptLowerPriority` high-priority replica requiring the same node's capacity → low-priority replica evicted (graceful), high-priority replica placed, `preemption_events` row recorded
* Disruption budget on the low-priority deployment (`min_available` equal to its current running count) → preemption of its last replica is declined; preemptor falls back to pending
* Rolling update (`07.03`) with a budget set → old-version replicas removed no faster than the budget allows, verified via deployment history timestamps
* Evicted victim reschedules automatically once capacity exists elsewhere; if it never does, it surfaces as `pending`

## Acceptance criteria

* `PriorityClass` is a real, cluster-scoped resource; every placement resolves to one, defaulting to `default` (`Never`) with zero behavior change for existing workloads
* A `PreemptLowerPriority` placement that cannot otherwise fit evicts the minimal strictly-lower-priority victim set on a topologically-eligible node, gracefully, and is placed
* Disruption budgets are honored by every voluntary removal path (preemption here; drains in epic 24; rollouts in epic 07/27) and never checked on involuntary node loss
* Every preemption is durably audited: victim, preemptor, priorities, node, reason
* Long-stuck pending placements are eventually promoted (anti-starvation) rather than waiting forever behind a constant stream of higher-priority arrivals
