## Goal

Make WireGuard optional rather than mandatory: nodes that already share a Docker daemon
(`docker` mode) or a provider's private network (`provider-private` mode) reach each other
directly, and Forge Network only sets up a WireGuard tunnel (`wireguard` mode, `22.03`) for
node pairs that share neither. A mixed cluster picks the transport **per node pair**, and a
workload's experience — resolve a name, connect — is identical regardless of which
transport carries the packets.

## Why this step exists

`22.03` can wire every node to every other node over WireGuard, but that is unnecessary
overhead (and an extra thing to keep converged) when two nodes are containers on the same
Docker host, or VMs already sitting on the same Hetzner/AWS/Azure private network. The
portability contract requires identical behavior across install targets without forcing
the most expensive transport everywhere; this step is what makes `docker` and
`provider-private` first-class instead of "WireGuard everywhere, always."

## Dependencies

* Step `22.01` — address plan (all three modes still assign addresses from it)
* Step `22.03` — WireGuard, used only for pairs that resolve to `wireguard`

## Scope

* Three transport modes: `docker`, `provider-private`, `wireguard`
* `docker`: nodes are containers on one Compose network; Forge Network still assigns
  overlay addresses (via a per-node Docker user-defined network with a matching
  `--subnet`) for naming consistency; no WireGuard interface starts
* `provider-private`: overlay addresses are still assigned by Forge Network; instead of a
  tunnel, each node adds a route for peer `/24`s via its private-network NIC (the
  provider's private network already carries L3 traffic)
* Per-node `network_membership` tag (e.g. `hetzner-private-fsn1`); two nodes with the same
  non-empty tag use `provider-private`; nodes on the same Docker daemon (detected via a
  colocation flag) use `docker`; everything else uses `wireguard`
* Per-pair transport computation, exposed for observability/debugging
* Identical reachability contract: workloads always resolve peers by internal DNS name
  (`22.06`) to an overlay IP; the transport underneath is invisible above the network layer

## Explicitly out of scope

* Automatic fallback when a declared `network_membership` turns out not to actually route
  (see Failure handling — this is a surfaced, operator-fixed condition, not a silent
  failover)
* `NetworkPolicy` (`22.05`)
* DNS integration itself (`22.06`)

## Architecture

```text
For each ordered pair (from_node, to_node):
  same docker-colocation flag?          → transport = docker
  else same non-empty network_membership? → transport = provider-private
  else                                    → transport = wireguard

docker            : per-node /24 realized as a Docker user-defined network; direct L2/L3
provider-private   : route added via the node's private NIC to the peer's /24; no tunnel
wireguard          : 22.03's mesh; encrypted tunnel per pair
```

## Components

| Component | Responsibility |
|---|---|
| `TransportResolver` | Pure function: (node A membership, node B membership, colocation) → transport |
| `network_routes` cache | Debug/observability record of computed transport per pair |
| Runtime `network/route.rs` | Applies `provider-private` routes (no tunnel) or hands off to `wireguard.rs` |
| `NodeMembershipRoute` | `PATCH /v1/nodes/{id}/network-membership` |

## Contracts

* `PATCH /v1/nodes/{id}/network-membership`:

```json
{ "membership": "hetzner-private-fsn1" }
```

* `GET /v1/networks/{name}/transport?from=node-a&to=node-b`:

```json
{ "from": "node-a", "to": "node-b", "transport": "provider-private" }
```

## Data model

```sql
ALTER TABLE nodes
  ADD COLUMN network_membership TEXT,       -- null = no shared provider network declared
  ADD COLUMN docker_colocated   BOOLEAN NOT NULL DEFAULT false;

CREATE TABLE network.network_routes (
  network_id  TEXT NOT NULL,
  from_node   TEXT NOT NULL,
  to_node     TEXT NOT NULL,
  transport   TEXT NOT NULL,   -- docker | provider-private | wireguard
  computed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (network_id, from_node, to_node)
);
```

`network_routes` is a computed cache for observability/debugging, not authoritative state
— `TransportResolver` is a pure function and can always regenerate it.

## Configuration

```text
FORGE_NETWORK_MODE_DEFAULT       # cluster-level fallback when no membership is declared; default wireguard
FORGE_NODE_NETWORK_MEMBERSHIP    # set per node/NodePool, e.g. hetzner-private-fsn1
FORGE_NODE_DOCKER_COLOCATED      # demo/dev convenience flag; the compose harness sets this true
```

## Security considerations

* `provider-private` routes ride inside the provider's own private network, which Forge
  does **not** encrypt — documented explicitly as a trust assumption consistent with most
  providers' own private-network guidance
* `wireguard` pairs are always encrypted regardless of mode elsewhere in the cluster
* Mixed clusters therefore have mixed per-hop encryption guarantees; this is stated
  plainly in the epic's portability contract rather than implied, so operators can decide
  whether a given install target's private network meets their requirements
* `network-membership` can only be set by platform-authenticated calls (epic `09`); a
  workload cannot influence its own node's declared membership

## Observability

* Metric: `forge_network_transport_pairs_total{transport=docker|provider-private|wireguard}`
* Log the chosen transport whenever it changes for a pair (membership edit, colocation
  flag change, node join/leave)
* `GET .../transport` used directly for debugging a specific pair

## Failure handling

* A node declares a `network_membership` it cannot actually route to (misconfiguration) →
  the connectivity probe introduced in `22.06` fails and the pair's status surfaces
  `False` with reason `TransportUnreachable`; the operator corrects the membership tag or
  forces `wireguard` for that pair explicitly — **no automatic silent fallback**, to avoid
  masking a misconfigured private network
* `docker_colocated` incorrectly set true for nodes that are not actually on the same
  daemon → same `TransportUnreachable` surfacing, same manual correction path
