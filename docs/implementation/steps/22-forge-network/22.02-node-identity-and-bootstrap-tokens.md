## Goal

Give every Runtime node a join sequence: it authenticates to Control with a single-use,
scoped, expiring bootstrap token; generates its own key pair locally (private key never
leaves the node); and, once Control verifies the token, receives a network assignment from
`forge-network` (`22.01`) before it is marked schedulable. This is the node registration
flow the epic exists to deliver: *VM starts → Runtime receives bootstrap token → Runtime
generates node key pair → Control verifies token → Forge Network assigns node address →
WireGuard peers distributed → node joins → node becomes schedulable.*

## Why this step exists

`04.02` gave a node a stable local identity; `08.02` let any node calling `POST
/v1/nodes/register` join the fleet — fine when every node is a trusted container on the
developer's own machine, not fine once a node is a VM on someone else's cloud reachable
over the public Internet. A join needs proof the node is allowed to be here (the bootstrap
token) before it gets a network identity (a key pair) and a place in the overlay (an
address + peers). This step is the seam every later join-related feature (autoscaler node
provisioning `24`, HA control plane `35`) calls into.

## Scope

* Bootstrap tokens: Control-issued, single-use, scoped to an organization and optional
  `NodePool`, expiring (default 15 minutes)
* Node-side key generation: an X25519 key pair generated on first start, persisted
  alongside the existing node id (`04.02`'s data dir), private key never included in any
  API payload or log
* Extended registration: `POST /v1/nodes/register` accepts `bootstrap_token` and
  `wireguard_public_key`; Control verifies and consumes the token, then calls
  `forge-network` for a node-block lease and an initial peer set
  before responding
* Join-sequence node statuses: `pending-network` → `joining` → `online` (additive to
  `08.02`'s `status` enum)
* Node certificate issuance seam: a documented, currently-stubbed extension point
  (`IssueNodeCertificate(nodeId, publicKey) → cert`) for epic `34`'s internal CA
* Revocation: revoke an unconsumed bootstrap token (blocks future registration); separately,
  remove a node's WireGuard key from the peer registry (evicts an already-joined node)
* Re-join: a node restarting with its persisted node id and key pair resumes via heartbeat
  alone (no new token needed); a node whose key was evicted needs a fresh bootstrap token
  to rejoin

## Explicitly out of scope

* WireGuard peer *computation* beyond an initial peer set returned at join
  (full peer management is `22.03`)
* Network mode / transport selection (`22.04`)
* `NetworkPolicy` (`22.05`)
* Actual certificate issuance (epic `34`) — seam only
* CLI command for token issuance (assumption: Control API only in this epic; a CLI
  subcommand can call it later without a contract change)

## Architecture

```text
VM starts
  │
  ▼
Runtime: load or generate node id (04.02) + X25519 key pair (this step; private key stays on disk, 0600)
  │
  ▼
Runtime → Control: POST /v1/nodes/register { node_id, bootstrap_token, wireguard_public_key, capacity }
  │
  ▼
Control: verify token (unexpired, unconsumed, scope matches) → consume it (atomic)
  │  status = pending-network
  ▼
Control → forge-network: POST /v1/networks/{name}/node-leases { node_id }
  │                       (22.01 address plan)
  ▼
forge-network → Control: { cidr, gateway }
  │  status = joining
  ▼
Control → Runtime (register response): { network: {cidr, gateway}, peers: [...] }
  │
  ▼
Runtime: apply network + peers (interface config lands fully in 22.03)
  │
  ▼
Runtime heartbeats normally (08.02) → Control flips status = online (schedulable)
```

## Data model

```sql
ALTER TABLE nodes
  ADD COLUMN wireguard_public_key TEXT,
  ADD COLUMN network_cidr         CIDR,
  ADD COLUMN network_gateway      INET,
  ADD COLUMN joined_at            TIMESTAMPTZ;
-- status enum gains 'pending-network', 'joining' alongside existing
-- 'online' | 'offline' | 'draining' (08.02); no existing value removed

CREATE TABLE bootstrap_tokens (
  id               TEXT PRIMARY KEY,
  token_hash       TEXT NOT NULL UNIQUE,   -- sha256, never store plain
  organization     TEXT NOT NULL,
  node_pool        TEXT,
  expires_at       TIMESTAMPTZ NOT NULL,
  consumed_at      TIMESTAMPTZ,
  consumed_by_node TEXT,
  revoked_at       TIMESTAMPTZ,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX bootstrap_tokens_single_use
  ON bootstrap_tokens (id) WHERE consumed_at IS NULL AND revoked_at IS NULL;
```

## Security considerations

* Private key never leaves the node: generated in Runtime, written `0600`, never appears
  in an API request/response body or a log line
* Tokens are hashed at rest (same treatment as session/API tokens in epic `09`) and
  single-use: consumption is an atomic `UPDATE ... WHERE consumed_at IS NULL` guarded by
  the partial unique index, so a race between two registration attempts with the same
  token can only succeed once
* Token scope (organization/`NodePool`) restricts which nodes can register under it;
  registration with a token outside the requester's scope is rejected
* Revoking a node's key is independent of revoking a token — a node can be evicted from
  the network without touching token history, and vice versa

## Observability

* Log join lifecycle transitions: `{node_id, from_status, to_status}` at each of
  `pending-network → joining → online`
* Metrics: `forge_node_join_total{result=success|invalid_token|expired_token|no_address}`,
  `forge_node_join_duration_seconds`
* Trace span `node.join` spanning register → verify → lease → respond

## Failure handling

* Expired, consumed, or revoked token → `401` typed `InvalidBootstrapToken`; Runtime
  retries registration with a freshly issued token (no crash loop)
* `forge-network` unreachable during join → node stays `pending-network` (distinct from
  `offline` in the fleet view), Control retries the lease call with backoff
* Restart with a persisted identity and no revocation → registration is skipped entirely;
  the node resumes via heartbeat with its existing `network_cidr`/peers intact
* Evicted node attempting to heartbeat → rejected until it re-registers with a fresh
  bootstrap token

## Acceptance criteria

* Bootstrap tokens are single-use, scoped, and expiring; a consumed or expired token is
  rejected
* Registration without a valid token fails; registration with a valid token assigns a
  network lease and moves the node to `joining`
* Node private keys are generated on the node and never appear in any API payload or log
* A restarted node with unrevoked identity resumes without a new token
* An evicted node cannot heartbeat again until it registers with a new bootstrap token
