## Goal

Connect the Forge overlay network to Discovery's internal DNS so workloads can resolve and reach services on other nodes through stable `.svc.forge` names, independent of whether the nodes are Docker containers, local VMs, or cloud VMs.

## Why this step exists

Steps `22.01`-`22.05` create addresses, node identity, WireGuard peers, and policy enforcement. This step makes those pieces usable by applications: DNS answers must point at routable overlay addresses, policy must still apply, and endpoint health must drive records.

## Dependencies

* `21.04` - internal DNS zone and naming convention.
* `22.03` - WireGuard peer mesh and route distribution.
* `22.05` - policy enforcement.

## Scope

* Runtime DNS resolver configuration so workloads use Forge DNS for `.svc.forge`.
* Discovery endpoint records include overlay addresses, not provider public IPs.
* Split-horizon behavior: `.svc.forge` stays internal; public domains remain Gateway/DNS epic 34.
* Cross-node connectivity test app.
* Drift check comparing Discovery endpoints, Network leases, and Runtime routes.

## Explicitly out of scope

* Public DNS, ACME, and customer domains.
* Cross-region routing.
* Service mesh L7 policy.

## Contracts

```text
users.production.shop.svc.forge -> A/AAAA records for Ready overlay endpoints
```

Runtime node DNS configuration:

```text
nameserver <forge-dns-overlay-ip>
search production.shop.svc.forge
```

## Observability

* Metric `forge_network_dns_resolution_total{result}`.
* Metric `forge_network_route_drift_total`.
* Span `network.dns.resolve`.
* Log drift with `{endpoint_id, expected_overlay_ip, observed_route}`.

## Failure handling

* DNS unavailable -> Runtime keeps existing resolver config and marks node `Degraded`.
* Endpoint has no current overlay lease -> Discovery excludes it from answers.
* Policy denies cross-service traffic -> DNS still resolves; connection is denied and logged by `22.05`.
