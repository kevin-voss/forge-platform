## Goal

Introduce the `NetworkPolicy` kind — target an application, allow ingress from named
services, allow egress to named databases/queues on specific ports — enforced on each node
by Runtime and observable when a connection is denied. Segmentation is layered:
cross-environment traffic is always denied; within an environment, an explicit
`NetworkPolicy` narrows what the environment's own default (`allow-within-environment` or
`deny-all`) permits.

## Why this step exists

Every prior step in this epic makes nodes reachable; nothing yet says which workloads are
*allowed* to reach which other workloads. Without `NetworkPolicy`, a flat overlay is a flat
trust domain — any workload in any project could otherwise reach any other. This step adds
the org/project/environment segmentation the cross-cutting requirements call for on the
network axis specifically, and gives operators an auditable, denyable, observable
allow-list model instead of relying on the transport layer alone.

## Scope

* `NetworkPolicy` kind (project/environment-scoped): `target.application`, `ingress[].from`
  (service), `egress[].to` (database/queue), `ports` per rule
* Per-environment default: `allow-within-environment` (default) or `deny-all`
* Cross-environment traffic denied unconditionally, regardless of any `NetworkPolicy`
* Compilation: `forge-network` cross-references scheduler placements (epic `08`) + workload
  addresses (`22.01`) + policy specs into a per-node rule set
* Enforcement point: Runtime's network module applies the rule set via nftables, scoped to
  each workload's overlay address
* Policy evaluation order, explicit and documented
* Observability of denies: counted at the enforcement point, sampled into detailed logs,
  emitted as a platform event

## Explicitly out of scope

* A general admission/policy engine beyond L3/L4 allow/deny by service and port (epic
  `33-forge-policy`)
* Policy for non-network concerns (resource quotas, image provenance, etc.)
* UI for authoring policies (epic `40-forge-console`)

## Architecture

```text
NetworkPolicy (project/environment scoped)
       │
       ▼
forge-network: compile(policy, placements[08], workload_leases[22.01])
       │  → per-node rule set, generation N
       ▼
GET /v1/nodes/{node_id}/network-policy-rules   (polled by Runtime)
       │
       ▼
Runtime network module: nftables atomic table replace (forge-policy table)
       │
       ▼
Denied connection → nftables counter → sampled log + forge_network_policy_denied_total
                                       + platform event network.policy.denied
```

Evaluation order (most to least specific):

1. Cross-environment traffic → always denied.
2. Explicit `NetworkPolicy` rule matching the destination application → deny takes
   precedence over allow at equal specificity.
3. No matching explicit rule → the environment's default (`allow-within-environment` or
   `deny-all`).

| Component | Responsibility |

## Contracts

* `NetworkPolicy` resource:

```yaml
apiVersion: forge.dev/v1
kind: NetworkPolicy
metadata:
  name: invoice-api-policy
  organization: forge-labs
  project: invoice-platform
  environment: production
spec:
  target:
    application: invoice-api
  ingress:
    - from: { service: invoice-frontend }
      ports: [{ port: 8080, protocol: tcp }]
  egress:
    - to: { database: invoice-db }
      ports: [{ port: 5432, protocol: tcp }]
    - to: { queue: invoice-jobs }
      ports: [{ port: 4222, protocol: tcp }]
status:
  phase: Ready
  observedGeneration: 2
  conditions:
    - type: Enforced
      status: "True"
      reason: AppliedToAllNodes
      message: "3/3 nodes running invoice-api have current rules"
```

```http
POST/GET/PUT/PATCH/DELETE /v1/projects/{project}/environments/{environment}/network-policies[/{name}]
```

## Data model

```sql
CREATE TABLE network.network_policies (
  id                 TEXT PRIMARY KEY,
  organization       TEXT NOT NULL,
  project            TEXT NOT NULL,
  environment        TEXT NOT NULL,
  name               TEXT NOT NULL,
  target_application TEXT NOT NULL,
  spec_json          JSONB NOT NULL,
  generation         INT NOT NULL DEFAULT 1,
  resource_version   BIGINT NOT NULL DEFAULT 1,
  phase              TEXT NOT NULL DEFAULT 'Ready',
  created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (organization, project, environment, name)
);

CREATE TABLE network.environment_network_defaults (
  organization   TEXT NOT NULL,
  project        TEXT NOT NULL,
  environment    TEXT NOT NULL,
  default_policy TEXT NOT NULL DEFAULT 'allow-within-environment', -- or deny-all
  PRIMARY KEY (organization, project, environment)
);
```

## Security considerations

* Enforcement fails closed: if a node cannot fetch current rules, it keeps its
  last-known-good rule set rather than opening up
* Rules are replaced atomically per `generation` (nftables atomic table swap) — there is
  never a window with no rules applied
* Cross-environment denial cannot be overridden by any `NetworkPolicy` — it is enforced
  ahead of policy evaluation, not as a rule an operator could accidentally weaken
* `NetworkPolicy` mutations go through the same platform authz as every other resource
  (epic `09`); a project's members cannot write policy for another project

## Observability

* Metric: `forge_network_policy_denied_total{from_workload=...,to_workload=...,port=...}`
* Metric: `forge_network_policy_rules_generation{node=...}` (for drift/convergence)
* Platform event `network.policy.denied` (event id, resource id, generation, timestamp,
  producer, schema version, trace id — per the platform event schema every controller
  emits) for denies above the sample threshold
* Span `network.policy.compile`

## Failure handling

* nftables apply failure on a node → node reports `Degraded` with a reason, keeps the
  previous rule set (fail closed, not fail open)
* Policy targets a nonexistent application → `Failed` condition, reason `TargetNotFound`,
  no partial enforcement of the rest of the policy
* `PolicyCompiler` failure for one node does not block compilation/distribution for other
  nodes (isolated per-node compilation)

## Unit tests

* `PolicyCompiler`: an ingress rule from a named service produces an `allow` keyed to that
  service's current workload addresses only; unmatched traffic within the environment
  falls through to the environment default
* Cross-environment traffic is denied even when an (incorrectly scoped) explicit policy
  would otherwise allow it
* `deny-all` environment default with no explicit policy denies all same-environment
  traffic to the target application

## Integration tests

* Apply a `NetworkPolicy` allowing `invoice-frontend → invoice-api:8080`; a call from an
  unlisted service to `invoice-api:8080` is denied and counted
* Switching an environment's default from `allow-within-environment` to `deny-all` closes
  previously-open same-environment paths that have no explicit policy
* A denied connection produces both the counter metric and a `network.policy.denied`
  platform event
