## Goal

Make Forge Network own WireGuard peer configuration end to end: a key registry seeded by
`22.02`'s join, a peer-set computation that gives every node the `allowed_ips` it needs to
reach every other node's `/24` block, incremental distribution so a join/leave only
updates the nodes it affects, and key rotation that never drops traffic. Runtime applies
what it is given — the kernel WireGuard interface when available, a userspace
implementation otherwise.

## Why this step exists

`22.02` gets a node's public key registered and hands back an (empty) peer list at join
time. Nothing yet computes or maintains that list as the fleet changes, handles the
keepalives NAT'd nodes need, or lets an operator rotate a compromised key without an outage.
This step is the actual WireGuard control plane the rest of the epic's connectivity
promises depend on.

## Dependencies

* Step `22.02` — node public keys registered during join
* Step `22.01` — node `/24` blocks (`allowed_ips` source)

## Scope

* Key registry: node id ↔ public key ↔ endpoint (extends the join-time registration)
* Peer set computation: full mesh by default — every online node peers with every other
  online node, `allowed_ips` = peer's `/24`
* Incremental distribution: a join/leave/rotation recomputes and republishes only the
  affected nodes' peer lists, tracked by a `peer_version` per network
* Endpoint/NAT handling: nodes report a reachable `host:port` endpoint; `persistent-
  keepalive` (default 25s) for NAT'd peers
* MTU: default `1420` (1500 − 60 bytes WireGuard overhead), configurable
* Key rotation without dropping traffic: register a new key, run both keys as valid peers
  during a rotation window, retire the old key only after peers confirm the new one
* Convergence/drift detection: compare each node's last-applied `peer_version` against the
  network's current version
* Mesh vs. hub topology: implement full mesh; document a hub option (all nodes peer only
  to designated relay nodes) with a recommendation to defer it until node counts make
  O(n²) tunnels impractical (roughly >50 nodes)

## Explicitly out of scope

* Network mode / transport selection — this step assumes WireGuard is the transport for
  the pairs it configures; `22.04` decides *which* pairs use WireGuard at all
* `NetworkPolicy` enforcement (`22.05`)
* Hub topology implementation (documented only)

## Architecture

```text
forge-network
  wireguard_peers table  (node_id, public_key, endpoint, status, peer_version)
       │
       │ on join/leave/rotate → recompute affected nodes' peer sets, bump peer_version
       ▼
GET /v1/networks/{name}/nodes/{node_id}/peers   (polled by Runtime, 04.07-style poll loop)
       │
       ▼
Runtime network module: diff against last-applied peer_version → apply via wg (kernel) or
                         wireguard-go/boringtun (userspace) → report applied version back
```

## Components

| Component | Responsibility |
|---|---|
| `PeerRegistry` | Node id ↔ public key ↔ endpoint, status (`active`/`rotating`/`retiring`) |
| `PeerSetComputer` | Full-mesh (or hub) peer set per node; bumps `peer_version` |
| `PeerDistributionRoute` | `GET /v1/networks/{name}/nodes/{id}/peers` |
| `KeyRotation` | Dual-key window management |
| Runtime `network/wireguard.rs` | Applies peer config via kernel `wg` or userspace backend |

## Data model

```sql
CREATE TABLE network.wireguard_peers (
  network_id  TEXT NOT NULL REFERENCES network.networks(id),
  node_id     TEXT NOT NULL,
  public_key  TEXT NOT NULL,
  endpoint    TEXT,                              -- host:port; null if inbound-only via NAT
  status      TEXT NOT NULL DEFAULT 'active',     -- active | rotating | retiring
  rotates_to  TEXT,                               -- new public_key during rotation window
  applied_peer_version BIGINT NOT NULL DEFAULT 0, -- last version Runtime confirmed
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (network_id, node_id)
);

ALTER TABLE network.networks ADD COLUMN peer_version BIGINT NOT NULL DEFAULT 0;
```

## Security considerations

* Only public keys and endpoints ever cross the wire or land in the database; private
  keys stay on the node (`22.02`)
* Peer distribution API requires the same platform auth as every internal API (epic `09`
  tokens) — a node can only fetch its own peer list
* Rotation requires the node to re-register with proof of its existing key (the request is
  authenticated as that node) before a new key is trusted; a rotation cannot be triggered
  by a third party
* Convergence data (`applied_peer_version`) is used only for observability, never as an
  authorization signal

## Observability

* Metrics: `forge_network_peers_total{network=...}`,
  `forge_network_peer_drift_total{node=...}` (nodes whose `applied_peer_version` lags the
  network's current version), `forge_wireguard_handshake_age_seconds{node=...}` (reported
  by Runtime from the kernel/userspace WireGuard handshake timestamp)
* Trace span `network.peers.compute`
* Log key rotation start/complete and any peer permanently unreachable

## Failure handling

* Rotation interrupted mid-way (Control or node restarts) → the old key remains valid
  until an explicit retire call; connectivity is never dropped by an incomplete rotation
* A peer's endpoint becomes permanently unreachable → surfaced as a `False` condition on
  that node pair (consumed by `22.06`'s route health), not retried indefinitely by the
  peer computer itself
* Runtime falls back to `userspace` backend automatically when the kernel WireGuard module
  or `/dev/net/tun` is unavailable, logging the fallback once at startup

## Unit tests

* `PeerSetComputer`: N online nodes produce N-1 peers each with correct `allowed_ips`; an
  offline node is excluded from all peer sets
* Rotation: old key stays `active` alongside the new key until an explicit retire; retire
  removes only the old key
* Drift metric: a node whose `applied_peer_version` is behind the network's current
  version is counted; a converged node is not

## Integration tests

* Three nodes join → each receives peers for the other two with correct `allowed_ips`
* A fourth node joins → only nodes whose peer set actually changed get a bumped
  `peer_version` (incremental, not a full broadcast)
* Rotate a node's key → connectivity (measured via a running WireGuard handshake) is
  unbroken through the rotation window

## Acceptance criteria

* Every online node's peer list contains every other online node with correct
  `allowed_ips`
* Joining or leaving a node updates only the affected nodes' `peer_version`, not the whole
  fleet
* Key rotation never produces a window with zero valid peer keys for a node
* Drift (a node running a stale `peer_version`) is observable via metric
* Runtime applies peer config via the kernel backend when available and via the userspace
  backend otherwise, without operator intervention
