# Step 09.07: CLI `forge login` + token profile

## Status

Not started

## Epic

`09-forge-identity` → [`../../epics/09-forge-identity.md`](../../epics/09-forge-identity.md)

## Goal

Add `forge login` to the CLI: authenticate against Identity, obtain a token, and store it securely in the CLI profile so all subsequent `forge` commands send `Authorization: Bearer` to Control automatically. Add `forge logout` and `forge whoami`, and make the CLI surface `401/403` clearly.

## Why this step exists

With Control now enforcing auth (09.06), the CLI must supply credentials or every command fails. `specs.md` names `forge login` as the entry point. This makes the developer happy path work end to end against a secured control plane and stores the token profile the rest of the CLI relies on.

## Dependencies

* Step `09.06` — Control enforces auth (so the CLI must authenticate)
* Step `09.03`/`09.05` — login + tokens to obtain and store
* Epic `03-forge-cli` — CLI skeleton, profiles, endpoint config (`03.01`)

## Scope

* `forge login` (interactive password prompt or `--token` for CI) → store token in profile
* Token storage: OS keychain when available, else `~/.config/forge/credentials` with `0600` perms
* Automatic `Authorization` header injection on all Control calls
* `forge logout` (revoke session/token + clear local profile) and `forge whoami` (introspect current token)
* Clear error UX on `401` (not logged in) / `403` (insufficient role) with the required action surfaced

## Explicitly out of scope

* SSO/browser login flows
* Managing other users' tokens from the CLI (identity admin UX)
* Multiple simultaneous identities beyond named profiles (profiles already exist from 03.01)

## Architecture

```text
forge login → prompt → Identity /v1/auth/login → session token
            (or --token <pat> for service accounts / CI)
            → store in credential store keyed by profile
forge <cmd> → load token → attach Bearer → Control
forge logout → Identity revoke + clear local
forge whoami → Identity introspect → print principal + project + role
```

## Components

| Component | Responsibility |
|---|---|
| `login` command | Obtain + persist token |
| `CredentialStore` | Keychain / file (0600) per profile |
| `AuthenticatedClient` | Inject Bearer on Control requests |
| `whoami` / `logout` | Introspect / revoke + clear |

## Contracts

* Consumes Identity `POST /v1/auth/login`, `POST /v1/auth/introspect`, `POST /v1/auth/logout` (and token endpoints for `--token`)
* No new server API; CLI-only behavior

## Data model

Local only:

```text
~/.config/forge/credentials   (mode 0600)
  [profile:default]
  identity_url = http://localhost:4002
  token = <opaque>            # or stored in OS keychain reference
```

## Configuration

```text
FORGE_PROFILE                 # active profile (from 03.01)
FORGE_IDENTITY_URL            # identity endpoint (default per profile)
FORGE_TOKEN                   # env override for CI (non-interactive)
FORGE_CREDENTIALS_BACKEND     # keychain|file (auto-detect default)
```

## Security considerations

* Credentials file is `0600`; keychain preferred where available
* Password read without echo; never written to disk or logs
* `--token`/`FORGE_TOKEN` supports CI without interactive prompt (service-account tokens)
* `forge logout` revokes server-side, not just local delete

## Observability

* CLI prints request IDs on failures (from `03.04`)
* No token values in verbose/debug logs (redacted)
* Non-zero exit codes: `4` for auth errors (documented alongside `03.04` exit codes)

## Failure handling

* Wrong password → clear message, exit non-zero, no partial profile write
* Expired/invalid stored token → command prints "session expired, run forge login", exit auth code
* `403` → prints required role/action from Control's error body
* Identity unreachable → actionable error, non-zero exit
