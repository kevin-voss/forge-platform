## Goal

Add machine identity: **API tokens** (issued to users or service accounts, scoped to a project + role) and **service accounts** (non-human principals that authenticate without a session), plus **revocation** so a token stops working immediately. Introspection (09.03) is extended to resolve tokens to a principal + role.

## Why this step exists

CLI (`09.07`), CI, and later platform services need non-interactive credentials. `specs.md` Step 09 explicitly requires API tokens, service accounts, and token revocation, and that "revoked tokens become unusable" and "service accounts work without human sessions." This is the machine analog of sessions.

## Dependencies

* Step `09.03` — introspection to extend for tokens
* Step `09.04` — roles the token carries
* Step `09.02` — project membership scope

## Scope

* `service_accounts`: named machine principals within a project (role `service-account` or a specific project role)
* `api_tokens`: opaque tokens (shown once at creation, stored hashed), owned by a user or service account, scoped to project + role, with optional expiry
* Token creation, list (metadata only, never plaintext), and revoke APIs
* Introspection resolves a token → `{ principal_type: user|service_account, principal_id, project_id, role, active }`
* Immediate revocation (introspection returns inactive after revoke, honoring the short cache in Control)

## Explicitly out of scope

* Control-side enforcement (09.06)
* Fine-grained scopes beyond project + role
* Token auto-rotation (secrets rotation is epic 10)

## Architecture

```text
create token → generate opaque secret (prefix + random) → store sha256 hash + metadata → return plaintext ONCE
introspect(token) → hash → lookup → active? principal + project + role
revoke(token_id) → set revoked_at → introspection inactive
service account → is a principal that owns tokens; no password/session
```

Token format: `forge_pat_<random>` (users) / `forge_sat_<random>` (service accounts); only prefix + id retrievable later.

## Components

| Component | Responsibility |
|---|---|
| `ServiceAccountStore` | Persist machine principals |
| `TokenStore` | Create/list/revoke tokens (hashed) |
| `TokenIntrospector` | Resolve token → principal + role (extends 09.03 introspect) |
| `TokenRoute` / `ServiceAccountRoute` | HTTP surface |

## Contracts

* `POST /v1/service-accounts` `{ "project_id","name","role" }` → service account
* `POST /v1/tokens` `{ "owner": {"type":"user|service_account","id":...}, "project_id","role","expires_in_s?" }` → `{ token_id, token, prefix, expires_at }` (**token shown once**)
* `GET /v1/tokens?owner=...` → metadata list (`token_id, prefix, project_id, role, created_at, revoked_at`) — never plaintext
* `POST /v1/tokens/{id}/revoke` → `204`
* `POST /v1/auth/introspect` now accepts session **or** API tokens

## Data model

```sql
CREATE TABLE service_accounts (
  id         TEXT PRIMARY KEY,
  project_id TEXT NOT NULL REFERENCES projects(id),
  name       TEXT NOT NULL,
  role       TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (project_id, name)
);
CREATE TABLE api_tokens (
  id           TEXT PRIMARY KEY,
  prefix       TEXT NOT NULL,
  token_hash   TEXT UNIQUE NOT NULL,
  owner_type   TEXT NOT NULL,          -- user | service_account
  owner_id     TEXT NOT NULL,
  project_id   TEXT NOT NULL REFERENCES projects(id),
  role         TEXT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at   TIMESTAMPTZ,
  revoked_at   TIMESTAMPTZ
);
```

## Security considerations

* Token plaintext returned exactly once; only prefix + hash stored — list APIs cannot leak secrets
* Revocation is immediate at Identity; Control's introspection cache TTL is short (default 10s) and bypassed on explicit revoke where feasible
* Service accounts have no password and cannot escalate beyond their granted role
* Token values never logged (log token_id/prefix only)

## Observability

* Log token create/revoke `{token_id, prefix, owner, project, role}` (no secret)
* Metrics: `forge_tokens_active` gauge, `forge_token_revocations_total`, `forge_introspect_total{principal_type}`
* Trace span `token.introspect`

## Failure handling

* Introspect of revoked/expired token → `active:false`
* Creating a token for a principal without membership in the project → `403`
* Listing never returns plaintext even for the creator
* Duplicate service-account name in a project → `409`

## Manual verification

```bash
make dev
# create a developer token for a user in prj_1
T=$(curl -s -X POST localhost:4002/v1/tokens \
  -d '{"owner":{"type":"user","id":"usr_dev"},"project_id":"prj_1","role":"developer"}' \
  -H 'content-type: application/json')
TOK=$(echo "$T" | jq -r .token); TID=$(echo "$T" | jq -r .token_id)
curl -s -X POST localhost:4002/v1/auth/introspect -d "{\"token\":\"$TOK\"}" -H 'content-type: application/json' | jq '{active,role,project_id}'  # active,developer
curl -s -X POST localhost:4002/v1/tokens/$TID/revoke -o /dev/null -w '%{http_code}\n'  # 204
curl -s -X POST localhost:4002/v1/auth/introspect -d "{\"token\":\"$TOK\"}" -H 'content-type: application/json' | jq .active  # false
curl -s "localhost:4002/v1/tokens?owner=usr_dev" | jq '.[0] | has("token")'   # false (no plaintext)
```
