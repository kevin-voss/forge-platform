## Goal

Define the platform role model ‚Äî `organization-owner`, `project-admin`, `developer`, `viewer`, `service-account` ‚Äî and a permission matrix mapping `(role, action)` to allow/deny. Expose an authorization decision API (`POST /v1/authz/check`) that resolves a principal's role in a project scope and answers whether an action is permitted. This is the brain Control consults in 09.06.

## Why this step exists

Sessions/tokens prove identity; roles decide capability. `specs.md` Step 09 requires "project roles are enforced" with a defined initial role set. Centralizing the decision in Identity keeps the policy in one place and testable, and lets Control stay a thin enforcement point.

## Dependencies

* Step `09.02` ‚Äî memberships with a role column
* Step `09.03` ‚Äî introspection returning the principal + memberships (session)
* Step `09.05` provides token principals later; the matrix here is principal-agnostic

## Scope

* Canonical role enum + ordering (owner > admin > developer > viewer; service-account is a distinct machine role)
* Permission matrix: map platform actions (e.g. `project.read`, `deployment.create`, `secret.write`, `member.manage`) to allowed roles
* `POST /v1/authz/check` `{ principal, project_id, action }` ‚Üí `{ allow, role, reason }`
* Role resolution: derive effective role from org-owner (implies admin on all org projects) + project membership
* Deny-by-default: unknown action or no membership ‚Üí deny

## Explicitly out of scope

* Enforcing decisions inside Control (09.06)
* Custom/user-defined roles (fixed set only)
* Per-resource ACLs beyond project scope

## Architecture

```text
authz/check(principal, project, action):
  role = resolveRole(principal, project)      # org-owner > project role > none
  allow = MATRIX[action].contains(role)
  return { allow, role, reason }
```

## Contracts

* `POST /v1/authz/check`:

```json
{ "principal": { "type": "user", "id": "usr_1" },
  "project_id": "prj_1", "action": "deployment.create" }
```

response:

```json
{ "allow": true, "role": "developer", "reason": "developer may deployment.create" }
```

* Published matrix: `GET /v1/authz/matrix` (read-only, for docs/tests)

## Data model

## Security considerations

* Deny-by-default: any action not explicitly granted is denied
* Org-owner escalation is explicit and bounded to their org's projects
* Service-account role cannot manage members or other tokens (least privilege)
* Decision responses never leak other principals' data

## Observability

* Log denials at info with `{principal, project, action, role}` (for audit + debugging)
* Metrics: `forge_authz_checks_total{allow}`, `forge_authz_denies_total{action}`
* Trace span `authz.check`

## Failure handling

* Principal with no membership in project ‚Üí `allow:false, role:none`
* Unknown action ‚Üí `allow:false, reason:"unknown action"` (fail closed)
* Missing project ‚Üí `404`
