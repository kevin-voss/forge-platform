# Step 09.03: Registration, login, sessions

## Status

Not started

## Epic

`09-forge-identity` → [`../../epics/09-forge-identity.md`](../../epics/09-forge-identity.md)

## Goal

Add human authentication to Identity: register a user with a password (hashed with Argon2id), log in to obtain an opaque session token, use the session to identify the caller, and log out to invalidate it. Sessions expire and can be validated via an introspection endpoint that Control will later call.

## Why this step exists

Tenancy (09.02) established who exists; this step proves who is calling. Sessions are the human-facing authentication primitive and the model that token introspection (09.05/09.06) generalizes. Password hashing and session lifecycle are explicitly required by `specs.md` Step 09 tests.

## Dependencies

* Step `09.02` — users to authenticate
* Step `09.01` — service + DB

## Scope

* `credentials` storage: Argon2id password hash + params, separate from the user row
* Registration: `POST /v1/auth/register` (creates user + credential)
* Login: `POST /v1/auth/login` → opaque session token + expiry
* Session validation: `POST /v1/auth/introspect` → principal (user id, memberships summary), `active` flag
* Logout: `POST /v1/auth/logout` invalidates the session
* Session expiry + sliding vs fixed lifetime (fixed for simplicity)
* Rate limiting / lockout hook for repeated failed logins (basic counter)

## Explicitly out of scope

* API tokens + service accounts (09.05)
* Role/permission evaluation (09.04) — introspect returns memberships, not a decision
* Control enforcement (09.06)
* Password reset / email verification

## Architecture

```text
register → hash(password, Argon2id) → credentials row
login    → verify hash → create session (opaque token, hashed at rest) → return token
introspect(token) → lookup session by token hash → active? principal + memberships
logout   → mark session revoked
```

## Contracts

* `POST /v1/auth/register` `{ "email","password","display_name" }` → `201 { user_id }`
* `POST /v1/auth/login` `{ "email","password" }` → `200 { session_token, expires_at }`
* `POST /v1/auth/introspect` `{ "token" }` → `200 { active, principal_type:"user", user_id, memberships }`
* `POST /v1/auth/logout` `Authorization: Bearer <session>` → `204`

## Data model

```sql
CREATE TABLE credentials (
  user_id      TEXT PRIMARY KEY REFERENCES users(id),
  hash         TEXT NOT NULL,          -- Argon2id encoded
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE sessions (
  id           TEXT PRIMARY KEY,
  user_id      TEXT NOT NULL REFERENCES users(id),
  token_hash   TEXT UNIQUE NOT NULL,   -- sha256 of opaque token
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at   TIMESTAMPTZ NOT NULL,
  revoked_at   TIMESTAMPTZ
);
CREATE TABLE login_attempts (
  email        CITEXT NOT NULL,
  at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  success      BOOLEAN NOT NULL
);
```

## Configuration

```text
FORGE_SESSION_TTL_S           # default 86400
FORGE_ARGON2_MEMORY_KB        # default 65536
FORGE_ARGON2_ITERATIONS       # default 3
FORGE_LOGIN_MAX_FAILS         # lockout threshold (default 5 / 15min)
```

## Security considerations

* Passwords hashed with Argon2id; plaintext never stored or logged
* Session tokens are high-entropy opaque strings; only their SHA-256 hash is stored (steal-resistant DB)
* Introspection reveals principal + memberships only, never the password hash
* Failed-login counter mitigates brute force; responses are uniform to avoid user enumeration timing leaks
* Token values never appear in logs (log session id, not token)

## Observability

* Log auth events: `{event=login|logout|register, user_id, success}` — no credentials
* Metrics: `forge_login_total{result=success|fail}`, `forge_active_sessions` gauge
* Trace spans `auth.login`, `auth.introspect`

## Failure handling

* Wrong password / unknown user → uniform `401` (no enumeration)
* Expired/revoked session → introspect `active:false`
* Too many failures → temporary lockout `429`
* Hash verify error → treated as failure, logged as anomaly

## Manual verification

```bash
make dev
curl -s -X POST localhost:4002/v1/auth/register -d '{"email":"a@x.com","password":"s3cret!!","display_name":"A"}' -H 'content-type: application/json'
TOK=$(curl -s -X POST localhost:4002/v1/auth/login -d '{"email":"a@x.com","password":"s3cret!!"}' -H 'content-type: application/json' | jq -r .session_token)
curl -s -X POST localhost:4002/v1/auth/introspect -d "{\"token\":\"$TOK\"}" -H 'content-type: application/json' | jq '{active,user_id}'   # active true
curl -s -X POST localhost:4002/v1/auth/logout -H "Authorization: Bearer $TOK" -o /dev/null -w '%{http_code}\n'  # 204
curl -s -X POST localhost:4002/v1/auth/introspect -d "{\"token\":\"$TOK\"}" -H 'content-type: application/json' | jq .active  # false
# ensure no plaintext password in logs
docker logs forge-identity 2>&1 | grep -c 's3cret' | grep -q '^0$' && echo no-plaintext-in-logs
```

## Acceptance criteria

* Registration stores an Argon2id password hash; plaintext is never persisted or logged
* Login returns an opaque session token with expiry; only its hash is stored
* Introspection reports active principal + memberships and `active:false` for expired/revoked/unknown tokens
* Logout invalidates the session immediately
* Repeated failed logins trigger a lockout; wrong credentials return a uniform 401
