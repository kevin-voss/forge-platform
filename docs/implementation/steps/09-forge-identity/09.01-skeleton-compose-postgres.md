## Goal

Create the Forge Identity service: a Kotlin/Ktor application on host port `4002` that satisfies the runtime contract (health endpoints, structured logs, env config, graceful shutdown), connects to its own Postgres database via a migration framework, and runs under Compose alongside the rest of the platform.

## Why this step exists

Every identity capability (users, sessions, tokens, roles) needs a running service with database migrations and health/observability wiring. Building the skeleton first mirrors the epic-02 Control skeleton and gives later steps a stable home and CI target.

## Dependencies

* Epic `01-runtime-contract` — health/logs/env conventions the service must follow
* Foundation `00` — Compose, Postgres, port map, Make targets
* No prior steps in this epic

## Scope

* Ktor application: `GET /health/live`, `GET /health/ready`, identity `GET /` (service/language/status), structured JSON logs, `PORT`/`FORGE_*` env config, SIGTERM graceful shutdown
* Postgres connection + migration framework (Flyway) with a baseline empty migration
* Dockerfile + Compose service `forge-identity` on port `4002` with a dedicated `identity` Postgres database
* Makefile wiring (build/test/run for the service)
* Port reservation confirmation (`4002`) in `docs/operations/ports.md`
* Skeleton OpenAPI file for the Identity API surface

## Explicitly out of scope

* Any user/org/token domain logic (09.02+)
* Authentication endpoints (09.03)
* Control authz integration (09.06)

## Contracts

* `GET /health/live` → 200 when process alive
* `GET /health/ready` → 200 when DB reachable + migrations applied
* `GET /` → identity JSON `{ "service": "forge-identity", "language": "kotlin", "status": "running" }`
* `contracts/openapi/forge-identity.openapi.yaml` — skeleton declaring the three paths

## Data model

No domain tables yet. Baseline migration establishes the schema/version table only (Flyway history).

## Configuration

```text
PORT                     # default 4002
FORGE_SERVICE_NAME       # forge-identity
FORGE_LOG_LEVEL          # info
FORGE_IDENTITY_DB_URL    # jdbc:postgresql://postgres:5432/forge_identity
FORGE_IDENTITY_DB_USER
FORGE_IDENTITY_DB_PASSWORD
```

## Security considerations

* DB credentials via env only; never logged
* `/` and health endpoints unauthenticated (contract requirement); no sensitive data in identity payload
* TLS terminated upstream (Gateway) for local dev; document

## Observability

* Structured JSON logs to stdout with service name + level
* OTEL tracing/metrics wiring consistent with epic 02 Control (spans on requests) — recommended, mirrors platform
* Readiness reflects DB + migration state

## Failure handling

* DB unreachable at startup → readiness stays non-200; liveness stays 200; retries connection with backoff
* Migration failure → service refuses ready; logs the failing migration
* SIGTERM → stop accepting new requests, drain, exit 0 within grace (default 10s)
