## Goal

Model the core tenancy entities in Identity — users, organizations, projects, and memberships (who belongs to which org/project) — with Postgres persistence and CRUD APIs. This is the data foundation for authentication (09.03) and authorization (09.04).

## Why this step exists

Authorization decisions require knowing which principals exist and which org/project they belong to. Building the persistence + read/write APIs before login/roles keeps the schema stable and testable, and lets Control eventually resolve a principal's project memberships.

## Dependencies

* Step `09.01` — Identity service skeleton + migrations
* Epic `02-forge-control` — project identifiers Identity memberships reference (projects are created in Control; Identity mirrors/links by project id)

## Scope

* Entities: `users`, `organizations`, `projects` (identity-side reference), `org_memberships`, `project_memberships`
* CRUD APIs: create/get/list users; create/get/list orgs; add/remove org members; add/remove project members
* Uniqueness + referential integrity (unique email, membership unique per (user, scope))
* Consistent error format (reuse Control's `02.06` error shape)
* OpenAPI for the new endpoints

## Explicitly out of scope

* Password/credential storage + login (09.03)
* Role semantics + permission evaluation (09.04) — membership rows carry a role column but enforcement is 09.04
* Tokens/service accounts (09.05)

## Architecture

```text
users ──< org_memberships >── organizations
  │                                  │
  └──────< project_memberships >── projects (id links to Control project)
```

Identity is the source of truth for identity + membership; Control owns the project resource and shares its id.

## Components

| Component | Responsibility |
|---|---|
| `UserStore` / `UserRoute` | User persistence + CRUD |
| `OrgStore` / `OrgRoute` | Org persistence + membership |
| `ProjectMembershipStore` | Project membership persistence |
| error mapping | Shared error format |

## Contracts

* `POST /v1/users` `{ "email": "...", "display_name": "..." }` → user (no credentials here)
* `GET /v1/users/{id}` / `GET /v1/users?email=`
* `POST /v1/orgs` `{ "name": "..." }` → org; `POST /v1/orgs/{id}/members` `{ "user_id":"...", "role":"organization-owner" }`
* `POST /v1/projects/{project_id}/members` `{ "user_id":"...", "role":"developer" }`
* `GET /v1/users/{id}/memberships` → orgs + projects the user belongs to

## Data model

```sql
CREATE TABLE users (
  id           TEXT PRIMARY KEY,
  email        CITEXT UNIQUE NOT NULL,
  display_name TEXT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE organizations (
  id         TEXT PRIMARY KEY,
  name       TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE org_memberships (
  org_id  TEXT NOT NULL REFERENCES organizations(id),
  user_id TEXT NOT NULL REFERENCES users(id),
  role    TEXT NOT NULL,             -- semantics enforced in 09.04
  PRIMARY KEY (org_id, user_id)
);
CREATE TABLE projects (
  id       TEXT PRIMARY KEY,          -- shared with Control project id
  org_id   TEXT NOT NULL REFERENCES organizations(id),
  name     TEXT NOT NULL
);
CREATE TABLE project_memberships (
  project_id TEXT NOT NULL REFERENCES projects(id),
  user_id    TEXT NOT NULL REFERENCES users(id),
  role       TEXT NOT NULL,
  PRIMARY KEY (project_id, user_id)
);
```

## Security considerations

* Emails stored case-insensitively (CITEXT) with uniqueness to prevent duplicate accounts
* No credentials in this step; user rows carry no secrets
* Membership APIs will be protected once authz lands (09.04/09.06); until then guarded by `FORGE_AUTH_MODE=dev`

## Observability

* Log create/delete of users/orgs/memberships (ids only, never PII beyond email domain if configured)
* Metric: `forge_identity_users_total`, `forge_identity_orgs_total`
* Trace spans on each store operation

## Failure handling

* Duplicate email → `409` typed error
* Membership referencing missing user/org/project → `404`/`422`
* Duplicate membership → idempotent (return existing)
