# Step 17.03: Upsert + cosine NN query

## Status

Not started

## Epic

`17-forge-memory` → [`../../epics/17-forge-memory.md`](../../epics/17-forge-memory.md)

## Goal

Implement vector upsert (insert/update by id) and a brute-force cosine nearest-neighbor query returning the top-k most similar records with scores and metadata, plus metadata filtering and record deletion.

## Why this step exists

Nearest-neighbor retrieval is the core capability of the service (`specs.md` Step 17: cosine similarity, brute-force first, filtering, deletion). Correctness on fixtures is a hard acceptance criterion.

## Dependencies

* `17.02` (collections + storage)

## Scope

* `POST /v1/collections/{name}/upsert` — batch upsert of `{id, vector, metadata}`
* `POST /v1/collections/{name}/query` — `{ vector, top_k, filter? }` → ranked results
* Brute-force cosine over the collection (L2-normalize; dot product)
* Metadata filtering (equality/`in` predicates) applied before/after scoring
* `DELETE /v1/collections/{name}/records/{id}` (tombstone) + compaction on boot
* Documented benchmark at fixture scale

## Explicitly out of scope

* Namespaces/ACL (17.04)
* HNSW/ANN (later optimization)
* Models/Agents integration (17.05)

## Architecture

```text
upsert: validate dim → normalize → write vector (new offset or overwrite) → upsert metadata
query:  normalize query → iterate live vectors → cosine (dot of normalized) → top-k heap
        apply filter (metadata) → return [{id, score, metadata}]
delete: mark tombstone; boot/periodic compaction reclaims space
```

## Components

| Component | Purpose |
|---|---|
| Upsert handler | Batch validate + store |
| Cosine scorer | Normalized dot product |
| Top-k selector | Bounded heap |
| Filter evaluator | Metadata predicates |
| Compactor | Reclaim tombstoned slots |

## Contracts

* `POST .../upsert` `{ "records":[{ "id","vector":[...],"metadata":{} }] }` → `200 { "upserted":N }`
* `POST .../query` `{ "vector":[...],"top_k":5,"filter":{"type":"deploy"} }` → `200 { "results":[{ "id","score","metadata" }] }` (score desc)
* `422` on dim mismatch; `404` on missing collection
* Extend OpenAPI

## Data model

Reuses 17.02 tables; deletion sets `records.deleted=1`. Optional `norm` cached per vector to speed scoring.

## Configuration

```text
FORGE_MEMORY_MAX_TOP_K        # default 100
FORGE_MEMORY_MAX_UPSERT_BATCH # default 512
FORGE_MEMORY_COMPACT_ON_BOOT  # default true
```

## Security considerations

* Upsert/query dim-validated
* Batch/top_k caps prevent resource exhaustion
* Filter predicates are safe structured filters (no code eval)

## Observability

* Log upsert counts + query latency + candidates scanned
* Metric: `memory_query_latency_seconds`, `memory_upserts_total`, `memory_query_candidates`

## Failure handling

* Dim mismatch → `422`
* top_k/batch over cap → `422`
* Query on empty collection → `200` empty results
* Corrupt vector on scan → error surfaced (not silent wrong result)
