# Step 11.06: Idempotency keys + consumer identity

## Status

Not started

## Epic

`11-forge-events` → [`../../epics/11-forge-events.md`](../../epics/11-forge-events.md)

## Goal

Make delivery safe under at-least-once semantics: **idempotency keys** so publishing the same event twice does not create duplicates and consumers can dedup redeliveries, and **consumer identity** so each durable consumer is a named, optionally-authenticated principal whose progress and permissions are tracked. Together these guarantee "duplicate-event handling" works.

## Why this step exists

`specs.md` Step 11 requires "idempotency keys" and "consumer identities", and acceptance that duplicate events are handled. JetStream is at-least-once, so without dedup a redelivered or double-published event is processed twice (e.g. two deployments). Consumer identity is also needed for auditing and for future authorization.

## Dependencies

* Step `11.03` — durable consumers + redelivery (source of duplicates)
* Step `11.02` — publish path (source of double-publish)
* Epic `09-forge-identity` (optional) — service-account tokens for authenticated consumer identity

## Scope

* Publish idempotency: accept an `Idempotency-Key` (or `event_id`) used as the NATS msg-id so JetStream dedups within a window; duplicate publish returns the original result
* Consumer-side dedup: a seen-key store per consumer so a consumer can mark an event processed and skip duplicates on redelivery
* Consumer identity: durable consumers carry an identity (name + optional service-account token); consume/ack require the matching identity
* Optional auth: when `FORGE_AUTH_MODE=enforce`, publish/consume require a valid Identity service-account token; consumer identity is derived from it
* Expose processed/dedup metrics + a `processed` marker API

## Explicitly out of scope

* Exactly-once across arbitrary side effects (idempotency is best-effort dedup)
* Cross-consumer shared dedup (dedup is per consumer)
* New event types/schemas (11.05)

## Architecture

```text
publish(Idempotency-Key=K):
  msg-id = K → JetStream dedup window drops duplicates → same {event_id,seq}

consume(consumer=C): message carries event_id
process → POST /v1/processed {consumer:C, event_id} → seen-store marks done
redelivery of same event_id for C → client checks seen-store (or server filters) → skip

consumer identity:
  durable consumer C bound to principal (service-account) when auth enabled
  consume/ack must present C's token
```

## Components

| Component | Responsibility |
|---|---|
| publish dedup | Map Idempotency-Key → NATS msg-id (dedup window) |
| `SeenStore` | Per-consumer processed-event keys |
| `ConsumerIdentity` | Bind durable consumer to a principal/token |
| auth (optional) | Enforce identity on publish/consume/ack |

## Contracts

* `POST /v1/events` accepts header `Idempotency-Key`; duplicate within the window → same `202 {event_id, seq}` (not a new message)
* `POST /v1/processed` `{ consumer, event_id }` → `204` (mark processed)
* `GET /v1/processed?consumer=&event_id=` → `{ processed: bool }`
* Consumer creation (`11.03`) gains optional `identity` / requires token when auth enforced

## Data model

```sql
CREATE TABLE processed_events (
  consumer   TEXT NOT NULL,
  event_id   TEXT NOT NULL,
  at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (consumer, event_id)
);
CREATE INDEX idx_processed_at ON processed_events(at);   -- for TTL cleanup
```

Publish dedup uses the NATS JetStream duplicate window (configured per stream) keyed by msg-id.

## Configuration

```text
FORGE_DEDUP_WINDOW_S           # JetStream duplicate window (default 120)
FORGE_SEEN_STORE_TTL_S         # processed_events retention (default 86400)
FORGE_AUTH_MODE                # enforce → require consumer identity tokens
FORGE_IDENTITY_URL             # when auth enforced
```

## Security considerations

* When enforced, publish/consume/ack require a valid service-account token; consumer identity is derived from the token, not client-claimed
* Idempotency keys are opaque; a malicious duplicate key cannot suppress a legitimately different event (key is caller-scoped where auth enabled)
* Seen-store keyed by (consumer, event_id) prevents cross-consumer interference

## Observability

* Log dedup hits `{event_id, consumer|publisher}` and processed marks
* Metrics: `forge_publish_dedup_hits_total`, `forge_consumer_dedup_skips_total{consumer}`, `forge_processed_events_total`
* Trace spans `events.dedup`, `events.processed`

## Failure handling

* Duplicate publish within window → returns original result (no new message), no error
* Duplicate publish after window expiry → treated as new (documented limitation of at-least-once)
* Processed-mark race (two workers) → PK conflict handled idempotently (first wins)
* Auth enabled + missing/invalid token → `401`; wrong consumer identity → `403`

## Implementation tasks

1. Configure JetStream duplicate window per stream; map `Idempotency-Key` → msg-id in publish.

3. Add consumer identity to durable consumer config; bind to a principal when auth enabled.
4. Implement optional auth middleware (introspect service-account token; derive consumer identity).
5. Add dedup-skip helper so consumers can check-then-process idempotently.
6. Add retention cleanup for the seen store.

## Acceptance criteria

* Publishing with the same idempotency key within the dedup window yields a single stored message and identical result
* Consumers can mark events processed and skip duplicate redeliveries (no double processing)
* Durable consumers have a tracked identity; when auth is enforced, publish/consume/ack require a valid service-account token
* Wrong consumer identity is rejected (`403`); missing token → `401`
* Processed marks persist across consumer restarts; seen-store retention is bounded
