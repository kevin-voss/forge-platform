## Goal

Expose the Forge Events abstraction over JetStream: a `publish` endpoint that accepts an event (subject + payload) and persists it to the right stream, and a `consume`/subscribe mechanism that lets a client pull events for a subject. This is the minimal messaging surface before durability, retries, and DLQ are layered on.

## Why this step exists

11.01 gave streams but no way to send or receive events. Publish/subscribe is the core API every producer/consumer uses. Keeping it separate from durable-consumer semantics (11.03) isolates the wire contract from delivery guarantees.

## Dependencies

* Step `11.01` — service + JetStream streams
* No auth yet (added optionally in 11.06)

## Scope

* `POST /v1/events` publish: `{ subject, data, headers? }` → stored to the stream matching the subject family; returns `{ event_id, stream, seq }`
* Subject validation: subject must match a known family (`build.*`, etc.); unknown → `400`
* Pull consume: `POST /v1/consume` `{ subject, batch, consumer? }` → returns a batch of messages with delivery metadata (ack handled in 11.03; here auto-ack or explicit-ack stub)
* Optional push: register a webhook URL for a subject (basic, best-effort; durable version in 11.03)
* Envelope: standard event envelope (id, subject, time, source, data) wrapping the payload

## Explicitly out of scope

* Durable named consumers + retry/ack policy (11.03)
* DLQ (11.04)
* Schema validation (11.05) — payload is opaque bytes/JSON here
* Idempotency keys + consumer identity (11.06)

## Architecture

```text
POST /v1/events → validate subject → js.Publish(subject, envelope) → {event_id, seq}
POST /v1/consume → js pull subscribe (ephemeral) → batch of envelopes
envelope: { id, subject, time, source, data }
```

## Components

| Component | Responsibility |
|---|---|
| `Publisher` | Validate subject; publish envelope to JetStream |
| `Envelope` | Standard event wrapper |
| `Consumer` (pull) | Fetch a batch for a subject |
| `PublishRoute` / `ConsumeRoute` | HTTP surface |

## Contracts

* `POST /v1/events`:

```json
{ "subject": "application.crashed",
  "data": { "service": "demo", "reason": "oom" },
  "source": "runtime" }
```

→ `202 { "event_id": "evt_1", "stream": "application", "seq": 42 }`

* `POST /v1/consume` `{ "subject": "application.crashed", "batch": 10 }` → `{ "messages": [ { "event_id", "subject", "time", "data", "ack_token" } ] }`

## Data model

State in JetStream. Envelope stored as the message payload; `event_id` is the envelope id (also used as NATS msg-id for later dedup).

## Configuration

```text
FORGE_CONSUME_MAX_BATCH        # default 100
FORGE_CONSUME_WAIT_MS          # long-poll wait for pull (default 2000)
```

## Security considerations

* Subject allow-list prevents publishing to arbitrary/unknown streams
* Payload size limit to prevent abuse (`FORGE_EVENT_MAX_BYTES`, default 256KB)
* No auth yet; documented; added in 11.06 when Identity is enabled

## Observability

* Log publish `{event_id, subject, stream, seq, bytes}` and consume `{subject, delivered}`
* Metrics: `forge_events_published_total{subject}`, `forge_events_consumed_total{subject}`, publish latency histogram
* Trace spans `events.publish`, `events.consume` with subject attributes

## Failure handling

* Unknown/invalid subject → `400`
* Oversized payload → `413`
* JetStream publish failure → `503` (retryable); not acknowledged to the caller
* Consume with no messages → empty batch after long-poll wait (not an error)
