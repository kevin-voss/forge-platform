# Step 11.05: Event JSON Schemas

## Status

Not started

## Epic

`11-forge-events` → [`../../epics/11-forge-events.md`](../../epics/11-forge-events.md)

## Goal

Define JSON Schemas for the platform event types (`build.*`, `deployment.*`, `runtime.node.offline`, `application.crashed`, `agent.run.*`), register them in Forge Events, and **validate published events against their schema** so malformed events are rejected at publish time. Schemas are versioned and published under `contracts/events/`.

## Why this step exists

`specs.md` Step 11 requires "event schemas" and that "malformed events are rejected." Schemas turn implicit event shapes into an enforced contract so producers and polyglot consumers (Go, Elixir, etc.) agree on structure — critical for the cross-language demo and for later consumers in Agents/Workflows.

## Dependencies

* Step `11.02` — publish path to add validation to
* Step `11.01` — streams/subjects

## Scope

* JSON Schema files under `contracts/events/` for the named event types (Draft 2020-12)
* Schema registry in Forge Events: load schemas at startup, map subject → schema (+ version)
* Publish-time validation: reject events whose `data` fails schema validation (`422` with error details)
* Schema versioning: `subject` + `schema_version` header; support additive evolution
* Schema listing API: `GET /v1/schemas` and `GET /v1/schemas/{subject}`

## Explicitly out of scope

* Idempotency + consumer identity (11.06)
* Consumer-side schema enforcement (producers validated here; consumers may re-validate but it's optional)
* Auto-generating client types from schemas (nice-to-have later)

## Architecture

```text
startup: load contracts/events/*.json → registry (subject → schema[])
publish(subject, data): registry.validate(subject, data) → ok | 422
GET /v1/schemas: list registered schemas + versions
```

## Components

| Component | Responsibility |
|---|---|
| Schema files | JSON Schema per event type in `contracts/events/` |
| `SchemaRegistry` | Load + index schemas by subject/version |
| `Validator` | Validate event data against schema |
| `SchemaRoutes` | List schemas |

## Contracts

* Schemas: `contracts/events/build.completed.schema.json`, `deployment.completed.schema.json`, `runtime.node.offline.schema.json`, `application.crashed.schema.json`, `agent.run.completed.schema.json`, etc.
* `POST /v1/events` now validates `data`; on failure → `422 { error, subject, violations:[...] }`
* `GET /v1/schemas` → registered subjects + versions

## Data model

Schemas are files (source of truth) loaded into memory. Optional registry table for versions:

```sql
CREATE TABLE event_schemas (
  subject        TEXT NOT NULL,
  schema_version INT  NOT NULL,
  schema_json    JSONB NOT NULL,
  registered_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (subject, schema_version)
);
```

## Configuration

```text
FORGE_EVENT_SCHEMA_DIR         # default /contracts/events
FORGE_SCHEMA_VALIDATION        # strict|warn (default strict — reject invalid)
```

## Security considerations

* Strict validation prevents malformed/oversized/unexpected fields from entering streams
* Schema files are read-only at runtime; no untrusted schema upload endpoint in this epic
* Validation errors do not echo the full payload (avoid leaking sensitive data in error responses/logs)

## Observability

* Log rejected publishes `{subject, violations_count}` (not full payload)
* Metrics: `forge_events_rejected_total{subject,reason}`, `forge_schemas_loaded` gauge
* Trace span `events.validate`

## Failure handling

* Missing schema for a subject → configurable: strict rejects publish (`422 unknown schema`), warn allows + logs
* Schema file parse error at startup → service logs + refuses ready (fail fast)
* Validation failure → `422` with structured violations
* Schema version mismatch header → validate against the requested version if present, else latest

```bash
curl -s localhost:4105/v1/schemas | jq 'keys'
```
