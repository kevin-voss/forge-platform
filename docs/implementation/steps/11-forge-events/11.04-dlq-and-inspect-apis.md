# Step 11.04: DLQ + inspect APIs

## Status

Not started

## Epic

`11-forge-events` → [`../../epics/11-forge-events.md`](../../epics/11-forge-events.md)

## Goal

Route messages that exhaust their retry budget to a **dead-letter queue** and provide APIs to inspect, and optionally redeliver, DLQ messages. Operators can see what failed, why, and how many times, and can replay a message after fixing the consumer.

## Why this step exists

`specs.md` Step 11 requires "dead-letter handling" and that "dead-letter events are inspectable." Without a DLQ, a poison message either blocks a consumer or is silently lost. This turns the parked/terminal state from 11.03 into an observable, actionable queue.

## Dependencies

* Step `11.03` — max-deliveries terminal state + delivery count
* Step `11.01`/`11.02` — streams + envelope

## Scope

* DLQ stream per source family (e.g. `dlq.application`) capturing messages that exceed `max_deliveries`
* Automatic routing: on terminal failure, republish the envelope to the DLQ with failure metadata (original subject, consumer, delivery_count, last_error, first_failed_at)
* Inspect API: `GET /v1/dlq?subject=&consumer=` list; `GET /v1/dlq/{id}` detail
* Redeliver API: `POST /v1/dlq/{id}:redeliver` republishes to the original subject (for replay after a fix)
* Purge/ack DLQ entry: `DELETE /v1/dlq/{id}` (acknowledge handled)

## Explicitly out of scope

* Schema validation (11.05)
* Idempotency (11.06) — though redeliver reuses the original event id so consumers can dedup
* Automatic retry from DLQ (manual redeliver only)

## Architecture

```text
11.03 terminal (delivery_count > max) → DLQRouter:
   publish envelope + failure meta → dlq.<family>
inspect: GET /v1/dlq (list), GET /v1/dlq/{id} (detail)
redeliver: POST /v1/dlq/{id}:redeliver → publish to original subject (same event_id)
delete: DELETE /v1/dlq/{id} → ack/remove from DLQ
```

## Components

| Component | Responsibility |
|---|---|
| `DLQRouter` | Move terminal messages to the DLQ stream with metadata |
| `DLQStore` | List/get/delete DLQ messages |
| `Redeliverer` | Republish a DLQ message to its original subject |
| `DLQRoutes` | inspect/redeliver/delete API |

## Contracts

* `GET /v1/dlq?subject=application.crashed&consumer=crash-worker` → `[ { dlq_id, event_id, original_subject, consumer, delivery_count, last_error, first_failed_at } ]`
* `GET /v1/dlq/{dlq_id}` → full envelope + failure metadata
* `POST /v1/dlq/{dlq_id}:redeliver` → `202 { republished_to, event_id }`
* `DELETE /v1/dlq/{dlq_id}` → `204`

## Data model

DLQ messages live in JetStream `dlq.*` streams; failure metadata carried in message headers. Optional index table for fast listing:

```sql
CREATE TABLE dlq_index (
  dlq_id           TEXT PRIMARY KEY,
  event_id         TEXT NOT NULL,
  original_subject TEXT NOT NULL,
  consumer         TEXT NOT NULL,
  delivery_count   INT  NOT NULL,
  last_error       TEXT,
  first_failed_at  TIMESTAMPTZ NOT NULL,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## Configuration

```text
FORGE_DLQ_ENABLED           # default true
FORGE_DLQ_RETENTION_DAYS    # default 7
```

## Security considerations

* DLQ payloads may contain sensitive event data → same access controls as normal events (auth in 11.06); do not log payloads
* Redeliver preserves original event id so downstream idempotency (11.06) still applies (no accidental duplicate processing)
* Delete/redeliver are explicit operator actions (audited via logs)

## Observability

* Log DLQ routing `{event_id, original_subject, consumer, delivery_count, last_error}` (no payload)
* Metrics: `forge_dlq_messages_total{subject}`, `forge_dlq_redelivered_total`, `forge_dlq_size` gauge
* Trace spans `events.dlq.route`, `events.dlq.redeliver`

## Failure handling

* DLQ publish failure → keep the message parked + retry routing; never drop silently
* Redeliver of a non-existent DLQ id → `404`
* Redeliver failure (original subject stream down) → `503`, DLQ entry retained
* DLQ retention cleanup removes old entries (configurable)

## Manual verification

```bash
make dev
# using crash-worker (max_deliveries=3, ack_wait small) from 11.03, publish a message the consumer always naks
curl -s -X POST localhost:4105/v1/events -d '{"subject":"application.crashed","data":{"service":"poison"},"source":"test"}' -H 'content-type: application/json'
# let it exhaust retries (consumer naks / never acks)
sleep 15
curl -s "localhost:4105/v1/dlq?subject=application.crashed" | jq '.[0] | {original_subject, consumer, delivery_count, last_error}'
DID=$(curl -s "localhost:4105/v1/dlq?subject=application.crashed" | jq -r '.[0].dlq_id')
curl -s -X POST localhost:4105/v1/dlq/$DID:redeliver | jq
```

## Acceptance criteria

* Messages exceeding `max_deliveries` are routed to a DLQ stream with failure metadata (original subject, consumer, delivery count, last error)
* DLQ messages are listable and inspectable via API
* A DLQ message can be redelivered to its original subject, preserving the original event id
* DLQ entries can be deleted/acknowledged and are subject to retention cleanup
* DLQ routing never silently drops a poison message
