## Goal

Add delivery guarantees: named **durable consumers** that survive restarts, explicit **acknowledgement** (ack/nak) so unprocessed messages are redelivered, and a **retry policy** (max deliveries + backoff) so transient failures are retried a bounded number of times. A message that is not acked is redelivered; a consumer that restarts resumes from its last acked position.

## Why this step exists

11.02 delivered messages but without durability a consumer restart loses in-flight work and there is no retry. `specs.md` Step 11 requires "durable subscriptions", "acknowledgement", "retries", and that "events survive consumer restart". This is the reliability core the DLQ (11.04) builds on.

## Scope

* Durable consumer registration: `POST /v1/consumers` `{ name, subject, ack_wait_s, max_deliveries }` → creates a JetStream durable consumer
* Explicit ack API: `POST /v1/ack` `{ ack_token }` and `POST /v1/nak` `{ ack_token, delay_s? }`
* Ack-wait redelivery: unacked messages redelivered after `ack_wait_s`
* Retry policy: `max_deliveries` with backoff; delivery count surfaced in message metadata
* Consume from a durable (not ephemeral) consumer so position persists
* Consumer restart resumes from last acked sequence

## Explicitly out of scope

* DLQ routing when max deliveries exceeded (11.04) — this step exposes the delivery count + terminal state; routing is next
* Schema validation (11.05)
* Idempotency keys (11.06)

## Architecture

```text
create durable consumer (name) on stream/subject with ack_wait + max_deliveries
consume(name) → messages with {ack_token, delivery_count}
ack(token)   → JetStream ack → advances position
nak(token)   → redeliver (optionally delayed)
no ack within ack_wait → automatic redelivery (delivery_count++)
delivery_count > max_deliveries → terminal (DLQ handling in 11.04)
```

## Security considerations

* Ack tokens are opaque, single-use, and tied to a specific message delivery (cannot ack arbitrary messages)
* Consumer names namespaced to avoid collisions; creating an existing consumer is idempotent (compatible config) or `409` (conflicting)
* No plaintext payloads in ack/nak logs

## Observability

* Log ack/nak/redelivery with `{consumer, event_id, delivery_count}`
* Metrics: `forge_events_acked_total`, `forge_events_redelivered_total{consumer}`, `forge_consumer_pending` gauge
* Trace spans `events.ack`, `events.redeliver`

## Failure handling

* No ack within `ack_wait_s` → automatic redelivery (delivery_count++)
* `nak` → immediate/delayed redelivery
* Delivery count exceeds `max_deliveries` → message is not redelivered to the consumer (held for DLQ handling in 11.04); until 11.04 lands, such messages are parked + logged
* Ack of an expired/unknown token → `410`/`404`
* Consumer restart → resumes from last acked sequence (no reprocessing of acked messages)

## Acceptance criteria

* Named durable consumers persist and resume from their last acked position across restarts
* Unacked messages are redelivered after `ack_wait_s`; `nak` triggers (optionally delayed) redelivery
* Retry is bounded by `max_deliveries`; delivery count is visible in message metadata
* A repeatedly failing message stops being redelivered after max deliveries (parked for DLQ handling)
* Events survive a consumer restart (no loss of unacked, no reprocessing of acked)
