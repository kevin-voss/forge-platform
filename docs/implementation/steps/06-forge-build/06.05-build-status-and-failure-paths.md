# Step 06.05: Build status + failure paths

## Status

Not started

## Epic

`06-forge-build` → [`../../epics/06-forge-build.md`](../../epics/06-forge-build.md)

## Goal

Finalize the build status model and failure handling: a durable `GET /v1/builds/{id}` status with clear phases and error details, guaranteed workspace cleanup, and the invariant that failed builds never yield a deployable image — so downstream deploy steps only ever see successful, pushed images.

## Why this step exists

The spec requires accessible build status, that failed builds do not create deployments, and that temporary files are removed. This step hardens the lifecycle from 06.03/06.04 into a reliable, queryable state machine with proper cleanup and failure semantics.

## Dependencies

* Prior steps in this epic: `06.03` (build engine), `06.04` (tag/push)
* Cross-epic prerequisites: none

## Scope

* Durable build record persistence (survives service restart) with a defined status machine
* Phases: `queued → running(cloning|building|pushing) → succeeded | failed | canceled`
* `GET /v1/builds/{id}` returns phase, timestamps, commit, image (if succeeded), and structured error (if failed)
* `GET /v1/builds` list with basic filtering (status, service)
* Guaranteed workspace cleanup on all exit paths (success/failure/cancel/crash-recovery)
* Cancel endpoint `POST /v1/builds/{id}/cancel`
* Invariant: `image` is set only on `succeeded`

## Explicitly out of scope

* Control image-ref recording (06.06)
* Retention/GC policy tuning (basic retention documented)
* Metrics dashboards (epic 12)

## Architecture

```text
status machine:
  queued → running → {cloning → building → pushing} → succeeded
                                   │                └→ failed
                                   └→ failed (build)
        (any) → canceled (on cancel)
persistence: build records in a small store (file/embedded DB) so status survives restart
cleanup: defer workspace removal on every terminal transition; sweep orphans on startup
```

## Components

| Component | Purpose |

## Contracts

```text
GET  /v1/builds/{buildId}
  200: {buildId, status, phase, commit?, image?, digest?, startedAt, finishedAt?,
        error?:{code,message}}
GET  /v1/builds?status=&service=       → [BuildRecord]
POST /v1/builds/{buildId}/cancel       → 202 {status:"canceling"} | 409 if terminal
```

Invariant (contract-tested): `image` present ⟺ `status == succeeded`.

## Data model

```text
BuildRecord{ id, repo, ref, commit, status, phase, image?, digest?,
             startedAt, finishedAt?, error?{code,message}, workspacePath }
```

Persisted to a durable store (e.g. embedded KV / JSON file / Postgres — implementer choice; must survive restart).

## Configuration

```text
FORGE_BUILD_STORE_DIR=/var/lib/forge-build
FORGE_BUILD_RETENTION_HOURS=72
FORGE_BUILD_CLEANUP_ON_START=true
```

## Security considerations

* Error details are safe (no secrets, no full env dumps).
* Workspace cleanup prevents source/artifact leakage across builds.
* Cancel only affects the caller's build id; no bulk cancel via API.

## Observability

* Log every phase transition with build id + duration in phase.
* Log cleanup actions and startup orphan sweeps.
* Count builds by terminal status (metric later).

## Failure handling

* Any build error sets `status=failed` with a structured `error{code,message}`; `image` remains unset.
* Service crash mid-build → on restart, mark orphaned `running` builds as `failed` (interrupted) and clean their workspaces.
* Cancel on a terminal build → `409`.
* Cleanup failure → logged and retried by the sweeper; never blocks status reporting.

## Implementation tasks

1. Implement the durable build store (survives restart).
2. Formalize the phase/status machine with the `image ⟺ succeeded` invariant.
3. Add `GET /v1/builds/{id}`, `GET /v1/builds`, and `POST /v1/builds/{id}/cancel`.
4. Guarantee workspace cleanup on every terminal transition.
5. Add a startup sweeper: fail orphaned running builds + remove stale workspaces.
6. Enforce retention policy for old records/logs.

## Acceptance criteria

* Build status is queryable with clear phases, timestamps, commit, and (on success) image + digest.
* Failed builds never expose a deployable image (`image ⟺ succeeded`).
* Workspaces are cleaned on all terminal paths, including crash recovery.
* Status survives a service restart.
* Builds can be canceled; canceling a terminal build returns `409`.
