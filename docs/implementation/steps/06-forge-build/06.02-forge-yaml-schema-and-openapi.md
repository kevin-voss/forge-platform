## Goal

Define the `forge.yaml` manifest schema (service name/port + build dockerfile/context) and the build-job HTTP API contract (create build, get status, stream logs) as an OpenAPI document, plus a parser/validator for `forge.yaml`.

## Why this step exists

Before running builds, both the manifest that describes how to build a repo and the API used to submit/track builds must be specified and validated. Contract-first keeps the build engine (06.03+) and Control integration (06.06) aligned and testable.

## Dependencies

* Prior steps in this epic: `06.01` (service + workspace)
* Cross-epic prerequisites: `contracts/` tree (epic 00); error-envelope convention from Control (`02.06`)

## Scope

* `forge.yaml` schema (JSON Schema) + a Go parser/validator matching the spec example
* `contracts/openapi/forge-build.openapi.yaml` for the build-job API (create/status/logs)
* Build request/response DTOs (Go types) mirroring the contract
* Validation errors surfaced via the shared error envelope
* Example `forge.yaml` + example build request/response fixtures

## Explicitly out of scope

* Actually cloning/building (06.03)
* Push (06.04)
* Persisting build state beyond types (06.05)
* Control image-ref recording (06.06)

## Architecture

```text
forge.yaml (in repo root)
  service: { name, port }
  build:   { dockerfile, context }
        │ parse + validate (JSON Schema + Go)
        ▼
Build API (OpenAPI):
  POST /v1/builds            → create build (repo, ref, forgeYamlPath?)
  GET  /v1/builds/{id}       → status
  GET  /v1/builds/{id}/logs  → logs (stream)
```

## Contracts

`forge.yaml` (matches the spec):

```yaml
service:
  name: api
  port: 8080
build:
  dockerfile: Dockerfile
  context: .
```

Build API:

```text
POST /v1/builds
  body: {"repo":"file:///fixtures/app","ref":"main","forgeYamlPath":"forge.yaml"}
  202:  {"buildId":"<id>","status":"queued"}
GET  /v1/builds/{buildId}          → {buildId,status,image?,commit?,startedAt,finishedAt?,error?}
GET  /v1/builds/{buildId}/logs     → streamed/plain build logs
```

Validation failures use the shared `{"error":{...}}` envelope.

## Data model

Types only (persistence in 06.05):

```text
Manifest{ Service{name,port}, Build{dockerfile,context} }
BuildRequest{ repo, ref, forgeYamlPath? }
BuildRecord{ id, status, image?, commit?, startedAt, finishedAt?, error? }
```

## Configuration

```text
FORGE_DEFAULT_FORGE_YAML=forge.yaml
```

## Security considerations

* Validate `forge.yaml` fields (name pattern, port range, context/dockerfile paths without traversal outside the repo).
* Reject manifests requesting a build context outside the checked-out repo.
* No secrets in examples.

## Observability

* Log manifest parse results (service name, dockerfile) at build submit (in 06.03); here, log validation outcomes in tests.

## Failure handling

* Missing/invalid `forge.yaml` → validation error envelope with the offending field.
* Port out of range / bad name → `400`.
* Malformed build request → `400`.
