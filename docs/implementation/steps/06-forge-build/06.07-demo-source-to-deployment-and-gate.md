## Goal

Deliver `demos/06-source-to-deployment`, which takes a local fixture Git repository through the full pipeline — build → push → record image on Control → deploy via Runtime → access via Gateway — proving a Git commit can become a running, routed service, and serving as the epic acceptance gate.

## Why this step exists

The spec defines the source-to-deployment demo and acceptance criteria (a commit becomes a running routed service, build logs accessible, failed builds create no deployments, tags include commit+build IDs, temp files removed, demo passes). This step composes Build with Control, Runtime, and Gateway into one falsifiable end-to-end check.

## Dependencies

* Prior steps in this epic: `06.01`–`06.06` (full build service)
* Cross-epic prerequisites: local registry (00); Control (02); Runtime (04); Gateway (05); `make demo` runner

## Scope

* `demos/06-source-to-deployment/` with `README.md`, `run.sh`, and a fixture Git repo (with `Dockerfile` + `forge.yaml`)
* Script: bring up registry + Control + Runtime + Gateway + Build; create app/service in Control; submit a build for the fixture repo; stream/verify logs; verify push to `localhost:5000`; verify image recorded on the service; create/await a deployment; access the service via the Gateway
* Negative case: a build with a broken Dockerfile → `failed`, no deployment created
* `make demo DEMO=06` wiring; epic acceptance

## Explicitly out of scope

* Rolling updates / rollback (epic 07)
* Multi-node (epic 08)
* Remote Git / private registry auth

## Architecture

```text
make demo DEMO=06
   │
   ▼
demos/06-source-to-deployment/run.sh
   ├── compose up registry control runtime gateway build
   ├── create project/env/app/service in Control
   ├── POST /v1/builds (fixture repo) → stream logs → succeeded
   ├── verify image in localhost:5000 + recorded on Control service
   ├── create deployment (or autoDeploy) → Runtime runs it → ready
   ├── curl service via Gateway hostname → 200
   └── negative: broken Dockerfile build → failed, no deployment
```

## Observability

* `run.sh` prints each phase + pass/fail; on failure dumps build logs, Control/Runtime/Gateway logs, registry catalog, and `docker ps`.

## Failure handling

* Any assertion failure → non-zero exit with context.
* Build/deploy/route timeouts → fail with logs.
* Cleanup on exit (trap): delete deployment, remove images/workspaces, `compose down`.
