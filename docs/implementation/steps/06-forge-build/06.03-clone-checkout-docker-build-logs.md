## Goal

Implement the build engine core: accept a build request, clone the repo and checkout the ref into an isolated workspace, read/validate `forge.yaml`, run `docker build` against the specified Dockerfile/context with a timeout, and stream build logs to clients in real time.

## Why this step exists

This is the heart of the build service — turning source into an image. Tag/push (06.04), status (06.05), and Control integration (06.06) all operate on the result of this step.

## Dependencies

* Prior steps in this epic: `06.01` (Docker + workspace), `06.02` (manifest + API contract)
* Cross-epic prerequisites: a fixture Git repo with a Dockerfile + `forge.yaml`

## Scope

* `POST /v1/builds` (async): enqueue a build, return a build id + `queued/running`
* Clone the repo (local path / `file://`) and checkout the commit/ref into `workspace/<buildId>`
* Read + validate `forge.yaml` (06.02)
* Run `docker build` (Docker client or `docker` CLI) with `--file`/context from the manifest, tagging a temporary local tag
* Stream logs via `GET /v1/builds/{buildId}/logs` (chunked/SSE) and capture them for later retrieval
* Enforce a build timeout; cancel the build on timeout
* Bounded concurrency (worker pool)

## Explicitly out of scope

* Tagging with commit/build IDs + push to registry (06.04)
* Full status model + failure taxonomy (06.05 — this step reports basic running/succeeded/failed)
* Control image-ref recording (06.06)
* Workspace cleanup policy details (basic cleanup here; finalized in 06.05)

## Architecture

```text
POST /v1/builds → enqueue → worker:
  ├── clone repo → checkout ref → workspace/<buildId>
  ├── read+validate forge.yaml
  ├── docker build -f <dockerfile> <context>  (timeout)
  │      └── stream stdout/stderr → log buffer + live subscribers
  └── on success: image built (local temp tag)   → 06.04 tags/pushes

GET /v1/builds/{id}/logs  → stream from buffer + live
```

## Contracts

```text
POST /v1/builds
  body: {"repo":"file:///fixtures/app","ref":"main","forgeYamlPath":"forge.yaml",
         "service":"api","project":"acme"}
  202:  {"buildId":"<id>","status":"queued"}
GET  /v1/builds/{buildId}/logs   → streamed build logs (text/event-stream or chunked)
```

(Status endpoint fully specified in 06.05; a minimal `GET /v1/builds/{id}` returning current state is acceptable here.)

## Data model

In-memory (persisted/finalized in 06.05):

```text
BuildRecord{ id, repo, ref, commit, status(queued|running|succeeded|failed),
             localImageTag, startedAt, finishedAt?, error?, logRef }
```

Logs captured to a file under the workspace and/or an in-memory ring buffer.

## Configuration


## Security considerations

* Clone only allowed sources (local path / `file://` in this epic); reject arbitrary remote URLs unless explicitly enabled.
* Build context restricted to the checked-out workspace (no host path escape).
* Do not echo environment secrets into build logs; build args with secrets are out of scope here.
* Each build is isolated in its own workspace dir.

## Observability

* Log build lifecycle: queued, cloning, checked-out commit, build start/end, duration.
* Stream logs with the build id; count active build jobs.

## Failure handling

* Clone/checkout failure (bad repo/ref) → build `failed` with a clear error; workspace cleaned.
* `docker build` non-zero exit → build `failed`; logs retained; no image produced.
* Timeout → cancel the build process, mark `failed` (timeout), clean up.
* Concurrency limit reached → builds wait in the queue (not rejected).

## Acceptance criteria

* A build request clones + checks out the ref, validates `forge.yaml`, and runs `docker build` producing a local image on success.
* Build logs stream in real time and are retained for later retrieval.
* A build timeout cancels the build and marks it failed.
* Failed clone/checkout/build produce a `failed` status with useful logs and no image.
* Each build runs in an isolated workspace that is cleaned up.
