## Goal

Tag a successful build's image with a deterministic reference that encodes the commit SHA and build id, push it to the local OCI registry at `localhost:5000`, and expose the resulting fully-qualified image reference on the build record for later deployment.

## Why this step exists

Runtime deploys images by reference from a registry. The spec requires image tags that include commit and build IDs and a push to the local registry. This step turns the locally-built image (06.03) into a deployable, addressable artifact.

## Dependencies

* Prior steps in this epic: `06.03` (successful local build), `06.02` (manifest → service name), `06.01`
* Cross-epic prerequisites: local registry at `localhost:5000` (epic 00)

## Scope

* Compute image reference(s): `localhost:5000/<project>-<service>:<shortSha>-<buildId>` plus a moving `:latest`
* `docker tag` the local build image to the computed reference(s)
* `docker push` to `localhost:5000` (anonymous, dev)
* Record the pushed `image` reference on the build record
* Handle push retries/verification (image digest recorded)

## Explicitly out of scope

* Private registry auth (deferred)
* Image signing / SBOM / scanning
* Control image-ref recording on the service (06.06)
* Full status/failure taxonomy (06.05 — this step reports push success/failure)

## Architecture

```text
build succeeded (local image) →
  ref = localhost:5000/<project>-<service>:<shortSha>-<buildId>
  latest = localhost:5000/<project>-<service>:latest
  docker tag localImage ref ; docker tag localImage latest
  docker push ref ; docker push latest
  verify (digest) → record BuildRecord.image = ref, digest
```

## Components

| Component | Purpose |

## Contracts

Build record (extended) now includes on success:

```text
{ "buildId","status":"succeeded","image":"localhost:5000/acme-api:ab12cd3-<buildId>",
  "digest":"sha256:...","commit":"ab12cd3...", ... }
```

No new HTTP endpoint; `GET /v1/builds/{id}` reflects the pushed image.

## Data model

Adds `image`, `digest` to the `BuildRecord` (from 06.03). Persistence finalized in 06.05.

## Configuration

```text
FORGE_REGISTRY=localhost:5000
FORGE_IMAGE_NAME_PATTERN={project}-{service}
FORGE_PUSH_LATEST=true
FORGE_PUSH_RETRIES=3
```

## Security considerations

* Anonymous push to the local dev registry only; do not push to external registries in this epic.
* Validate computed tags to avoid injection via project/service names (sanitize to registry-safe chars).

## Observability

* Log tag(s) computed, push start/finish per tag, and resulting digest.
* Log push retries.

## Failure handling

* `docker tag` failure → build `failed` (post-build) with cause.
* `docker push` failure → retry up to `FORGE_PUSH_RETRIES`, then mark failed; do not report a usable image.
* Registry unreachable → clear error; no partial `image` recorded.
