## Goal

Connect Build to Control: when a build succeeds, record the resulting image reference against the target Control service so a deployment can be created from it â€” making the built artifact a first-class, deployable input to the rest of the platform.

## Why this step exists

A pushed image is only useful if the platform knows about it. The spec integrates Build with Control; recording the image ref on the service is what lets the demo (06.07) deploy the freshly-built image via Runtime and route it via Gateway.

## Dependencies

* Prior steps in this epic: `06.04` (pushed image), `06.05` (durable status)
* Cross-epic prerequisites: Control services API (`02.04`) + deployments API (`02.05`); the shared error envelope (`02.06`)

## Scope

* On successful build, call Control to record the image ref on the target service (via a service update or a build-artifact record)
* Build request carries `project`/`service` (or `serviceId`) so Build knows the target
* Optional: create a deployment automatically when the build request asks for it (`autoDeploy: true`) â€” otherwise just record the image and let the demo/CLI create the deployment
* Idempotent recording (safe retries) using Control idempotency (`02.06`)
* Surface the linked `serviceId` + recorded image on the build record

## Explicitly out of scope

* Runtime deployment execution (epic 04 does this when a deployment is created)
* Gateway routing (epic 05)
* Reconciliation (epic 07)
* Auth between services (epic 09; dev mode)

## Architecture

```text
build succeeded (image ref from 06.04) â†’
  Control.recordImage(serviceId, image, digest, commit, buildId)   [idempotent]
  if request.autoDeploy:
     Control.createDeployment(serviceId, image, environmentId)     [idempotent]
  update BuildRecord.serviceId + linkedDeploymentId?
```

> If Control lacks a dedicated "record image on service" endpoint, this step documents the expected contract (below) and uses the closest existing mechanism: create/patch the service's current image field, or create a deployment referencing the image. The chosen mechanism is recorded here as the interim.

## Contracts

Build â†’ Control (consumed / documented):


## Security considerations

* Dev-mode auth to Control (epic 09 later).
* Only record images Build actually produced+pushed; never accept an arbitrary externally-supplied image to record.
* Idempotency keys derived from build id prevent duplicate records/deploys on retry.

## Observability

* Log the Control call (record image / create deployment) with build id + serviceId + resulting ids.
* Log and retry transient Control failures.

## Failure handling

* Control unreachable â†’ build remains `succeeded` (image is pushed) but flagged `imageRecorded=false`; retried with backoff; surfaced on the build record.
* Recording failure does not delete the pushed image.
* Auto-deploy failure â†’ surfaced distinctly from build failure (the build itself succeeded).
* Idempotent retries never create duplicate deployments (relies on Control 02.06).
