## Goal

Publish a language-agnostic runtime contract that every Forge deployable must satisfy: listen port, health endpoints, identity/info response, structured logs, configuration via environment variables, and graceful shutdown.

## Why this step exists

Later demo apps and the shared validator need a single written source of truth. Documenting first avoids five divergent interpretations of `specs.md` §2.2 / Step 01.

## Dependencies

* Epic `00-repository-foundation` complete (docs tree, `contracts/` placeholders, port map)

## Scope

* Human-readable runtime contract under `docs/contracts/`
* Architecture note linking containers as the runtime boundary
* Machine-readable OpenAPI for the required HTTP surface
* JSON Schema for structured log lines
* Example env files / response payloads under `contracts/examples/`
* Host port reservations for demo apps in `docs/operations/ports.md`
* Explicit decisions for ambiguous spec items (see Open questions in the epic)

## Explicitly out of scope

* Any demo application source code
* Contract test runner implementation
* OpenTelemetry instrumentation requirements for demos
* Platform service SDKs
* `/metrics` as a hard requirement (document as recommended only)

## Architecture

```text
docs/contracts/runtime-contract.md   ← human contract
contracts/openapi/runtime-contract.openapi.yaml
contracts/examples/runtime-*/...
        │
        ▼
Future: tools/contract-validator + demos/01-* apps
```

Workload contract (normative for this epic):

```text
Listen: PORT (integer; process binds 0.0.0.0:PORT)
GET /health/live   → 200 when process is alive
GET /health/ready  → 200 when ready to receive traffic
GET /             → 200 JSON identity (service, language, status, …)
Logs: stdout JSON lines with required fields
Signals: SIGTERM → drain and exit 0 within grace period
```

## Data model

No persistence. Identity response (minimum fields):

```json
{
  "service": "demo-go-api",
  "language": "go",
  "status": "running"
}
```

Recommended additional fields (document as optional for epic 01): `version`, `uptime_seconds`.

## Configuration

Normative demo/workload env vars for epic 01:

```text
PORT                 # listen port (required; 12-factor style)
FORGE_SERVICE_NAME   # service identity in logs /info
FORGE_SERVICE_VERSION
FORGE_LOG_LEVEL      # debug|info|warn|error
FORGE_ENV            # development|… (optional for demos, documented)
```

Document that platform services later use the fuller `FORGE_*` set from `specs.md` §5.5; demos need only the subset above. If both `PORT` and `FORGE_HTTP_PORT` appear, **`PORT` wins** for binding (record this decision in the contract doc).

## Security considerations

* No secrets in examples
* Health and `/` are unauthenticated by design for local demos
* Do not require TLS in epic 01

## Observability

* Structured JSON logs to stdout are required
* OTEL is **recommended / deferred** — mention in docs, not mandatory for demo apps in this epic

## Failure handling

Document expected behavior:

* not ready → `/health/ready` returns non-200
* invalid `PORT` → process exits non-zero at startup
* after SIGTERM → no new work; exit within documented grace (default **10s** for demos)
