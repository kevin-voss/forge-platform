# Step 08.03: First-fit and least-allocated placement strategies

## Status

Not started

## Epic

`08-multi-node-scheduler` → [`../../epics/08-multi-node-scheduler.md`](../../epics/08-multi-node-scheduler.md)

## Goal

Replace the single-node placeholder with real placement across the fleet: implement **first-fit** (place on the first online node with enough free capacity) and **least-allocated** (place on the online node with the most free capacity), selectable by configuration. Placement respects advertised capacity and never over-commits a node.

## Why this step exists

`specs.md` Step 08 requires deterministic distribution and rejecting overloaded nodes. With a live fleet (08.02) available, the scheduler can now make capacity-aware decisions. Two strategies give a simple/predictable option (first-fit) and a balancing option (least-allocated, used by the demo for even distribution).

## Dependencies

* Step `08.02` — node fleet with capacity + allocation + liveness
* Step `08.01` — `Scheduler` interface + placement persistence

## Scope

* `FirstFitScheduler` and `LeastAllocatedScheduler` implementing the `Scheduler` interface
* Capacity check: candidate must have `free.slots >= request.requirements.slots` (plus optional cpu/mem if provided)
* Deterministic tie-breaking (stable node ordering by id) for reproducible tests/demo
* Strategy selection via `FORGE_SCHEDULER_STRATEGY`
* Over-commit protection: reserve the placement's capacity atomically so concurrent placements in one reconcile pass don't double-book a node
* Reflect chosen strategy + node in the placement record and reconcile status

## Explicitly out of scope

* Anti-affinity / spread (08.04)
* Pending queue when no node fits (08.04)
* Reschedule on node loss (08.05)
* Rebalancing existing replicas

## Architecture

```text
Scheduler.place(req):
  nodes = fleet.online()
  candidates = nodes.filter(free >= req.requirements)
  if empty → NoNodeAvailable    (queueing added in 08.04)
  pick:
    first-fit       → candidates.sortedBy(id).first()
    least-allocated → candidates.maxBy(free.slots) (tie → lowest id)
  reserve(node, req)            # atomic in-memory + persisted allocation bump
  return decision(node, strategy, reason)
```

## Components

| Component | Responsibility |
|---|---|
| `FirstFitScheduler` | First online node that fits (stable order) |
| `LeastAllocatedScheduler` | Online node with most free capacity |
| `CapacityReservation` | Atomically reserve slots to prevent double-booking within a pass |
| `SchedulerFactory` | Select impl from `FORGE_SCHEDULER_STRATEGY` |

## Contracts

* `POST /v1/placements` unchanged in shape; `strategy` now `first-fit`/`least-allocated` and `node_id` chosen from the fleet
* `reason` explains the pick (e.g. `least-allocated: node-b free=4`)
* No new endpoints

## Data model

Reuses `placements` and `nodes`. Placement reserves capacity by incrementing `nodes.allocation_json.slots` in the same transaction as inserting the placement row (released later on stop/reschedule).

## Configuration

```text
FORGE_SCHEDULER_STRATEGY    # first-fit | least-allocated (default least-allocated)
```

## Security considerations

* Placement remains internal; no secrets
* Capacity reservation is transactional to avoid race-induced over-commit

## Observability

* Log per placement with candidate set sizes and chosen node
* Metrics: `forge_placement_decisions_total{strategy,node}`, `forge_placement_rejected_no_capacity_total`
* Trace span `scheduler.place` with attributes `strategy`, `candidates`, `node`

## Failure handling

* No candidate fits → `NoNodeAvailable` (queue behavior lands in 08.04; here it returns a typed rejection the reconciler surfaces)
* Concurrent placements → capacity reservation transaction ensures the second sees updated allocation
* Node goes offline between selection and reserve → reservation fails; retry selection excluding it
