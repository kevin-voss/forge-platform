# Step 08.05: Reschedule on node offline

## Status

Not started

## Epic

`08-multi-node-scheduler` → [`../../epics/08-multi-node-scheduler.md`](../../epics/08-multi-node-scheduler.md)

## Goal

When a node is marked `offline` (heartbeat lost), release its placements and reschedule the replicas it was running onto healthy nodes — or queue them if the remaining cluster has no capacity. The deployment's desired replica count is restored on the surviving fleet without operator intervention, and duplicate replicas are never created if the node comes back.

## Why this step exists

`specs.md` Step 08 requires "offline nodes stop receiving workloads" and "lost workloads are recreated elsewhere." Node failure is the whole point of multi-node scheduling; this step closes the loop between liveness detection (08.02), placement (08.03/08.04), and the reconciler (epic 07) so the platform self-heals across nodes.

## Dependencies

* Step `08.02` — offline detection (liveness monitor)
* Step `08.03` — capacity-aware placement
* Step `08.04` — pending queue for the no-capacity case
* Epic `07-deployment-reconciliation` — reconcile loop that recreates replicas

## Scope

* Node-offline handler: on `online→offline`, mark that node's placements as `lost` and free their reserved capacity
* Reschedule: for each lost replica still desired, request a fresh placement (respecting anti-affinity + queue) and let the reconciler create it on the new node
* Fencing/duplicate protection: a returning node's stale replicas are reconciled away (adopted/stopped) so desired count is not exceeded
* Node recovery: when an offline node heartbeats again, it returns to `online` and is eligible for new placements; it does not "reclaim" already-rescheduled replicas
* Emit reschedule transitions to deployment history (epic 07)

## Explicitly out of scope

* Graceful node drain command (nice-to-have; can be a follow-up)
* Preemption/priority during reschedule
* Cross-node data migration (stateless replicas assumed)

## Architecture

```text
LivenessMonitor: node-b online→offline
  → NodeOfflineHandler:
       placements(node-b) → status=lost ; release capacity
       for each lost replica desired: scheduler.place() → node-a or pending
  → reconciler creates replicas on new placements (07.02)

node-b returns (heartbeat):
  → status=online
  → reconciler observes node-b's stale containers; if replica already
    satisfied elsewhere → stop the stale one (no over-count)
```

## Components

| Component | Responsibility |
|---|---|
| `NodeOfflineHandler` | React to offline; mark lost; free capacity; request reschedule |
| `RescheduleRequest` | Fresh placement for a lost replica |
| `StaleReplicaFencer` | On node return, remove replicas that would exceed desired |
| history hook | Record `rescheduled` transitions |

## Security considerations

* Only placements belonging to the offline node are touched
* Fencing prevents a flapping node from silently doubling replica counts
* No secrets involved

## Observability

* Log: `{event=node_offline, node, lost_replicas}` and per reschedule `{replica, from_node, to_node|pending}`
* Metrics: `forge_reschedule_total{result=placed|pending}`, `forge_node_offline_total`, `forge_stale_replicas_fenced_total`
* Trace span `scheduler.reschedule`

## Failure handling

* No capacity on surviving nodes → lost replicas become `pending` (08.04), placed when capacity returns
* Node flaps offline→online within grace → no reschedule triggered (grace timer)
* Reschedule creates on node-a, then node-b returns with old replica → fencer stops the surplus so desired is not exceeded
* Control restart during reschedule → derived from placements (`lost`) + desired; idempotent

## Implementation tasks

1. Add `NodeOfflineHandler` hooked to `LivenessMonitor` online→offline transition (with grace timer).
2. Mark offline node's placements `lost`; release reserved capacity.
3. Request fresh placements for lost-but-desired replicas (via 08.03/08.04 path).
4. Let the epic-07 reconciler create replicas on new placements; record `rescheduled` history.
5. Implement `StaleReplicaFencer` for node recovery to prevent over-count.
6. Add `rescheduled_from_node` column + auditing.

## Manual verification

```bash
make dev   # node-a(4), node-b(4)
DPL=$(curl -s -X POST localhost:4001/v1/deployments \
  -d '{"service_id":"svc_demo","image":"registry.local/demo:v1","desired_replicas":4}' \
  -H 'content-type: application/json' | jq -r .id)
sleep 10
docker ps --filter "label=forge.node=node-b" --filter "label=forge.deployment=$DPL" -q | wc -l   # ~2

docker stop forge-runtime-b
sleep 25   # past heartbeat timeout + grace
curl -s localhost:4001/v1/nodes | jq '.[]|select(.id=="node-b").status'   # "offline"
# all 4 desired replicas now on node-a (capacity permitting)
docker ps --filter "label=forge.node=node-a" --filter "label=forge.deployment=$DPL" -q | wc -l   # expect 4
```

## Acceptance criteria

* An offline node's placements are marked lost and its capacity is freed
* Lost-but-desired replicas are rescheduled onto healthy nodes, or queued pending when no capacity exists
* Desired replica count is restored on the surviving fleet where capacity allows
* A recovered node does not cause replica counts to exceed desired (stale replicas fenced)
* Reschedule transitions are recorded in deployment history; the flow is idempotent across a Control restart
