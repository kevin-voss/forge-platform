## Goal

Add two placement refinements: **anti-affinity** so replicas of the same service spread across distinct nodes when possible, and a **pending queue** so that when the cluster has no node with capacity, unplaceable replicas are queued (not dropped or force-fit) and placed automatically once capacity frees up.

## Why this step exists

`specs.md` Step 08 lists anti-affinity and pending workload handling and requires that "overloaded nodes are rejected." Even distribution improves resilience (a node loss takes down fewer replicas of one service), and queueing prevents over-commit while keeping intent recoverable when capacity returns.

## Dependencies

* Step `08.03` — capacity-aware first-fit / least-allocated strategies + reservation
* Step `08.02` — fleet + free capacity

## Scope

* Anti-affinity filter: prefer nodes not already running a replica of the same service; `soft` (fall back to co-location) vs `hard` (queue instead of co-locating)
* Per-service `anti_affinity: soft|hard` (default soft)
* Pending queue: unplaceable placement requests recorded as `pending` with reason; a queue processor retries placement each scheduler tick
* Automatic drain of the queue when capacity frees (node added, replica stopped)
* Surface pending replicas in reconcile status and node fleet views

## Explicitly out of scope

* Reschedule on node offline (08.05) — though rescheduled replicas will reuse this queue when no capacity exists
* Priority/preemption of pending workloads
* Topology/zone affinity

## Architecture

```text
place(req):
  candidates = fitByCapacity(fleet.online)
  preferred  = candidates.filter(no same-service replica)   # anti-affinity
  if preferred nonempty → pick from preferred (08.03 strategy)
  else if soft          → pick from candidates (co-locate)
  else (hard) or none   → enqueue(req, reason)  → status=pending

queue processor (each tick):
  for pending in fifo order: try place; on success remove from queue
```

## Components

| Component | Responsibility |
|---|---|
| `AntiAffinityFilter` | Prefer/require distinct nodes per service |
| `PendingQueue` | Persist unplaceable requests; FIFO retry |
| `QueueProcessor` | Attempt placement of pending items each tick |
| placement status | `placed` vs `pending` with reason |

## Contracts

* `POST /v1/placements` may now return `{ "status": "pending", "reason": "no node with 1 free slot" }` (HTTP `202`)
* `GET /v1/placements?deployment=...&status=pending` lists queued items
* `PlacementRequest` gains optional `anti_affinity: soft|hard`

## Data model

```sql
ALTER TABLE placements
  ADD COLUMN status TEXT NOT NULL DEFAULT 'placed',   -- placed|pending
  ADD COLUMN anti_affinity TEXT NOT NULL DEFAULT 'soft';
CREATE INDEX idx_placements_pending ON placements(status) WHERE status='pending';
```

## Security considerations

* Queue stores placement intent only (no secrets)
* Hard anti-affinity cannot be used to leak capacity of other projects; placement is per-service within Control's trust boundary

## Observability

* Log enqueue/dequeue with reason; log anti-affinity fallback (soft co-location)
* Metrics: `forge_placements_pending` gauge, `forge_anti_affinity_fallback_total`, `forge_queue_drain_total`
* Trace span `scheduler.queue.process`

## Failure handling

* Queue full → reject new placement with typed error (never silently drop)
* Item unplaceable indefinitely → remains pending (visible), retried each tick; no starvation beyond FIFO fairness
* Node added / replica stopped → queue processor runs and drains eligible items

## Manual verification

```bash
make dev   # node-a(2 slots), node-b(2 slots) => capacity 4
# deploy 6 replicas: 4 placed, 2 pending
DPL=$(curl -s -X POST localhost:4001/v1/deployments \
  -d '{"service_id":"svc_demo","image":"registry.local/demo:v1","desired_replicas":6}' \
  -H 'content-type: application/json' | jq -r .id)
sleep 8
curl -s "localhost:4001/v1/placements?deployment=$DPL" | jq 'group_by(.status)|map({status:.[0].status,count:length})'
# expect placed:4, pending:2

# free capacity → a pending item drains
curl -s -X PATCH localhost:4001/v1/deployments/$DPL -d '{"desired_replicas":5}' -H 'content-type: application/json'
sleep 6
curl -s "localhost:4001/v1/placements?deployment=$DPL&status=pending" | jq length   # expect 1
```

## Acceptance criteria

* Replicas of the same service spread across distinct nodes when capacity allows (soft anti-affinity)
* Hard anti-affinity queues rather than co-locating when no distinct node is available
* Requests that no node can satisfy become `pending` (never over-commit, never silently dropped)
* Pending items are placed automatically when capacity frees (stop or new node)
* Pending placements are visible via API and reconcile status
