## Goal

Model a fleet of Runtime nodes in Control: multiple Runtime agents register with a unique node ID and advertised capacity (slots, optional CPU/memory), send periodic heartbeats, and report current allocation. Control maintains node liveness (`online`/`offline`) from heartbeat freshness and exposes a node fleet API the scheduler will read.

## Why this step exists

08.01 gave a placement seam but only one node. Real placement needs a live inventory of nodes with capacity and liveness. This step turns Runtime's single-node identity (`04.02`) into a multi-node fleet with heartbeats and resource accounting — the input every scheduling strategy consumes.

## Dependencies

* Step `08.01` — scheduler module + placement persistence
* Epic `04-forge-runtime` — node identity + registration/heartbeat API (`04.02`); this step generalizes it to N nodes
* Epic `02-forge-control` — Postgres

## Scope

* `nodes` fleet table: node id, address, advertised capacity (slots + optional cpu_millis/mem_mb), last_heartbeat_at, status
* Registration API: `POST /v1/nodes/register` (idempotent by node id)
* Heartbeat API: `POST /v1/nodes/{id}/heartbeat` carrying current allocation + free capacity
* Liveness evaluator: mark nodes `offline` when heartbeat older than `FORGE_NODE_HEARTBEAT_TIMEOUT_S`
* Node fleet read API: `GET /v1/nodes` (with capacity + allocation + status)
* Runtime agent config to run with a distinct node id + slot capacity (enables demo simulation)

## Explicitly out of scope

* Placement strategies using this data (08.03)
* Anti-affinity + pending queue (08.04)
* Rescheduling on offline (08.05) — this step only marks offline
* Enforcing resource limits inside containers (Runtime concern)

## Architecture

```text
forge-runtime (node A) ─register─▶ Control /v1/nodes
forge-runtime (node B) ─register─▶      nodes table
        │ heartbeat (alloc, free) every N s
        ▼
Control liveness loop: now - last_heartbeat > timeout → status=offline
scheduler (08.03+) reads GET /v1/nodes (online + free capacity)
```

## Contracts

* `POST /v1/nodes/register`:

```json
{ "node_id": "node-a", "address": "http://runtime-a:4102",
  "capacity": { "slots": 4, "cpu_millis": 4000, "mem_mb": 4096 } }
```

* `POST /v1/nodes/{id}/heartbeat`:

```json
{ "allocated": { "slots": 2 }, "free": { "slots": 2 }, "running_replicas": ["dpl_1:0","dpl_1:1"] }
```

* `GET /v1/nodes` → array with `status: online|offline`, capacity, allocation, `last_heartbeat_at`

## Data model

```sql
CREATE TABLE nodes (
  id                TEXT PRIMARY KEY,
  address           TEXT NOT NULL,
  capacity_json     JSONB NOT NULL,
  allocation_json   JSONB NOT NULL DEFAULT '{}',
  status            TEXT NOT NULL DEFAULT 'online',   -- online|offline|draining
  last_heartbeat_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  registered_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
```


## Security considerations

* Registration is internal; a node id collision returns the existing record (no takeover of address without matching id)
* No secrets in fleet payloads
* Auth deferred to epic 09; documented

## Observability

* Log registration + status transitions (`online→offline`)
* Metrics: `forge_nodes_total{status=...}`, `forge_node_free_slots{node=...}`, `forge_node_heartbeat_age_seconds`
* Trace span `node.heartbeat`

## Failure handling

* Missed heartbeats → `offline` after timeout (does not delete the node row)
* Heartbeat from an unknown node → auto-register (log warning) or reject based on `FORGE_NODE_STRICT_REGISTER` (default lenient for demo)
* Control restart → fleet reloaded from DB; liveness recomputed from `last_heartbeat_at`

## Manual verification

```bash
make dev   # brings up two runtime agents node-a, node-b
sleep 6
curl -s localhost:4001/v1/nodes | jq '[.[] | {id,status,free:.free}]'
# expect two online nodes

# simulate node-b loss
docker stop forge-runtime-b
sleep 20
curl -s localhost:4001/v1/nodes | jq '.[] | select(.id=="node-b") | .status'
# expect "offline"
```
