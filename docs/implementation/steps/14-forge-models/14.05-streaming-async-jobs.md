# Step 14.05: Streaming + async jobs

## Status

Not started

## Epic

`14-forge-models` → [`../../epics/14-forge-models.md`](../../epics/14-forge-models.md)

## Goal

Add streaming generation responses (token/chunk streaming over SSE) and an asynchronous job mode for long-running inference, so clients can either stream output or submit-and-poll.

## Why this step exists

`specs.md` Step 14 requires streaming output and asynchronous execution of long-running jobs. Workflows/Agents benefit from streaming diagnoses and from submitting long jobs without holding a connection.

## Dependencies

* `14.04` (synchronous generate/classify/summarize)

## Scope

* `POST /v1/models/{model}/generate?stream=true` → SSE chunk stream
* Async job API: `POST /v1/jobs` (submit), `GET /v1/jobs/{id}` (status/result), `DELETE /v1/jobs/{id}` (cancel)
* In-memory job store with states: `queued|running|succeeded|failed|cancelled`
* Cancellation stops a running job promptly
* Backpressure/timeout handling on streams

## Explicitly out of scope

* Durable job persistence across restart (documented non-goal; in-memory)
* Usage metrics export (14.06)
* Distributed job queue

## Architecture

```text
Streaming: generate(stream=true) → async generator → SSE `data: {chunk}` ... `data: [DONE]`
Async:     POST /v1/jobs {model,task,input} → job_id (queued)
           worker task runs adapter → updates job state/result
           GET /v1/jobs/{id} → {status,result?}
           DELETE /v1/jobs/{id} → cancel
```

## Components

| Component | Purpose |
|---|---|
| SSE streamer | Emit incremental generation chunks |
| `JobStore` | In-memory job registry + state transitions |
| Job worker | Background task executing submitted jobs |
| Cancellation | Cooperative cancel via job flag / task cancel |

## Contracts

* `POST .../generate?stream=true` → `200 text/event-stream`, events `data: {"delta":"..."}` then `data: [DONE]`
* `POST /v1/jobs` `{ "model","task":"generate|classify|summarize|embed","input":... }` → `202 { "job_id","status":"queued" }`
* `GET /v1/jobs/{id}` → `200 { "status","result"?,"error"? }`
* `DELETE /v1/jobs/{id}` → `200 { "status":"cancelled" }` | `409` if terminal
* Extend OpenAPI

## Data model

In-memory:

```text
Job { id, project_id, model, task, input, status, result?, error?, created_at, updated_at }
```

## Configuration

```text
FORGE_MODELS_STREAM_TIMEOUT_SECONDS   # default 60
FORGE_MODELS_JOB_TTL_SECONDS          # default 3600 (completed jobs GC'd)
FORGE_MODELS_MAX_CONCURRENT_JOBS      # default 4
```

## Security considerations

* Jobs scoped to submitting project; cross-project job read → `404`
* Concurrency cap prevents resource exhaustion
* Stream timeout closes stuck connections

## Observability

* Log stream start/end (chunks, duration) and job lifecycle transitions
* Metric: `models_stream_active`, `models_jobs_total{status}`, job duration histogram

## Failure handling

* Stream client disconnect → generation task cancelled
* Job exceeds timeout → `failed` with `timeout` error
* Cancel of terminal job → `409`
* Worker crash → job marked `failed` (not stuck `running`)

* A job exceeding its timeout ends as `failed` (never stuck `running`)
* Jobs are project-scoped; concurrency capped
* Streams cancel work on client disconnect
* OpenAPI updated and valid
