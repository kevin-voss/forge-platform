# Step 14.03: Local embeddings adapter

## Status

Not started

## Epic

`14-forge-models` → [`../../epics/14-forge-models.md`](../../epics/14-forge-models.md)

## Goal

Implement a fully local embeddings adapter and `POST /v1/models/{model}/embed` that produces fixed-dimension vectors deterministically with no external API, satisfying the "at least one local model runs without an external API" acceptance criterion.

## Why this step exists

Embeddings are the first concrete inference capability and a hard dependency of Memory (17). Doing embeddings first gives a testable, deterministic vertical slice that proves the adapter → registry → HTTP path.

## Dependencies

* `14.02` (registry + adapter interface)

## Scope

* `LocalEmbeddingAdapter` producing L2-normalized fixed-dim vectors
* `POST /v1/models/{model}/embed` (single + batch input)
* Deterministic output for CI (same text → same vector)
* Dimension reported by registry matches produced vectors
* Input validation (non-empty, max length, batch size cap)

## Explicitly out of scope

* Generation/classification/summarization (14.04)
* Streaming/async (14.05)
* Usage metrics (14.06)

## Architecture

```text
POST /embed {input: str|[str]} → adapter.embed(texts) → [[float; dim]]
```

Two selectable implementations behind the same adapter:
* `fake`/CI: deterministic hashing → seeded pseudo-random vector, L2-normalized (fully reproducible, no ML deps).
* `local` (optional): a small sentence-transformer loaded on startup for realistic local demos.

Both report the same `embedding_dim` as the registry entry.

## Components

| Component | Purpose |
|---|---|
| `LocalEmbeddingAdapter` | Deterministic + optional real embeddings |
| `embed` route | Validate + dispatch to adapter |
| Normalizer | L2-normalize vectors for cosine use |

## Contracts

* `POST /v1/models/{model}/embed` `{ "input": "text" }` or `{ "input": ["a","b"] }`
  → `200 { "model","embeddings":[[...]], "dim":384, "usage":{"input_count":N} }`
* `404` unknown model; `422` if model lacks `embed` capability or invalid input
* Extend OpenAPI

## Data model

No persistence. Vector is `list[float]` length = registry `embedding_dim`.

## Configuration

```text
FORGE_MODELS_EMBED_MAX_BATCH      # default 64
FORGE_MODELS_EMBED_MAX_CHARS      # default 8192 per input
FORGE_MODELS_LOCAL_MODEL_PATH     # optional; enables real model when set
```

## Security considerations

* Input size caps prevent resource exhaustion
* No external network calls in CI backend

## Observability

* Log embed calls with model, input_count, latency
* Metric: `models_embed_requests_total`, `models_embed_latency_seconds`

## Failure handling

* Empty/oversized input → `422`
* Batch over cap → `422 batch_too_large`
* Model missing `embed` capability → `422 capability_unsupported`

## Manual verification

```bash
BASE=localhost:4300
curl -fsS -XPOST $BASE/v1/models/local-embed-small/embed \
  -H 'content-type: application/json' -d '{"input":"hello forge"}' \
  | python3 -c 'import sys,json;d=json.load(sys.stdin);print(len(d["embeddings"][0])==d["dim"])'
# determinism
a=$(curl -fsS -XPOST $BASE/v1/models/local-embed-small/embed -H 'content-type: application/json' -d '{"input":"x"}')
b=$(curl -fsS -XPOST $BASE/v1/models/local-embed-small/embed -H 'content-type: application/json' -d '{"input":"x"}')
test "$a" = "$b" && echo "deterministic OK"
```

* Output is deterministic for identical input
* Produced dimension matches the registry's `embedding_dim`
* Batch input supported up to a configurable cap; invalid input → `422`
* OpenAPI updated and valid
