## Goal

Introduce, inside Forge Control, a persisted model of **desired** deployment state (image + replica count) and **actual** deployment state (observed replicas), plus a reconciliation controller skeleton that runs on a fixed interval, reads both, computes a diff, and logs the diff **without acting on it yet**. Expose a read-only reconciliation status API so the diff and controller health are observable.

## Why this step exists

Every later step in this epic (reconcile loop, rolling update, rollback, history) needs a shared, durable representation of desired vs actual and a controller lifecycle to hang logic on. Building the model and an inert loop first keeps the risky convergence logic out of the initial change and gives a testable seam (diff computation) before any container is touched.

* Epic `02-forge-control` — Deployments (desired state) API + audit records (`02.05`), Postgres migrations infra (`02.02`)

* No prior steps in this epic

## Scope

* A `DesiredState` model per service/deployment: target image ref, target replica count, rollout policy fields (batch size, timeout) with defaults
* An `ActualState` view: list of observed replicas with per-replica status (`pending`, `running`, `ready`, `failed`, `stopped`)
* Postgres migrations for desired-state persistence and a `reconcile_status` snapshot table
* A `ReconciliationController` skeleton: background loop on a configurable interval that loads desired + actual, computes a `ReconcilePlan` (list of intended actions), and **only logs** it
* A clean extract seam: controller depends on `RuntimeClient` and `DeploymentStore` interfaces, not concrete Control internals
* Read-only API: `GET /v1/deployments/{id}/reconcile` returns desired, actual, computed plan, last-run timestamp
* OpenAPI additions for the new read endpoint

## Explicitly out of scope

* Actually creating, stopping, or restarting any workload (07.02)
* Rolling update / traffic shift (07.03)
* Rollback (07.04)
* Deployment history persistence beyond the current-status snapshot (07.05)
* Multi-node placement (epic 08)

## Architecture

```text
Forge Control (Kotlin/Ktor)
├── deployment/DeploymentStore        (desired state; Postgres)
├── reconcile/ReconciliationController (interval loop; injectable clock)
│     ├── loadDesired()  ← DeploymentStore
│     ├── loadActual()   ← RuntimeClient (04.04/04.07)
│     ├── computePlan(desired, actual) → ReconcilePlan   ← pure, unit-tested
│     └── log(plan)      (NO execution in 07.01)
└── api/ReconcileStatusRoute  GET /v1/deployments/{id}/reconcile
```

The controller loop is start/stop-controlled by the Ktor application lifecycle. `computePlan` is a **pure function** so it can be exhaustively unit-tested without a database or Docker.

## Components

| Component | Responsibility |
|---|---|
| `DesiredState` | Target image + replicas + rollout policy for a deployment |
| `ActualState` / `ReplicaObservation` | Observed replicas and their status from Runtime |
| `ReconcilePlan` | Ordered list of intended actions (`StartReplica`, `StopReplica`, `NoOp`) — not executed yet |
| `computePlan` | Pure diff: desired vs actual → plan |
| `ReconciliationController` | Interval loop; loads state, computes plan, logs, writes snapshot |
| `RuntimeClient` (interface) | Read actual state (mockable) |
| `ReconcileStatusRoute` | Read-only status API |

## Data model

Postgres (Control DB), new migration:

```sql
-- desired rollout config lives with the deployment row (extend 02 schema)
ALTER TABLE deployments
  ADD COLUMN desired_replicas   INT      NOT NULL DEFAULT 1,
  ADD COLUMN rollout_batch_size INT      NOT NULL DEFAULT 1,
  ADD COLUMN rollout_timeout_s  INT      NOT NULL DEFAULT 120;

CREATE TABLE reconcile_status (
  deployment_id  TEXT PRIMARY KEY REFERENCES deployments(id),
  last_run_at    TIMESTAMPTZ NOT NULL,
  desired_json   JSONB NOT NULL,
  actual_json    JSONB NOT NULL,
  plan_json      JSONB NOT NULL,
  controller_healthy BOOLEAN NOT NULL DEFAULT TRUE
);
```

Actual replica status enum: `pending | running | ready | failed | stopped`.

## Security considerations

* Reconcile status API is read-only; no secrets in the payload (image refs only, no credentials)
* Controller runs with the same trust boundary as Control; no new external surface beyond one read endpoint
* Auth deferred to epic 09 via `FORGE_AUTH_MODE=dev`; documented as temporary

## Observability

* Structured log line per tick: `{deployment_id, desired_replicas, actual_replicas, plan_size, tick_duration_ms}`
* Metric (if metrics wired from 02.07): `forge_reconcile_ticks_total`, `forge_reconcile_plan_actions` gauge
* Trace span `reconcile.tick` around each loop iteration

## Failure handling

* Runtime unreachable → controller logs a warning, sets `controller_healthy=false` in the snapshot, keeps the last plan, and retries next tick (no crash)
* Malformed desired state → skip that deployment, log error, continue others
* Controller loop exception → caught per-deployment so one bad deployment cannot stall the whole loop

* Controller catches a per-deployment exception and still processes the next deployment
