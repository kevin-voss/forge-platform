## Goal

Persist a durable, ordered **deployment history** of every state transition (requested, deploying, deployed, rolling_back, rolled_back, failed) and prove the controller is **restart-safe**: after a Control process restart at any point (mid-rollout, mid-rollback), the controller resumes from persisted state and converges without duplicating containers or losing history.

## Why this step exists

07.02–07.04 keep only a current-status snapshot. `specs.md` Step 07 requires "deployment history records transitions" and "deployment survives controller restart." An auditable timeline is needed for debugging, for the demo's assertions, and as the source later epics (workflows/observe) read. Restart safety must be validated explicitly because reconcile logic runs continuously and crashes are inevitable.

## Dependencies

* Steps `07.02`, `07.03`, `07.04` — the transitions being recorded
* Epic `02-forge-control` — Postgres + audit record patterns (`02.05`)

## Scope

* Append-only `deployment_events` table capturing each transition with timestamp, from/to status, image, replica counts, and reason
* Emit a history event at every controller-driven transition (create, ready, shift, stop, timeout, rollback, complete)
* `GET /v1/deployments/{id}/history` read API (chronological)
* Restart-safety hardening: on startup the controller rebuilds in-memory state from persisted desired/actual + last history entry; running rollouts/rollbacks resume idempotently
* Startup reconciliation guard: adopt existing labeled containers rather than recreating them

## Explicitly out of scope

* UI for history (CLI/Gateway concerns; not this epic)
* Retention/pruning policy (documented as future; keep all rows for now)
* Emitting to the Events service (epic 11) — history is DB-local here
* Multi-node considerations (epic 08)

## Architecture

```text
transition() helper:
  write deployment_events row (append-only)
  update deployments.status

startup:
  load deployments where status in (deploying, rolling_back)
  observe actual via Runtime (adopt by label)
  resume reconcile → converges to desired or completes rollback
```

## Data model

```sql
CREATE TABLE deployment_events (
  id             BIGSERIAL PRIMARY KEY,
  deployment_id  TEXT NOT NULL REFERENCES deployments(id),
  at             TIMESTAMPTZ NOT NULL DEFAULT now(),
  from_status    TEXT NOT NULL,
  to_status      TEXT NOT NULL,
  image          TEXT,
  desired_replicas INT,
  actual_replicas  INT,
  reason         TEXT
);
CREATE INDEX idx_deployment_events_dpl_at ON deployment_events(deployment_id, at);
```

## Security considerations

* History stores image refs + reasons only; no secrets or tokens
* Append-only design prevents silent rewriting of the audit trail

## Observability

* Each transition logged and recorded; log includes the `deployment_events.id`
* Metric: `forge_deployment_transitions_total{to_status=...}`
* Startup recovery logs a summary: `{adopted_containers, resumed_deployments}`

## Failure handling

* Status update and history append happen in one DB transaction → no status change without a history row (and vice versa)
* On startup, a deployment in `deploying`/`rolling_back` with all-ready target → mark `deployed`/complete rollback; otherwise continue converging
* Orphan container (labeled but no live deployment) → logged and stopped (garbage collection)

## Manual verification

```bash
make dev
# after a v1→v2 rollout and a broken v3 rollback (DPL known):
curl -s localhost:4001/v1/deployments/$DPL/history | jq '.events[] | {from,to,image}'
# expect a chronological trail including deployed(v2) and rolled_back

# restart safety: begin a rollout then restart control
curl -s -X PATCH localhost:4001/v1/deployments/$DPL -d '{"image":"registry.local/demo:v2b"}' -H 'content-type: application/json'
docker restart forge-control
sleep 15
BEFORE=$(docker ps --filter "label=forge.deployment=$DPL" -q | wc -l | tr -d ' ')
sleep 5
AFTER=$(docker ps --filter "label=forge.deployment=$DPL" -q | wc -l | tr -d ' ')
test "$BEFORE" = "$AFTER" && echo "no duplicate containers after restart"
```

## Acceptance criteria

* Every controller-driven transition is recorded append-only with timestamp, from/to, image, replica counts, reason
* `GET /v1/deployments/{id}/history` returns the ordered transition trail
* Status and history are updated atomically (never one without the other)
* After a Control restart mid-rollout/rollback, the controller adopts existing containers by label and converges without duplicates
* Orphaned labeled containers with no deployment are cleaned up on startup
