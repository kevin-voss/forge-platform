# Step 07.04: Unhealthy rollout → automatic rollback

## Status

Not started

## Epic

`07-deployment-reconciliation` → [`../../epics/07-deployment-reconciliation.md`](../../epics/07-deployment-reconciliation.md)

## Goal

When a rolling update cannot reach readiness within the deployment timeout, the controller automatically **rolls back** to the last known-good version: it stops the failed new-version replicas, restores the previous version's replica count, and marks the deployment `rolled_back`. The system never gets stuck serving a broken release or a half-migrated state.

## Why this step exists

Rolling update (07.03) holds when a new replica won't become ready, but holding forever is not acceptable. `specs.md` Step 07 requires "unhealthy release rolls back" and a "deployment timeout." This step adds the timeout, the last-good tracking, and the automated recovery that makes deploys safe.

## Dependencies

* Step `07.03` — rolling update with readiness gate + traffic shift
* Step `07.02` — idempotent start/stop used to restore old version
* Epic `04-forge-runtime` — readiness/status for detecting failure

## Scope

* Track `last_healthy_deployment` (image + replica count) per service, updated when a rollout fully succeeds
* Deployment timeout: if a rollout does not complete within `rollout_timeout_s` (default 120s), declare it failed
* Rollback procedure: stop new-version replicas, ensure old-version replicas back to desired count, shift traffic back, set deployment status `rolled_back`
* Rollback must itself be safe (no total downtime; old version is known-good and already ran)
* Idempotent: re-running after restart during rollback does not double-act

## Explicitly out of scope

* Human-approval gate before rollback (that richer flow is workflows, epic 16)
* Notifying via events (epic 11) — a log line + status is enough here
* Deployment history table (07.05) — this step sets status; history persistence is next
* Multi-node reschedule (epic 08)

## Architecture

```text
rollout start → timer(rollout_timeout_s)
  if all replicas reach target version & ready before timeout:
      status = deployed
      last_healthy = target
  else (timeout OR replica marked failed):
      status = rolling_back
      stop target-version replicas
      restore last_healthy image to desired replicas   (reuse 07.02/07.03 mechanics)
      shift traffic back to last_healthy
      status = rolled_back
```

## Components

| Component | Responsibility |
|---|---|
| `RolloutTimer` | Track elapsed rollout time vs `rollout_timeout_s` (injectable clock) |
| `HealthEvaluator` | Decide rollout success/failure from readiness + failures |
| `Rollbacker` | Execute rollback to `last_healthy` safely |
| `LastHealthyStore` | Persist/read last known-good deployment per service |

## Contracts

* `GET /v1/deployments/{id}/reconcile` gains `status: deploying|deployed|rolling_back|rolled_back|failed` and `last_healthy_image`
* No new external mutation API; rollback is automatic

## Security considerations

* Rollback only ever restores a version that previously ran successfully (`last_healthy`), never an arbitrary image
* No new external surface

## Observability

* Log rollout outcome: `{deployment_id, outcome=deployed|rolled_back, elapsed_ms, target_image, restored_image}`
* Metrics: `forge_rollout_result_total{result=deployed|rolled_back}`, `forge_rollback_duration_ms`
* Trace span `reconcile.rollback` with the restore steps

## Failure handling

* Timeout fires but new replica becomes ready in the same tick → prefer success (evaluate readiness before declaring failure)
* Rollback itself fails to restore (Runtime down) → retry next tick; status stays `rolling_back` until restored; never leaves zero ready replicas if old ones still exist
* Controller restart during rollback → re-derive: target replicas still failing + `last_healthy` known → continue rollback idempotently

## Implementation tasks

1. Add `status` to deployments and `last_healthy_deployment_id` to services (migration).
2. Implement `RolloutTimer` with injectable clock; start on rollout begin.
3. Implement `HealthEvaluator` (success = all target replicas ready; failure = timeout or a target replica `failed`).
4. On success: set `deployed`, update `last_healthy`.
5. Implement `Rollbacker`: stop target replicas, restore `last_healthy` image to desired count via existing reconcile mechanics, shift traffic, set `rolled_back`.
6. Guard the whole flow for idempotency across restarts.

## Manual verification

```bash
make dev
# assume v2 currently healthy at replicas=2 (from 07.03 flow), DPL known
# deploy a broken v3 whose /health/ready always returns 503
curl -s -X PATCH localhost:4001/v1/deployments/$DPL \
  -d '{"image":"registry.local/demo:v3-broken"}' -H 'content-type: application/json'

# wait past the timeout
sleep 130
curl -s localhost:4001/v1/deployments/$DPL/reconcile | jq '{status, last_healthy_image}'
# expect: status "rolled_back", last_healthy_image ...demo:v2

# traffic serves v2 again
curl -s -H 'Host: demo.localhost' localhost:4000/ | jq -r .version   # expect v2
# no v3 containers survive
docker ps --filter "label=forge.image=registry.local/demo:v3-broken" -q | wc -l   # expect 0
```

## Acceptance criteria

* A rollout that does not reach readiness within `rollout_timeout_s` is declared failed
* The controller automatically restores the last known-good version at the desired replica count
* Traffic returns to the healthy version; no broken-version replicas remain
* Deployment status transitions `deploying → rolling_back → rolled_back` (or `deployed` on success)
* Rollback is idempotent across a controller restart and never leaves zero ready replicas when a healthy version exists
