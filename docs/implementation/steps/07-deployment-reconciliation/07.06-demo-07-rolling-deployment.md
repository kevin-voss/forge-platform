# Step 07.06: Demo `07-rolling-deployment` + epic gate

## Status

Not started

## Epic

`07-deployment-reconciliation` → [`../../epics/07-deployment-reconciliation.md`](../../epics/07-deployment-reconciliation.md)

## Goal

Assemble the end-to-end demo `demos/07-rolling-deployment` that proves the whole epic: a healthy rolling update from v1 → v2 with no failed requests, and an intentionally unhealthy v3 that is automatically rolled back to v2. Wire it into `make demo DEMO=07` as the epic acceptance gate.

## Why this step exists

Individual steps prove pieces in isolation; the epic is only "done" when a single command demonstrates zero-downtime rollout and automatic rollback against the real Control + Runtime + Gateway stack. This is the gate referenced by `MASTER_PLAN.md` demo ownership map.

## Dependencies

* Steps `07.01`–`07.05` — full reconciliation controller with history + restart safety
* Epic `04-forge-runtime`, `05-forge-gateway`, `02-forge-control` running under Compose
* Epic `06-forge-build` (optional) or pre-built demo images `demo:v1`, `demo:v2`, `demo:v3-broken`

## Scope

* `demos/07-rolling-deployment/` Compose overlay bringing up Control + Runtime + Gateway + demo service
* Three demo image versions: v1, v2 (visible response change), v3-broken (readiness always fails)
* Acceptance script running Scenario A (rolling update, no-downtime probe) and Scenario B (rollback)
* `Makefile` demo target `make demo DEMO=07`
* README documenting how to run and what it proves

## Explicitly out of scope

* New controller features (all landed in 07.01–07.05)
* CI tuning of image build times (tracked as an open question in the epic)
* Multi-node behavior (epic 08 demo)

## Architecture

```text
demos/07-rolling-deployment/
├── docker-compose.yml         # control, runtime, gateway, demo service
├── apps/demo/                  # tiny app with /,/health/live,/health/ready + VERSION env
│   ├── v1, v2, v3-broken image build args or tags
├── run.sh                      # orchestrates scenarios + assertions
└── README.md
```

Demo app reuses the epic-01 runtime contract (PORT, `/health/*`, identity `/`). v3-broken returns 503 from `/health/ready`.

## Components

| Component | Responsibility |
|---|---|
| Demo service app | Contract-compliant app returning its `VERSION` on `/` |
| `run.sh` | Deploy v1 → v2 with a no-downtime probe; deploy v3 → assert rollback |
| No-downtime probe | Loop of gateway requests; fails the demo on any non-2xx |
| History assertion | Verify `v1→v2 deployed` and `v3 rolled_back to v2` |

## Contracts

* Uses Control deployment + reconcile + history APIs (07.x)
* Uses Gateway host routing (`demo.localhost`) from epic 05
* No new contracts introduced

## Data model

None beyond what 07.x created. Demo asserts against `deployment_events` via the history API.

## Security considerations

* Demo runs with `FORGE_AUTH_MODE=dev` (auth arrives in epic 09); documented in README
* No secrets in demo images or compose files

## Observability

* `run.sh` prints each phase; on failure prints controller logs + `GET /history`
* Demo asserts controller emitted expected transitions (observability doubles as the assertion source)

## Failure handling

* Any failed probe request during Scenario A → demo exits non-zero
* Rollback not observed within timeout in Scenario B → demo exits non-zero with controller logs
* Cleanup trap tears down Compose + removes demo containers on exit

## Implementation tasks

1. Create the demo app (contract-compliant) with `VERSION` env and a build path for v1/v2/v3-broken.
2. Write `docker-compose.yml` wiring Control, Runtime, Gateway, and the demo service on the demo network.
3. Write `run.sh`: deploy v1 replicas=2; start no-downtime probe; roll to v2; assert no probe failures and `/` returns v2.
4. Extend `run.sh`: deploy v3-broken; wait past timeout; assert status `rolled_back`, `/` returns v2, no v3 containers.
5. Assert history via `GET /v1/deployments/{id}/history`.
6. Add `make demo DEMO=07` wiring + README.

## Acceptance criteria

* `make demo DEMO=07` runs both scenarios and exits 0
* Scenario A performs a rolling update with zero failed probe requests and ends on v2
* Scenario B deploys a broken v3, and the controller automatically rolls back to v2 with no surviving v3 containers
* Deployment history reflects both the successful rollout and the rollback
* Demo app satisfies the epic-01 runtime contract
* Compose stack is fully torn down on exit
