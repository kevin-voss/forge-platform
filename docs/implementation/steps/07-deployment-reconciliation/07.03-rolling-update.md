## Goal

Support changing a deployment's image version with **zero total downtime**. When desired image differs from actual, the controller starts a new-version replica, waits for it to become ready, shifts traffic to it (via Gateway health-aware selection), then stops one old-version replica — repeating one replica at a time until all replicas run the new version.

## Why this step exists

07.02 converges a single version but a version change would naively stop everything and restart. Rolling updates are the headline capability of `specs.md` Step 07 ("healthy rolling deployment avoids total downtime") and are the substrate rollback (07.04) reverts along.

## Dependencies

* Step `07.02` — idempotent create/stop/observe of replicas
* Epic `05-forge-gateway` — health-aware upstream selection so only ready replicas receive traffic (`05.04`), route sync from Control (`05.03`)
* Epic `04-forge-runtime` — per-workload readiness (`04.04`)

## Scope

* Version-aware plan: `computeRollingPlan(desired, actual)` yields ordered steps (start new, wait ready, stop old) honoring `rollout_batch_size`
* Readiness gate: a new replica counts as "shiftable" only when Runtime reports `ready`
* Traffic shift: ensure the ready new replica is registered as a Gateway upstream and old replicas are drained before stop
* Maintain a minimum-available invariant: never drop below `desired_replicas - batch_size` ready replicas during rollout
* Surface rollout progress in `reconcile_status` (e.g. `phase: rolling`, `updated_replicas: 1/2`)

## Explicitly out of scope

* Rollback on failure/timeout (07.04)
* Deployment history persistence (07.05)
* Canary percentage or weighted splitting (not in scope for the epic)
* Multi-node placement (epic 08)

## Architecture

```text
desired.image = v2, replicas = 2
actual = [v1, v1]

rolling steps (batch_size=1):
  1. start v2#0
  2. wait until v2#0 ready (poll Runtime health)
  3. register v2#0 in Gateway; confirm receiving traffic
  4. drain + stop v1#0
  5. repeat for v2#1 / v1#1
  → actual = [v2, v2]
```

Invariant checked each tick: `readyReplicas >= desired_replicas - rollout_batch_size`.

## Security considerations

* Only ready replicas are advertised to Gateway (avoids routing to half-started workloads)
* Draining prevents dropping in-flight requests on stop where Runtime supports graceful stop (`04.06`)
* No secrets involved

## Observability

* Log per rollout step: `{deployment_id, from_image, to_image, replica_index, step, ready_wait_ms}`
* Metrics: `forge_rollout_in_progress` gauge, `forge_rollout_step_total`
* Trace: span `reconcile.rolling_update` with child spans per replica swap

## Failure handling

* New replica never becomes ready within `FORGE_READINESS_MAX_WAIT_S` → do **not** stop old replica; hold rollout and mark degraded (rollback decision is 07.04)
* Gateway unreachable → do not stop old replicas (fail safe: keep serving from old until traffic can shift)
* Controller restart mid-rollout → re-derive from actual (some new, some old) and continue; invariant prevents downtime

## Manual verification

```bash
make dev
# deploy v1 replicas=2
DPL=$(curl -s -X POST localhost:4001/v1/deployments \
  -d '{"service_id":"svc_demo","image":"registry.local/demo:v1","desired_replicas":2}' \
  -H 'content-type: application/json' | jq -r .id)
sleep 6

# start a no-downtime probe against the gateway
( for i in $(seq 1 200); do curl -sf -H 'Host: demo.localhost' localhost:4000/ >/dev/null || echo "FAIL@$i"; sleep 0.1; done ) &
PROBE=$!

# roll to v2
curl -s -X PATCH localhost:4001/v1/deployments/$DPL -d '{"image":"registry.local/demo:v2"}' -H 'content-type: application/json'
sleep 12
wait $PROBE   # expect no FAIL lines printed

curl -s -H 'Host: demo.localhost' localhost:4000/ | jq -r .version   # expect v2
```

## Acceptance criteria

* Version change triggers a one-at-a-time rolling update (start new → ready → shift → stop old)
* At least `desired - batch_size` ready replicas exist at all times (no total downtime)
* Traffic only routes to ready replicas; old replicas are drained before stop
* A new replica that never becomes ready holds the rollout without stopping old replicas
* Rollout progress (`phase`, `updated_replicas`) is visible via the reconcile status API
