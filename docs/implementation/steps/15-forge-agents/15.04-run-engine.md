## Goal

Implement the agent run engine that drives a model-guided loop (call model → optionally invoke a tool → observe → repeat) with enforced `max_steps` and `timeout_seconds`, cancellation, and a persisted, auditable run history — using the Models fake backend for deterministic CI.

## Why this step exists

This is the heart of the runtime: it ties agents, tools, permissions, and the model together under hard execution limits. `specs.md` Step 15 requires enforced limits, run history, cancellation, and auditability.

## Dependencies

* `15.03` (tool registry + permissions)
* Epic `14-forge-models` (generate/classify from `14.04`; fake backend for determinism)

## Scope

* `POST /v1/agents/{name}/runs` → start a run (input/context) → `run_id`
* Run loop: bounded by `max_steps`; wall-clock `timeout_seconds`; each step may call one tool via the invoker
* `GET /v1/runs/{id}` → status + step history; `GET /v1/runs` → list (project-scoped)
* `POST /v1/runs/{id}/cancel`
* Persist run + steps (SQLite) for audit
* Deterministic dry-run mode with fake model + mock tools

## Explicitly out of scope

* Real platform tools (15.05)
* Human approval (15.06)
* Seed agents/CLI (15.07)

## Architecture

```text
start run → RunContext(scope, agent, limits)
loop while steps < max_steps and elapsed < timeout:
    model.decide(history) → {final | tool_call}
    if tool_call: invoker.call(...) → observation → append history
    if final: record result → done
enforce timeout via asyncio.wait_for; cancellation via flag/task cancel
persist every step (audit)
```

## Data model

```sql
CREATE TABLE runs (
  id TEXT PRIMARY KEY, project_id TEXT NOT NULL, agent TEXT NOT NULL,
  status TEXT NOT NULL, result TEXT, error TEXT,
  step_count INTEGER NOT NULL DEFAULT 0,
  started_at TEXT NOT NULL, ended_at TEXT
);
CREATE TABLE run_steps (
  id TEXT PRIMARY KEY, run_id TEXT NOT NULL REFERENCES runs(id),
  idx INTEGER NOT NULL, type TEXT NOT NULL,     -- model|tool|final
  tool TEXT, args TEXT, observation TEXT, decision TEXT, ts TEXT NOT NULL
);
```

## Security considerations

* Runs scoped to project; cross-project run read → `404`
* Limits are hard ceilings; a run cannot exceed `max_steps`/`timeout_seconds` even if the model loops
* Every tool decision recorded for audit

## Observability

* Log run lifecycle + per-step decisions
* Metric: `agent_runs_total{status}`, `agent_run_steps`, run duration histogram

## Failure handling

* Exceeds `max_steps` → run ends `failed`/`stopped` with reason `max_steps_exceeded`
* Exceeds timeout → `failed` reason `timeout`
* Model/tool error → step records error; run policy decides fail vs continue
* Cancel → prompt stop; terminal-run cancel → `409`
