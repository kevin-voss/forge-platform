# Step 15.07: Seed agents + CLI `forge agent`

## Status

Not started

## Epic

`15-forge-agents` → [`../../epics/15-forge-agents.md`](../../epics/15-forge-agents.md)

## Goal

Ship the five seed agent definitions (deployment-investigator, log-summarizer, docs-assistant, release-reviewer, infra-health) with least-privilege tool/permission sets, and add a thin `forge agent` CLI (`list`, `run`, `status`, `approve`/`deny`) over the API.

## Why this step exists

`specs.md` Step 15 names the initial agents and the final DX includes `forge agent run deployment-investigator`. This step turns the runtime into something usable and sets up the demo (15.08).

## Dependencies

* `15.06` (approval gate) — full runtime available
* Epic `03-forge-cli` for CLI framework (thin client patterns)

## Scope

* Five seed agent YAMLs with appropriate `tools`/`permissions`/`limits`
* deployment-investigator explicitly includes `runtime.restart` (destructive, gated) but is scoped read-only in permissions except via approval
* `forge agent list|run|status|approve|deny` CLI subcommands
* CLI output formats (table/JSON) + exit codes
* Docs for each seed agent

## Explicitly out of scope

* Demo assembly (15.08)
* Memory retrieval tool (17.05)
* New tools (15.05)

## Architecture

```text
agents/deployment-investigator.yaml
agents/log-summarizer.yaml
agents/docs-assistant.yaml
agents/release-reviewer.yaml
agents/infra-health.yaml

forge agent run <name> --input ... → POST /v1/agents/<name>/runs → poll GET /v1/runs/{id}
forge agent approve <approval-id>  → POST /v1/approvals/{id}/approve
```

## Components

| Component | Purpose |

## Contracts

## Configuration

```text

```

Seed agent examples:

```yaml
# deployment-investigator
name: deployment-investigator
model: local-general
tools: [deployment.read, logs.search, metrics.query, runtime.restart]
permissions: [project:read, deployment:read]   # restart requires approval
limits: { max_steps: 10, timeout_seconds: 120 }
```

## Security considerations

* Seed agents follow least privilege; only investigator declares a destructive tool, and it is approval-gated
* No agent has standing permission to mutate without approval

## Observability

* CLI surfaces run id + request id; logs unchanged from service side

## Failure handling

* CLI: non-zero exit on run failure/denial; clear message on `awaiting_approval`
* Unknown agent → CLI error mapped from `404`

4. Document each seed agent (purpose, tools, permissions).
5. Tests: seed definitions load; CLI commands hit correct endpoints (against a stub server).
