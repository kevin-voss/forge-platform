# Step 15.05: Platform tools

## Status

Not started

## Epic

`15-forge-agents` → [`../../epics/15-forge-agents.md`](../../epics/15-forge-agents.md)

## Goal

Implement the concrete read-oriented platform tools backed by Control, Runtime, Observe, Storage, Models, and Events, plus tag write/destructive tools (e.g. `runtime.restart`) — each with a permission and a fake fallback so runs work deterministically in CI.

## Why this step exists

Agents are only useful when their tools can inspect the real platform. `specs.md` Step 15 lists tools like `deployment.read`, `logs.search`, `metrics.query`, `runtime.restart`. This step wires them behind the enforcement pipeline from 15.03.

## Dependencies

* `15.04` (run engine)
* Backing services (minimum available capabilities): Control `02.05`, Runtime `04.04`/`04.05`, Observe `12.04`, Storage `13.03`, Models `14.03`/`14.04`, Events `11.02`. Where a backing service is absent in CI, the tool uses a deterministic fake.

## Scope

* Read tools: `deployment.read` (Control), `logs.search` (Observe/Runtime), `metrics.query` (Observe), `storage.get`/`storage.put` (Storage), `models.generate`/`models.embed` (Models), `events.publish` (Events)
* Write/destructive tools tagged `destructive: true`: `runtime.restart` (approval enforced in 15.06)
* Each tool: input/output schema, required permission, live client + fake mode
* Timeouts + error normalization per tool

## Explicitly out of scope

* Human approval gate mechanics (15.06 — here tools are just tagged destructive)
* Memory retrieval tool (epic 17.05)
* Seed agents/CLI (15.07)

## Architecture

```text
tools/
├── control.py    # deployment.read
├── observe.py    # logs.search, metrics.query
├── runtime.py    # runtime.restart (destructive)
├── storage.py    # storage.get/put
├── models.py     # models.generate/embed
└── events.py     # events.publish
Each tool: LiveClient (httpx) | FakeClient (fixtures) selected by FORGE_AGENTS_TOOLS_MODE
```

## Components

| Component | Purpose |
|---|---|
| Tool clients | httpx clients to each platform service |
| Fake fixtures | Deterministic responses for CI |
| Destructive tagging | Mark tools requiring approval |
| Error normalization | Uniform tool error shape |

## Contracts

* Tools registered in the `GET /v1/tools` listing with `destructive` flag + `required_permissions`
* Each tool documents its input/output JSON schema
* Tool errors normalized to `{ "tool","error_code","message" }`

## Data model

No new persistence (tool calls recorded as run steps by 15.04).

## Configuration

```text
FORGE_AGENTS_TOOLS_MODE      # fake|live

```

## Security considerations

* Each tool declares least-privilege `required_permissions`
* `runtime.restart` (and any mutation) marked `destructive: true`
* Tool clients pass the run's identity/project scope downstream
* No credentials embedded; sourced from env/Secrets

## Observability

* Log each tool call with target service, latency, outcome
* Metric: `agent_tool_backend_latency_seconds{tool}`, `agent_tool_backend_errors_total{tool}`

## Failure handling

* Backend unavailable (live) → tool error `backend_unavailable`; run records it, continues per policy
* Timeout → `tool_timeout`
* Fake mode always returns deterministic fixtures

## Implementation tasks

1. Implement each tool client (live + fake) with schemas + permissions.
2. Tag `runtime.restart` (and mutations) destructive.
3. Register tools in the registry; ensure enforcement pipeline covers them.
4. Add per-tool timeout + error normalization.
5. Provide CI fixtures for fake mode.
6. Extend OpenAPI (tool listing) + README; tests per tool.
