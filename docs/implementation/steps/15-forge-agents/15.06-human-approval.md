## Goal

Gate every destructive tool call behind an explicit human approval: when an agent attempts a `destructive: true` tool, the run pauses in an `awaiting_approval` state, records an approval request, and only proceeds (or is denied) after a human decision via API.

## Why this step exists

`specs.md` Step 15 requires that destructive actions require approval and that the demo agent must not restart anything without approval. This is the epic's headline safety guarantee.

## Dependencies

* `15.05` (platform tools, destructive tagging)

## Scope

* Approval request model + persistence (durable across service restart)
* Run engine pause: destructive tool call → create approval request → run state `awaiting_approval`
* `GET /v1/approvals`, `GET /v1/approvals/{id}`
* `POST /v1/approvals/{id}/approve` / `POST /v1/approvals/{id}/deny`
* On approve: resume run and execute the tool; on deny: record and continue without executing (or fail per policy)
* Approval expiry/timeout

## Explicitly out of scope

* Cross-restart workflow orchestration (Workflows 16.05 handles durable approvals in workflows)
* Seed agents/CLI (15.07)
* Notification delivery (Events integration deepens later)

## Architecture

```text
run loop → tool.destructive?  yes → create approval(run_id, tool, args) → pause (awaiting_approval)
                              no  → execute
approve → resume run → execute tool → continue loop
deny    → record denial → skip tool → continue/fail per policy
```

Approval state persisted so a restart mid-approval resumes correctly.

## Components

| Component | Purpose |
|---|---|
| `ApprovalStore` | Persist approval requests + decisions |
| Engine pause/resume | Suspend + continue a run around approval |
| `approvals` routes | List/get/approve/deny |
| Expiry | Auto-deny stale approvals |

## Contracts

* When destructive tool requested: run status → `awaiting_approval`; `GET /v1/runs/{id}` shows pending approval
* `GET /v1/approvals` → `200 { "approvals":[{ "id","run_id","tool","args","status","created_at","expires_at" }] }`
* `POST /v1/approvals/{id}/approve` → `200 { "status":"approved" }`; run resumes
* `POST /v1/approvals/{id}/deny` `{ "reason":"..." }` → `200 { "status":"denied" }`
* Approve/deny of terminal approval → `409`
* Extend OpenAPI

## Data model

```sql
CREATE TABLE approvals (
  id TEXT PRIMARY KEY, run_id TEXT NOT NULL REFERENCES runs(id),
  project_id TEXT NOT NULL, tool TEXT NOT NULL, args TEXT NOT NULL,
  status TEXT NOT NULL,          -- pending|approved|denied|expired
  decided_by TEXT, reason TEXT,
  created_at TEXT NOT NULL, expires_at TEXT NOT NULL, decided_at TEXT
);
```

## Security considerations

* Only authorized identities (project scope / role) may approve; approver recorded in `decided_by`
* Destructive tool cannot execute without a persisted `approved` record
* Cross-project approval access → `404`

## Observability

* Log approval created/approved/denied/expired with actor
* Metric: `agent_approvals_total{status}`, time-to-decision histogram

## Failure handling

* Restart while `awaiting_approval` → run resumes from persisted state on next poll/tick
* Expired approval → auto-`expired` (treated as deny), run continues without executing
* Deny → tool not executed; run records denial
