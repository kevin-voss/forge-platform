## Goal

Load workflow definitions, start runs via API, and persist run + step state durably (Ecto/Postgres) so that a service restart mid-run resumes execution without repeating completed steps.

## Why this step exists

Durability is the defining property of this epic (`specs.md` Step 16: workflows survive restart; completed steps not repeated). Establishing definitions + a persisted, idempotent state machine underpins every later primitive.

## Dependencies

* `16.01` (OTP skeleton)

## Scope

* Workflow definition schema (YAML) → internal struct with ordered/typed steps
* Ecto repo + migrations for `workflow_runs`, `workflow_steps`
* `POST /v1/workflows/{name}/runs`, `GET /v1/runs/{id}`, `GET /v1/runs`
* Per-run supervised process (GenServer) that persists state transitions before/after each step
* Resume-on-boot: reload in-flight runs and continue from last durable step
* Idempotency key `(run_id, step_id)` to prevent re-execution

## Explicitly out of scope

* Step primitives beyond a trivial `noop`/`log` step (16.03)
* Event triggers + agent steps (16.04)
* Approval (16.05), compensation (16.06)

## Architecture

```text
definitions/*.yaml → DefinitionLoader → %Workflow{steps: [...]}
POST /runs → persist run(queued) → DynamicSupervisor starts RunServer
RunServer: for each step → check completed? skip : execute → persist(step done) → advance
crash/restart → BootResumer loads non-terminal runs → restart RunServers → resume
```

## Components

| Component | Purpose |
|---|---|
| `DefinitionLoader` | Parse/validate YAML definitions |
| `RunServer` (GenServer) | Drive a single run, persist transitions |
| `RunSupervisor` (DynamicSupervisor) | One process per active run |
| `BootResumer` | Resume in-flight runs on startup |
| Ecto schemas | `workflow_runs`, `workflow_steps` |

## Contracts

* `POST /v1/workflows/{name}/runs` `{ "input":{...} }` → `202 { "run_id","status":"running" }`
* `GET /v1/runs/{id}` → `200 { "status","steps":[{id,type,status,output?}],"result"? }`
* `GET /v1/runs` → `200 { "runs":[...] }`
* `GET /v1/workflows` → list definitions
* Extend OpenAPI

## Data model

```sql
CREATE TABLE workflow_runs (
  id UUID PRIMARY KEY, workflow TEXT NOT NULL, project_id TEXT NOT NULL,
  status TEXT NOT NULL, input JSONB, result JSONB, error TEXT,
  current_step TEXT, inserted_at TIMESTAMPTZ, updated_at TIMESTAMPTZ
);
CREATE TABLE workflow_steps (
  id UUID PRIMARY KEY, run_id UUID REFERENCES workflow_runs(id),
  step_id TEXT NOT NULL, type TEXT NOT NULL, status TEXT NOT NULL,
  output JSONB, error TEXT, attempt INT DEFAULT 0,
  inserted_at TIMESTAMPTZ, updated_at TIMESTAMPTZ,
  UNIQUE(run_id, step_id)
);
```

## Security considerations

* Runs project-scoped; cross-project read → `404`
* Definitions validated; no arbitrary code execution
* DB credentials from env/Secrets, never committed

## Observability

* Log run/step transitions with `run_id`, `step_id`, `status`
* Metric: `workflow_runs_total{status}`, `workflow_steps_total{status}`

## Failure handling

* Malformed definition → load-time error
* RunServer crash → supervisor restart → resume from last durable step
* Persisted step marked done is never re-executed on resume (idempotency)
