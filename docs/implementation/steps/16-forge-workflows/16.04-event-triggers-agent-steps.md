# Step 16.04: Event triggers + agent steps

## Status

Not started

## Epic

`16-forge-workflows` → [`../../epics/16-forge-workflows.md`](../../epics/16-forge-workflows.md)

## Goal

Let workflows be triggered by platform events (e.g. `deployment.failed`) via a durable Events subscription, and add an `agent` step type that invokes a Forge Agents run and incorporates its result into the workflow.

## Why this step exists

The demo workflow is event-driven and includes an agent diagnosis step (`specs.md` Step 16 example). This step connects Events (11) and Agents (15) to the durable engine.

## Dependencies

* `16.03` (step primitives)
* Epic `11-forge-events` durable subscribe (`11.03`)
* Epic `15-forge-agents` run API + approval (`15.06`)

## Scope

* Event trigger registration: map event types → workflow definitions
* Durable consumer on Events (JetStream) that starts runs idempotently (dedupe by event id)
* `agent` step type: call `POST /v1/agents/{name}/runs`, poll to completion, capture result; handle agent `awaiting_approval`
* Trigger + agent-step definition schema
* Backpressure/failure handling for the consumer

## Explicitly out of scope

* Workflow-level human approval steps (16.05 — agent's own approval is surfaced but the workflow approval step is next)
* Compensation/rollback (16.06)

## Architecture

```text
Events(deployment.failed) → EventConsumer (durable) → find triggers → start run (idempotent by event_id)
agent step → AgentClient.run(name, input) → poll GET /v1/runs/{id}
           → if agent awaiting_approval: surface + wait (full workflow approval in 16.05)
           → capture diagnosis into workflow context
```

## Components

| Component | Purpose |
|---|---|
| `EventConsumer` | Durable subscription; idempotent run start |
| `TriggerRegistry` | event type → workflow(s) |
| `Steps.Agent` | Invoke + poll an agent run |
| `AgentClient` | HTTP client to forge-agents |

## Contracts

* Trigger config in definition:

```yaml
trigger:
  event: deployment.failed
steps:
  - id: diagnose
    type: agent
    agent: deployment-investigator
    input: { deployment: "${event.deployment_id}" }
```

* `POST /v1/triggers/test` (optional) to inject a synthetic event for testing
* Extend OpenAPI

## Data model

Add `event_dedup` table (`event_id UNIQUE`) to guarantee idempotent triggering (migration `0003`). Agent step output stored in `workflow_steps.output`.

## Configuration

```text
FORGE_EVENTS_URL / NATS connection   # events subscription
FORGE_AGENTS_URL                     # http://forge-agents:4301
FORGE_WORKFLOWS_AGENT_POLL_MS        # default 1000
FORGE_WORKFLOWS_AGENT_STEP_TIMEOUT   # default 300000 ms
```

## Security considerations

* Consumer verifies event source/schema before triggering
* Agent step passes project scope; cannot escalate beyond workflow's scope
* Duplicate events do not start duplicate runs

## Observability

* Log event received → trigger matched → run started (with `event_id`, `run_id`)
* Metric: `workflow_triggers_total{event}`, `workflow_agent_steps_total{status}`

## Failure handling

* Unmatched event → ack + ignore (logged)
* Agent unavailable → agent step retries per policy (from 16.03) then fails
* Duplicate event → deduped (no second run)
* Agent step timeout → step failed
