# Step 16.05: Human approval across restarts

## Status

Not started

## Epic

`16-forge-workflows` → [`../../epics/16-forge-workflows.md`](../../epics/16-forge-workflows.md)

## Goal

Add an `approval` step type that durably pauses a workflow awaiting a human decision, with approval state persisted so that a service restart while awaiting approval resumes correctly and a subsequent decision continues the workflow.

## Why this step exists

`specs.md` Step 16 requires that approval state is durable and workflows survive restart. Before a workflow can roll back a deployment (16.06), it must gate that action behind a persisted human approval that survives crashes.

## Dependencies

* `16.04` (event triggers + agent steps)

## Scope

* `approval` step type: pause run, create durable approval request, resume on decide
* `GET /v1/runs/{id}/approvals`, `GET /v1/approvals/{id}`
* `POST /v1/approvals/{id}/approve` / `deny`
* Restart safety: `awaiting_approval` runs reload and remain pending; decision after restart resumes
* Approval expiry policy

## Explicitly out of scope

* Compensation/rollback action (16.06)
* Demo assembly (16.07)

## Architecture

```text
approval step → persist approval(pending) + run status awaiting_approval → RunServer parks
decide(approve) → persist decision → RunServer resumes next step
decide(deny)    → persist decision → run follows deny branch / fails per definition
restart while awaiting → BootResumer loads run in awaiting_approval, parks again (no step repeat)
```

## Components

| Component | Purpose |
|---|---|
| `Steps.Approval` | Create approval + park run |
| `ApprovalStore` (Ecto) | Durable approvals |
| `approvals` routes | List/get/approve/deny |
| Resume-on-decide | Continue parked RunServer |
| Expiry sweeper | Auto-resolve stale approvals |

## Contracts

* Definition:

```yaml
  - id: approve-rollback
    type: approval
    prompt: "Approve rollback of ${event.deployment_id}?"
    on_deny: close
```

* Run in `awaiting_approval` exposes pending approval via `GET /v1/runs/{id}`
* `POST /v1/approvals/{id}/approve` → `200`; run resumes
* `POST /v1/approvals/{id}/deny` `{reason}` → `200`; run follows `on_deny`
* Approve/deny terminal → `409`
* Extend OpenAPI

## Data model

```sql
CREATE TABLE workflow_approvals (
  id UUID PRIMARY KEY, run_id UUID REFERENCES workflow_runs(id),
  step_id TEXT NOT NULL, project_id TEXT NOT NULL, prompt TEXT,
  status TEXT NOT NULL,                 -- pending|approved|denied|expired
  decided_by TEXT, reason TEXT,
  inserted_at TIMESTAMPTZ, expires_at TIMESTAMPTZ, decided_at TIMESTAMPTZ,
  UNIQUE(run_id, step_id)
);
```

## Configuration

```text
FORGE_WORKFLOWS_APPROVAL_TTL_SECONDS   # default 86400; expiry policy configurable
```

## Security considerations

* Only authorized identities may decide; `decided_by` recorded
* Approval bound to a specific run+step (no cross-run reuse)
* Cross-project approval access → `404`

## Observability

* Log approval created/approved/denied/expired with actor + run
* Metric: `workflow_approvals_total{status}`, time-to-decision histogram

## Failure handling

* Restart while awaiting → run stays `awaiting_approval`, resumes on decision (step not repeated)
* Expiry → `expired` (treated per `on_deny` or fail)
* Terminal decision → `409`
