# Step 19.04: Models + Agents + Memory for diagnosis

## Status

Not started

## Epic

`19-full-platform-demo` → [`../../epics/19-full-platform-demo.md`](../../epics/19-full-platform-demo.md)

## Goal

Wire the AI layer for incident diagnosis: seed historical incidents into Memory (embedded via Models), configure the deployment-investigator agent with the platform + `memory.search` tools, and verify the agent can diagnose a problem using real telemetry and cite a similar historical incident from Memory.

## Why this step exists

`specs.md` Step 19 scenario steps 10–12 require the agent to inspect logs/metrics/deployment state, have Memory return a similar historical incident, and prepare a diagnosis referencing real telemetry. This assembles the AI loop before the failure/rollback scenario.

## Dependencies

* `19.03` (foundations wired)
* Models `14.07`, Agents `15.08`, Memory `17.06`

## Scope

* Seed historical incidents (fixtures) into a project Memory collection (embedded via Models)
* Configure deployment-investigator with tools: `deployment.read`, `logs.search`, `metrics.query`, `memory.search`, and the approval-gated `runtime.restart`
* Verify agent diagnosis references real Observe telemetry + cites a Memory record
* Ensure classification service / product signals feed the agent's context

## Explicitly out of scope

* Failure injection + Workflows orchestration (19.05)
* Acceptance suite (19.06)

## Architecture

```text
Memory ← seed historical incidents (Models embeddings)
Agent(deployment-investigator): deployment.read + logs.search + metrics.query + memory.search
   → diagnosis citing telemetry + similar incident
   → recommends restart (approval-gated; executed only via Workflow in 19.05)
```

## Components

| Component | Purpose |
|---|---|
| Memory seeding | Historical incidents for retrieval |
| Agent config | Tool + permission set for investigator |
| Telemetry access | Agent reads real Observe data |
| Diagnosis output | Structured, cited result |
