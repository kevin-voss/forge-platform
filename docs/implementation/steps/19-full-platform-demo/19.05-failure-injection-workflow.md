# Step 19.05: Failure injection + Workflows approval/rollback

## Status

Not started

## Epic

`19-full-platform-demo` → [`../../epics/19-full-platform-demo.md`](../../epics/19-full-platform-demo.md)

## Goal

Assemble the full recovery loop: deploy a deliberately broken release whose readiness fails, have the failure emit an event that triggers a Workflow which collects diagnostics, runs the investigator agent (with Memory), requests human approval, and on approval rolls back the deployment and stores a final report — restoring product health.

## Why this step exists

This is the north-star scenario (`specs.md` Step 19 main scenario steps 7–16). It ties Reconciliation, Events, Workflows, Agents, Memory, and Control rollback into one automated, human-gated recovery.

## Dependencies

* `19.04` (AI diagnosis loop)
* Workflows `16.07`, Reconciliation/rollback `07.06`, Events `11.07`

## Scope

* A deterministic "broken release" v2 image (readiness fails via flag/env)
* Readiness failure → `deployment.failed` event
* Capstone workflow definition: collect logs+metrics → agent diagnose → approval → rollback → store report → emit completion event
* Approval gate (durable) before rollback
* Rollback via Control/reconciliation restores healthy v1; product healthy again

## Explicitly out of scope

* Acceptance suite packaging + docs hardening (19.06)
* New platform features

## Architecture

```text
deploy v2 (broken) → readiness fails → deployment.failed event
   → Workflow(incident-response) triggered
   → collect logs+metrics (parallel) → investigator agent (Memory-assisted) diagnosis
   → human approval → rollback deployment (Control) → store report (Storage) → emit event
   → product healthy again (v1)
```

## Components

| Component | Purpose |

## Contracts

* Workflow triggered by documented `deployment.failed` event
* Rollback via documented Control rollback API
* Report stored in Storage + completion event on Events
* All via documented contracts

## Data model

Workflow run + approval + saga log (from epic 16). Final report artifact in Storage; run result references it.

## Configuration

```text

```

## Security considerations

* Rollback only after human approval (no autonomous destructive action)
* All actions auditable (workflow + agent + approval + rollback)
* Approval authz enforced

## Observability

* Full trace/audit from failure → recovery
* Metrics on workflow/agent/rollback

## Failure handling

* If approval denied → no rollback; documented alternate path (e.g. hold/escalate)
* Rollback failure → workflow records failure + report reflects it
* Restart mid-workflow → resumes without repeating steps (epic 16 durability)
