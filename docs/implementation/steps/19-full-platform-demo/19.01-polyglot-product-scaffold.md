## Goal

Scaffold the capstone incident-management product as five runtime-contract-compliant services — Go API, Kotlin admin, Rust log-processing worker, Python classification service, Elixir notification worker — each with health endpoints, structured logs, `forge.yaml`, and Dockerfile, under `demos/09-full-platform/product/`. No platform integration yet.

## Why this step exists

The capstone needs a real, minimal polyglot product before it can be deployed and integrated. Building the services against the runtime contract (epic 01) first ensures they are schedulable/routable by the platform in later steps.

## Dependencies

* Epic `01-runtime-contract` conventions (health, logs, `PORT`, shutdown)
* Language demo patterns from epic `01` (`01.03`–`01.07`)

## Scope

* Five service directories under `demos/09-full-platform/product/`
* Each: `/health/live`, `/health/ready`, `/` identity, structured logs, Dockerfile, `forge.yaml`
* A shared product README describing service responsibilities + interactions (contract-level)
* Local `docker compose` (product-only) to run all five for smoke checks
* Deterministic behaviors needed later (e.g. classification service returns stable labels)

## Explicitly out of scope

* Platform deploy path (19.02)
* Identity/Secrets/Observe/Storage/DB wiring (19.03)
* Models/Agents/Memory (19.04)
* Failure injection + Workflows (19.05)
* Acceptance suite (19.06)

## Architecture

```text
demos/09-full-platform/product/
├── api-go/            # public incident API
├── admin-kotlin/      # admin operations
├── log-worker-rust/   # processes log stream
├── classify-python/   # classifies incidents (deterministic)
└── notify-elixir/     # sends notifications
Each: health + identity + logs + Dockerfile + forge.yaml
```

Inter-service contracts documented (who calls whom) but wired through Gateway/Events in 19.02.
