# Step 19.02: Deploy path: Build→Runtime→Gateway→Events

## Status

Not started

## Epic

`19-full-platform-demo` → [`../../epics/19-full-platform-demo.md`](../../epics/19-full-platform-demo.md)

## Goal

Deploy the capstone product through the real platform path: Build creates images from source, Control records desired state, Runtime starts the workloads, Gateway exposes the product on stable hostnames, and the services publish/consume platform events — so the product is reachable and event-driven end to end.

## Why this step exists

`specs.md` Step 19 scenario steps 1–5 are the deploy/expose/event path. This proves the product runs on the platform (not just standalone compose) before layering identity/data/AI.

## Dependencies

* `19.01` (product services)
* Build `06.07`, Control `02.05`, Runtime `04.08`, Gateway `05.07`, Events `11.07` (minimum capabilities)

## Scope

* Product deployed via Build → registry → Control deployment → Runtime workloads
* Gateway routes: stable `*.demo.localhost` (or documented) hostnames per service
* Events wiring: product emits/consumes at least one platform-relevant event (e.g. `incident.created`, `deployment.*`)
* Capstone Compose (`demos/09-full-platform/compose.yaml`) brings up the platform + product
* `forge deploy` (CLI) path exercised

## Explicitly out of scope

* Identity/Secrets/Observe/Storage/DB (19.03)
* Models/Agents/Memory (19.04)
* Failure injection + Workflows (19.05)
* Full acceptance suite (19.06)

## Architecture

```text
source → Build → image(registry) → Control(desired) → Runtime(workloads) → Gateway(routes)
product services → Events (publish/subscribe)
```

## Components

| Component | Purpose |
|---|---|
| Capstone compose | Platform services + product |
| Deploy orchestration | Build→Control→Runtime→Gateway |
| Event wiring | Product ↔ Events |
| Route config | Gateway hostnames |

## Contracts

* Uses documented Build/Control/Runtime/Gateway/Events contracts only
* Product reachable via Gateway hostnames
* At least one documented event flows product→Events→consumer

## Security considerations

## Observability

* Deploy path logs correlated by request/deployment id

## Failure handling

* Build failure → no deploy (surfaced)
* Runtime unhealthy workload → reflected in Control status
* Gateway skips unready upstream

## Implementation tasks

1. Author capstone `compose.yaml` (platform + product) or an orchestration script.
2. Wire Build to produce product images; deploy via Control→Runtime.
3. Configure Gateway routes for the five services.
4. Add product Events publish/subscribe for a core event.
5. Exercise `forge deploy`; document the path in README.
6. Smoke test: product reachable via Gateway + event observed.
