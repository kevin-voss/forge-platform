# Step 18.02: Create instance/database/credentials

## Status

Not started

## Epic

`18-managed-postgresql` → [`../../epics/18-managed-postgresql.md`](../../epics/18-managed-postgresql.md)

## Goal

Implement the real (local) provisioner that creates an isolated Postgres instance (container), creates a database and a least-privilege role/credential inside it, and records credentials as Secrets references — with the product database fully isolated from Control's own database.

## Why this step exists

This is the core provisioning capability: turning a resource record into a real, isolated Postgres a product can use. `specs.md` Step 18 requires isolated databases with created credentials.

## Dependencies

* `18.01` (resource model + provisioner interface)
* Epic `10-forge-secrets` (minimum: secret set/store `10.02`) to hold generated credentials

## Scope

* `LocalProvisioner`: start a Postgres container per instance on the managed docker network
* Create database + a dedicated role with least privilege on that database
* Generate strong credentials; store as a Secret; keep only `secret_ref` in Control
* Instance/database health check (connection ping) → `available`
* `POST /v1/databases/instances/{id}/databases` + credential creation

## Explicitly out of scope

* Attach + URL injection (18.03)
* Backup/restore (18.04)
* Rotation/deletion protection (18.05)

## Architecture

```text
create instance → LocalProvisioner.start_container(name) on FORGE_DB_MANAGED_NETWORK
create database → CREATE DATABASE + CREATE ROLE ... LIMITED GRANTS
credentials     → generate → store in Secrets → db_credential.secret_ref
health          → connect + SELECT 1 → status available
```

Each product instance is a separate container with its own superuser, distinct from Control's DB — enforced by construction (separate container, separate network alias).

## Components

| Component | Purpose |
|---|---|
| `LocalProvisioner` | Manage product Postgres containers |
| Role/DB creator | Least-privilege database + role |
| Credential generator | Strong random creds |
| Secrets client | Persist credentials as secrets |
| Health checker | Connection readiness |

## Contracts

* `POST /v1/databases/instances/{id}/databases` `{ "name":"appdb" }` → `201 { database }`
* Credentials returned only as references + one-time reveal at creation (documented), never re-listed in plaintext
* `GET .../databases/{id}` includes `status`, `host`, `port`, `secret_ref` (no plaintext password)
* Extend OpenAPI

## Data model

Uses 18.01 tables. `db_instance` gains `host`, `port`, `container_id` (migration). `db_credential.secret_ref` points at a Secrets entry.

## Configuration

```text
FORGE_DB_PROVISIONER=local
FORGE_DB_MANAGED_NETWORK        # docker network for product DBs
FORGE_DB_POSTGRES_IMAGE         # e.g. postgres:16
FORGE_SECRETS_URL               # store credentials
```

## Security considerations

* Product role has privileges ONLY on its own database (no superuser to product)
* Passwords generated with sufficient entropy; stored only in Secrets
* Control persists references, not plaintext
* Product DB container isolated on managed network, not reachable as Control's DB

## Observability

* Log provisioning steps (container start, db/role create, health) without secrets
* Metric: `managed_db_provision_duration_seconds`, `managed_db_provision_errors_total`

## Failure handling

* Container start failure → instance `error` with reason; cleanup partial resources
* DB/role creation failure → rollback (drop partial) + `error`
* Health never passes within timeout → `error`
