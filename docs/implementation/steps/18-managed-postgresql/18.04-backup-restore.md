## Goal

Implement on-demand database backup and restore: capture a consistent dump of a managed database, store the archive durably (Forge Storage or a backup volume), and restore it into a database such that known fixture data is recovered.

## Why this step exists

`specs.md` Step 18 requires backup and restore, with acceptance that a restore recovers known fixture data. This protects product data and is exercised in the demo.

## Dependencies

* `18.03` (attached, working databases)
* Epic `13-forge-storage` optional (`13.03` streamed upload) for backup archive storage; falls back to a named volume

## Scope

* `POST /v1/databases/{database_id}/backups` → create backup (`pg_dump`) → store archive → record `db_backup`
* `GET /v1/databases/{database_id}/backups` / `GET .../backups/{id}`
* `POST /v1/databases/backups/{id}/restore` → restore into target database
* Backup integrity (checksum) + status tracking
* Consistent snapshot (dump within a transaction/`--serializable-deferrable` or `pg_dump` default consistency)

## Explicitly out of scope

* Point-in-time recovery / WAL archiving (documented as later)
* Scheduled/automatic backups (on-demand only this step)
* Rotation/deletion protection (18.05)

## Architecture

```text
backup  → provisioner.pg_dump(db) → archive → Storage.put(db-backups/<id>) or volume → record + checksum
restore → fetch archive → verify checksum → pg_restore/psql into target db → verify
```

## Components

| Component | Purpose |
|---|---|
| Backup runner | `pg_dump` → archive |
| Archive store | Forge Storage bucket or volume |
| Restore runner | `pg_restore`/`psql` into target |
| Integrity | Checksum on store + verify on restore |

## Contracts

* `POST /v1/databases/{database_id}/backups` → `202 { backup, status:"running" }` → `succeeded|failed`
* `GET .../backups` list; `GET .../backups/{id}` status + location + checksum
* `POST /v1/databases/backups/{id}/restore` `{ target_database_id }` → `202` → `succeeded|failed`
* Extend OpenAPI

## Data model

Uses `db_backup` (18.01); add `checksum`, `size_bytes`, `completed_at` (migration).

## Configuration

```text
FORGE_DB_BACKUP_TARGET       # storage|volume (default storage if available)
FORGE_DB_BACKUP_BUCKET       # default db-backups (Storage)
FORGE_DB_BACKUP_DIR          # volume path fallback
```

## Security considerations

* Backups are project-scoped; cross-project backup/restore → `404`
* Archive access controlled (Storage signed access or restricted volume)
* Restore into a different project's database is forbidden

## Observability

* Log backup/restore start/end (db, size, duration, checksum)
* Metric: `managed_db_backups_total{status}`, `managed_db_restore_total{status}`

## Failure handling

* `pg_dump` failure → backup `failed` with reason; partial archive removed
* Checksum mismatch on restore → abort with `integrity_error`
* Restore failure → target left unchanged where possible; status `failed`
