## Goal

Support credential rotation (issue new credentials, update the injected secret, invalidate old credentials) and enforce deletion protection so a protected instance/database cannot be deleted without explicitly disabling protection and forcing the delete.

## Why this step exists

`specs.md` Step 18 requires credential rotation (rotated credentials invalidate old ones) and deletion protection. These are the safety/lifecycle guarantees for production data.

## Dependencies

* `18.04` (backups exist for safe deletes)
* Epic `10-forge-secrets` (secret update) + Runtime redeploy path

## Scope

* `POST /v1/databases/{database_id}/rotate-credentials` → new role/password → update Secrets → old credentials invalidated
* Attached apps pick up new URL on next deploy (and old is revoked in Postgres)
* Deletion protection flag (default on) on instance + database
* Delete requires `deletion_protection=false` + `?force=true`; otherwise `409`
* Optional pre-delete backup

## Explicitly out of scope

* CLI/demo (18.06)
* Automatic scheduled rotation (on-demand only)

## Architecture

```text
rotate → create new role/password → update Secrets(secret_ref) → drop/disable old role (grace)
delete → if deletion_protection → 409; require disable + force → backup(optional) → drop db/stop container
```

Rotation issues the new credential before revoking the old to avoid an outage window; old is revoked after the new URL is delivered (documented grace).

## Components

| Component | Purpose |

## Contracts

* `POST /v1/databases/{database_id}/rotate-credentials` → `200 { credential, secret_ref }`
* `PATCH /v1/databases/instances/{id}` `{ "deletion_protection": false }`
* `DELETE /v1/databases/instances/{id}?force=true` → `204` (only if protection disabled) | `409`
* `DELETE /v1/databases/{database_id}?force=true` similarly guarded
* Extend OpenAPI

## Data model

`db_credential` gains `rotated_at`, `revoked_at`; `db_instance.deletion_protection` (already present) toggled via PATCH.

## Configuration

```text
FORGE_DB_ROTATION_GRACE_SECONDS   # default 60 (old cred valid until new delivered)
FORGE_DB_PREDELETE_BACKUP         # default true
```

## Security considerations

* Old credentials MUST be invalid after rotation completes (verify revocation)
* Deletion protection prevents accidental data loss; force is explicit + audited
* Rotation + delete are privileged, project-scoped, audited

## Observability

* Log rotation (old revoked, new delivered) + delete (protection state, force, backup id)
* Metric: `managed_db_rotations_total`, `managed_db_deletes_total{forced}`

## Failure handling

* Rotation failure mid-way → keep old credentials valid, mark rotation `failed` (no outage)
* Delete without disabling protection → `409`
* Delete of attached database → require detach or cascade documented; default `409` if attachments exist
