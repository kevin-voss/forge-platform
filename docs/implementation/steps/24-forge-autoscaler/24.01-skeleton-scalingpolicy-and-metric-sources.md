## Goal

Stand up `forge-autoscaler` as a Go service on port `4112` hosting the new `ScalingPolicy` resource kind (full CRUD, status, watch, per `docs/concepts/resource-model.md`), and build the pluggable `MetricSource` interface with a deterministic fake implementation. The evaluation loop runs on a tick, fetches metrics, and records a recommendation on each policy's status — but performs **no actuation**. This step proves the shape everything else in the epic fills in.

## Why this step exists

Every later step (utilization math, request-rate math, queue math, schedules, node autoscaling) needs the same three things already in place: a resource to configure against, a way to fetch a metric without caring where it comes from, and somewhere to record what was computed. Building the seam and an inert evaluator first — recording recommendations without ever writing a replica count — keeps the interface honest and gives `24.02` a deterministic fake source to test dampening/clamping math against before any real metric backend is wired in.

* `20-declarative-resource-api` — resource envelope shape (metadata/spec/status, optimistic concurrency, conditions) this step's `ScalingPolicy` implements

## Scope

* `ScalingPolicy` resource kind (environment-scoped): full envelope, `spec` matching the normative example exactly (`targetRef`, `minReplicas`, `maxReplicas`, `metrics[]`, `behavior.scaleUp`/`scaleDown` with `stabilizationWindowSeconds` + `maxReplicasPerMinute`, `schedules[]`)
* CRUD + status + watch routes: `POST/GET/PUT/PATCH/DELETE /v1/projects/{project}/environments/{environment}/scalingpolicies[/{name}]`, `PUT .../{name}/status`, `GET /v1/watch/scalingpolicies?since=...`
* `MetricSource` interface: `Fetch(ctx, targetRef, metricSpec) (Sample, error)`
* Four adapter stubs — `ObserveSource`, `GatewaySource`, `QueueSource`, `RuntimeSource` — wired to their upstream URLs but returning a typed `ErrNotImplemented` sample until `24.02`–`24.04` fill in the real queries
* `FakeSource` — deterministic, script-driven (a queue of values per `(target, metric type)` key) so unit tests never depend on wall-clock or a real backend
* Evaluation loop: on each tick, for every `ScalingPolicy`, fetch each configured metric, append a bounded recommendation entry to `status.recommendations`, set `conditions` (`AbleToScale`, `ScalingActive`) — never writes to any other resource
* Health/readiness/liveness endpoints and graceful shutdown (cross-cutting requirement)

## Explicitly out of scope

* Any actual replica math or actuation (`24.02`)
* Request-rate/latency/error-rate metrics (`24.03`), queue metrics (`24.04`)
* Schedules, manual override, deployment freeze, metric-outage fallback (`24.05`)
* `NodePool` resource and node autoscaling (`24.06`, `24.07`)
* Real Observe/Gateway/Queue/Runtime queries (adapters exist but return `ErrNotImplemented`; only `FakeSource` is exercised by tests this step)

## Contracts

`ScalingPolicy` spec, matching the normative example exactly:

```yaml
apiVersion: forge.dev/v1
kind: ScalingPolicy
metadata:
  name: invoice-api-scaling
  project: invoice-platform
  environment: production
spec:
  targetRef:
    kind: Application
    name: invoice-api
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: cpu
      targetAverageUtilization: 65
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0
      maxReplicasPerMinute: 4
    scaleDown:
      stabilizationWindowSeconds: 300
      maxReplicasPerMinute: 2
  schedules: []
status:
  phase: Ready
  observedGeneration: 1
  currentReplicas: 0
  desiredReplicas: 0
  recommendations: []
  conditions:
    - type: AbleToScale
      status: "True"
      reason: ReadyForScaling
      lastTransitionTime: 2026-07-22T10:00:00Z
```

* `POST /v1/projects/{project}/environments/{environment}/scalingpolicies` — create; `201` + envelope
* `GET .../scalingpolicies/{name}` — `404` if absent
* `PUT .../scalingpolicies/{name}` — full spec replace; requires `metadata.resourceVersion`; `409` on stale write
* `PUT .../scalingpolicies/{name}/status` — status-only write, internal to this service's own evaluator
* `GET /v1/watch/scalingpolicies?since={resourceVersion}` — SSE `ADDED`/`MODIFIED`/`DELETED` frames

## Data model

Persisted in `forge-autoscaler`'s own Postgres schema (same foundation Postgres instance, port `5001`, separate `forge_autoscaler` database) — `ScalingPolicy` is a resource this service owns end to end, per `docs/concepts/resource-model.md` §5 (`ScalingPolicy → Autoscaler`):

```sql
CREATE TABLE scaling_policies (
  id                TEXT PRIMARY KEY,             -- sp_<ULID>
  name              TEXT NOT NULL,
  project           TEXT NOT NULL,
  environment       TEXT NOT NULL,
  generation        INT NOT NULL DEFAULT 1,
  resource_version  BIGINT NOT NULL DEFAULT 1,
  spec_json         JSONB NOT NULL,
  status_json       JSONB NOT NULL DEFAULT '{}',
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (project, environment, name)
);
```

`status_json.recommendations` is a bounded ring buffer (last 10 entries), each:

```json
{ "metricType": "cpu", "metricValue": 81.4, "targetValue": 65.0,
  "recommendedReplicas": null, "reason": "recorded, no actuation in 24.01",
  "computedAt": "2026-07-22T10:00:03Z" }
```

## Security considerations

* `ScalingPolicy` CRUD is internal platform-trust-boundary API; no secrets in spec/status
* `/status` writes are restricted to this service's own evaluator process (the only writer); user/CLI writes touch `spec` only, matching the resource-model status-subresource rule
* `FORGE_AUTH_MODE=dev` bypass documented as temporary (epic 09)

## Observability

* Log per evaluation: `{policy_id, target_kind, target_name, metric_type, metric_value, target_value, reason}`
* Metrics: `forge_autoscaler_evaluations_total{policy}`, `forge_autoscaler_metric_fetch_errors_total{source}`
* Trace span `autoscaler.evaluate` with attributes `policy_id`, `metric_type`, `source`

## Failure handling

* Metric source returns `ErrNotImplemented` or a transport error → recommendation entry records the failure reason; `ScalingActive` condition set to `"Unknown"` with reason `MetricFetchFailed`; loop continues to the next policy (one failing policy never blocks others)
* Stale `resourceVersion` on write → `409 Conflict` with the canonical error envelope
* Service restart → policies reloaded from Postgres; the evaluation loop resumes on its own tick with no replay needed (nothing was actuated yet, so there is nothing to recover)

## Unit tests

* `FakeSource.Fetch` returns scripted values deterministically per `(targetRef, metricType)` and errors clearly once exhausted
* `policy.Store` CRUD is idempotent under `Idempotency-Key`; stale `resourceVersion` write returns a typed conflict
