## Goal

Scale runtime nodes up when the scheduler cannot place workloads or cluster reservation exceeds safe thresholds. The autoscaler chooses a matching `NodePool` and asks Forge Infrastructure to create nodes; it never starts workloads directly.

## Dependencies

* `23-forge-infrastructure` - `NodePool`, provider adapters, node lifecycle.
* `25.01` - resource requests, limits, and capacity accounting.
* `08.04` - pending queue and unschedulable placement signals.

## Scope

* Pending workload signal ingestion from Scheduler.
* Cluster reservation thresholds for CPU, memory, and GPU capacity.
* NodePool selection by labels, region, architecture, GPU, and policy constraints.
* Scale-up recommendation status on `NodePool`.
* Idempotent call to Infrastructure with operation id.
* Cooldown and max-node safeguards.

## Explicitly out of scope

* Scale-down and draining.
* Cost-aware provider choice beyond static NodePool eligibility.
* Multi-region balancing.

## Observability

* Metric `forge_node_autoscaler_pending_workloads_total`.
* Metric `forge_node_autoscaler_scale_up_requests_total{nodepool,result}`.
* Span `autoscaler.node.scale_up`.

## Failure handling

* No matching NodePool -> pending workload condition `NoEligibleNodePool`.
* Provider quota/capacity error -> NodePool condition `ProviderCapacityBlocked`.
* Duplicate scale-up ticks reuse the same operation id for the same unschedulable demand window.

## Implementation tasks

1. Consume pending workload and cluster reservation signals.
2. Implement NodePool eligibility and selection.
3. Create scale-up operation records and call Infrastructure.
4. Update NodePool status with desired, creating, ready, and failed counts.
5. Add tests for CPU/memory/GPU demand and no-eligible-pool cases.

## Acceptance criteria

* Pending workloads trigger node creation in an eligible NodePool.
* New nodes register and pending workloads become schedulable.
* The autoscaler never starts containers directly.
* Duplicate evaluations do not create duplicate nodes.
