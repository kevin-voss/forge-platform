## Goal

Add application autoscaling based on traffic pressure: requests per second, active connections, latency, and error rate from Gateway and Observe. This lets network-facing services scale on customer-visible load, not only container resource saturation.

## Scope

* Metric types: `httpRequests`, `activeConnections`, `p95Latency`, `errorRate`.
* Gateway admin metric adapter.
* Observe query fallback for historical windows.
* Per-replica target calculation for request and connection metrics.
* Guardrail: latency/error metrics can recommend scale-up or hold, never scale-down by themselves.
* Status reason codes that explain which metric drove the decision.

## Explicitly out of scope

* Canary analysis and rollout promotion.
* Synthetic checks.
* Queue depth and worker autoscaling.

## Observability

* Metric `forge_autoscaler_metric_source_latency_seconds{source}`.
* Log dominant metric per decision with `{metric_type, observed, target, recommended_replicas}`.
* Span `autoscaler.metrics.gateway`.

## Failure handling

* Missing Gateway metrics for a route marks that metric unavailable while allowing CPU/memory metrics to continue.
* Latency spike with low request volume requires a minimum sample count before scaling.
* Error-rate scale-up is capped by policy behavior and never triggers scale-down.

## Implementation tasks

1. Implement `GatewaySource` queries for request rate and active connections.
2. Add Observe queries for p95 latency and error rate.
3. Extend metric evaluation to combine multiple metrics by taking the highest safe replica recommendation.
4. Add sample-count and low-traffic guards.
5. Add tests for mixed metric decisions and degradation paths.

## Acceptance criteria

* Request-rate policies scale up under generated load and scale down after stabilization.
* Latency/error-rate policies can hold or scale up but cannot reduce replicas.
* Missing one metric source does not stop evaluation of the remaining metrics.
