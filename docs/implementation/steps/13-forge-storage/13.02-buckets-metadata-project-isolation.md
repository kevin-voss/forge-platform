## Goal

Introduce buckets and an object metadata index scoped to a project, with create/list/delete bucket APIs and object metadata records, enforcing that a caller in one project can never see or address another project's buckets or objects.

## Why this step exists

Storage must be multi-tenant safe before any bytes flow. Establishing the metadata index and project isolation now means streaming (13.03) and later steps operate inside a trusted namespace boundary.

## Dependencies

* `13.01` (service skeleton + FS backend)
* Epic `09-forge-identity` for project scope (minimum: verified project id). `FORGE_AUTH_MODE=dev` with `X-Forge-Project` header allowed until Identity enforced.

## Scope

* Persistent metadata index under `FORGE_STORAGE_ROOT/meta/` (embedded SQLite per epic assumption)
* Bucket APIs: create, list, get, delete (empty only)
* Object metadata records (no payload bytes yet — placeholder rows for upload in 13.03)
* Project scoping middleware: derive `project_id`, filter every query
* Naming/validation rules for bucket and object keys

## Explicitly out of scope

* Streaming upload/download bytes (13.03)
* Checksums, ranges (13.04)
* Signed tokens (13.05)
* Quotas + non-empty bucket delete semantics (13.06)

## Architecture

```text
Request → auth/project middleware (project_id) → handler → MetadataStore (SQLite)
                                                              │
                                                    tables: buckets, objects
```

`MetadataStore` wraps SQLite at `meta/index.db`. Every statement includes `project_id` in its `WHERE`/`INSERT`. Buckets are unique per `(project_id, name)`.

## Components

| Component | Purpose |
|---|---|
| `project.rs` middleware | Extract + validate `project_id` (Identity token or dev header) |
| `MetadataStore` | SQLite-backed CRUD for buckets/objects, always project-scoped |
| `buckets` handlers | REST for bucket lifecycle |
| Validation | Bucket/key charset, length, reserved-name checks |

## Contracts

* `POST /v1/buckets` `{ "name": "artifacts" }` → `201 {bucket}`
* `GET /v1/buckets` → `200 {"buckets":[...]}` (only caller's project)
* `GET /v1/buckets/{bucket}` → `200 {bucket}` | `404`
* `DELETE /v1/buckets/{bucket}` → `204` if empty, `409` if not empty (full cascade in 13.06)
* Project context via `Authorization: Bearer <token>` (Identity) or `X-Forge-Project: <id>` in dev mode
* Extend `contracts/openapi/forge-storage.openapi.yaml` with these paths

## Data model

```sql
CREATE TABLE buckets (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  name TEXT NOT NULL,
  created_at TEXT NOT NULL,
  UNIQUE(project_id, name)
);
CREATE TABLE objects (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  bucket_id TEXT NOT NULL REFERENCES buckets(id),
  key TEXT NOT NULL,
  size_bytes INTEGER NOT NULL DEFAULT 0,
  sha256 TEXT,                       -- filled in 13.04
  content_type TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  UNIQUE(project_id, bucket_id, key)
);
CREATE INDEX idx_objects_project ON objects(project_id, bucket_id);
```

## Security considerations

* Every query MUST be filtered by `project_id`; add a test that a cross-project read returns 404, not 403 (avoid existence leaks)
* Reject bucket/key names with path traversal (`..`, leading `/`, NUL)
* Dev-mode header bypass logged as a warning

## Observability

* Log bucket create/delete with `project_id`, `bucket`, `request_id`
* Metric counters: `storage_buckets_total`, `storage_objects_total` (per project label optional)

## Failure handling

* Duplicate bucket name → `409`
* Missing/invalid project context (enforced mode) → `401`
* Delete non-empty bucket → `409` with object count

```bash
curl -fsS -XPOST $BASE/v1/buckets -H 'X-Forge-Project: proj-a' \
  -H 'content-type: application/json' -d '{"name":"artifacts"}'

```
