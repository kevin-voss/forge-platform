## Goal

Issue short-lived, HMAC-signed access tokens for specific object operations and enforce expiry, so clients (and other services) can be granted scoped, time-boxed access to an object without holding platform credentials.

## Why this step exists

`specs.md` Step 13 requires signed access tokens and that expired tokens fail. Signed URLs let products and services hand out download/upload links (e.g. a browser fetching a user upload) without proxying credentials.

## Dependencies

* `13.04` (objects fully addressable with integrity)

## Scope

* `POST /v1/buckets/{bucket}/objects/{key}/sign` → returns a signed token/URL for a given method + TTL
* Token = HMAC-SHA256 over canonical `(method, project_id, bucket, key, expiry)` using `FORGE_STORAGE_SIGNING_KEY`
* Token acceptance on GET/PUT via `?token=` query param or `Authorization: Bearer <token>`
* Strict expiry enforcement (reject `now > expiry`)
* Constant-time signature comparison; clock-skew tolerance configurable

## Explicitly out of scope

* Quotas + delete (13.06)
* Key rotation infrastructure (single active key + optional previous key for validation only)
* Public CDN / caching semantics

## Architecture

```text
POST .../sign {method, ttl_seconds} → build canonical string → HMAC(sign_key) → token
GET/PUT with token → recompute HMAC → compare (constant time) → check expiry + method + scope match
```

Token payload (URL-safe base64): `{v, method, project_id, bucket, key, exp}` + `sig`. Server recomputes `sig` from its key; payload is not encrypted (no secrets inside), only signed.

## Components

| Component | Purpose |
|---|---|
| `signing.rs` | Canonicalization + HMAC sign/verify |
| Sign handler | Issue tokens for a scoped operation |
| Token auth extractor | Accept signed token as an alternative to project auth |
| Expiry/skew check | Reject expired or not-yet-valid tokens |

## Contracts

* `POST /v1/buckets/{bucket}/objects/{key}/sign` `{ "method":"GET", "ttl_seconds":300 }` → `200 { "token":"...", "url":"...", "expires_at":"..." }`
* `GET .../objects/{key}?token=<t>` → `200/206` if valid+unexpired+method matches, else `401`/`403`
* Expired token → `401` with error `token_expired`
* Method/scope mismatch (e.g. GET token used for PUT) → `403`
* Extend OpenAPI

## Data model

No new tables required (stateless tokens). Optionally log issued tokens' `jti` for audit (add `signed_tokens_audit` table if audit needed in 13.06). `sha256` binding optional: a token may pin the expected `sha256` so it invalidates if content changes.

## Configuration

```text
FORGE_STORAGE_SIGNING_KEY        # required; HMAC secret (from Secrets epic later)
FORGE_STORAGE_SIGNING_KEY_PREV   # optional; accepted for verify during rotation
FORGE_STORAGE_MAX_TTL_SECONDS    # default 3600; reject larger TTL requests
FORGE_STORAGE_CLOCK_SKEW_SECONDS # default 30
```

## Security considerations

* Constant-time HMAC comparison (no early-exit compare)
* Reject TTL beyond `FORGE_STORAGE_MAX_TTL_SECONDS`
* Signing key never logged; sourced from env/Secrets, not committed
* Token scope is method- and object-specific (no wildcard tokens)
* Refuse to start if `FORGE_STORAGE_SIGNING_KEY` unset in enforced mode

## Observability

* Log token issuance (project, bucket, key, method, exp) — never the signature
* Metric: `storage_tokens_issued_total`, `storage_token_rejections_total{reason}`

## Failure handling

* Missing/garbled token → `401 invalid_token`
* Expired → `401 token_expired`
* Wrong method/scope → `403`
* Signing key unset → startup failure (enforced) / warning (dev)

## Manual verification

```bash
BASE=localhost:4107; P='-H X-Forge-Project:proj-a'
TOK=$(curl -fsS $P -XPOST -H 'content-type: application/json' \
  -d '{"method":"GET","ttl_seconds":2}' \
  "$BASE/v1/buckets/artifacts/objects/big.bin/sign" | grep -o '"token":"[^"]*"' | cut -d'"' -f4)
curl -fsS "$BASE/v1/buckets/artifacts/objects/big.bin?token=$TOK" -o /dev/null && echo "valid token OK"
sleep 3
code=$(curl -s -o /dev/null -w '%{http_code}' "$BASE/v1/buckets/artifacts/objects/big.bin?token=$TOK")
test "$code" = 401 && echo "expired token rejected OK"
```

## Acceptance criteria

* A signed token grants scoped access to a single object + method without project credentials
* Expired tokens are rejected with `401 token_expired`
* Tampered tokens fail signature verification
* A token scoped to GET cannot be used for PUT (`403`)
* TTL is clamped to `FORGE_STORAGE_MAX_TTL_SECONDS`
* Signature comparison is constant-time; signing key never logged
* OpenAPI updated and valid
