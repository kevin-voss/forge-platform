## Goal

Stand up the `forge-storage` Rust service with health endpoints, structured logging, graceful shutdown, a Docker image, Compose wiring on host port `4107`, and a durable local filesystem storage root — with no object/bucket semantics yet.

## Why this step exists

Every later storage capability (buckets, streaming, integrity, signing, quotas) needs a running, contract-compliant service and a persistent on-disk root. Establishing the skeleton and FS layout first keeps subsequent steps focused on behavior rather than scaffolding.

## Dependencies

* Epic `00-repository-foundation` complete (Compose, Make, ports, docs tree)
* Epic `01-runtime-contract` conventions (health endpoints `/health/live` + `/health/ready`, `PORT`, JSON logs, SIGTERM drain)
* No cross-epic runtime dependency yet (Identity wired in `13.02`)

## Scope

* Rust crate `services/forge-storage` (Axum + tokio)
* `GET /health/live`, `GET /health/ready`, `GET /` identity JSON
* Config via environment (`PORT`, `FORGE_STORAGE_ROOT`, `FORGE_LOG_LEVEL`)
* Local FS backend abstraction that creates/validates the storage root on boot
* Dockerfile (multi-stage) + Compose service + named volume for durability
* Makefile targets and port reservation in `docs/operations/ports.md`

## Explicitly out of scope

* Buckets, objects, metadata (13.02)
* Upload/download (13.03)

* Quotas/delete (13.06)

## Architecture

```text
services/forge-storage/
├── src/
│   ├── main.rs          # bootstrap, router, graceful shutdown
│   ├── config.rs        # env parsing (PORT, FORGE_STORAGE_ROOT, ...)
│   ├── health.rs        # live/ready handlers
│   └── backend/
│       ├── mod.rs        # StorageBackend trait
│       └── local_fs.rs   # LocalFsBackend: root init/validation
├── Cargo.toml
├── Dockerfile
└── README.md
```

The service binds `0.0.0.0:$PORT`. `LocalFsBackend::init` ensures `FORGE_STORAGE_ROOT` exists, is writable, and contains an `objects/` and `meta/` subtree; readiness returns 200 only after the root is confirmed writable.

## Components

| Component | Purpose |
|---|---|
| `main.rs` | Router assembly + graceful shutdown on SIGTERM |
| `config.rs` | Typed config from env with validation |
| `health.rs` | Liveness always-200; readiness gated on backend init |
| `StorageBackend` trait | Seam for future backends; `LocalFsBackend` first impl |

## Contracts

* `GET /health/live` → `200 {"status":"live"}`
* `GET /health/ready` → `200 {"status":"ready"}` once root writable, else `503`
* `GET /` → `200` identity JSON: `{"service":"forge-storage","language":"rust","status":"running","version":"<v>"}`
* Add `contracts/openapi/forge-storage.openapi.yaml` with only these three paths (extended in later steps)

## Data model

No object metadata yet. On-disk layout initialized:

```

FORGE_SERVICE_VERSION
```

## Security considerations

* No auth yet; service is internal-only (not exposed publicly by Gateway)
* Refuse to start if `FORGE_STORAGE_ROOT` resolves outside an allowed base or is world-writable
* No secrets logged

## Observability

* Structured JSON logs to stdout with `timestamp`, `level`, `service`, `message`
* Log root path and backend readiness transition
* OTEL export recommended, not required this step

## Failure handling

* Root not writable → readiness `503`, log error, do not crash-loop needlessly (retry with backoff)
* Invalid `PORT` → exit non-zero at startup
* SIGTERM → stop accepting connections, finish in-flight, exit 0 within 10s
