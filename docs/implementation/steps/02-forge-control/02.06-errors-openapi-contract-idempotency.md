## Status

## Epic

## Goal

Formalize the platform-wide error envelope, publish a complete OpenAPI document for the Control API, add contract tests that assert responses match the spec, and implement `Idempotency-Key` support on create endpoints so retries do not create duplicates.

## Why this step exists

The spec requires a shared error format, an OpenAPI contract, idempotent creates, and contract validation. Doing this once — after the endpoints exist (02.03–02.05) — lets the CLI (03) and later services rely on a stable, documented, machine-readable contract and safe retries.

## Dependencies

* Prior steps in this epic: `02.03`, `02.04`, `02.05` (all endpoints exist)
* Cross-epic prerequisites: `contracts/` tree from epic `00`; error-format convention seeded by epic `01`

## Scope

* Canonical error envelope `{"error":{"code","message","details?,"requestId"}}` applied uniformly

* Contract tests validating live responses against the OpenAPI schema
* `Idempotency-Key` header support on all `POST` creates (store key → resulting resource; replay returns the original)

## Architecture

```text
Ktor StatusPages plugin
   └── maps DomainError → {error:{code,message,details,requestId}} + HTTP status

Idempotency:
  POST + header "Idempotency-Key: <k>"
   ├── first time  → execute, persist (key, request-hash, resource-id, response)
   └── replay      → return stored response (200/201) without re-creating

Contract tests
   └── run requests → validate response bodies/status against
       contracts/openapi/forge-control.openapi.yaml
```

## Contracts

Error envelope (normative for the platform):

```json
{
  "error": {
    "code": "validation_error",
    "message": "name must not be blank",
    "details": [{"field": "name", "issue": "blank"}],
    "requestId": "req_01H..."
  }
}
```

Status mapping: `400 validation_error`, `404 not_found`, `409 conflict`, `422 unprocessable` (optional), `500 internal_error`. Idempotency: `POST` with `Idempotency-Key` replays the original response for identical requests; a conflicting body reuse returns `409 idempotency_key_conflict`.

## Data model

New table:

```text
idempotency_keys(key text pk, request_hash text, resource_type text,
                 resource_id UUID, response_status int, response_body jsonb,
                 created_at timestamptz)
```

Added as migration `V2__idempotency_keys.sql`. Keys optionally expire (documented TTL; cleanup deferred).

## Configuration

```text
FORGE_IDEMPOTENCY_TTL_HOURS=24   # documented; cleanup job optional
```

## Security considerations

* `requestId` is generated server-side and echoed; never trust a client-supplied id for security decisions.
* Stored idempotency responses must not include secrets (none exist in this epic).
* OpenAPI examples contain no credentials.

## Observability

* Every error response logs `code`, `requestId`, and cause at appropriate level.
* Idempotency replays log `idempotency_replay=true` with the key hash (not raw sensitive data).

## Failure handling

* Unknown error → `500 internal_error` with generic message + `requestId` (details logged, not returned).
* Idempotency store failure → fail safe: do not double-create; return `503`/`500` with envelope.
* Malformed `Idempotency-Key` → `400`.

## Manual verification

```bash
# idempotent create
K=$(uuidgen)
A=$(curl -sf -X POST http://127.0.0.1:4001/v1/projects -H "Idempotency-Key: $K" -H 'content-type: application/json' -d '{"name":"idem"}')
B=$(curl -sf -X POST http://127.0.0.1:4001/v1/projects -H "Idempotency-Key: $K" -H 'content-type: application/json' -d '{"name":"idem"}')
[ "$A" = "$B" ] && echo "idempotent OK"
# error envelope
curl -s http://127.0.0.1:4001/v1/projects/00000000-0000-0000-0000-000000000000 | grep -q '"code"' && echo "envelope OK"
```
