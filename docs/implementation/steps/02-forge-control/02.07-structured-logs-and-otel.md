# Step 02.07: Structured logs + OTEL

## Status

Not started

## Epic

`02-forge-control` → [`../../epics/02-forge-control.md`](../../epics/02-forge-control.md)

## Goal

Make Control fully observable: structured JSON logs with the platform field set, request/trace correlation via a request id, and OpenTelemetry traces + basic metrics exported to the foundation OTEL Collector.

## Why this step exists

The spec lists OpenTelemetry and structured logging as required integrations for Control, and epic 12 (Observe) builds dashboards and correlation on top of consistent instrumentation. Doing it now, after endpoints exist, gives every request a trace and every log a correlatable id.

## Dependencies

* Prior steps in this epic: `02.01`–`02.06` (service + endpoints + request id from 02.06)
* Cross-epic prerequisites: epic `00` OTEL Collector (OTLP `4317`/`4318`), Grafana/Tempo/Loki/Prometheus stack

## Scope

* Structured JSON logs to stdout with platform fields (`timestamp`, `level`, `service`, `message`, `requestId`, `traceId`, `spanId`, plus contextual ids)
* OTEL SDK wiring: tracer + meter providers exporting via OTLP to the Collector
* HTTP server spans per request; DB spans around repository calls
* Basic metrics: request count, request duration histogram, error count
* `traceId`/`spanId` injected into log lines for correlation
* Log level controlled by `FORGE_LOG_LEVEL`

## Explicitly out of scope

* Grafana dashboards (epic 12)
* Alert rules (epic 12)
* Log aggregation queries (epic 12)
* Custom business metrics beyond request/DB basics

## Architecture

```text
Request → Ktor
   ├── OTEL server span (traceId/spanId)
   ├── logging interceptor → JSON line with traceId/spanId/requestId
   ├── repository call → DB client span
   └── metrics: http_server_requests, http_server_duration, http_server_errors
        │ OTLP 4317/4318
        ▼
   OTEL Collector → Tempo (traces) / Prometheus (metrics) / Loki (logs)
```

## Components

| Component | Purpose |
|---|---|
| `Telemetry.kt` | Configure OTEL tracer + meter providers + OTLP exporters |
| JSON log encoder | Logback/logstash (or kotlinx) JSON layout with platform fields |
| MDC/context bridge | Put `requestId`/`traceId`/`spanId` into log context |
| HTTP span interceptor | Server span per request |
| DB span wrapper | Span per repository operation |
| Metrics registry | Counters/histograms |

## Contracts

No HTTP contract change. Observability contract: logs are valid JSON lines containing at least `timestamp`, `level`, `service`, `message`, `requestId`; when a trace is active, `traceId` and `spanId` are present. OTLP export targets the Collector.

## Data model

None.

## Configuration

```text
FORGE_LOG_LEVEL=info
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317
OTEL_SERVICE_NAME=forge-control
OTEL_TRACES_SAMPLER=parentbased_always_on   # dev default
FORGE_OTEL_ENABLED=true                      # allow disabling in unit tests/CI
```

## Security considerations

* No secrets or request bodies with sensitive data in logs or span attributes.
* Redact/omit `Authorization` and any credential headers from span/log attributes (relevant once auth lands).
* Sampling configurable to limit data volume.

## Observability

This step *is* the observability wiring. Verification: traces visible in Tempo, metrics scraped by Prometheus, logs shipped/structured for Loki. A single request produces one server span with correlated log lines.

## Failure handling

* Collector unreachable → export failures must not crash the service; degrade to local logging (log a warning, drop/retry spans per SDK defaults).
* `FORGE_OTEL_ENABLED=false` disables exporters cleanly (for hermetic tests).
* Malformed log context never throws in the request path.

## Acceptance criteria

* Logs are structured JSON with the required platform fields and include `requestId`; when a trace is active, `traceId`/`spanId` are present and correlate to the span.
* OTEL traces for HTTP requests and DB calls are exported to the Collector and visible in Tempo.
* Request count, duration, and error metrics are exported.
* `FORGE_LOG_LEVEL` controls verbosity; `FORGE_OTEL_ENABLED=false` disables export without errors.
* Collector downtime does not crash the service.
