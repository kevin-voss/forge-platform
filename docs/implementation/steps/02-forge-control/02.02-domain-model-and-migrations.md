## Goal

Define the Control domain model — `Project`, `Environment`, `Application`, `Service`, `Deployment` — as Kotlin types plus a versioned PostgreSQL schema created by automatic migrations on startup, with a connection pool, a DB-backed readiness check, and repository interfaces ready for the API steps.

## Why this step exists

The API steps (02.03–02.05) need entities, tables, foreign keys, and repositories to read and write. Establishing the schema, UUID scheme, timestamps, and migration mechanism now keeps later steps focused on HTTP behavior and validation rather than persistence plumbing.

## Dependencies

* Prior steps in this epic: `02.01` (bootable service, config, health)
* Cross-epic prerequisites: PostgreSQL from epic `00` (host `5001`)

## Scope

* Schema `control` (or database `forge_control`) with tables for all five entities plus an `audit_log` table
* Versioned migrations (Flyway/Liquibase or equivalent) run automatically on startup and via a `make` target
* HikariCP connection pool configured from `DATABASE_*` env
* Repository interfaces + JDBC/SQL implementations for CRUD (used by later steps)
* Readiness check upgraded to verify DB connectivity
* Kotlin domain types (data classes) with UUID ids and created/updated timestamps

## Explicitly out of scope

* HTTP endpoints for the entities (02.03–02.05)
* Relationship validation rules exposed over the API (02.04)
* Idempotency keys (02.06)
* OTEL spans around DB calls (02.07)

## Architecture

```text
Ktor startup
   │
   ├── open Hikari pool (DATABASE_URL/USER/PASSWORD)
   ├── run migrations (V1__init.sql … )  ← automatic + `make migrate`
   └── /health/ready now also checks `SELECT 1`

schema control
├── projects        (id, name, slug, created_at, updated_at)
├── environments    (id, project_id → projects, name, created_at, updated_at)
├── applications    (id, project_id → projects, name, created_at, updated_at)
├── services        (id, application_id → applications, name, port, created_at, updated_at)
├── deployments     (id, service_id → services, environment_id → environments,
│                    image, desired_replicas, status, created_at, updated_at)
└── audit_log       (id, entity_type, entity_id, action, actor, at, detail jsonb)
```

## Contracts

No HTTP contract change. Internal repository contracts:

* `create(entity)`, `findById(id)`, `list(parentId?)`, `update(id, patch)`, `delete(id)` per entity as needed by later steps.
* All ids are UUID; all rows carry `created_at`/`updated_at` (UTC).

## Data model

```text
Project(id UUID pk, name text, slug text unique, created_at, updated_at)
Environment(id UUID pk, project_id fk→projects, name text, created_at, updated_at,
            unique(project_id, name))
Application(id UUID pk, project_id fk→projects, name text, created_at, updated_at,
            unique(project_id, name))
Service(id UUID pk, application_id fk→applications, name text, port int,
        created_at, updated_at, unique(application_id, name))
Deployment(id UUID pk, service_id fk→services, environment_id fk→environments,
           image text, desired_replicas int default 1, status text default 'pending',
           created_at, updated_at)
audit_log(id UUID pk, entity_type text, entity_id UUID, action text,
          actor text, at timestamptz, detail jsonb)
```

Foreign keys use `ON DELETE RESTRICT` (invalid deletes rejected). `desired_replicas >= 0` and `port` range checked via constraints.

## Configuration

```text
DATABASE_URL=jdbc:postgresql://postgres:5432/forge   # foundation Postgres
DATABASE_USER=forge
DATABASE_PASSWORD=forge
DATABASE_SCHEMA=control
DATABASE_POOL_MAX=10
DATABASE_MIGRATE_ON_START=true
```

`make migrate` (service Makefile) runs migrations without starting the HTTP server.

## Security considerations

* DB credentials only from env/`.env`; never logged.
* Migrations are checked into source and reviewed; no ad-hoc runtime DDL.
* Least-privilege DB user acceptable later; foundation `forge` user used for local dev.

## Observability

* Log migration version applied at startup (`from → to`).
* Log pool initialization (size) without credentials.
* Readiness failure logs the DB error cause.

## Failure handling

* Migration failure → service fails to become ready and exits non-zero (fail fast).
* DB unreachable at startup → readiness `503`; bounded retry/backoff before exit is acceptable.
* Constraint violations surface as typed repository errors for the API layer to translate (02.04/02.06).
