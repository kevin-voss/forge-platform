## Goal

Expose the first curlable slice of the control plane: create/list/get projects and create/list environments within a project, persisted via the repositories from 02.02, returning stable UUIDs and consistent JSON.

## Why this step exists

Projects and environments are the root of the hierarchy; everything else (applications, services, deployments) hangs off them. Delivering these two resources first provides the earliest end-to-end verification (`curl` create â†’ read back) and the parent records that 02.04/02.05 require.

## Dependencies

* Prior steps in this epic: `02.01` (service + health), `02.02` (domain model + repositories)
* Cross-epic prerequisites: none beyond epic `00` Postgres

## Scope

* `POST /v1/projects`, `GET /v1/projects`, `GET /v1/projects/{projectId}`
* `POST /v1/projects/{projectId}/environments`, `GET /v1/projects/{projectId}/environments`, `GET /v1/environments/{environmentId}`
* Request validation (non-blank name, slug uniqueness) with clear rejections
* JSON request/response DTOs with kotlinx.serialization
* Write audit rows for create actions

## Explicitly out of scope

* Applications, services, deployments (02.04/02.05)
* The formal shared error envelope + OpenAPI (introduced in 02.06 â€” this step uses a provisional error shape aligned with the plan)
* Idempotency keys (02.06)
* Update/delete endpoints (not required by the demo; may be added in 02.06 if trivial)

Error responses use the provisional envelope `{"error":{"code","message","details?"}}` (formalized in 02.06).

## Data model

Uses `projects` and `environments` tables from 02.02. No schema changes. `slug` is unique per platform; environment `name` is unique per project.

## Configuration

No new configuration. Uses `DATABASE_*` and service config from 02.01/02.02.

## Security considerations

* `FORGE_AUTH_MODE=dev`: requests attributed to a synthetic `dev` actor recorded in `audit_log`.
* Input length limits on `name`/`slug` to avoid oversized payloads.
* No secrets in requests or responses.

## Observability

* One structured log line per request (method, path, status, duration, request id).
* Audit row written on successful create (`action=create`, `actor=dev`).

## Failure handling

* Invalid body / blank name â†’ `400` with error envelope.
* Duplicate slug / duplicate environment name in project â†’ `409` conflict.
* Unknown `projectId` on environment create â†’ `404`.
* Repository/DB error â†’ `500` with safe message (no internals leaked).

## Implementation tasks

1. Define request/response DTOs for projects and environments.
2. Implement project service (validation, slug derivation, uniqueness, audit).
3. Implement environment service (project existence check, uniqueness, audit).
4. Register `/v1/projects` and nested `/v1/projects/{id}/environments` routes plus `/v1/environments/{id}`.
5. Map repository/constraint errors to `400/404/409/500` using the provisional envelope.
6. Emit audit rows on create.

## Acceptance criteria

* A project can be created and read back with a stable UUID.
* Environments can be created under a project and listed.
* Duplicate slug and duplicate environment name are rejected with `409`.
* Blank/invalid input returns `400` with an error envelope.
* Unknown parent project returns `404`.
* Created records survive a restart.
