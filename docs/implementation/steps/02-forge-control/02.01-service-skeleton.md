## Goal

Stand up the `forge-control` Kotlin + Ktor service as a runnable, health-checked container wired into Docker Compose on host port `4001`, with a Makefile, Dockerfile, environment-based configuration, and graceful shutdown — but no domain logic yet.

## Why this step exists

Every later Control step (domain model, APIs, audit, telemetry) needs a compiling, bootable service with the platform operational contract already satisfied. Building the skeleton first means subsequent steps only add handlers and persistence, never plumbing.

## Dependencies

* Prior steps in this epic: none (first step of epic 02)
* Cross-epic prerequisites:
  * Epic `00` — Make interface, Compose stack, PostgreSQL on `5001`, service template, ports doc
  * Epic `01` — health endpoint, structured-log, config, and graceful-shutdown conventions (`01.01`)

## Scope

* `services/forge-control/` Ktor module that builds and runs
* `GET /health/live` and `GET /health/ready` per the runtime contract
* Environment-based configuration loader (`PORT`, `FORGE_*`)
* Graceful shutdown on `SIGTERM` within a documented grace period
* `Dockerfile` and service-local `Makefile` targets (`make dev`, build, test)
* Docker Compose wiring so `make service-run SERVICE=forge-control` starts it on host `4001`
* Reservation of host port `4001` in `docs/operations/ports.md`

## Explicitly out of scope

* Domain entities, database schema, or migrations (02.02)
* Any `/v1/*` business endpoints (02.03+)
* OpenAPI document (02.06)
* OTEL exporter wiring beyond a no-op placeholder (02.07)
* Authentication logic beyond reading `FORGE_AUTH_MODE` (09.x)

## Architecture

```text
make service-run SERVICE=forge-control
        │
        ▼
Docker Compose service `forge-control`
        │  host 4001 → container PORT (8080)
        ▼
Ktor application
├── /health/live   → 200 {"status":"live"}
├── /health/ready  → 200 {"status":"ready"} (ready once started)
└── SIGTERM        → stop accepting, drain, exit 0
```

Control depends on the foundation PostgreSQL container but does not yet open a connection pool in this step (readiness is process-only until 02.02 adds a DB check).

## Contracts

Implements the epic-01 runtime-contract health surface:

* `GET /health/live` → `200` JSON `{"status":"live"}`
* `GET /health/ready` → `200` JSON `{"status":"ready"}` when started; `503` before

No `/v1` contract in this step. OpenAPI document lands in 02.06.

## Data model

None. No persistence is opened in this step.

## Configuration

```text
PORT=8080                 # in-container listen port (host publishes 4001)
FORGE_SERVICE_NAME=forge-control
FORGE_SERVICE_VERSION=0.1.0
FORGE_LOG_LEVEL=info      # debug|info|warn|error
FORGE_ENV=development
FORGE_AUTH_MODE=dev       # dev bypass until Identity 09.06
FORGE_SHUTDOWN_GRACE_SECONDS=10
# Reserved for 02.02 (not consumed yet):
# DATABASE_URL, DATABASE_USER, DATABASE_PASSWORD
```

Host publish: `4001:8080`. If both `PORT` and `FORGE_HTTP_PORT` are set, `PORT` wins (per `01.01`).

## Security considerations

* Health endpoints are unauthenticated by design.
* `FORGE_AUTH_MODE=dev` is read and recorded in logs at startup so its presence is auditable; enforcement is deferred to epic 09.
* Run as non-root in the container image where practical.
* No secrets committed; local DB credentials come from `.env`.

## Observability

* Structured JSON logs to stdout with fields `timestamp`, `level`, `service`, `message` (full platform field set formalized in 02.07).
* Startup log line records effective config (excluding secrets) and `FORGE_AUTH_MODE`.
* OTEL is deferred to 02.07; a no-op tracer/placeholder is acceptable.

## Failure handling

* Missing/invalid `PORT` → process exits non-zero at startup.
* `SIGTERM` → stop accepting connections, drain in-flight, exit `0` within `FORGE_SHUTDOWN_GRACE_SECONDS`.
* `/health/ready` returns `503` until the server has fully started.
