## Goal

Add applications (under a project) and services (under an application) to the control plane, with explicit relationship validation so that invalid hierarchies are rejected before persistence, extending the curlable slice from 02.03.

## Why this step exists

Deployments (02.05) target a service. Services belong to applications, which belong to projects. This step completes the parent chain and codifies the relationship rules the spec requires ("invalid relationships are rejected").

## Dependencies

* Prior steps in this epic: `02.03` (projects & environments), `02.02` (repositories), `02.01` (service)
* Cross-epic prerequisites: none

## Scope

* `POST /v1/projects/{projectId}/applications`, `GET` list, `GET /v1/applications/{id}`
* `POST /v1/applications/{applicationId}/services`, `GET` list, `GET /v1/services/{id}`
* Relationship validation: application must reference an existing project; service must reference an existing application; names unique within their parent
* Service `port` validation (1–65535)
* Audit rows on create

## Explicitly out of scope

* Deployments (02.05)
* Cross-project moves / reparenting
* OpenAPI + idempotency (02.06)
* Update/delete (optional, only if trivial and additive)

## Architecture

```text
POST /v1/projects/{projectId}/applications
        │ validate project exists
        ▼
ApplicationService → ApplicationRepository

POST /v1/applications/{applicationId}/services
        │ validate application exists, port in range, name unique
        ▼
ServiceService → ServiceRepository
```

## Contracts

```text
POST /v1/projects/{projectId}/applications
  body: {"name":"web"}
  201:  {"id","projectId","name","createdAt","updatedAt"}
  404:  unknown projectId
  409:  duplicate application name in project

GET  /v1/projects/{projectId}/applications  → 200 [ApplicationResponse]
GET  /v1/applications/{id}                  → 200 | 404

POST /v1/applications/{applicationId}/services
  body: {"name":"api","port":8080}
  201:  {"id","applicationId","name","port","createdAt","updatedAt"}
  400:  port out of range
  404:  unknown applicationId
  409:  duplicate service name in application

GET  /v1/applications/{applicationId}/services → 200 [ServiceResponse]
GET  /v1/services/{id}                          → 200 | 404
```

## Failure handling

* Unknown parent (`projectId`/`applicationId`) → `404`.
* Duplicate name within parent → `409`.
* Invalid `port` or blank name → `400`.
* DB constraint violation → mapped to `409`/`400` as appropriate, never `500` for known cases.

## Acceptance criteria

* Applications can be created under a project and listed.
* Services can be created under an application with a valid port and listed.
* Invalid relationships (unknown parent) are rejected with `404`.
* Duplicate names within a parent return `409`; invalid ports return `400`.
* The full project→application→service chain persists and reads back with stable UUIDs.
