# Step 02.05: Deployments desired-state API + basic audit

## Status

Not started

## Epic

`02-forge-control` → [`../../epics/02-forge-control.md`](../../epics/02-forge-control.md)

## Goal

Add the deployments resource that records **desired state** for a service (image, replicas, target environment) and returns it, plus consistent audit records for all mutations, completing the hierarchy the demo reads back. No containers run — Control only stores intent.

## Why this step exists

Deployments are the artifact later epics act on: Runtime (04) materializes them, the reconciler (07) converges toward them, the CLI (03) creates them. This step makes desired state a first-class, queryable record and finalizes the audit trail.

## Dependencies

* Prior steps in this epic: `02.04` (services), `02.03`, `02.02`, `02.01`
* Cross-epic prerequisites: none

## Scope

* `POST /v1/services/{serviceId}/deployments` — create desired deployment
* `GET /v1/deployments/{deploymentId}` — read one
* `GET /v1/services/{serviceId}/deployments` — list for a service
* Optional read model: `GET /v1/projects/{projectId}?expand=tree` returning the full hierarchy (used by the demo)
* Deployment fields: `image`, `desiredReplicas` (default 1), `environmentId`, `status` (starts `pending`)
* Audit records for deployment creation and any status change

## Explicitly out of scope

* Actual container execution / reconciliation (epics 04, 07)
* Rolling updates, rollback (epic 07)
* Idempotency-Key handling (02.06 — though the schema anticipates it)
* Authn/authz (epic 09)

## Architecture

```text
POST /v1/services/{serviceId}/deployments
   │ validate service + environment exist
   ▼
DeploymentService
   ├── insert deployment (status=pending)
   └── audit_log append (deployment.create)

GET /v1/deployments/{id}                 → DeploymentResponse
GET /v1/projects/{id}?expand=tree        → { project, environments[], applications[ services[ deployments[] ] ] }
```

`status` transitions are stored but not driven by Control in this epic (Runtime/reconciler update actual status later via 04.07/07.x). For epic 02 the value remains `pending` unless set explicitly by an internal test hook.

## Contracts

```text
POST /v1/services/{serviceId}/deployments
  body: {"image":"localhost:5000/demo-go:latest","desiredReplicas":1,"environmentId":"<uuid>"}
  201:  {"id","serviceId","environmentId","image","desiredReplicas","status":"pending","createdAt","updatedAt"}
  400:  missing image / bad replicas
  404:  unknown serviceId or environmentId

GET /v1/deployments/{id}                    → 200 DeploymentResponse | 404
GET /v1/services/{serviceId}/deployments    → 200 [DeploymentResponse]
GET /v1/projects/{projectId}?expand=tree    → 200 hierarchical ProjectTreeResponse | 404
```

## Data model

Uses `deployments` and `audit_log` from 02.02. `desired_replicas >= 0`; `status ∈ {pending, active, failed, stopped}` (enforced in code/constraint). `environment_id` must belong to the same project as the service's application (cross-checked in the service layer).

## Configuration

None new.

## Security considerations

* `FORGE_AUTH_MODE=dev` actor recorded on every audit row.
* Validate `image` is a non-empty string; do not attempt to pull or resolve it.
* No credentials stored on deployments in this epic (registry is local/anonymous).

## Observability

* Structured log per request with `deploymentId`, `serviceId`.
* Audit rows: `deployment.create` and (if used) `deployment.status_change` with `detail` JSON capturing old/new.

## Failure handling

* Unknown `serviceId`/`environmentId` → `404`.
* Environment not in the service's project → `400` (relationship error).
* Missing `image` or negative replicas → `400`.
* DB error → `500` with safe message.

## Acceptance criteria

* A desired deployment can be created for a service and read back with a stable UUID and `status=pending`.
* Deployments reference a valid service and environment within the same project; mismatches are rejected.
* `expand=tree` returns the complete hierarchy.
* Audit rows are written for all create actions.
* Desired state survives a restart.
* No container execution is triggered.
