# Step 10.05: CLI `forge secret` / `forge config`

## Status

Not started

## Epic

`10-forge-secrets` → [`../../epics/10-forge-secrets.md`](../../epics/10-forge-secrets.md)

## Goal

Add developer-facing CLI commands for secrets and config: `forge secret set/list/rotate`, `forge config set/show`, scoped to the active project + environment, authenticated with the login token from `09.07`. Secret values are entered without echo or via stdin; `list` shows names/versions only.

## Why this step exists

`specs.md` Step 10 explicitly specifies these commands (`forge secret set/list/rotate`, `forge config set`). They are the primary way developers manage secrets and are needed to drive the `demos/10-secrets` scenario.

## Dependencies

* Step `10.02`/`10.03` — secret + config APIs with auth
* Step `09.07` — CLI auth/token profile
* Epic `03-forge-cli` — CLI framework, output formatting, exit codes

## Scope

* `forge secret set NAME` (prompt without echo, or `--from-stdin` / `--from-file`)
* `forge secret list` (names + current version, never values)
* `forge secret rotate NAME` (prompt/stdin new value → new version)
* `forge config set NAME=VALUE` and `forge config show` (values shown — config is non-secret)
* Project/env selection via flags/profile (`--project`, `--env`, defaults from profile)
* Machine-readable `--json` output; correct exit codes; request-id surfacing on error

## Explicitly out of scope

* Runtime injection (10.04)
* Audit/masking internals (10.06)
* Bulk import/export of secrets (nice-to-have later)

## Architecture

```text
forge secret set DB_PW           → read value (no echo) → PUT .../secrets/DB_PW
forge secret list                → GET .../secrets (metadata only)
forge secret rotate DB_PW        → read new value → PUT (creates new version)
forge config set FEATURE_X=true  → PUT .../config/FEATURE_X
forge config show                → GET .../config (values shown)
```

## Components

| Component | Responsibility |

| `secret` command group | set/list/rotate |

## Contracts

* Consumes `PUT/GET .../secrets`, `.../secrets/{name}` metadata, `PUT/GET .../config` (10.02/10.03)
* No new server API

## Data model

None (CLI). Uses the credential profile from `09.07` and project/env from the CLI config.

## Configuration

```text
FORGE_PROJECT / --project
FORGE_ENV     / --env       # default from profile (e.g. production)
FORGE_SECRETS_URL           # secrets endpoint (from profile)
```

## Security considerations

* Secret values read without echo; never written to shell history when using stdin/file
* `list` never prints values; `rotate` never prints the old or new value
* Values not logged even in verbose mode (redacted)
* `--from-file` reads a file the user controls; warn if world-readable

## Observability

* Print request IDs on failures (from `03.04`)
* No secret values in any CLI log/telemetry
* Exit codes consistent with `03.04` (auth=4, not-found, etc.)

## Failure handling

* Not logged in / expired token → prompt to `forge login`, non-zero exit
* `403` (wrong project/role) → clear message with required action
* Empty value → rejected before sending; `--from-file` missing → clear error
* Secrets service down → actionable error, non-zero exit
