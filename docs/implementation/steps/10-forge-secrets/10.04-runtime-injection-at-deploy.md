## Goal

Deliver a service's resolved secrets + config into its workload as environment variables at deploy time. Control/Reconciler requests the env bundle for a service+environment from Secrets, and Runtime injects it when creating the container. Plaintext exists only inside the workload's environment â€” never persisted by Control and never written to logs.

## Why this step exists

Secrets are useless until workloads receive them. `specs.md` Step 10 requires "runtime delivery" and that "runtime receives the correct secret." This connects the encrypted store (10.02/10.03) to Runtime's env injection (`04.03`) via a bindings model that says which secrets/config a service consumes.

## Dependencies

* Step `10.02`/`10.03` â€” encrypted store + `:access` reveal + config, with authorization
* Epic `04-forge-runtime` â€” workload env injection at create (`04.03`)
* Epic `07-deployment-reconciliation` â€” the reconciler that creates workloads (calls Runtime)

## Scope

* Service bindings: which secret names + config names a service consumes in a given environment (`bindings` table)
* Resolve endpoint: `POST /v1/projects/{pid}/envs/{env}/services/{svc}/resolve` â†’ env bundle `{ KEY: value }` combining decrypted secrets + config (authorized, audited)
* Reconciler integration: on workload create, fetch the resolved bundle (service-account token) and pass it to Runtime env injection
* Redeploy semantics: changing a secret version or binding triggers a new deployment so the workload picks up the new value
* Ensure Control never stores the resolved plaintext (pass-through only)

## Explicitly out of scope

* CLI (10.05)
* Audit trail + masking details (10.06) â€” resolve is audited via the 10.02 hook, full masking is 10.06
* File-based secret mounts (env vars only this epic)

## Architecture

```text

  bundle = Secrets.resolve(service, env)      # via service-account token
  Runtime.createWorkload(image, env = bundle) # 04.03 env injection
```

## Security considerations

* Resolve requires a service-account token scoped to the project (`09.05`); Control uses least-privilege
* Plaintext bundle is transient: held in memory by Control only long enough to hand to Runtime; never persisted or logged
* Runtime treats injected env as sensitive: not echoed in workload status/logs
* Resolve is audited (10.06) so secret access at deploy is traceable

## Observability

* Log resolve requests with `{project, env, service, keys:[names], version_fingerprint}` â€” names only, never values
* Metric: `forge_secret_resolves_total`, `forge_injected_keys` histogram
* Trace span `secrets.resolve` and reconciler `deploy.inject` (no values in attributes)

## Failure handling

* Missing bound secret â†’ resolve `409`/`422` listing missing names; reconciler holds the deploy (does not start a workload with absent required secrets)
* Secrets service down â†’ deploy fails safe (no start without required secrets), retried by reconciler
* Value contains characters invalid for env keys/values â†’ validated at set time (10.02) so resolve is safe
