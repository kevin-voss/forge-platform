# Step 10.02: Encrypted store + key versioning + metadata APIs

## Status

Not started

## Epic

`10-forge-secrets` → [`../../epics/10-forge-secrets.md`](../../epics/10-forge-secrets.md)

## Goal

Store secret values encrypted at rest (per-project data key), version secrets so rotation preserves history, and expose metadata APIs that list secret names/versions **without ever returning plaintext**. A single explicit "reveal" path exists only for authorized retrieval (used by runtime injection), gated separately from listing.

## Why this step exists

This is the core of the epic: encryption-at-rest, versioning, and the guarantee that plaintext is neither stored nor leaked via list APIs (`specs.md`: "plaintext secrets are not stored", "not returned by list APIs"). Key versioning is required for rotation (10.05) and master-key rotation.

## Dependencies

* Step `10.01` — service + key hierarchy (master key wraps per-project data keys)
* Epic `09-forge-identity` — project scope + authorization (enforced fully in 10.03)

## Scope

* `secrets` table: encrypted value (AEAD ciphertext + nonce), project/env scope, name, version, data-key version
* Set/create secret: `PUT /v1/projects/{pid}/envs/{env}/secrets/{name}` (creates a new version if it exists)
* List metadata: `GET /v1/projects/{pid}/envs/{env}/secrets` → names, current version, created_at, updated_at — **no values**
* Get metadata for one: `GET .../secrets/{name}` → metadata + version history (no values)
* Authorized reveal (internal): `POST .../secrets/{name}:access` returns decrypted value — audited (audit in 10.06), used by injection (10.04)
* Data-key + master-key versioning fields so re-wrap/rotation is possible

## Explicitly out of scope

* Non-secret config APIs (10.03)
* Full project-isolation enforcement via Identity (10.03) — this step scopes by path + a basic guard
* Runtime injection (10.04)
* Audit logging + masking (10.06) — reveal endpoint records a placeholder hook now

## Architecture

```text
PUT secret:
  data_key = unwrap(project_data_key)         # in memory only
  (ciphertext, nonce) = AEAD.encrypt(data_key, value)
  insert secrets(version = max+1, ciphertext, nonce, data_key_version)

GET list/metadata: SELECT name, version, timestamps  (never ciphertext/plaintext)

POST access (reveal): decrypt latest (or requested) version → return value  (audited)
```

## Components

| Component | Responsibility |
|---|---|
| `SecretStore` | Persist encrypted versions; query metadata |
| `SecretCipher` | AEAD encrypt/decrypt with the project data key |
| `SecretRoutes` | set / list / metadata / access |
| `VersionManager` | Compute next version; list history |

## Contracts

* `PUT /v1/projects/{pid}/envs/{env}/secrets/{name}` `{ "value": "..." }` → `201 { name, version }`
* `GET /v1/projects/{pid}/envs/{env}/secrets` → `[ { name, version, updated_at } ]` (no values)
* `GET /v1/projects/{pid}/envs/{env}/secrets/{name}` → `{ name, current_version, versions:[{version,created_at}] }`
* `POST /v1/projects/{pid}/envs/{env}/secrets/{name}:access` → `{ name, version, value }` (authorized + audited)

## Data model

```sql
CREATE TABLE secrets (
  id               BIGSERIAL PRIMARY KEY,
  project_id       TEXT NOT NULL,
  environment      TEXT NOT NULL,
  name             TEXT NOT NULL,
  version          INT  NOT NULL,
  ciphertext       BYTEA NOT NULL,
  nonce            BYTEA NOT NULL,
  data_key_version INT  NOT NULL,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (project_id, environment, name, version)
);
CREATE INDEX idx_secrets_scope ON secrets(project_id, environment, name);
```

## Configuration

```text
FORGE_SECRETS_AEAD_ALG        # aes-256-gcm | xchacha20poly1305 (default aes-256-gcm)

```

## Security considerations

* Values encrypted with a fresh nonce per write; ciphertext + nonce stored, plaintext never
* List/metadata endpoints structurally cannot return values (separate `:access` path)
* `:access` is the only decrypt path and is audited (10.06) — least exposure
* Reject oversized values; validate names (no injection into env var keys)

## Observability

* Log set/list/metadata with `{project, env, name, version}` — never the value
* Metric: `forge_secrets_total{project}`, `forge_secret_access_total`
* Trace spans without plaintext attributes

## Failure handling

* Decrypt failure (wrong/rotated key) → `500` typed crypto error, logged without plaintext; never returns garbage
* Unknown secret on access → `404`
* Value too large → `413`
* Version conflict → new version always wins (monotonic)

```bash
make dev   # secrets service up with master key
P=prj_1; E=production
curl -s -X PUT localhost:4104/v1/projects/$P/envs/$E/secrets/DATABASE_PASSWORD -d '{"value":"s3cret"}' -H 'content-type: application/json'

```

## Acceptance criteria

* Secret values are stored encrypted (AEAD) with per-write nonces; plaintext is never persisted
* Setting an existing secret creates a new version; history is queryable
* List and metadata APIs never return plaintext values
* A separate authorized `:access` endpoint decrypts and returns the value (used by injection)
* Decrypt failures are handled cleanly without leaking plaintext or garbage
