# Step 10.03: Config vs secrets APIs; project isolation

## Status

Not started

## Epic

`10-forge-secrets` → [`../../epics/10-forge-secrets.md`](../../epics/10-forge-secrets.md)

## Goal

Separate non-secret **configuration** (stored plaintext, listable with values) from **secrets** (encrypted, values hidden), each project/environment scoped, and enforce strict **project isolation** via Identity authorization so no caller can read or write another project's secrets or config.

## Why this step exists

`specs.md` Step 10 requires "configuration values separate from secrets" and that "unauthorized projects cannot access secrets." 10.02 built encrypted storage; this step adds the config counterpart and turns path-based scoping into enforced, Identity-backed authorization.

## Dependencies

* Step `10.02` — encrypted secret store + APIs
* Epic `09-forge-identity` — `authz/check` with `secret.read`/`secret.write`/`config.read`/`config.write` actions (`09.04`), token introspection (`09.05`)

## Scope

* `config` store + APIs: `PUT/GET/DELETE .../config/{name}` and `GET .../config` (values **are** returned — config is non-secret)
* Auth middleware in Secrets: introspect the bearer token via Identity, resolve principal + project role, authorize each action, and verify the token's project scope matches the path project
* Project isolation: even a valid token cannot access a project it has no membership in (server-side enforced, not client-trusted)
* Distinct actions for config vs secrets so a `viewer` might read config but not reveal secrets (per matrix)
* Consistent error format; `401`/`403` on auth failures

## Explicitly out of scope

* Runtime injection (10.04)
* CLI (10.05)
* Audit + masking (10.06) — authorization decisions logged, full audit trail is 10.06

## Architecture

```text
request → SecretsAuth:
  token = bearer or 401
  principal = Identity.introspect(token)
  if principal.project_id != path.project_id → 403 (isolation)
  action = map(method, path)  # secret.read | secret.write | config.read | config.write
  Identity.authz/check(principal, project, action) allow ? proceed : 403
```

## Components

| Component | Responsibility |
|---|---|
| `ConfigStore` | Persist plaintext config per project/env |
| `ConfigRoutes` | CRUD config (values returned) |
| `SecretsAuth` | Introspect + isolation + authorize |
| `IdentityClient` | Call introspect + authz/check (cached) |

## Contracts

* `PUT /v1/projects/{pid}/envs/{env}/config/{name}` `{ "value": "true" }` → `201`
* `GET /v1/projects/{pid}/envs/{env}/config` → `[ { name, value, updated_at } ]` (values included)
* `DELETE .../config/{name}` → `204`
* All secret + config endpoints now require `Authorization: Bearer` and enforce project scope

## Data model

```sql
CREATE TABLE config_values (
  project_id  TEXT NOT NULL,
  environment TEXT NOT NULL,
  name        TEXT NOT NULL,
  value       TEXT NOT NULL,          -- plaintext (non-secret by definition)
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (project_id, environment, name)
);
```

## Security considerations

* Project isolation enforced from the introspected token's project scope, never from client-supplied identity
* Config is explicitly non-secret; documentation warns not to put secrets in config
* Secret reveal (`:access`) requires `secret.read`; config read requires `config.read` — different privilege levels
* Fail-closed on Identity outage for writes/reveals

## Observability

* Log authorization decisions `{principal, project, action, allow}`
* Metric: `forge_secrets_authz_total{action,result}`, `forge_config_values_total`
* Trace spans linking to Identity calls

## Failure handling

* Token project ≠ path project → `403` (isolation), logged as a potential cross-tenant attempt
* Missing/invalid token → `401`
* Identity unreachable → writes/reveals `503` (fail-closed); config reads may use cache within TTL
* Deleting missing config → `404` (idempotent option documented)

```bash
make dev   # identity + secrets in enforce mode
P=prj_1; E=production
# config (non-secret) is listable with values
curl -s -X PUT localhost:4104/v1/projects/$P/envs/$E/config/FEATURE_X -H "Authorization: Bearer $DEV_A" -d '{"value":"true"}' -H 'content-type: application/json'

# unauthenticated → 401
curl -s -o /dev/null -w '%{http_code}\n' localhost:4104/v1/projects/$P/envs/$E/secrets  # 401
```

## Acceptance criteria

* Config values are stored separately from secrets and are returned by config list APIs; secret values remain hidden
* Every secret/config endpoint requires authentication and enforces the token's project scope (isolation)
* A token for one project cannot access another project's secrets or config (`403`)
* Config vs secret read/write map to distinct authorization actions honored by the Identity matrix
* Identity outage fails closed for secret writes/reveals
