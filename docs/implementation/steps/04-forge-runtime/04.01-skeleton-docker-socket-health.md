# Step 04.01: Skeleton + Docker socket + health

## Status

Not started

## Epic

`04-forge-runtime` → [`../../epics/04-forge-runtime.md`](../../epics/04-forge-runtime.md)

## Goal

Stand up the `forge-runtime` Rust service as a health-checked container on host port `4102` that can reach the Docker Engine over the mounted socket, with env-based configuration, structured logs, graceful shutdown, and a readiness check that verifies Docker connectivity — but no workload logic yet.

## Why this step exists

Workload lifecycle steps (04.03–04.06) need a bootable service with a working Docker client and the platform operational contract in place. Establishing the skeleton and Docker connectivity first isolates environment/socket issues from workload logic.

## Dependencies

* Prior steps in this epic: none (first step of epic 04)
* Cross-epic prerequisites: epic `00` (Docker Engine, Compose, ports); epic `01` health/log conventions

## Scope

* `services/forge-runtime/` Rust service (Tokio + Axum) that builds and runs
* `GET /health/live` and `GET /health/ready` (ready requires Docker ping success)
* Docker Engine client wired to `/var/run/docker.sock`
* Env-based config (`PORT`, `FORGE_*`, `DOCKER_HOST`/socket path)
* Structured JSON logs + graceful shutdown on `SIGTERM`
* `Dockerfile`, service `Makefile`, Compose wiring (socket mount), port `4102` reserved

## Explicitly out of scope

* Node registration/heartbeat (04.02)
* Any workload create/start/stop (04.03+)
* Log streaming (04.05)
* Control integration (04.07)

## Architecture

```text
Compose service `forge-runtime` (host 4102 → PORT 8080)
  mounts /var/run/docker.sock
        │
        ▼
Axum app
├── /health/live   → 200
├── /health/ready  → 200 iff docker.ping() ok, else 503
└── Docker client (bollard) → Docker Engine
```

## Contracts

Runtime-contract health surface: `GET /health/live` → `200`; `GET /health/ready` → `200` when Docker is reachable, else `503`. No workload API yet.

## Data model

None persisted in this step (node id persistence arrives in 04.02).

## Configuration

```text
PORT=8080
FORGE_SERVICE_NAME=forge-runtime
FORGE_SERVICE_VERSION=0.1.0
FORGE_LOG_LEVEL=info
FORGE_ENV=development
FORGE_AUTH_MODE=dev
DOCKER_HOST=unix:///var/run/docker.sock
FORGE_SHUTDOWN_GRACE_SECONDS=10
```

Host publish `4102:8080`.

## Security considerations

* Mounting the Docker socket grants broad host control; documented as a dev-only privileged mount and flagged for later hardening.
* Health endpoints unauthenticated by design.
* Run the process as a user in the `docker` group where feasible; no secrets logged.

## Observability

* Structured JSON logs (`tracing` + JSON subscriber) with `timestamp`, `level`, `service`, `message`.
* Startup logs Docker Engine version from `ping`/`version`.
* OTEL export deferred (aligns with the platform pattern; wired alongside 04.07 or a later observe step).

## Failure handling

* Docker unreachable at startup → readiness `503`; bounded retry before optionally exiting non-zero (documented).
* Invalid `PORT` → exit non-zero.
* `SIGTERM` → drain + exit `0` within grace period.
