## Goal

Implement the core workload lifecycle: accept a workload definition, pull the image, create and start a Docker container with injected environment variables, a published port, and deterministic name/labels derived from the deployment id.

## Why this step exists

This is the heart of Runtime — turning a desired workload into a running container. Everything else (health probing, logs, stop/delete, Control integration) operates on containers created here.

## Dependencies

* Prior steps in this epic: `04.01` (Docker client + health), `04.02` (node id + labels)
* Cross-epic prerequisites: a deployable image in the local registry (`01.03` Go demo → `localhost:5000/demo-go:latest`)

## Scope

* `POST /v1/workloads` accepting the spec's workload definition (deployment_id, image, port, environment)
* Image pull (from `localhost:5000` / any reachable registry)
* Container create with env injection, port publish (ephemeral host port), deterministic name `forge-<deploymentId>`, labels (`forge.deployment_id`, `forge.node_id`, `forge.managed=true`)
* Container start; report assigned host port back in the response
* `GET /v1/workloads/{deploymentId}` returning current container mapping/state

## Explicitly out of scope

* Health probing / status derivation (04.04)
* Log streaming (04.05)
* Stop/delete + dedupe-on-retry (04.06 — though naming here enables it)
* Control push/pull integration (04.07)

## Architecture

```text
POST /v1/workloads {deployment_id, image, port, environment}
   │
   ├── docker pull image
   ├── docker create:
   │      name  = forge-<deployment_id>
   │      env   = environment map
   │      ports = container `port` → ephemeral host port
   │      labels= forge.deployment_id, forge.node_id, forge.managed=true
   ├── docker start
   └── read back host port → 201 {deploymentId, containerId, hostPort, state}
```

## Contracts

```text
POST /v1/workloads
  body: {"deployment_id":"deployment-123",
         "image":"localhost:5000/demo-go:latest",
         "port":8080,
         "environment":{"FORGE_ENV":"development"}}
  201:  {"deploymentId":"deployment-123","containerId":"<docker id>",
         "hostPort":49xxx,"state":"starting"}
GET /v1/workloads/{deploymentId}
  200:  {"deploymentId","containerId","hostPort","state","image"}
  404:  no such workload
```

## Data model

No database. Source of truth for actual state is Docker (queried by label/name). Response DTOs mirror inspected container facts.

## Configuration

```text
FORGE_PULL_TIMEOUT_SECONDS=120
FORGE_DEFAULT_REGISTRY=localhost:5000   # informational; images are fully qualified
```

## Security considerations

* Validate `image` reference format; pull only from reachable registries (local dev anonymous).
* Do not log full environment maps if they could contain sensitive values (mask values, log keys) — anticipates Secrets epic 10.
* Containers run with `forge.managed=true` so only managed containers are ever touched by stop/delete (04.06).

## Observability

* Log pull start/finish (image, duration), create, and start with `deploymentId` + `containerId`.
* Emit host-port assignment.

## Failure handling

* Image pull failure → `502`/`422` with a clear error; no container left dangling.
* Create/start failure → clean up any partially created container; return error with cause.
* Invalid spec (missing image/port) → `400`.
* Port publish failure → error; do not leave a created-but-unstarted container.

## Implementation tasks
