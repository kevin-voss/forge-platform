# Step 04.07: Control integration (desired→actual)

## Status

Not started

## Epic

`04-forge-runtime` → [`../../epics/04-forge-runtime.md`](../../epics/04-forge-runtime.md)

## Goal

Close the loop between Control's desired state and Runtime's actual state: Runtime learns about deployments targeted at its node, materializes them via the 04.03–04.06 lifecycle, and reports actual status (and the published endpoint) back to Control so a deployment's status becomes `active`/`failed`.

## Why this step exists

Until now Runtime is driven only by its own API. The spec requires Runtime to receive desired workload instructions and report actual state to Control. This integration is what makes the CLI→Control→Runtime demo (04.08) and Gateway route sync (05.03) possible.

## Dependencies

* Prior steps in this epic: `04.03`–`04.06` (full lifecycle + status)
* Cross-epic prerequisites: Control deployments API (`02.05`) and a status/endpoint update path (added or extended here on the Control side is out of scope — Runtime uses existing Control endpoints; if a status-update endpoint is missing, this step defines the contract Runtime expects and uses a documented interim: Runtime exposes state for Control to pull)

## Scope

* Runtime obtains desired deployments for its node — via one of: (a) polling a Control endpoint, or (b) accepting pushes from Control. Chosen: **poll Control** for deployments assigned to this node (simple, robust) + expose current actual state for pull.
* Reconcile the fetched desired set against actual containers using idempotent create + delete (04.03/04.06)
* Report actual status + published `hostPort`/endpoint back to Control (push on change) or expose via `GET /v1/node/state` for Control to read
* Map Runtime status vocabulary → Control deployment status (`pending`/`active`/`failed`/`stopped`)

## Explicitly out of scope

* Continuous multi-replica reconciliation with rolling updates (epic 07 — this step is a single-shot converge, not a full controller)
* Multi-node assignment/scheduling (epic 08)
* Auth between services (epic 09; dev mode until then)

## Architecture

```text
loop (interval):
  desired = Control.GET deployments for node   (or pushed set)
  actual  = docker managed containers (by label)
  converge:
    desired\actual → POST /v1/workloads (create)   [idempotent]
    actual\desired → DELETE /v1/workloads/{id}
  report:
    for each: map Runtime status → Control status
    Control.UPDATE deployment status + endpoint (push)  OR  expose GET /v1/node/state
```

Runtime remains authoritative for actual state; Control remains authoritative for desired state.

## Components

| Component | Purpose |
|---|---|
| `control_client.rs` | Talk to Control (fetch desired, report actual) |
| `converge.rs` | Single-shot desired↔actual convergence |
| status mapper | Runtime status → Control status |
| `/v1/node/state` | Expose actual state for pull (interim/fallback) |

## Contracts

Runtime → Control (consumed):

```text
GET  {CONTROL}/v1/deployments?nodeId=<id>&desired=true   → desired deployments for this node
POST {CONTROL}/v1/deployments/{id}/status                → {status, endpoint:{hostPort,...}, nodeId}
```

Runtime → local (exposed):

```text
GET /v1/node/state → { nodeId, workloads:[{deploymentId, status, hostPort, image}] }
```

> If Control lacks a status-update endpoint at this point, the pull model (`/v1/node/state`) is the interim contract; the push endpoint shape above is documented for Control to implement (tracked as a cross-epic note).

## Data model

No new persistence. Desired set is fetched; actual set is Docker-derived.


## Security considerations

* Dev-mode auth (`FORGE_AUTH_MODE=dev`) on Control calls; real service auth in epic 09.
* Runtime only reports state for deployments it manages; it will not fabricate status for unknown ids.

## Observability

* Log each converge cycle: created/deleted/updated counts.
* Log status transitions reported to Control with `deploymentId`.

## Failure handling

* Control unreachable → skip the cycle, log a warning, retry next interval (no crash, no local churn).
* Report failure → retry with backoff; local actual state remains correct.
* Desired references an image that fails to pull → workload `failed`, reported as `failed` (not retried infinitely without backoff).

## Acceptance criteria

* Runtime obtains desired deployments for its node and converges actual containers to match.
* Actual status + published endpoint are reported to Control (or exposed for pull); a healthy workload drives the deployment to `active`, a failed one to `failed`.
* Deleting a deployment in Control causes Runtime to remove the container.
* Control downtime does not cause local churn or crashes.
* Convergence is idempotent (no duplicate containers across cycles).
