## Goal

Probe each workload's health (via the runtime-contract endpoints and Docker container state) and derive a normalized status model — `starting`, `running`, `ready`, `unhealthy`, `stopped`, `failed` — that later steps report to Control and Gateway consumes for routing.

## Why this step exists

Deployment status must become `active` only when the workload is actually ready. Gateway (05.04) must skip unready upstreams. A consistent status model derived from real probes is the basis for both.

## Dependencies

* Prior steps in this epic: `04.03` (running workloads with a host port), `04.01`/`04.02`
* Cross-epic prerequisites: workloads implement the epic-01 health contract (`/health/live`, `/health/ready`)

## Scope

* Periodic health probing of managed workloads on their published port (`/health/ready`, `/health/live`)
* Combine Docker container state + probe results into a normalized status
* `GET /v1/workloads/{deploymentId}/status` returning the status model + last probe details
* Status transitions with timestamps; readiness gating (`ready` only when `/health/ready` 200)
* Configurable probe interval, timeout, and failure threshold

## Explicitly out of scope

* Reporting status to Control (04.07)
* Log streaming (04.05)
* Restart/rollback policies (epic 07)

## Architecture

```text
prober (interval):
  for each managed workload:
    docker state  → created/running/exited
    GET :hostPort/health/ready  → 200? 
    GET :hostPort/health/live   → 200?
    derive status:
      exited/error            → failed
      running & ready 200     → ready
      running & live 200      → running
      running & !live (N x)   → unhealthy
      just started            → starting
      stopped by us           → stopped

GET /v1/workloads/{id}/status → {status, since, lastProbe:{live,ready,at}, restarts}
```

## Contracts

```text
GET /v1/workloads/{deploymentId}/status
  200: {"deploymentId","status":"ready","since":"...",
        "lastProbe":{"live":true,"ready":true,"at":"..."},"restarts":0}
```

Status vocabulary is normative for Runtime and reused by 04.07 (Control mapping) and 05.04 (Gateway).

## Data model

In-memory status cache keyed by deployment id, rebuilt from Docker on restart (probe resumes). No database.

## Configuration

```text
FORGE_PROBE_INTERVAL_SECONDS=5
FORGE_PROBE_TIMEOUT_SECONDS=2
FORGE_PROBE_FAILURE_THRESHOLD=3
FORGE_PROBE_READY_PATH=/health/ready
FORGE_PROBE_LIVE_PATH=/health/live
```

## Security considerations

* Probes hit only managed workloads on their mapped host/loopback port.
* Probe responses are not logged verbatim beyond status codes (avoid capturing app payloads).

## Observability

* Log status transitions with old→new + reason.
* Expose probe latency/failure counts (metrics wired with observability later).

## Failure handling

* Probe timeout/connection refused counts toward the failure threshold before marking `unhealthy`.
* Container exited unexpectedly → `failed` (distinct from operator-initiated `stopped`).
* Prober never crashes the process; per-workload probe errors are isolated.

## Implementation tasks

1. Define the status enum + derivation rules in `status.rs`.
2. Implement the probe HTTP client (ready/live) with timeout.
3. Implement the periodic prober over all managed workloads (discovered by label).
4. Maintain an in-memory status cache with transition timestamps.
5. Add `GET /v1/workloads/{id}/status`.
6. Make interval/timeout/threshold configurable.
7. On service restart, rediscover managed containers and resume probing.
