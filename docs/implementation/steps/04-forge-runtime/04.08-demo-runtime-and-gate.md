## Goal

Deliver `demos/04-runtime`, which deploys the Step-01 Go demo image end-to-end via the CLI→Control→Runtime path: the container starts, becomes ready, the deployment status flips to `active`, logs are readable, deleting the deployment removes the container, and duplicate deploys create no duplicates — serving as the epic acceptance gate.

## Why this step exists

The spec defines a concrete demo and acceptance criteria for Runtime (deploy one arbitrary OCI image, report actual state, failed startup → failed state, no duplicate containers, deterministic names/labels, demo passes). This step composes all prior Runtime steps into one falsifiable check.

## Dependencies

* Prior steps in this epic: `04.01`–`04.07` (all)
* Cross-epic prerequisites: Control (epic 02); Go demo image (`01.03`) present in `localhost:5000`; CLI optional (`03.03`) to drive the deploy; `make demo` runner

## Scope

* `demos/04-runtime/` with `README.md` and `run.sh`
* Script: bring up Postgres, Control, Runtime; build/tag the Go demo image into `localhost:5000`; create hierarchy + deployment (via CLI or Control API); assert container running + ready + status `active`; read logs; delete; assert removal; assert idempotency
* Negative case: a deployment with a bad image → deployment `failed`
* `make demo DEMO=04` wiring; epic acceptance

## Explicitly out of scope

* Gateway routing (epic 05)
* Rolling updates / rollback (epic 07)
* Multi-node (epic 08)

## Architecture

```text
make demo DEMO=04
   │
   ▼
demos/04-runtime/run.sh
   ├── compose up postgres forge-control forge-runtime (+ registry from 00)
   ├── ensure localhost:5000/demo-go:latest exists (build/tag/push if needed)
   ├── create project→env→app→service (CLI or API)
   ├── deployment create → wait for status active
   ├── curl app via reported hostPort; read logs
   ├── delete deployment → assert container gone
   ├── negative: deploy bad image → assert deployment failed
   └── idempotency: deploy same twice → one container
```

## Observability

* `run.sh` prints each phase + a final pass/fail; on failure dumps Runtime + Control logs and `docker ps -a`.

## Failure handling

* Any assertion failure → non-zero exit with context.
* Readiness/convergence timeout → fail with logs.
* Cleanup on exit (trap): delete deployment, `docker rm` stragglers, `compose down`.
