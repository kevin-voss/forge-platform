## Goal

Expose a workload's container logs through the Runtime API — both a bounded fetch (last N lines / since timestamp) and a follow/stream mode — so operators, the CLI (`forge logs`), and later Observe can read what a deployment is doing.

## Why this step exists

The Step-04 acceptance criteria require that "logs can be read." Log access is also the foundation for `forge logs --follow` (12.05) and incident diagnosis by agents (epic 15).

## Dependencies

* Prior steps in this epic: `04.03` (running workloads)
* Cross-epic prerequisites: none

## Scope

* `GET /v1/workloads/{deploymentId}/logs` — bounded fetch with `tail`, `since`, `stdout/stderr` selection
* `GET /v1/workloads/{deploymentId}/logs?follow=true` — streaming (chunked/SSE) tail
* Multiplex stdout/stderr from the Docker log stream
* Backpressure-safe streaming with client-disconnect handling

## Explicitly out of scope

* Log persistence/aggregation (epic 12 / Loki)
* Log parsing/indexing/search (epic 12)
* CLI wiring of `forge logs` (epic 12.05; Runtime provides the API here)

## Architecture

```text
GET /v1/workloads/{id}/logs?tail=100&since=...        → bounded, one response
GET /v1/workloads/{id}/logs?follow=true               → streamed chunks (SSE/chunked)
   │
   ▼
docker logs (container by label/name) → demux stdout/stderr → client
   client disconnect → cancel docker stream
```

## Components

| Component | Purpose |
|---|---|
| `logs.rs` | Docker log fetch + stream demux |

## Contracts

```text
GET /v1/workloads/{deploymentId}/logs
  query: tail=<n> (default 100), since=<rfc3339>, streams=stdout|stderr|all
  200 (bounded): text/plain or application/x-ndjson lines
GET /v1/workloads/{deploymentId}/logs?follow=true
  200: streamed (Content-Type: text/event-stream or chunked); ends on stop/disconnect
  404: unknown workload
```

Each line optionally annotated with stream source and timestamp when `ndjson` is requested.

## Data model

None persisted. Logs are read directly from Docker on demand.

## Configuration

```text
FORGE_LOG_DEFAULT_TAIL=100
FORGE_LOG_STREAM_BUFFER=8192
```

## Security considerations

* Only managed workloads (`forge.managed=true`) are readable.
* No auth yet (dev mode); flagged that log access will be authorized in epic 09.
* Avoid unbounded memory: stream rather than buffering entire log history.

## Observability

* Log the open/close of follow streams with `deploymentId` and duration.
* Count active log streams (metric later).

## Failure handling

* Unknown workload → `404`.
* Container removed mid-stream → stream ends cleanly with a terminal marker.
* Client disconnect → cancel the underlying Docker stream promptly (no leak).

* Follow mode streams new lines and stops on stop/disconnect without leaking resources.
* stdout and stderr are distinguishable.
