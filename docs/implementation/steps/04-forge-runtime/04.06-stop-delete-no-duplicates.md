## Goal

Complete the lifecycle: gracefully stop and delete a workload's container, and make create/start idempotent so repeated deploy commands for the same deployment never produce duplicate containers.

## Why this step exists

Step-04 acceptance requires that deleting a deployment removes the container and that "duplicate commands do not produce duplicate containers." Idempotent lifecycle operations are also essential for the reconciler (epic 07) which repeatedly issues desired state.

## Dependencies

* Prior steps in this epic: `04.03` (create/start), `04.04` (status)
* Cross-epic prerequisites: none

## Scope

* `DELETE /v1/workloads/{deploymentId}` — graceful stop (SIGTERM + grace) then remove container
* Idempotent `POST /v1/workloads`: if a container for the deployment id already exists, return the existing one (or reconcile config) instead of creating a duplicate
* Deterministic lookup by name/label to detect existing containers
* Orphan cleanup helper (remove managed containers with no known deployment on request)

## Explicitly out of scope

* Reconciliation loop / desired-vs-actual convergence (epic 07)
* Rolling replace (epic 07)
* Multi-replica handling (epic 07/08)

## Architecture

```text
DELETE /v1/workloads/{id}
   ├── docker stop (SIGTERM, wait grace) → kill if needed
   ├── docker rm
   └── 204 (idempotent: already-gone → 204)

POST /v1/workloads (idempotent):
   lookup by name forge-<id> / label forge.deployment_id
   ├── exists & running & same image → return existing (200)
   ├── exists & stopped              → start it (or recreate) 
   └── absent                        → create+start (201)  [04.03 path]
```

## Contracts

```text
DELETE /v1/workloads/{deploymentId}
  204: removed (also 204 if already absent — idempotent)
POST /v1/workloads (idempotent)
  201: created new
  200: already existed (returns existing mapping)
  409: exists with conflicting image/config (documented decision) — or recreate
```

## Data model

None persisted; Docker (by label/name) remains the source of truth.

## Configuration

```text
FORGE_STOP_GRACE_SECONDS=10
FORGE_ON_CONFIG_CONFLICT=recreate    # or 'reject' → 409 (documented default: recreate)
```

## Security considerations

* Only containers labeled `forge.managed=true` may be stopped/removed — never touch foreign containers.
* Deletion is irreversible; require the exact deployment id (no wildcard delete via API).

## Observability

* Log stop (with grace used), kill-if-needed, and remove events.
* Log idempotent short-circuits (`existing container reused`).

## Failure handling

* Stop timeout → force kill, then remove; report the escalation.
* Remove failure (container busy) → retry/backoff, then error with cause.
* Concurrent create for the same id → single winner (locking by deployment id); loser returns the existing container.

## Manual verification

```bash
# idempotent create
curl -sf -X POST http://127.0.0.1:4102/v1/workloads -H 'content-type: application/json' -d '{"deployment_id":"deployment-123","image":"localhost:5000/demo-go:latest","port":8080,"environment":{}}'
curl -sf -X POST http://127.0.0.1:4102/v1/workloads -H 'content-type: application/json' -d '{"deployment_id":"deployment-123","image":"localhost:5000/demo-go:latest","port":8080,"environment":{}}'
[ "$(docker ps -aq --filter label=forge.deployment_id=deployment-123 | wc -l | tr -d ' ')" = "1" ] && echo "no duplicates OK"
# delete
curl -sf -X DELETE http://127.0.0.1:4102/v1/workloads/deployment-123 -o /dev/null -w '%{http_code}\n'
curl -sf -X DELETE http://127.0.0.1:4102/v1/workloads/deployment-123 -o /dev/null -w '%{http_code}\n'   # still 204
```

## Acceptance criteria

* `DELETE` gracefully stops then removes the container; repeated deletes are idempotent (`204`).
* Repeated `POST` for the same deployment id never creates a duplicate container.
* Concurrent creates for the same id result in a single container.
* Only `forge.managed=true` containers are ever stopped/removed.
* Stop escalates to force-kill after the grace period.
