## Goal

Give the Runtime node a stable identity that persists across restarts, and expose registration/heartbeat so the node advertises itself and its liveness — locally now, and to Control/scheduler later.

## Why this step exists

Workloads must be attributable to a node, and multi-node scheduling (epic 08) and Control integration (04.07) need a stable node id and heartbeat. Establishing identity before workloads means every container can be labeled with its node from the start.

## Dependencies

* Prior steps in this epic: `04.01` (service + Docker + health)
* Cross-epic prerequisites: none required now; Control registration endpoint consumed later (04.07 / epic 08)

## Scope

* Generate a stable node id once and persist it to a data directory
* `GET /v1/node` returning node identity + basic capacity/info (id, hostname, docker version, cpu/mem best-effort)
* Internal heartbeat mechanism (timestamped liveness) exposed via `GET /v1/node/heartbeat` or included in `/v1/node`
* Node metadata used later as container labels (`forge.node_id`)
* Optional outbound registration hook (no-op/stub until Control endpoint exists)

## Explicitly out of scope

* Registering with a remote Control/scheduler for real (04.07 / epic 08)
* Resource scheduling decisions (epic 08)
* Workload lifecycle (04.03+)

## Architecture

```text
startup:
  node_id = read(data_dir/node_id) OR generate uuid → persist
  node_info = { id, hostname, docker_version, cpu, mem, started_at }

GET /v1/node            → node_info + last_heartbeat
GET /v1/node/heartbeat  → { node_id, at, healthy }

(optional) registration hook: POST to Control (stub until 04.07)
```

## Contracts

```text
GET /v1/node
  200: {"id":"<uuid>","hostname":"...","dockerVersion":"...",
        "cpu":<n>,"memoryBytes":<n>,"startedAt":"...","lastHeartbeat":"..."}
GET /v1/node/heartbeat
  200: {"nodeId":"<uuid>","at":"...","healthy":true}
```

## Data model

Persisted node identity:

```text
<data_dir>/node_id        # plain uuid string, created once
```

In-memory: `NodeInfo`, `Heartbeat{at, healthy}`. No database in Runtime (state is Docker + small local files).

## Security considerations

* Node id is non-secret but stable; stored `0600`.
* Info endpoints expose only non-sensitive host facts; no secrets or full env dumps.

## Observability

* Log node id (and whether generated vs loaded) at startup.
* Heartbeat transitions (healthy↔unhealthy) logged.

## Failure handling

* Unwritable data dir → fail startup with a clear error (identity must persist).
* Docker version lookup failure → still serve identity with `dockerVersion: unknown` and log a warning.
* Heartbeat task panic → restart the task; never crash the process.
