# Step 05.04: Health-aware upstreams

## Status

Not started

## Epic

`05-forge-gateway` → [`../../epics/05-forge-gateway.md`](../../epics/05-forge-gateway.md)

## Goal

Route traffic only to ready upstreams: combine the readiness/status reported by Runtime (04.04) with the gateway's own lightweight passive/active checks so unhealthy or not-yet-ready workloads receive no traffic, and recovered ones are automatically re-added.

## Why this step exists

The spec requires that "unhealthy workloads receive no traffic." Round-robin over all upstreams (05.02) would send requests to dead pods. Health-awareness makes routing correct during startup, failure, and rolling changes (which the reconciler in epic 07 relies on).

## Dependencies

* Prior steps in this epic: `05.02` (proxy/balancer), `05.03` (synced routes with status)
* Cross-epic prerequisites: Runtime status model (`04.04`) exposing per-workload readiness

## Scope

* Mark each upstream ready/unready using: (1) Runtime-reported status from sync, and (2) gateway active checks (`/health/ready` probe) and/or passive checks (recent 5xx/connection errors)
* Balancer skips unready upstreams; if none are ready → `503` with a clear body
* Automatic re-add when an upstream becomes ready again
* Configurable check interval, failure/success thresholds, and probe path

## Explicitly out of scope

* Rolling traffic shift orchestration (epic 07 owns the rollout; gateway just honors health)
* Circuit-breaker/retry budgets beyond basic passive marking (documented as future)
* Metrics dashboards (epic 12)

## Architecture

```text
upstream state: ready | unready (with lastChange)
sources:
  - sync status (Runtime 04.04) sets initial/authoritative readiness
  - active probe: GET upstream/health/ready every interval
  - passive: N consecutive connection errors/5xx → unready

balancer.pick(route):
  candidates = route.upstreams where state==ready
  if empty → 503 no_healthy_upstream
  else round-robin among candidates
```

## Components

| Component | Purpose |
|---|---|
| `internal/health` (extend) | Active prober for upstreams |
| upstream state store | ready/unready per upstream w/ thresholds |
| balancer (extend) | Filter to ready upstreams |
| passive marker | Hook proxy errors → failure counter |

## Contracts

No new public contract. Behavior contract: only `ready` upstreams receive traffic; zero ready → `503 {error:{code:"no_healthy_upstream"}}`.

## Data model

In-memory per-upstream state: `{state, consecutiveFailures, consecutiveSuccesses, lastProbeAt}`.

## Configuration

```text
FORGE_UPSTREAM_PROBE_INTERVAL_SECONDS=5
FORGE_UPSTREAM_PROBE_PATH=/health/ready
FORGE_UPSTREAM_FAILURE_THRESHOLD=3
FORGE_UPSTREAM_SUCCESS_THRESHOLD=2
FORGE_UPSTREAM_TRUST_RUNTIME_STATUS=true   # use Runtime-reported readiness as authoritative
```

## Security considerations

* Probes hit only configured upstreams.
* Probe results not logged verbatim (status only).

## Observability

* Log upstream state transitions (ready↔unready) with reason (probe/passive/sync).
* Expose ready/unready counts per route (metric later).

## Failure handling

* All upstreams unready → `503` (not `502`) with `no_healthy_upstream`.
* Probe flapping dampened by success/failure thresholds.
* Passive failures decay/reset on success so recovery is automatic.

## Implementation tasks

1. Add per-upstream health state with thresholds.
2. Implement the active prober (`/health/ready`) on an interval.
3. Wire passive marking from proxy errors (connection refused / 5xx).
4. Integrate Runtime-reported status from sync as authoritative when trusted.
5. Filter the balancer to ready upstreams; return `503` when none.
6. Ensure automatic re-add on recovery.

## Acceptance criteria

* Only ready upstreams receive traffic; unready ones are skipped.
* When no upstream is ready, the gateway returns `503 no_healthy_upstream`.
* Upstreams are automatically re-added when they recover.
* Runtime-reported readiness is honored (authoritative when trusted); active/passive checks catch gaps.
* Health-state changes require no gateway restart.
