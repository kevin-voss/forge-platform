## Goal

Build routes automatically from platform state: periodically read active service endpoints (from Control, with Runtime `/v1/node/state` as an interim source) and update the route table so that deployed workloads become reachable by a stable hostname without manual `PUT /admin/routes`.

## Why this step exists

Manual route configuration doesn't scale and doesn't reflect reality. The spec requires the gateway to read active service endpoints and update routes dynamically. This step turns actual deployments into live routes.

## Dependencies

* Prior steps in this epic: `05.02` (route table + proxy)
* Cross-epic prerequisites: Runtime published endpoints/status (`04.04`/`04.07`); Control service/deployment data (`02.05`) — the source that maps a service to a hostname + upstream

## Scope

* A sync loop that fetches active endpoints on an interval and rebuilds the route table
* Route derivation: service/deployment → hostname (`<service>.<project>.demo.localhost` or configured pattern) → upstream(s) = Runtime host port(s)
* A manual `POST /admin/routes/refresh` to force an immediate sync
* Atomic route-table swap (reuse 05.02) so updates need no restart
* Source selection: Control primary; Runtime `/v1/node/state` fallback/interim (configurable)

## Explicitly out of scope

* Health-aware filtering of upstreams (05.04 — sync includes all known, health gate filters)
* Event-driven push updates (epic 11)
* Header/timeout behavior (05.05)

## Architecture

```text
sync loop (interval):
  endpoints = Control.GET active endpoints        (or Runtime /v1/node/state)
  routes    = map endpoints → Route{host, upstreams:[hostPort]}
  routeTable.Replace(routes)   # atomic (05.02)

POST /admin/routes/refresh → run one sync now
```

Hostname pattern is configurable; default `<service>.<project>.demo.localhost`.

## Components

| Component | Purpose |
|---|---|
| `internal/sync` | Fetch endpoints + derive routes + schedule |
| endpoint client | Control (and Runtime fallback) HTTP clients |
| host pattern | Build hostnames from service/project |
| refresh route | `POST /admin/routes/refresh` |

## Contracts

Consumes (one of):

```text
GET {CONTROL}/v1/endpoints            → [{host, service, project, upstreams:[{url}], ready}]
GET {RUNTIME}/v1/node/state           → { workloads:[{deploymentId, hostPort, status, ...}] }  (interim)
```

> If Control does not yet expose an endpoints read model, this step documents the expected shape above and uses the Runtime `/v1/node/state` interim source, joining with Control service/deployment metadata for hostnames.

Exposes: `POST /admin/routes/refresh → 200 {routesLoaded:N}`.

## Data model

In-memory route table (05.02). No persistence.

## Configuration

```text
FORGE_CONTROL_URL=http://forge-control:8080
FORGE_RUNTIME_URL=http://forge-runtime:8080
FORGE_ROUTE_SOURCE=control            # control|runtime
FORGE_ROUTE_SYNC_INTERVAL_SECONDS=10
FORGE_HOST_PATTERN={service}.{project}.demo.localhost
```

## Security considerations

* Dev-mode auth to Control/Runtime (epic 09 later).
* Only endpoints reported by trusted platform sources become routes (no client-driven route injection via the data plane).

## Observability

* Log each sync: sources queried, routes built, added/removed diffs.
* Log sync failures without dropping the existing (last-good) route table.

## Failure handling

* Source unreachable → keep the last-good route table, log a warning, retry next interval.
* Malformed endpoint data → skip the bad entry, keep the rest.
* Refresh endpoint reports partial success with counts.

## Acceptance criteria

* Routes are built automatically from platform state and updated on an interval without restarting the gateway.
* A newly deployed workload becomes reachable by its stable hostname after a sync (or refresh).
* A removed workload's route is dropped on the next sync.
* Source downtime does not wipe existing routes (last-good retained).
* `POST /admin/routes/refresh` forces an immediate sync and reports counts.
