## Goal

Make proxied requests correct and traceable: generate/propagate a request id, set standard forwarded headers (`X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Host`, `Forwarded`), and enforce configurable connect/response timeouts so slow or hung upstreams don't tie up the gateway.

## Why this step exists

The spec requires request IDs, forwarded headers, and timeouts. Request-id propagation ties gateway logs to Control/Runtime/app logs (one trace across the platform); forwarded headers let upstreams see the real client; timeouts protect the gateway.

## Dependencies

* Prior steps in this epic: `05.02` (proxy core), `05.04` (health-aware) — this layers cross-cutting behavior on the proxy
* Cross-epic prerequisites: request-id convention aligned with Control (`02.06` `X-Request-Id`)

## Scope

* Generate `X-Request-Id` if absent; propagate the incoming one if present; echo it on the response
* Set forwarded headers correctly (append to `X-Forwarded-For`, set proto/host, RFC 7239 `Forwarded`)
* Configurable timeouts: connect, response header, and overall request; return `504` on timeout
* Ensure the request id is present in gateway access logs and passed upstream (so app logs correlate)

## Explicitly out of scope

* WebSocket/SSE special handling (05.06 — but timeouts must not kill long-lived streams; documented interplay)
* Retries/hedging (future)
* TLS/`X-Forwarded-Proto=https` termination (dev is http)

## Architecture

```text
inbound request
  ├── ensure X-Request-Id (reuse or generate)
  ├── set X-Forwarded-For (append client IP), X-Forwarded-Proto/Host, Forwarded
  ├── context timeout (overall) + transport timeouts (connect/response header)
  ▼
ReverseProxy → upstream
  ├── upstream sees request id + forwarded headers
  └── on timeout → 504 gateway_timeout (request id in body/log)
response: echo X-Request-Id
```

## Components

| Component | Purpose |
|---|---|
| request-id middleware | reuse/generate + echo |
| forwarded-headers setter | X-Forwarded-*/Forwarded |
| transport config | connect + response-header timeouts |
| context timeout | overall request deadline |

## Contracts

Header contract: every proxied request carries `X-Request-Id` (echoed to client and forwarded upstream) and correct forwarded headers. Timeout contract: overall deadline exceeded → `504 gateway_timeout`.

## Data model

None.

## Configuration

```text
FORGE_REQUEST_ID_HEADER=X-Request-Id
FORGE_PROXY_CONNECT_TIMEOUT_SECONDS=5
FORGE_PROXY_RESPONSE_HEADER_TIMEOUT_SECONDS=15
FORGE_PROXY_OVERALL_TIMEOUT_SECONDS=30
FORGE_TRUST_INBOUND_XFF=false     # whether to trust client-provided X-Forwarded-For
```

## Security considerations

* Do not blindly trust inbound `X-Forwarded-For`; append the observed client IP and control trust via config to avoid spoofing.
* Do not forward hop-by-hop headers; strip per RFC 7230.
* Request id is for correlation only, never an authz signal.

## Observability

* Access log per request includes `requestId`, host, path, upstream, status, duration.
* Timeout events logged distinctly with the upstream and elapsed time.

## Failure handling

* Overall timeout → `504` with `gateway_timeout` envelope + request id.
* Connect failure → `502` (health marking handled in 05.04).
* Ensure long-lived streaming routes (05.06) are exempt from the overall timeout (per-route override).

## Implementation tasks

1. Add request-id middleware (reuse/generate/echo) using the configured header.
2. Implement forwarded-header handling with configurable XFF trust; strip hop-by-hop headers.
3. Configure transport connect + response-header timeouts.
4. Add an overall request deadline via context; map timeout → `504`.
5. Ensure request id + forwarded headers reach the upstream.
6. Add a per-route flag to exempt streaming routes from the overall timeout (used by 05.06).
