## Goal

Stand up the `forge-gateway` Go service as a health-checked container on host port `4000`, with env-based configuration, structured logs, graceful shutdown, and Compose wiring — but no routing yet.

## Why this step exists

The routing, sync, and proxy steps need a bootable service with the platform operational contract in place. Building the skeleton first isolates plumbing from proxy logic.

## Dependencies

* Prior steps in this epic: none (first step of epic 05)
* Cross-epic prerequisites: epic `00` (Compose, ports); epic `01` health/log conventions

## Scope

* `services/forge-gateway/` Go service that builds and runs
* `GET /health/live` and `GET /health/ready`
* Env-based config; structured JSON logs; graceful shutdown on `SIGTERM`
* `Dockerfile`, service `Makefile`, Compose entry; host port `4000` reserved

## Explicitly out of scope

* Route table / reverse proxy (05.02)
* Route sync (05.03)
* Health-aware balancing (05.04)
* Headers/timeouts/WS/SSE (05.05/05.06)

## Architecture

```text
Compose service `forge-gateway` (host 4000 → PORT 8080)
        │
        ▼
net/http server
├── /health/live   → 200
├── /health/ready  → 200 (ready once started)
└── SIGTERM        → graceful shutdown
```

## Security considerations

* Health endpoints unauthenticated.
* No edge auth yet (epic 09); documented.
* No secrets logged.

## Observability

* Structured JSON logs with `timestamp`, `level`, `service`, `message`.
* Startup logs effective config (non-secret).

## Failure handling

* Invalid `PORT` → exit non-zero.
* `SIGTERM` → drain + exit `0` within grace.
* Readiness `503` until server started.
