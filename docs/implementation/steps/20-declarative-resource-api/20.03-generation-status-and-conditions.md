## Goal

Make `metadata.generation` bump only when `spec` actually changes, split `spec` writes from `status` writes into two enforced-separate endpoints, and ship the shared `Condition` type plus a merge helper and an optional phase-derivation helper that owning controllers use to report convergence.

## Why this step exists

Without this step, "declarative" is just a JSON blob with no way for a client to tell "my edit hasn't taken effect yet" from "my edit was applied and failed" from "nothing changed." Generation vs `observedGeneration` is the mechanism that answers that question, and it only works if ordinary clients cannot bump `status` and controllers cannot bump `spec` — hence the hard split enforced here, not left to convention.

## Dependencies

* `20.02` — generic CRUD, `resourceVersion` guard
* `20.01` — envelope types (`status` already exists as a JSONB column)

## Scope

* Generation bump rule: any `PUT`/`PATCH` that changes `spec` (byte-for-byte JSON comparison after canonicalization) increments `generation` by 1; a write that touches only `metadata.labels`/`annotations`/`finalizers` or `status` does not
* `PUT /v1/.../{plural}/{name}/status` — status-only subresource; body may contain only `status` (and optionally `metadata.resourceVersion` for the same optimistic-concurrency guard); any `spec` field present → `400 status_subresource_spec_forbidden`
* Main resource `PUT`/`PATCH` rejects a `status` field in the body → `400 spec_endpoint_status_forbidden`
* `Condition` type: `type`, `status` (`True`/`False`/`Unknown`), `reason`, `message`, `lastTransitionTime`
* `mergeCondition(existing, next)` — replaces same-`type` condition; updates `lastTransitionTime` only when `status` actually changes value, otherwise preserves it and updates `reason`/`message` in place
* Optional `derivePhase(conditions, deletionTimestamp, generation, observedGeneration)` helper implementing the six-phase enum (`Pending`/`Progressing`/`Ready`/`Degraded`/`Failed`/`Terminating`); controllers may call it or set `status.phase` directly
* `/status` writer identity convention: `X-Forge-Controller: <name>` header checked against `KindDescriptor.owningController` (soft, documented, not real authz — see epic open questions)

## Explicitly out of scope

* Real service-identity enforcement of the `/status` writer (epic 09)
* Label selectors / listing (`20.04`)
* Watch (`20.05`)
* Finalizers / deletion-timestamp semantics (`20.06`) — `deletionTimestamp` is read here only insofar as `derivePhase` accepts it as an input
* Wiring any real controller (e.g. the epic-07 reconciler) to call `/status` — that lands in `20.07` once `Deployment` is a registered kind

## Architecture

```text
PUT   .../{plural}/{name}          spec + metadata only → generation bump on spec change
PUT   .../{plural}/{name}/status   status only, X-Forge-Controller checked → resourceVersion bump, no generation bump
                     │
                     ▼
              mergeCondition() → status.conditions
              derivePhase()   → status.phase (optional helper)
```

## Contracts

```http
PUT /v1/projects/p/environments/e/widgets/sample/status
X-Forge-Controller: widget-controller
Content-Type: application/json

{
  "metadata": { "resourceVersion": "1042" },
  "status": {
    "phase": "Ready",
    "observedGeneration": 3,
    "conditions": [
      { "type": "Available", "status": "True",
        "reason": "MinimumReplicasAvailable", "message": "3/3 replicas ready" }
    ]
  }
}
```

Response: full envelope, `metadata.generation` unchanged, `metadata.resourceVersion` incremented, `lastTransitionTime` server-populated (`now()`) if the condition's `status` value changed from the prior stored condition of the same `type`.

Rejection examples:

```json
{ "error": { "code": "status_subresource_spec_forbidden",
             "message": "the /status subresource accepts status only", "requestId": "..." } }
{ "error": { "code": "spec_endpoint_status_forbidden",
             "message": "status is read-only on this endpoint; use /status", "requestId": "..." } }
```

## Data model

No new migration. `generation` and `status` already exist on `control.resources` from `20.01`; this step is entirely write-path policy over existing columns. `status.conditions` is an array inside the existing `status JSONB` column, not a separate table — condition history beyond the current value is not retained (matches Kubernetes: conditions are current-state, not an audit trail; the audit trail is `resource_events`, landing in `20.05`).

## Configuration

```text
FORGE_STATUS_CONTROLLER_HEADER_ENFORCED   # default true; set false only for local single-controller dev convenience
```

## Security considerations

* `X-Forge-Controller` is a convention, not authentication — documented as temporary and superseded by epic 09 service identity; this is stated in code comments and in the epic's open questions, not silently assumed
* Rejecting cross-writes (`spec` via `/status`, `status` via the main endpoint) closes the most common way controllers and clients trample each other even before real auth exists

## Observability

* Log on generation bump: `{kind, name, old_generation, new_generation}`
* Log on status write: `{kind, name, controller, phase, observed_generation}`
* Metric `forge_resource_generation_bumps_total{kind=...}`
* Metric `forge_resource_condition_transitions_total{kind=...,type=...,status=...}`
* Span `resource.status_write`

## Failure handling

* `spec` present in a `/status` body → `400 status_subresource_spec_forbidden` (no partial apply)
* `status` present in a main-endpoint body → `400 spec_endpoint_status_forbidden`
* `X-Forge-Controller` missing or mismatched `owningController` when enforcement is on → `403 status_writer_not_recognized`
* Stale `resourceVersion` on `/status` → same `409 resource_version_conflict` as `20.02`

## Unit tests

* `GenerationPolicy`: identical spec (including key-order differences) → no bump; changed spec → bump by exactly 1; label-only change → no bump
* `mergeCondition`: same status value twice → `lastTransitionTime` unchanged; status flips → `lastTransitionTime` updated, `reason`/`message` replaced
* `derivePhase`: each of the six phases produced from the documented input combination (e.g. `deletionTimestamp` set → `Terminating` regardless of conditions)

## Integration tests

* `PUT` changing `spec` bumps `generation`; a second identical `PUT` does not
* `PUT /status` with a `spec` key → `400`; main `PUT` with a `status` key → `400`
* `PUT /status` without a matching `X-Forge-Controller` → `403` when enforcement is on
* `observedGeneration` written via `/status` is readable via the main `GET` alongside the current `generation`

## Contract tests

* OpenAPI declares `/status` and the `Condition` schema; the shared brief's example condition JSON deserializes against it

## Acceptance criteria

* `generation` bumps exactly once per distinct `spec` change and never on a status-only or label-only write
* `/status` accepts only `status` (+ optional `resourceVersion`); the main endpoint accepts only `spec`/`metadata`
* `mergeCondition` preserves `lastTransitionTime` across non-transitioning updates and refreshes it on real transitions
* `derivePhase` produces all six documented phases from their stated inputs
