## Goal

Add a generic watch API and durable resource-event log so clients and controllers can move from "list then poll" to "list from a snapshot resourceVersion, then watch changes after it". This is the event backbone later controllers use for `Application`, `Node`, `ScalingPolicy`, `NetworkPolicy`, and every future resource kind.

## Why this step exists

Declarative controllers need a reliable change stream. Without it, every controller either polls aggressively or misses ordering details during restarts. `20.04` gives lists a snapshot high-water mark; this step makes that value actionable by emitting ordered, replayable events for create, update, status update, and delete transitions.

## Dependencies

* `20.02` - generic CRUD routes that create and mutate resources.
* `20.03` - generation/status rules that determine event payload fields.
* `20.04` - list `resourceVersion` used as the watch starting point.

## Scope

* `control.resource_events` table with monotonic `resource_version`, `event_type`, resource identity, generation, payload snapshot, timestamp, actor, request id.
* Events emitted inside the same database transaction as the resource mutation.
* `GET /v1/watch/{plural}?since={resourceVersion}` as Server-Sent Events.
* Replay from retained events, then live tail.
* `410 Gone` when `since` is older than the retained compaction window.
* Heartbeat comments so clients can detect idle-but-healthy streams.

## Explicitly out of scope

* A distributed event bus fanout to other services; that remains a later integration with Forge Events.
* Cross-cluster/global watch semantics.
* Per-field watches or JSONPath watches.

## Contracts

```http
GET /v1/watch/applications?since=1058
accept: text/event-stream
```

SSE frame:

```text
event: MODIFIED
id: 1061
data: {"type":"MODIFIED","resourceVersion":"1061","resource":{"apiVersion":"forge.dev/v1","kind":"Application","metadata":{"name":"invoice-api","generation":3},"spec":{},"status":{}}}
```

Event types:

* `ADDED`
* `MODIFIED`
* `STATUS_MODIFIED`
* `DELETED`

## Data model

```sql
CREATE TABLE control.resource_events (
  resource_version BIGINT PRIMARY KEY,
  event_id TEXT NOT NULL,
  event_type TEXT NOT NULL,
  kind TEXT NOT NULL,
  organization TEXT NOT NULL,
  project TEXT,
  environment TEXT,
  resource_id TEXT NOT NULL,
  resource_name TEXT NOT NULL,
  generation BIGINT NOT NULL,
  payload JSONB NOT NULL,
  actor TEXT,
  request_id TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX resource_events_kind_scope_idx
  ON control.resource_events(kind, organization, project, environment, resource_version);
```

## Observability

* Log each watch start/end with `{kind, since, replay_count, request_id}`.
* Metric `forge_resource_events_emitted_total{kind,type}`.
* Metric `forge_resource_watch_connections{kind}`.
* Trace span `resource.watch`.

## Failure handling

* Stale `since` below retained minimum -> `410 Gone` with `resource_version_too_old`.
* Client disconnect -> stream closes without affecting mutation flow.
* Event write failure -> whole resource mutation transaction rolls back.
