## Goal

Expose one generic set of HTTP routes — create, replace, patch (merge + JSON Patch), get, delete — that serve every kind currently in `KindRegistry`, dispatched dynamically by the `{plural}` path segment rather than one route per kind. Add `resourceVersion` optimistic concurrency (`409` on mismatch) and reuse epic 02's `Idempotency-Key` handling exactly as shipped.

## Why this step exists

`20.01` proved the storage shape works; this step proves the *generic* part of "generic resource API" — one route implementation that automatically serves any kind registered now or in the future, instead of a route per kind that every later epic would have to rewrite. Concurrency control has to land in the same step as writes: an API that lets two callers blindly overwrite each other's spec is not a declarative API, it is a race condition with JSON.

## Dependencies

* `20.01` — envelope types, `KindRegistry`, `ResourceRepository`, `control.resources`
* Epic `02-forge-control` — `ApiException`, `ErrorEnvelope`, `idempotentCreate`/`idempotentAction`, `IdempotencyStore`

## Scope

* Route dispatch by scope: `Cluster` → `/v1/{plural}`; `Project` → `/v1/projects/{project}/{plural}`; `Environment` → `/v1/projects/{project}/environments/{environment}/{plural}`
* `POST` create (idempotent via `Idempotency-Key`, reusing `call.idempotentCreate`)
* `PUT` full replace, requiring `metadata.resourceVersion` to match current
* `PATCH` merge-patch (`Content-Type: application/merge-patch+json`, RFC 7396) and JSON Patch (`application/json-patch+json`, RFC 6902); no `Content-Type` or `application/json` defaults to merge-patch
* `GET` single resource by name
* `DELETE` (unconditional hard behavior for this step only — finalizer-aware Terminating semantics arrive in `20.06`; this step's delete sets `deleted_at` immediately, full stop)
* `409` on `resourceVersion` mismatch, with the current `resourceVersion` echoed in `details` so the client can refetch-and-retry
* Migration widening `idempotency_keys.resource_id` from `UUID` to `TEXT` (new resource ids are ULID strings, not valid UUIDs)

## Explicitly out of scope

* `/status` subresource and generation-bump rules (`20.03`)
* Label selectors, field filters, pagination (`20.04`; `GET` list in this step is unfiltered, unpaginated, kind-scoped only)
* Watch (`20.05`)
* Finalizer-blocked delete, ownership cascade (`20.06`)
* Registering `Application`/`Service`/`Deployment` (`20.07`); this step's routes are exercised against the `20.01` test fixture kind and any kind future epics register — there is no product-facing kind live yet

## Architecture

```text
Request → /v1/[projects/{p}/[environments/{e}/]]{plural}[/{name}]
              │
              ▼
       KindRegistry.byPlural(plural)   404 kind_not_registered if absent
              │
              ▼
       ResourceRoutes (generic dispatch)
              │
              ▼
       ResourceRepository (20.01 interface; Jdbc- or legacy-backed per 20.07)
```

One Ktor route tree, not one route per kind — the kind is resolved from the registry at request time.

## Components

| Component | Responsibility |
|---|---|
| `ResourceRoutes.kt` | Generic CRUD dispatch keyed by `{plural}` + scope |
| `MergePatch.kt` | RFC 7396 merge-patch apply over `JsonObject` |
| `JsonPatch.kt` | RFC 6902 apply (`add`/`remove`/`replace`/`move`/`copy`/`test`) restricted to `spec`/`metadata.labels`/`metadata.annotations` |
| `ResourceVersionGuard.kt` | Compares supplied vs stored `resourceVersion`; raises typed conflict |
| `JdbcResourceRepository` (extended) | Adds `replace`, `patch`, `softDelete`, all `resourceVersion`-guarded |

## Security considerations

* `FORGE_AUTH_MODE=dev` continues platform-wide; no new bypass introduced
* `resourceVersion` guard prevents silent lost updates, which is a correctness property, not just a security one
* JSON Patch is restricted to `spec`/`metadata.labels`/`metadata.annotations` paths — `move`/`copy`/`test` against `metadata.id`, `generation`, or `resourceVersion` is rejected with `400 patch_path_forbidden`

## Observability

* Log per write: `{kind, name, project, environment, action, resource_version_old, resource_version_new}`
* Metric `forge_resources_total{kind=...}` (gauge) and `forge_resource_writes_total{kind=...,action=...}` (counter)
* Trace span `resource.write` with `kind`/`action` attributes

## Failure handling

* Unknown `{plural}` → `404 kind_not_registered`
* Stale `resourceVersion` on `PUT`/`PATCH` → `409 resource_version_conflict` (current version echoed)
* Reused `Idempotency-Key` with a different body hash → `409 idempotency_key_conflict` (unchanged from epic 02 behavior)
* Malformed JSON Patch (bad path, forbidden path) → `400 invalid_patch` / `400 patch_path_forbidden`

## Unit tests

* `MergePatch` removes a key on `null`, replaces scalars, deep-merges nested objects (RFC 7396 examples)
* `JsonPatch` applies `add`/`remove`/`replace`/`test`; rejects a path outside `spec`/`metadata.labels`/`metadata.annotations`
* `ResourceVersionGuard` passes on match, throws typed conflict on mismatch, and accepts create (no prior version)

## Integration tests

* `POST` creates a `Widget`, echoes generated `id`/`resourceVersion`
* `PUT` with the current `resourceVersion` succeeds and bumps it; with a stale one returns `409` with `currentResourceVersion` in `details`
* `PATCH` (merge) and `PATCH` (JSON Patch) both mutate `spec` correctly
* Repeated `POST` with the same `Idempotency-Key` and identical body returns the original response; same key with a different body returns `409 idempotency_key_conflict`
* `DELETE` removes the resource from subsequent `GET`s (soft: `deleted_at` set)

## Contract tests

* OpenAPI declares the generic paths for at least one scope shape and the `ResourceEnvelope`/error schemas; example payloads round-trip through `kotlinx.serialization`

## Manual verification

```bash
make dev
curl -s -X POST localhost:4001/v1/projects/demo/environments/dev/widgets \
  -H 'content-type: application/json' \
  -d '{"apiVersion":"forge.dev/v1","kind":"Widget","metadata":{"name":"sample"},"spec":{"size":"large"}}' \
  | tee /tmp/widget.json | jq '{id: .metadata.id, rv: .metadata.resourceVersion}'
RV=$(jq -r '.metadata.resourceVersion' /tmp/widget.json)
curl -s -o /dev/null -w '%{http_code}\n' -X PUT \
  localhost:4001/v1/projects/demo/environments/dev/widgets/sample \
  -d "{\"metadata\":{\"resourceVersion\":\"$((RV - 1))\"},\"spec\":{\"size\":\"small\"}}"
# expect 409
```
