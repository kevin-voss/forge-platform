## Goal

Bridge the existing epic 02-19 product APIs into the new declarative resource model without breaking current clients, then add `forge apply -f` for portable manifests. Existing endpoints remain supported; the new resource API becomes the future-facing path.

## Why this step exists

The current platform has concrete `Project`, `Environment`, `Application`, `Service`, and `Deployment` endpoints already being implemented. The future resource model must not interrupt that work. This step introduces a compatibility facade that projects the existing data into resource envelopes while keeping legacy tables and routes intact.

## Dependencies

* `20.01`-`20.06` - complete generic resource behavior.
* Epic `02-forge-control` - existing project/environment/application/service/deployment domain.
* Epic `03-forge-cli` - CLI command structure and config.

## Scope

* Register initial real kinds: `Organization`, `Project`, `Environment`, `Application`, `Service`, `Deployment`, `Revision`, `Route`, `Secret`, `Config`.
* Compatibility repository implementations that read/write existing tables for shipped kinds where applicable.
* Legacy endpoint parity tests: old routes continue to return the same shapes.
* `forge apply -f forge.yaml` accepting `Application` and related resources.
* Server-side dry run and diff summary.

## Explicitly out of scope

* Replacing legacy tables with `control.resources`.
* Removing old CLI commands.
* Full multi-document dependency ordering beyond parent-before-child by scope.

## Contracts

```bash
forge apply -f forge.yaml --target local
forge apply -f forge.yaml --dry-run
```


```json
{
  "dryRun": false,

  ]
}
```

## Architecture

```text
Generic Resource API
  -> KindRegistry
      -> native resource table repository for new kinds
      -> compatibility repository for shipped kinds
Existing /v1/projects, /v1/applications, /v1/deployments routes remain unchanged
```

## Observability

* Log apply operation with `{operation_id, resource_count, dry_run, changed_count}`.
* Metric `forge_apply_operations_total{dry_run,result}`.
* Audit entry per resource changed by apply.

## Failure handling

* A manifest containing provider-specific fields rejected with `portable_manifest_violation`.
* One invalid resource in a multi-resource apply rejects the whole operation before mutation.
* Stale resource versions return `409` with actionable CLI output.
