## Goal

Implement the `docker` provider: a `NodePool` with `providerRef` pointing at a `type: docker` `InfrastructureProvider` starts real `forge-runtime` containers, each of which registers as its own independent node with its own identity, address, and capacity — exactly as if it were a remote machine. This is the provider every demo and CI run uses by default, and the first adapter to make `23.01`'s inert `NodePoolController` actually converge.

## Why this step exists

Cloud providers (`23.05`, `23.06`) are opt-in and cost money; the platform's default and CI-gating path must be free, fast, and fully local while still exercising the real provider interface — not a special-cased mock. Building the Docker adapter first also proves the interface from `23.01` is sufficient before harder providers (SSH, Hetzner, AWS, Azure) implement it.

## Dependencies

* Step `23.01` — `Provider` interface, registry, operation ledger, `NodePoolController`
* Epic `04-forge-runtime` — the `forge-runtime` image and its self-registration/heartbeat (`04.02`), started here unmodified
* Epic `08-multi-node-scheduler` — node fleet capacity accounting (`08.02`) that these containers report into

## Scope

* `internal/provider/docker/` adapter implementing all 16 `Provider` methods against the Docker Engine API
* `CreateNode` starts a `forge-runtime` container: distinct name (`forge-node-<pool>-<suffix>`), a per-node named volume for Runtime's persisted node-id file (`04.02`), CPU/memory limits derived from `machineType`, attached to the existing `forge-platform` Compose network, labeled `forge.managed=true`, `forge.pool=<name>`, `forge.op_id=<op>`
* A static local machine-type table (`docker-small`, `docker-medium`, `docker-large`) mapping to CPU/memory/slot capacity, analogous to `FORGE_NODE_SLOTS` from epic 08
* `DeleteNode`: graceful stop (`SIGTERM` + grace period) then container + volume removal
* `RebootNode`: `docker restart`
* `GetNode`/`ListNodes`: `docker inspect`/`docker ps` filtered by the `forge.managed`/`forge.pool` labels
* `CreateNetwork`/`DeleteNetwork`: optional per-pool Docker bridge network (default: reuse the existing Compose network)
* `AttachDisk`/`DetachDisk`/`ResizeDisk`: Docker named volumes (create/remove; resize is a no-op with a documented size annotation, since Docker does not enforce volume quotas)
* `CreatePublicIP`/`DeletePublicIP`: no-op, returns the host bind address (`FORGE_INFRA_DOCKER_HOST_ADDRESS`) with a documented "no public IP concept locally" note
* `GetPricing`: always `{hourly: 0, currency: "USD"}`
* Orphan cleanup: a periodic pass lists containers labeled `forge.managed=true` and deletes any with no corresponding `Node` resource (crash between `CreateNode` succeeding and the `Node` resource being written)

## Explicitly out of scope

* Bootstrap token exchange / mTLS join (`23.03` — this step's `CreateNode` still uses `FORGE_AUTH_MODE=dev` registration, matching epic 04/08's current dev-mode default)
* SSH/bare-metal, Hetzner, AWS, Azure adapters (`23.04`–`23.06`)
* The full node-loss demo (`23.07`) — this step's tests exercise the adapter directly, not the end-to-end scenario

## Architecture

```text
services/forge-infrastructure/internal/provider/docker/
├── docker.go            Provider implementation (Docker Engine client over the mounted socket)
├── machinetypes.go       docker-small|medium|large -> {cpu, memoryMiB, slots}
├── labels.go             forge.managed, forge.pool, forge.op_id, forge.nodepool constants
└── reconcile_orphans.go  list managed containers -> diff against known Nodes -> delete strays

CreateNode(op, req):
  name := "forge-node-" + req.NodePool + "-" + shortID()
  vol  := docker volume create  name + "-data"
  container := docker create --name name \
      --network forge-platform_default \
      --cpus <machineType.cpu> --memory <machineType.memoryMiB>Mi \
      -v <vol>:/var/lib/forge-runtime \
      -l forge.managed=true -l forge.pool=<req.NodePool> -l forge.op_id=<op> \
      -e FORGE_CONTROL_URL=http://forge-control:8080 \
      -e FORGE_RUNTIME_DATA_DIR=/var/lib/forge-runtime \
      forge/forge-runtime:local
  docker start container
  return ProviderNode{ id: "docker:"+container.ID, address: container.NetworkSettings.IPAddress }
```

## Data model

No new tables; this step consumes `23.01`'s `provider_operations` ledger. The static machine-type table is compiled-in, not persisted:

```text
docker-small:  { cpu: "1",   memoryMiB: 1024, slots: 2 }
docker-medium: { cpu: "2",   memoryMiB: 2048, slots: 4 }
docker-large:  { cpu: "4",   memoryMiB: 4096, slots: 8 }
```


## Security considerations

* Mounting the Docker socket into `forge-infrastructure` is a privileged local-dev convenience (same documented tradeoff as Runtime's own socket mount in `04.01`); flagged for hardening (rootless Docker / a proxy socket) alongside that existing flag, not newly introduced here
* Node containers get no credentials at all (`docker` provider needs none); this keeps the "credentials never in the manifest" rule trivially true for the default/CI path

## Observability

* Log per container lifecycle event: `{container_id, node_pool, action, op_id}`
* Metric: `forge_infra_docker_containers_total{pool,state}`
* Metric: `forge_infra_docker_orphans_removed_total`
* Trace span `infra.provider.docker.create_node`

## Failure handling

* `docker create`/`docker start` failure → ledger row marked `failed` with the Docker daemon's error message; `NodePoolController` retries next interval (no partial container left running — a failed `create` is followed by a best-effort `docker rm` of any half-created container)
* Container exits unexpectedly (crash, OOM) → observed as a missed heartbeat via the existing `08.05` path, not by this adapter; the adapter's `GetNode`/`ListNodes` simply reports the container as stopped
* Orphan scan finds a managed container with no `Node` resource → deleted, logged with `reason=orphan_no_resource`, counted in the orphan metric

## Acceptance criteria

* A `docker`-type `NodePool` converges its running container count to `spec.replicas`, using the real `Provider` interface (no test-only shortcut)

* The orphan-reconciliation pass removes managed containers that have no corresponding `Node` resource
