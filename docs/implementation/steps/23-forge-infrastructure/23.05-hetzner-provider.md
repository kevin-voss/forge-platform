## Goal

Implement the first real, billable cloud provider: Hetzner Cloud. Servers, Private Networks, Volumes, and Floating/primary IPs map directly onto the `Provider` interface using only Hetzner's IaaS primitives â€” no managed services. This is also the first adapter where rate limits, provider-side eventual consistency, and cost-safety reconciliation are real constraints instead of theoretical ones.

## Why this step exists

The portability contract (epic goal) is only proven once at least one real cloud target works end-to-end, not just Docker. Hetzner is chosen first because its API is small, its primitives map almost one-to-one onto the `Provider` interface, and it has none of AWS/Azure's managed-service gravity pulling toward a non-portable design â€” a clean second data point before the more complex AWS/Azure step (`23.06`).

## Scope

* `internal/provider/hetzner/` adapter implementing all 16 `Provider` methods against `api.hetzner.cloud/v1`
* `CreateNode` â†’ `POST /servers` (server type = `machineType`, image = a Forge-maintained snapshot or stock image + `23.03` cloud-init `user_data`, `location` = region, `labels: {forge.managed: "true", forge.op_id: <op>, forge.nodepool: <name>}`)
* Since Hetzner's server-create API has no native idempotency key, retries first check `GET /servers?label_selector=forge.op_id==<op>` and adopt an existing match instead of creating a second server
* Networks â†’ Hetzner Private Networks (one per `NodePool` or shared per `InfrastructureProvider`, per config)
* Volumes â†’ Hetzner Volumes for `AttachDisk`/`DetachDisk`/`ResizeDisk`
* Public IP â†’ primary IP is returned automatically from `GetNode`; `CreatePublicIP`/`DeletePublicIP` manage a Floating IP for a stable address across node replacement
* Regions/machine types/pricing â†’ `GET /locations`, `GET /server_types` (server types embed hourly/monthly prices)
* Rate-limit handling: token-bucket limiter driven by the `RateLimit-Remaining`/`RateLimit-Reset` response headers, exponential backoff with jitter on `429`, and a bounded concurrent-in-flight cap so a NodePool scaling from 2â†’10 doesn't burst every request at once
* Teardown ordering: `DeleteNode` detaches/deletes exclusively-owned volumes and releases any assigned Floating IP before deleting the server; deleting the last node in a pool that owns a private network deletes the network last
* Label-based orphan reconciliation: periodically lists all `forge.managed=true` resources for this provider and deletes any with no matching `Node`/`NodePool` after a grace period

## Explicitly out of scope

* AWS/Azure adapters (`23.06`)
* Any Hetzner managed service (managed Kubernetes, managed load balancer as a required dependency â€” Hetzner Load Balancers are not used by this epic; Gateway remains Forge's own routing layer)
* The optional cloud-target demo script itself (`23.07` â€” this step only makes the adapter correct and tested)

```text
CreateNode(op, req):
  existing := GET /servers?label_selector=forge.op_id==<op>
  if existing: return existing            # idempotent retry, no duplicate server
  resp := POST /servers {
    server_type: req.MachineType, location: req.Region,
    image: forgeImageFor(req.Region), user_data: cloudInitFor(req, bootstrapToken),
    labels: { forge.managed: "true", forge.op_id: op, forge.nodepool: req.NodePool }
  }
  return ProviderNode{ id: "hetzner:"+resp.Server.ID, address: resp.Server.PublicNet.IPv4.IP }
```

## Security considerations

* The Hetzner API token is project-wide by design (Hetzner has no finer-grained scoping); the operational convention documented here is one dedicated Hetzner project per Forge organization so a leaked token's blast radius is bounded to that organization's infrastructure, not the whole platform
* Token is read from Forge Secrets per call (never cached to disk); `provider_operations.request`/`result` JSON has the token field scrubbed before persistence (same denylist mechanism as `23.01`)

## Failure handling

* `429 Too Many Requests` â†’ exponential backoff with jitter, respecting `RateLimit-Reset`; the call is retried by the same ledger-guarded operation, never re-issued as a new one
* Server creation partially succeeds (server created, cloud-init/join never completes) â†’ handled by `23.03`'s bootstrap timeout, which deletes the server; Hetzner-specific teardown ordering (volumes â†’ IP â†’ server) still applies
* Orphan found within the grace period â†’ left alone (a slow-to-register node isn't a false orphan); only past the grace period is it deleted, with an audit log entry naming the resource id and grace period

## Unit tests

* Idempotent-retry check: given a fake Hetzner API returning an existing server for a label selector, `CreateNode` adopts it and issues no second `POST /servers`
* `ratelimit.Limiter` backs off with increasing delay across repeated `429` responses and respects `RateLimit-Reset` when present
* `teardown.Delete` calls volume-detach, then floating-IP-release, then server-delete, then (only when it's the last node) network-delete, in that exact order (verified via a call-order-recording fake client)

## Integration tests

* Against a Hetzner sandbox/test account (or a recorded-fixture HTTP replay in CI): `CreateNode` produces a running server reachable at its public IP; `DeleteNode` removes server, volume, and floating IP with no leaked resource in `GET /servers|volumes|floating_ips`
* Two concurrent `CreateNode` calls for the same natural key (simulated controller restart mid-call) result in exactly one server, verified via the label selector
* `reconcile_orphans` deletes a manually-created (out-of-band) server labeled `forge.managed=true` with no matching `Node` once its grace period elapses, and leaves alone one still within the grace period
