## Goal

Add two adapters that share one idea — the machine already exists; Forge doesn't create or destroy it, it only claims and releases it: a generic `ssh` provider that turns any reachable host into a Forge node, and a `bare-metal` provider backed by a static, operator-declared inventory. `CreateNode`/`DeleteNode` become **adopt**/**release**, not provision/destroy, and capacity is whatever the operator listed — never fabricated.

## Why this step exists

Not every Forge deployment target is elastic compute. Racks, colo machines, and pre-existing VMs the operator already controls need a path into the same `NodePool`/`Node` model without pretending Forge can create more of them on demand. This step is also where "finite capacity" first becomes a real, tested condition — the Node Autoscaler (epic 24) must be able to ask for more nodes than exist here and get a clean, actionable answer instead of an infinite retry loop.

## Dependencies

* Steps `23.01`–`23.03` — `Provider` interface, ledger, bootstrap/join/drain phase machine (the SSH bootstrap path defined in `23.03` is used directly here)
* Epic `10-forge-secrets` — SSH private keys resolved via `credentialsSecretRef`, never inline

## Scope

* `internal/provider/ssh/` adapter: `InfrastructureProvider.spec.config.inventory` lists candidate hosts (`{address, sshUser, sshKeySecretRef}`); `CreateNode` picks the next unclaimed host, runs the `23.03` SSH bootstrap path, marks it claimed; `DeleteNode` drains, runs an uninstall/stop script over SSH, marks the host free again (the host itself is untouched)
* `internal/provider/baremetal/` adapter: same adopt/release semantics, but the inventory is the entirety of what the provider knows (no elastic fallback, no "just SSH anywhere") — modeling racked machines the operator has already imaged
* `ValidateCredentials` for both = an SSH connectivity check across the full inventory
* `ListRegions`/`ListMachineTypes` return static, fixed entries reflecting fixed inventory (`region: "on-prem"`; machine "types" derived from `nproc`/`/proc/meminfo` read once per host over SSH)
* `CreateNetwork`/`DeleteNetwork`/`AttachDisk`/`DetachDisk`/`ResizeDisk`/`CreatePublicIP`/`DeletePublicIP` return a typed `ErrNotSupported` (not fatal — see Failure handling) since neither provider provisions networks, disks, or IPs
* `NodePool.status` reports a capacity ceiling (`status.maxReplicas = len(inventory)`) and a `False/InventoryExhausted` condition (not an error, not a crash) when `spec.replicas` exceeds it

## Explicitly out of scope

* Hetzner/AWS/Azure adapters (`23.05`/`23.06`)
* Any mechanism to auto-discover new hosts to add to the inventory (the inventory is operator-declared, full stop)
* Rack-level power/PXE management — bare metal here means "already running an OS and reachable," not "bare"

## Architecture

```text
InfrastructureProvider(type: ssh | bare-metal).spec.config.inventory:
  - { address: 10.0.4.11, sshUser: forge, sshKeySecretRef: { name: rack1-ssh-key } }
  - { address: 10.0.4.12, sshUser: forge, sshKeySecretRef: { name: rack1-ssh-key } }
  - { address: 10.0.4.13, sshUser: forge, sshKeySecretRef: { name: rack1-ssh-key } }

CreateNode(op, req):
  host := next inventory entry where claimed == false
  if none: return ErrInventoryExhausted   # NodePool controller turns this into a status condition, not a crash
  claim(host, req.nodePoolRef)
  run 23.03 SSH bootstrap against host
  return ProviderNode{ id: "ssh:"+host.address, address: host.address, capacity: probed via SSH }

DeleteNode(op, nodeID):
  host := inventory entry for nodeID
  drain (23.03) then run uninstall script over SSH
  release(host)   # host stays up; just returned to the unclaimed pool
```

## Components

| Component | Responsibility |
|---|---|
| `ssh.Provider` / `baremetal.Provider` | Adopt/release semantics over a declared inventory |
| `inventory.Store` | Per-`InfrastructureProvider` claimed/unclaimed host bookkeeping |
| SSH connectivity prober | `ValidateCredentials` + capacity probing (`nproc`, `/proc/meminfo`) |
| `ErrNotSupported` convention | Shared typed error every non-applicable mutating method returns |

## Contracts

`InfrastructureProvider` config for the SSH/bare-metal providers:

```yaml
apiVersion: forge.dev/v1
kind: InfrastructureProvider
metadata: { name: rack1-baremetal }
spec:
  type: bare-metal
  config:
    inventory:
      - { address: 10.0.4.11, sshUser: forge, sshKeySecretRef: { name: rack1-ssh-key } }
      - { address: 10.0.4.12, sshUser: forge, sshKeySecretRef: { name: rack1-ssh-key } }
```

`NodePool.status` when capacity is exceeded:

```json
{
  "phase": "Degraded",
  "readyNodes": 2,
  "maxReplicas": 2,
  "conditions": [
    { "type": "Available", "status": "False", "reason": "InventoryExhausted",
      "message": "requested replicas=5 exceeds inventory size=2" }
  ]
}
```

```sql

  PRIMARY KEY (provider_name, address)
```


## Security considerations

* SSH keys are resolved from Forge Secrets per connection and never written to disk unencrypted longer than the single bootstrap/release session requires
* Inventory entries never carry a key inline — only a `sshKeySecretRef`, consistent with the epic-wide "credentials never in a manifest" rule
* A host address appearing in two `InfrastructureProvider` inventories simultaneously is rejected at admission (prevents double-claiming the same physical machine from two pools)

## Observability

* Log per claim/release: `{provider_name, address, node_pool, action}`
* Metric: `forge_infra_inventory_capacity{provider_name}` (gauge, total inventory size)
* Metric: `forge_infra_inventory_claimed{provider_name}` (gauge, currently claimed)
* Trace span `infra.provider.ssh.create_node` / `infra.provider.baremetal.create_node`

## Failure handling

* Inventory exhausted → `ErrInventoryExhausted`, surfaced as `NodePool.status` condition `False/InventoryExhausted`; `NodePoolController` stops retrying `CreateNode` for this pool until `spec.replicas` drops or the inventory grows, and does not busy-loop
* `ErrNotSupported` from an unsupported mutating method (network/disk/IP) → treated by callers as "skip this capability," not a failure; `NodePool` reconciliation for node count is unaffected
* SSH connection failure during `ValidateCredentials` sweep → the affected host is marked unreachable (excluded from future claims) without failing the whole provider; `InfrastructureProvider.status` reflects partial availability (`N/M hosts reachable`)

## Unit tests

* `inventory.Store` claim/release is exclusive: two concurrent claims for the last free host result in exactly one winner
* `ErrNotSupported` is returned (not panicked, not treated as success) from every unsupported mutating method on both adapters
* `NodePoolController` translates `ErrInventoryExhausted` into the documented `False/InventoryExhausted` condition and stops retrying that pool until replicas or inventory change

## Integration tests

```bash
curl -s http://127.0.0.1:4001/v1/nodepools/rack1-pool | python3 -c \

```

## Acceptance criteria

* `ssh` and `bare-metal` providers adopt/release existing hosts rather than creating/destroying machines
* Requesting more replicas than inventory size produces a stable `InventoryExhausted` condition, never a crash loop or fabricated capacity
* Every mutating method the two adapters don't support returns a typed `ErrNotSupported` that the `NodePoolController` treats as "skip," not "fail"
* SSH keys are always resolved via `sshKeySecretRef`; no inventory entry ever carries a raw key
* The same host address cannot be claimed by two `InfrastructureProvider` inventories at once
