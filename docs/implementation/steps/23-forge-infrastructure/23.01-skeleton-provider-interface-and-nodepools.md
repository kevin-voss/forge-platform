## Goal

Stand up `services/forge-infrastructure` as a Go service on host port `4111`, define the provider adapter interface every cloud/local backend will implement, and register the `NodePool`, `Node`, and `InfrastructureProvider` resource kinds against the epic-20 declarative resource API. Build the operation ledger that every later mutating provider call is required to use, and an inert `NodePoolController` that diffs desired vs. actual node count and calls the (not-yet-real) provider — so `23.02`'s Docker adapter is the first thing that makes it actually do something.

## Why this step exists

Every later step in this epic (Docker, SSH/bare-metal, Hetzner, AWS, Azure) implements the same interface. Defining that seam, the resource schemas, and the idempotency ledger first — before any real provider exists — keeps the contract honest and prevents each provider from inventing its own retry/idempotency story. This mirrors how `08.01` built the scheduler seam before `08.03` added a real strategy.

* Epic `20-declarative-resource-api` — generic resource envelope/registry (`/v1/{kind-plural}`, watch, status subresource) that stores these three kinds

## Scope

* New Go module `services/forge-infrastructure` (`forge.local/services/forge-infrastructure`), health endpoints, Compose wiring on host port `4111`
* `Provider` interface with exactly 16 methods: `ValidateCredentials`, `ListRegions`, `ListMachineTypes`, `CreateNetwork`, `DeleteNetwork`, `CreateNode`, `DeleteNode`, `RebootNode`, `GetNode`, `ListNodes`, `AttachDisk`, `DetachDisk`, `ResizeDisk`, `CreatePublicIP`, `DeletePublicIP`, `GetPricing`
* Provider registry: a `map[string]ProviderFactory` keyed by `InfrastructureProvider.spec.type` (`docker`, `ssh`, `bare-metal`, `hetzner`, `aws`, `azure`); default/no-op entry returns `ErrProviderNotConfigured`
* `NodePool`, `Node`, `InfrastructureProvider` resource schemas registered with the epic-20 registry (cluster-scoped, drop the project/environment prefix)
* `provider_operations` ledger: every mutating call is preceded by a persisted `pending` row keyed by an operation id (`op_<ULID>`) and a caller-supplied natural key, so a crash-and-retry finds the same row instead of calling the provider twice
* `NodePoolController`: watches `NodePool`, counts `Node`s it owns, diffs against `spec.replicas`, calls `Provider.CreateNode`/`DeleteNode` through the ledger — wired end-to-end but inert until `23.02` supplies a real provider
* `contracts/openapi/forge-infrastructure.openapi.yaml` — this service's own (small) debug/health API

## Explicitly out of scope

* Any real provider implementation (Docker `23.02`, SSH/bare-metal `23.04`, Hetzner `23.05`, AWS/Azure `23.06`)
* Bootstrap/join/drain phase machinery (`23.03`)

## Data model

Only the operation ledger is private to this service; `NodePool`/`Node`/`InfrastructureProvider` storage lives in epic 20's registry.

```sql
CREATE TABLE provider_operations (
  id             TEXT PRIMARY KEY,             -- op_<ULID>
  provider_name  TEXT NOT NULL,                -- InfrastructureProvider.metadata.name
  kind           TEXT NOT NULL,                -- create_node | delete_node | reboot_node | create_network |
                                                -- delete_network | attach_disk | detach_disk | resize_disk |
                                                -- create_public_ip | delete_public_ip
  target_kind    TEXT NOT NULL,                -- node_pool | node | network | disk | public_ip
  target_id      TEXT,                         -- resource id once known; null while pending
  natural_key    TEXT NOT NULL,                -- caller idempotency key, e.g. "local-docker-pool#3"
  request        JSONB NOT NULL,
  status         TEXT NOT NULL DEFAULT 'pending',   -- pending | succeeded | failed
  result         JSONB,
  error          TEXT,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  completed_at   TIMESTAMPTZ,
  UNIQUE (provider_name, natural_key)
);
CREATE INDEX provider_operations_status_idx ON provider_operations (status);
```

## Security considerations

* `InfrastructureProvider.spec` never carries a raw credential — only a `credentialsSecretRef`; the resource is rejected at admission if any field matching a credential shape (`token`, `secret`, `key`, `password`) appears inline
* The provider registry loads credentials on demand from Forge Secrets per call and never logs them; `provider_operations.request`/`result` JSON is scrubbed of any field named in a per-provider secret-field denylist before it is persisted
* `/v1/operations/{opId}` is an internal debug endpoint (trust boundary: platform operators only), not exposed through Gateway

## Observability

* Log per reconcile: `{nodepool, desired_replicas, ready_nodes, action, op_id}`
* Metric: `forge_infra_provider_operations_total{provider,kind,status}`
* Metric: `forge_infra_nodepool_ready_nodes{nodepool}` (gauge)
* Trace span `infra.nodepool.reconcile`, child span `infra.provider.<method>`

## Failure handling

* No provider registered for `spec.type` → `NodePool.status` condition `False, reason=ProviderNotConfigured`; controller retries on next interval, never crash-loops
* Ledger `Begin` finds an existing `pending` row for the same natural key → the reconcile skips calling the provider again this pass (avoids parallel double-create if two controller instances briefly overlap)
* Ledger `Begin` finds an existing `succeeded` row for the same natural key → returns the persisted result directly; provider is not called
* Provider call panics → recovered at the controller boundary, ledger row marked `failed` with the panic message, reconcile continues with the next pool

## Unit tests

* `noop` provider returns `ErrProviderNotConfigured` from every mutating method, and empty (not error) from every read method
* `operations.Ledger.Begin` is idempotent for the same `(provider_name, natural_key)`: second call returns the first row, does not insert a duplicate
* `NodePoolController.reconcile` with `replicas=3, ready=1` issues exactly one `CreateNode` call (single-slot-per-pass) and records one ledger row
* `NodePoolController.reconcile` with `replicas=1, ready=3` issues exactly one `DeleteNode` call targeting the most recently created node
