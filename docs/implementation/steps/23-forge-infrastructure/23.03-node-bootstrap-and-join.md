## Goal

Give every provider a common node lifecycle: `Provisioning → Bootstrapping → Joining → Ready → Draining → Deleting`. Bootstrap paths (cloud-init/user-data for VM providers, SSH for reachable hosts) install Forge Runtime, exchange a one-time bootstrap token from epic 22 for the node's real join credentials, and only then let the node register with Control through the existing `04.02`/`08.02` path. A node that never reaches `Ready` within a timeout is deleted automatically so a broken bootstrap script never leaves a billable machine running; deletion always drains first.

## Why this step exists

`23.02`'s Docker adapter starts a container that is instantly on the same trusted network as Control — real machines are not. VM and SSH-reachable nodes need an explicit install step, a way to prove they're allowed to join (epic 22's bootstrap token), and a bounded state machine so "half-provisioned" never means "invisible and still costing money." This step defines that machinery once, generically, so every provider from `23.04` onward plugs into the same phases instead of reinventing them.

## Scope

* `Node.status.phase` state machine and the controller that drives transitions: `Provisioning → Bootstrapping → Joining → Ready → Draining → Deleting`
* Bootstrap payload templating: cloud-init/user-data for VM-backed providers; a scripted SSH session for reachable-host providers (`23.04`)
* One bootstrap-token request per `CreateNode`, scoped to the target `NodePool`, single-use, requested from epic 22 and embedded in the bootstrap payload
* `FORGE_NODE_BOOTSTRAP_TIMEOUT_SECONDS`: a node stuck before `Ready` past this deadline is marked `Failed` (reason `BootstrapTimeout`) and deleted automatically
* Drain-before-delete: `Draining` signals the epic-07/08 reconciler to stop placing new replicas on the node and reschedule existing ones off it; `Deleting` (and the real `Provider.DeleteNode` call) only happens once the node is empty or a drain timeout elapses
* A private `node_bootstrap_timers` table (this service's own; not the epic-20 registry) tracking each node's deadline

## Explicitly out of scope

* The bootstrap token issuance API itself (owned by epic 22; this step is a consumer)
* SSH/bare-metal provider specifics (`23.04`) and cloud provider specifics (`23.05`/`23.06`) — this step defines the phase machine and payload shape they all use
* Changing `04.02`'s registration/heartbeat contract or `08.05`'s reschedule mechanism — both are reused unmodified

## Architecture

```text
Node.status.phase:

  Provisioning ---> Bootstrapping ---> Joining ---> Ready
       |                  |                |           |
       | (provider        | (install       | (token     | (heartbeat
       |  create fails     |  script        |  exchange   |  + capacity
       |  or times out)     |  fails/        |  fails/     |  observed)
       v                  |  times out)     |  times out) |
     Failed <-------------+-----------------+             |
       |                                                    v
       +------------------------- DeleteNode <---------- Draining
                                     ^                       |
                                     +----- drain complete ---+
                                                (or drain timeout)
                                                        |
                                                        v
                                                    Deleting -> (resource removed)
```

```text
node_controller.reconcile(node):
  switch node.status.phase:
    Provisioning:
      if provider reports the machine exists and is booted -> phase = Bootstrapping
      if deadline exceeded -> phase = Failed(reason=ProvisionTimeout); trigger delete
    Bootstrapping:
      if install script signals done (health probe reachable) -> phase = Joining
      if deadline exceeded -> phase = Failed(reason=BootstrapTimeout); trigger delete
    Joining:
      if 04.02 registration + first heartbeat observed -> phase = Ready
      if deadline exceeded -> phase = Failed(reason=JoinTimeout); trigger delete
    Ready:
      (steady state; scheduler/reconciler treat it as a normal node per 08.02)
    Draining:
      if workload count on node == 0 -> phase = Deleting
      if drain deadline exceeded -> phase = Deleting anyway (logged, not silent)
    Deleting:
      call Provider.DeleteNode through the ledger; on success, delete the Node resource
```

## Contracts

Bootstrap payload template (rendered once per `CreateNode`, provider-agnostic core):

```yaml
forge_bootstrap:
  control_url: http://forge-control:8080
  bootstrap_token: "<single-use token from epic 22, scoped to NodePool>"
  node_pool: local-hetzner-pool
  runtime_image: registry.forge.internal/forge/forge-runtime:1.4.0
```

VM cloud-init rendering (used by `23.05`/`23.06`):

```yaml
#cloud-config
write_files:
  - path: /etc/forge/runtime.env
    content: |
      FORGE_CONTROL_URL=http://forge-control:8080
      FORGE_BOOTSTRAP_TOKEN=<token>
runcmd:
  - curl -fsSL https://get.docker.com | sh
  - docker run -d --name forge-runtime --env-file /etc/forge/runtime.env \
      registry.forge.internal/forge/forge-runtime:1.4.0
```

SSH bootstrap (used by `23.04`): Infrastructure opens an SSH session with a key resolved from Forge Secrets, copies an install script + the rendered env file, executes it, then polls `GET http://<node-address>:4102/health/ready` until it answers or the `Bootstrapping` deadline passes.

## Security considerations

* Bootstrap tokens are single-use and scoped to one `NodePool`; a captured token cannot join a different pool or be replayed after first use (enforced by epic 22, consumed here)
* Rendered bootstrap payloads are never logged in full (the token field is masked before any log line, matching the epic-10 masking convention)
* SSH bootstrap always uses a key from Forge Secrets, never a key embedded in a `NodePool`/`InfrastructureProvider` spec

## Observability

* Log per transition: `{node_id, node_pool, from_phase, to_phase, reason}`
* Metric: `forge_infra_node_phase_seconds{phase}` (histogram, time spent per phase)
* Metric: `forge_infra_node_bootstrap_timeouts_total{phase}`
* Trace span `infra.node.bootstrap`, child spans per phase

## Failure handling

* Any phase timeout → `Failed` condition with a specific `reason` (`ProvisionTimeout`/`BootstrapTimeout`/`JoinTimeout`), then an automatic `Draining`→`Deleting` transition so the machine is never left running just because it never became useful
* `DeleteNode` itself fails (provider error) during cleanup-after-timeout → retried with backoff; the `Node` resource stays in `Deleting` (visible, not silently dropped) until the provider confirms deletion
* Drain deadline exceeded with workloads still present → node is deleted anyway (logged at `warn` with the stranded workload IDs) rather than blocking cleanup forever; the reconciler (epic 07) treats the abrupt loss the same as any node failure

## Acceptance criteria

* Every `Node` moves through `Provisioning → Bootstrapping → Joining → Ready` (or into `Failed` with a specific timeout reason) with no phase skipped and no silent hang
* A node that never reaches `Ready` within its configured timeout is deleted automatically; no orphaned billable machine remains
* Deletion always passes through `Draining` first; workloads are rescheduled off the node before the provider's `DeleteNode` is called, except when the drain timeout itself is exceeded (logged, not silent)
* Bootstrap tokens are requested per-node, single-use, and never appear unmasked in logs
