## Goal

Serve an authoritative DNS zone for `<service>.<environment>.<project>.svc.forge` directly out of Discovery's own store: `A`/`AAAA` records for every `Ready` endpoint's address, `SRV` records for named ports, TTLs short enough to track lease state, and correct negative-caching behavior when a service has no ready endpoints. Make the zone reachable from workload containers on local Docker without breaking normal name resolution, and describe the same wiring for bare metal/Hetzner/AWS/Azure node fleets.

## Why this step exists

The HTTP/SSE API (`21.03`) is enough for platform services and Go clients, but plain workload code (a product container doing a DNS lookup, an off-the-shelf library, a `curl` in a shell script) expects to resolve a name, not call an API. DNS is the lowest-common-denominator discovery interface every runtime already speaks, and it is the mechanism the internal naming convention (`<service>.<environment>.<project>.svc.forge`) was defined for.

## Dependencies

* `21.02`/`21.03` ÔÇö `Ready` endpoints and their addresses/ports to answer from
* Epic `00-repository-foundation` ÔÇö Docker's embedded DNS behavior this step must coexist with locally

## Scope

* UDP DNS server on `5053/udp`, authoritative for `svc.forge`
* `A`/`AAAA` records: one per `Ready` endpoint's address for `<service>.<environment>.<project>.svc.forge`
* `SRV` records: `_<port-name>._<protocol>.<service>.<environment>.<project>.svc.forge` Ôćĺ `priority weight port target`
* `CNAME`-equivalent resolution for `Service.spec.aliases` (an alias resolves to the same address set as the canonical name)
* TTL policy: answer TTL = `min(FORGE_DISCOVERY_DNS_TTL_SECONDS, remaining lease time)` so answers naturally go stale close to lease expiry
* Negative caching: `NXDOMAIN`/empty-answer responses use a short `FORGE_DISCOVERY_DNS_NEGATIVE_TTL_SECONDS` so a starting replica is picked up quickly
* Forwarding: anything outside `svc.forge` forwards to `FORGE_DISCOVERY_DNS_FORWARD_UPSTREAM` (Docker's embedded DNS, `127.0.0.11`, locally)
* Documented local wiring: how workload containers get this resolver without losing normal DNS
* Documented multi-node wiring: how a bare-metal/Hetzner/AWS/Azure node points its resolver at Discovery over the private network

## Explicitly out of scope

* DNSSEC / zone signing
* Public/external DNS delegation or TLS certificate issuance (epic 34)
* Split-horizon policy engines beyond "svc.forge here, everything else forwarded"
* Running more than one Discovery resolver instance / HA (epic 35 territory, noted as an epic-level open question)

## Components

| Component | Responsibility |
|---|---|
| `dns.Server` | UDP listener on `5053`, request parsing/response encoding (e.g. `miekg/dns`) |
| `dns.ZoneResolver` | Answers `svc.forge` queries from `discovery.endpoints`/`discovery.services` |
| `dns.Forwarder` | Passes non-`svc.forge` queries to the configured upstream |
| `dns.TTLPolicy` | Computes per-answer TTL from remaining lease time; negative TTL for empty results |

## Contracts

Example lookups (no HTTP contract; DNS wire protocol):

```bash

# ANSWER: 172.20.0.14  TTL <= FORGE_DISCOVERY_DNS_TTL_SECONDS
#         172.20.0.19  TTL <= FORGE_DISCOVERY_DNS_TTL_SECONDS

dig @127.0.0.1 -p 5053 _http._tcp.invoice-api.production.invoice-platform.svc.forge SRV
# ANSWER: 10 10 8080 invoice-api-7f3a2b1c-0.invoice-api.production.invoice-platform.svc.forge.

dig @127.0.0.1 -p 5053 legacy-invoice.production.invoice-platform.svc.forge A
# resolves identically to invoice-api.production.invoice-platform.svc.forge when
# "legacy-invoice" is declared in Service.spec.aliases

dig @127.0.0.1 -p 5053 nonexistent-service.production.invoice-platform.svc.forge A
# NXDOMAIN, TTL = FORGE_DISCOVERY_DNS_NEGATIVE_TTL_SECONDS

dig @127.0.0.1 -p 5053 google.com A
# forwarded to 127.0.0.11, answered normally
```

## Data model

No new tables; the resolver reads `discovery.endpoints` (`phase='Ready'`) and `discovery.services` (`aliases`) already defined in `21.01`/`21.02`. TTL is computed at answer time from `expires_at - now()`, not stored separately.

## Security considerations

* The resolver only answers authoritatively for `svc.forge`; it never claims authority over arbitrary forwarded names
* No recursive resolution is performed for `svc.forge` itself ÔÇö only exact zone data from Discovery's own store
* UDP port `5053` is internal-network-only (not published beyond the platform's private network in bare-metal/cloud deployments); local Docker exposes it only on the Compose network unless a human opts into a host port for `dig` debugging (documented, not required)

## Observability

* Log per query: `{event:"discovery.dns.query", qname, qtype, answer_count, zone_hit: true|false}` (sampled to avoid log volume at scale)
* Metrics: `forge_discovery_dns_queries_total{zone_hit=true|false}`, `forge_discovery_dns_nxdomain_total`, `forge_discovery_dns_forward_errors_total`
* Trace span `discovery.dns.resolve` (sampled)

## Failure handling

* Forward upstream unreachable Ôćĺ `SERVFAIL` for out-of-zone queries only; in-zone (`svc.forge`) answers are unaffected since they never touch the forwarder
* No `Ready` endpoints for a queried service Ôćĺ `NXDOMAIN` with negative TTL, not `SERVFAIL` ÔÇö a service scaling through zero is a normal, not exceptional, condition
* Malformed query Ôćĺ standard DNS `FORMERR`, never a crash
