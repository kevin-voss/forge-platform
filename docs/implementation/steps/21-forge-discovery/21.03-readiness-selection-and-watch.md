## Goal

Give products and platform services a read path that only ever returns `Ready` endpoints, selectable by project/environment/service and an optional `revision` label, plus a change feed: a synchronous `GET .../endpoints` list and an SSE watch consistent with epic 20's watch conventions. Ship a small client library so products can call Discovery directly instead of reimplementing polling/caching.

## Why this step exists

`21.02` makes endpoints exist and expire correctly, but nothing yet exposes them the way a consumer needs: filtered to `Ready`, scoped to a service, optionally pinned to a revision, and either point-in-time or streamed. This is the step that makes Discovery usable by anything other than its own internals — Gateway (`21.05`) and future product code both build on this contract.

## Dependencies

* `21.02` — endpoints exist with a `phase` and `revision` to filter/select on
* Epic `20-declarative-resource-api` — watch conventions (`GET /v1/watch/{kind-plural}?since=`) this step's SSE endpoint stays consistent with

## Scope

* `GET /v1/projects/{project}/environments/{environment}/services/{service}/endpoints` — synchronous, `Ready`-only by default, optional `?ready=false` to include all, optional `?revision=` filter
* `GET /v1/projects/{project}/environments/{environment}/services/{service}/endpoints/watch?since={resourceVersion}` — SSE stream of add/update/remove events, modeled on epic 20's generic watch shape but scoped to one service
* Discovery also supports the generic epic-20 watch (`GET /v1/watch/endpoints?since=&labelSelector=service=X`) by virtue of mirroring into Control (`21.02`) — this step's scoped watch is a low-latency, Discovery-native alternative that does not require Control to be up
* Client library (`forge-discovery-client`, Go) wrapping list + optional watch with local caching, for product code that talks to peers directly rather than through Gateway

## Explicitly out of scope

* DNS resolution of these endpoints (`21.04`)
* Gateway wiring itself to this API (`21.05`)
* Weighted/percentage selection among ready endpoints (epic 27); this step is presence/absence filtering only, consumer chooses its own balancing (e.g. round robin)

## Architecture

```text
                     ┌─────────────────────────────┐
Product code ───────▶│ GET .../services/x/endpoints │──▶ discovery.endpoints (phase='Ready')
(direct, no Gateway) │ GET .../endpoints/watch      │──▶ SSE: added/updated/removed since resourceVersion
                     └─────────────────────────────┘
                                  ▲
                     forge-discovery-client (Go)
                     list once, cache, follow watch, expose Resolve(service) → []Address
```

## Data model

No new tables. Reads are `SELECT ... FROM discovery.endpoints WHERE project=$1 AND environment=$2 AND service=$3 AND phase='Ready' [AND revision=$4]`, indexed by the existing `idx_endpoints_service_ready`-style composite from `21.01`/`21.02` (`project, environment, service, phase`). The watch broker keeps an in-memory ring buffer of recent change events keyed by `resource_version` for `since` replay; a `since` older than the buffer triggers a full list + `added` events (documented resync behavior, not a new table).

## Configuration

```text
FORGE_DISCOVERY_WATCH_BUFFER_SIZE       # default 500 (events retained per service for `since` replay)
FORGE_DISCOVERY_WATCH_MAX_CONNECTIONS   # default 1000 (SSE connection cap)
FORGE_DISCOVERY_WATCH_HEARTBEAT_SECONDS # default 15 (SSE keep-alive comment ping)
```

## Security considerations

* List/watch are internal (platform trust boundary); `FORGE_AUTH_MODE=dev` until epic 09
* A caller only ever sees endpoints for the project/environment/service in the request path — no cross-project leakage even in dev mode
* SSE connections are capped (`FORGE_DISCOVERY_WATCH_MAX_CONNECTIONS`) to bound resource use from misbehaving clients

## Observability

* Log per watch connect/disconnect: `{event:"discovery.watch.connected"|"discovery.watch.disconnected", service, since}`
* Metrics: `forge_discovery_list_requests_total{service=...}`, `forge_discovery_watch_connections_active`, `forge_discovery_watch_events_total{type=added|updated|removed}`
* Trace spans: `discovery.endpoints.list`, `discovery.endpoints.watch`

## Failure handling

* `since` older than the retained buffer → full resync (list all `Ready` + synthetic `added` events) instead of an error, so a slow/reconnecting client always converges
* Watch broker restart (process restart) → clients reconnect and resync from their last known `resourceVersion`; no missed-event guarantee is made across a Discovery restart, only within one broker lifetime
* List query timeout/DB hiccup → `503` with `Retry-After`; never returns a partial/incorrect Ready set
