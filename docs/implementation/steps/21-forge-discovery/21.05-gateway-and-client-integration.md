## Goal

Let Forge Gateway source its route table from Discovery instead of Control/Runtime, behind a flag that defaults to the shipped epic-05 behavior until parity is proven, and support `Service` aliases as additional hostnames Gateway will route. Flipping the flag is reversible: setting `FORGE_ROUTE_SOURCE` back to `control` restores exactly the epic-05 sync path with no other change.

## Why this step exists

The epic's whole value proposition collapses if Gateway can't actually use Discovery. This step is also the epic's compatibility contract in code: it proves the `sync.Source` facade from epic 05 absorbs a new source with zero changes to `routes.Table`, `DeriveRoutes`, or the existing sources, and that epic 05's own test suite is unaffected.

## Dependencies

* `21.03` — readiness-filtered list API this source polls
* `21.04` — alias resolution semantics this step's Gateway-side alias routing mirrors
* Epic `05-forge-gateway` — `sync.Source` interface, `Syncer`, `BuildSource` (`services/forge-gateway/internal/sync/`)

## Scope

* `sync.DiscoveryEndpointsSource` implementing the existing `Source` interface (`Fetch(ctx) ([]Endpoint, error)`, `Name() string`), calling Discovery's `GET .../services/{service}/endpoints` per known service (service list obtained from Discovery's `Service` listing) and shaping the result identically to `RuntimeInterimSource`'s output (`Service`, `Project`, `Upstreams`, `Ready`) so `DeriveRoutes` is unchanged
* `FORGE_ROUTE_SOURCE` gains a third accepted value: `discovery` (existing `control`/`runtime` behavior byte-for-byte unchanged); default remains `control`
* `BuildSource` gains a `discovery` branch requiring `FORGE_DISCOVERY_URL`
* Gateway-side alias routing: hostnames derived from `Service.spec.aliases` are added to the same route (same upstream set) as the canonical `{service}.{project}` host, additive to `DeriveRoutes`'s existing host-aggregation logic
* Parity checklist: run the existing epic-05 test suite (05.02–05.07) unmodified against `FORGE_ROUTE_SOURCE=discovery` fixtures to confirm no regression before recommending the flip

## Explicitly out of scope

* Changing the default away from `control` (a later, separate operational decision once parity is proven in production-like conditions — explicitly not part of this step)
* Any change to `routes.Table`, `proxy.Handler`, health-aware upstream tracking (`05.04`), or request ID/timeout/WebSocket handling (`05.05`/`05.06`)
* SSE-driven push updates in Gateway (Gateway keeps its existing poll-based `Syncer`; `21.03`'s watch stays available for other consumers only)

## Architecture

```text
services/forge-gateway/internal/sync/
├── endpoints.go     UNCHANGED: ControlEndpointsSource, RuntimeInterimSource, FallbackSource
├── discovery.go     NEW: DiscoveryEndpointsSource (implements Source)
└── sync.go          UNCHANGED except BuildSource gains a "discovery" case

FORGE_ROUTE_SOURCE=control    (default, unchanged)  → ControlEndpointsSource (+ Runtime fallback)
FORGE_ROUTE_SOURCE=runtime    (unchanged)            → RuntimeInterimSource
FORGE_ROUTE_SOURCE=discovery  (new)                  → DiscoveryEndpointsSource ──▶ forge-discovery:4109
```

```text
Reversibility: flipping FORGE_ROUTE_SOURCE=control ↔ discovery is the entire migration;
no code path is removed, no schema changes, no Gateway restart-required data loss.
```

## Components

| Component | Responsibility |
|---|---|
| `sync.DiscoveryEndpointsSource` | `Fetch` from Discovery's list API per known service; `Name()` returns `"discovery"` |
| `sync.BuildSource` (extended) | Adds the `discovery` case; existing cases untouched |
| `DeriveRoutes` (unchanged) | Continues aggregating by host; now also receives alias-derived hosts in the same `[]Endpoint` shape |
| Gateway alias expansion | Maps each `Service.spec.aliases` entry to an additional `Endpoint.Host` sharing the canonical service's upstreams |

## Contracts

Gateway → Discovery (reuses `21.03`'s contract, no new Discovery-side API):

```http
GET /v1/projects/{project}/environments/{environment}/services/{service}/endpoints
```

Gateway config (additive; existing values unchanged):

```text
FORGE_ROUTE_SOURCE=discovery
FORGE_DISCOVERY_URL=http://forge-discovery:8080
```

`sync.Endpoint` shape produced by `DiscoveryEndpointsSource.Fetch` (identical struct already defined in `services/forge-gateway/internal/sync/endpoints.go`):

```json

  "upstreams": [{"url": "http://172.20.0.14:8080"}], "ready": true }
```

Alias expansion example — `Service.spec.aliases: ["legacy-invoice"]` additionally yields:

```json
{ "host": "legacy-invoice.invoice-platform.demo.localhost",
  "service": "invoice-api", "project": "invoice-platform",
  "upstreams": [{"url": "http://172.20.0.14:8080"}], "ready": true }
```

## Security considerations

* Gateway↔Discovery traffic is internal platform traffic; `FORGE_AUTH_MODE=dev` until epic 09
* `DiscoveryEndpointsSource` performs read-only calls; it cannot register/renew/deregister endpoints
* No change to Gateway's edge-facing security posture (still no edge auth in this epic, per epic 05's own scope)

## Observability

* Reuses `sync.Result`/`sync.Syncer` logging unchanged (`route sync complete`, `added`, `removed`, `source: "discovery"`)
* Metric label `source="discovery"` flows through existing route-sync metrics with no new metric names required
* Log on alias expansion: `{event:"gateway.alias.route_added", alias, canonical_service}`

## Failure handling

* Discovery unreachable → `DiscoveryEndpointsSource.Fetch` returns an error → `Syncer.SyncOnce` retains the last-good route table (identical behavior to a `ControlEndpointsSource` failure today — no new failure mode introduced)
* Partial service list (one service's endpoint call fails) → that service's routes are omitted from the derived table for this cycle, logged, retried next interval; other services' routes are unaffected
* Flipping `FORGE_ROUTE_SOURCE` requires a Gateway restart (env var, read at startup) — documented as the only operational step in either direction
