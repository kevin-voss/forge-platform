## Goal

Stand up `services/forge-discovery` as a runnable Go service on host port `4109` with health/readiness/liveness, graceful shutdown, OTEL wiring, and Compose integration, and define the `Service` and `Endpoint` resource kinds against Control's declarative resource API (epic 20). Choose and migrate Discovery's own persistence: a `discovery` Postgres schema that is the fast, authoritative-for-serving store, distinct from the async mirror projected into Control's generic resource store.

## Why this step exists

Every later step in this epic (registration, leases, readiness selection, DNS, Gateway integration) needs a running service, a place to persist endpoints, and a resource-kind identity recognized by the rest of the platform. Building the skeleton and the model first — with no registration logic yet — keeps the persistence and kind-registration decisions isolated and reviewable before behavior is layered on.

## Dependencies

* Epic `20-declarative-resource-api` — kind registration, generic envelope (`metadata`/`spec`/`status`), watch conventions (**required**)
* Epic `00-repository-foundation` — Compose, shared Postgres instance, OTEL collector
* No prior steps in this epic

## Scope

* `services/forge-discovery/` Go module: `main.go`, HTTP server, `/health/live`, `/health/ready`, graceful shutdown on `SIGTERM`
* OTEL trace/metric wiring consistent with the platform convention (`FORGE_OTEL_ENABLED`, `OTEL_EXPORTER_OTLP_ENDPOINT`)
* `Service` and `Endpoint` kind registration against Control's generic kind registry (epic 20), including the cluster-scoped vs project/environment-scoped shape (`Endpoint`/`Service` are project/environment-scoped)
* `discovery` Postgres schema + migration for the base `services`/`endpoints` tables (no lease columns yet — added in `21.02`)
* Compose service block for `forge-discovery` (port `4109:8080`, DB env, depends on `postgres` + `forge-control`)
* OpenAPI skeleton: `contracts/openapi/forge-discovery.openapi.yaml` with health endpoints declared

## Data model

```sql
CREATE SCHEMA IF NOT EXISTS discovery;

CREATE TABLE discovery.services (
  id               TEXT PRIMARY KEY,             -- mirrors Control resource id (svc_...)
  project          TEXT NOT NULL,
  environment      TEXT NOT NULL,
  name             TEXT NOT NULL,
  ports            JSONB NOT NULL DEFAULT '[]',
  aliases          JSONB NOT NULL DEFAULT '[]',
  resource_version TEXT NOT NULL DEFAULT '0',    -- last resourceVersion mirrored to Control
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (project, environment, name)
);

CREATE TABLE discovery.endpoints (
  id               TEXT PRIMARY KEY,             -- = replica id (stable, from epic 07)
  project          TEXT NOT NULL,
  environment      TEXT NOT NULL,
  service          TEXT NOT NULL,
  node_id          TEXT NOT NULL,
  address_ip       TEXT NOT NULL,
  address_port     INT  NOT NULL,
  protocol         TEXT NOT NULL DEFAULT 'http',
  revision         TEXT,
  phase            TEXT NOT NULL DEFAULT 'Pending', -- Pending|Ready|Unready|Terminating
  resource_version TEXT NOT NULL DEFAULT '0',
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  FOREIGN KEY (project, environment, service) REFERENCES discovery.services (project, environment, name)
);
CREATE UNIQUE INDEX idx_endpoints_pk_scope ON discovery.endpoints (project, environment, service, id);
CREATE INDEX idx_endpoints_node ON discovery.endpoints (node_id);
```

## Security considerations

* `POST /v1/kinds` and Discovery's own APIs are internal (platform trust boundary); `FORGE_AUTH_MODE=dev` until epic 09 issues node/workload identities and mTLS
* Discovery's DB credentials are scoped to the `discovery` schema only (same pattern as `DATABASE_SCHEMA=control` on Forge Control) — no cross-schema grants
* No secrets ever stored in `Service`/`Endpoint` spec or status

## Observability

* Structured log on startup: `{event: "discovery.started", port, db_schema, control_url}`
* Log on kind registration: `{event: "discovery.kind_registered", kind, plural, result}`
* Metric: `forge_discovery_up{version=...}` gauge
* Trace span `discovery.startup`

## Failure handling

* Control unreachable at startup → retry kind registration with backoff (max 30s), log each attempt, `/health/ready` stays `503` until registration succeeds once; subsequent Control outages do not crash the process (registration is startup-only in this step)
* DB unreachable → `/health/ready` returns `503`; `/health/live` still returns `200` (process is alive, just not ready)
* Migration failure → process exits non-zero with a clear error (fail fast; never serve against an unmigrated schema)
