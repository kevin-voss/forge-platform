## Goal

Expose a correlation-aware log query API on Forge Observe that filters logs by `project`, `deployment`, `service`, `request_id`, and `trace_id` (and time range + free text), backed by Loki. A developer can pull all logs for one deployment or follow a single request/trace across services from one query.

## Why this step exists

`specs.md` Step 12 requires that "logs can be filtered by project and deployment" and correlation by request/trace IDs. Raw Loki queries require knowing label schemes; Observe provides a Forge-shaped API using the correlation contract (12.01) that services now emit (12.02). This backs the CLI tail (12.05) and demo (12.07).

## Dependencies

* Step `12.01` — correlation model + Loki client
* Step `12.02` — services emit correlation fields into logs
* Foundation `00` — Loki

## Scope

* `GET /v1/logs` query with filters: `project`, `deployment`, `service`, `request_id`, `trace_id`, `since`, `until`, `q` (text), `limit`, `direction`
* Translate filters into LogQL against Loki labels/fields
* Correlated retrieval: given a `trace_id`, return logs from all services in that trace ordered by time
* Pagination (cursor) + result caps to protect Loki
* Authorization: query scoped to projects the caller can access (via Identity when enforced)

## Explicitly out of scope

* Live tail/streaming (12.05 — this is point-in-time query)
* Alert rules (12.06)
* Metrics/trace query APIs (Grafana/Tempo used directly for those)

## Architecture

```text
GET /v1/logs?trace_id=T&project=P → build LogQL:
  {forge_project="P"} | json | trace_id="T"
→ Loki query_range → normalize → paginated results
authz: restrict project filter to caller's accessible projects
```

## Components

| Component | Responsibility |
|---|---|
| `LogQueryService` | Build LogQL from filters; query Loki |
| `FilterValidator` | Validate/normalize filter combos + caps |

## Contracts

* `GET /v1/logs?project=&deployment=&service=&request_id=&trace_id=&since=&until=&q=&limit=&cursor=` →

```json
{ "entries": [ { "time":"...", "service":"control", "trace_id":"...", "request_id":"...",
                 "level":"info", "message":"...", "deployment":"dpl_1" } ],
  "next_cursor": "..." }
```

## Data model

No storage; queries Loki. Result normalization to the correlation field set.

## Configuration

```text
FORGE_LOG_QUERY_MAX_LIMIT      # default 1000
FORGE_LOG_QUERY_MAX_RANGE_H    # default 24 (cap time range)
FORGE_LOKI_URL                 # from 12.01
FORGE_AUTH_MODE                # enforce → scope by project access
```

## Security considerations

* When enforced, a caller can only query logs for projects they belong to (Identity `authz`), preventing cross-tenant log access
* Free-text `q` is escaped to avoid LogQL injection
* Log entries may contain masked secrets already (per 10.06); Observe does not unmask
* Caps prevent expensive/unbounded Loki queries (DoS protection)

## Observability

* Log queries themselves logged with `{filters, result_count, duration_ms}` (no log content echoed)
* Metrics: `forge_log_queries_total`, `forge_log_query_duration_ms`, `forge_log_query_capped_total`
* Trace span `observe.logs.query`

## Failure handling

* Loki unavailable → `503` with a clear message (or degrade to Runtime log stream where feasible — noted for 12.05)
* Invalid filter combo (e.g. no project and no trace/request id) → `400` requiring at least one scoping filter
* Range/limit exceeds caps → clamp + indicate in response, or `400` (documented choice: clamp + warn)

## Acceptance criteria

* `GET /v1/logs` filters by project, deployment, service, request_id, and trace_id (plus time range + text)
* Querying by a single trace_id returns correlated logs from all services in that trace, time-ordered
* Queries require at least one scoping filter and are capped to protect Loki
* When auth is enforced, callers can only query logs for projects they can access
* Free-text search is injection-safe; Loki outage returns a clear `503`
