# Step 12.06: Basic alert rules

## Status

Not started

## Epic

`12-forge-observe` → [`../../epics/12-forge-observe.md`](../../epics/12-forge-observe.md)

## Goal

Provision, as code, basic alert rules that fire on **service down** and **elevated error rate**, evaluated against the metrics from 12.02, with alert state visible in Grafana and delivered to a local sink (log/webhook). Forge Observe exposes the current alert status via an API.

## Why this step exists

`specs.md` Step 12 lists "basic alerts" and acceptance that services emit health metrics usable for alerting. Alerts turn passive dashboards into active notification when something breaks — the last operational primitive before the observability demo.

## Dependencies

* Step `12.02` — `forge_service_up`, request/error metrics
* Step `12.03` — dashboards to display alert state
* Foundation `00` — Prometheus (alerting rules) / Grafana alerting / Alertmanager or Loki ruler

## Scope

* Alert rules as code:
  * `ServiceDown`: `forge_service_up == 0` for > `Xs`
  * `HighErrorRate`: error ratio (5xx / total) > threshold for > `Xs`
  * (optional) `NodeOffline` from runtime metrics
* Provision rules into Prometheus (or Grafana unified alerting)
* Local delivery sink: a webhook receiver (log/print) so firing alerts are observable without external paging
* Observe alert status API: `GET /v1/alerts` (active/pending alerts with labels) proxying the alerting backend
* Documented thresholds + how to tune them

## Explicitly out of scope

* SLOs / error budgets / burn-rate alerts
* External pager integrations (PagerDuty/OpsGenie)
* Auto-remediation (workflows epic 16 may consume alerts later)

## Architecture

```text
Prometheus rules (as code) evaluate forge_service_up / error ratio
   → firing → Alertmanager/webhook sink (local log receiver)
Grafana shows alert state on dashboards
Observe GET /v1/alerts → query alerting backend → normalized list
```

## Components

| Component | Responsibility |
|---|---|
| alert rules files | ServiceDown / HighErrorRate definitions |
| local alert sink | Receive + log firing alerts |
| `AlertStatusRoute` | `GET /v1/alerts` from Observe |
| rule provisioning | Load rules on startup |

## Contracts

* `GET /v1/alerts` → `[ { name, state:firing|pending, labels:{service,project,...}, since, value } ]`
* Alert rules query metric names from 12.02
* Webhook sink receives standard Alertmanager webhook payloads

## Data model

None persisted by Observe (queried live). Alert rules are provisioning files.

## Configuration

```text
FORGE_ALERT_SERVICE_DOWN_FOR      # e.g. 30s
FORGE_ALERT_ERROR_RATE_THRESHOLD  # e.g. 0.05
FORGE_ALERT_ERROR_RATE_FOR        # e.g. 60s
FORGE_ALERTMANAGER_URL            # backend for status queries
```

## Security considerations

* Alert labels must not include secrets/PII (bounded label set from 12.02)
* `GET /v1/alerts` read-only; authorized when auth enforced (project-scoped where labels allow)
* Local webhook sink is dev-only; documented as not for production paging

## Observability

* The alert pipeline is itself observable: a self-test alert can be triggered and observed in `/v1/alerts` + the sink
* Metrics: `forge_alerts_firing` gauge by alertname
* Log alert transitions (firing/resolved)

## Failure handling

* Alerting backend unreachable → `GET /v1/alerts` returns `503`; dashboards still render
* Rule file syntax error → provisioning fails the build (caught by test)
* Metric absent (service never came up) → `ServiceDown` still fires via `absent()`/up==0 semantics (documented)
