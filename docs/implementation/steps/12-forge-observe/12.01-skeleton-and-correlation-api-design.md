## Goal

Create the Forge Observe service: a Go HTTP service on host port `4106` satisfying the runtime contract, connected to the foundation telemetry backends (Loki/Tempo/Prometheus), and define the **correlation model** — the standard trace/request IDs and resource attributes (`forge.project`, `forge.deployment`, `forge.service`, `forge.node`) that every service will emit and that all queries key on.

## Why this step exists

Instrumentation (12.02), dashboards (12.03), and log query (12.04) all depend on a single agreed correlation model and a running Observe service that can reach the backends. Designing the correlation contract first prevents each service inventing incompatible labels.

## Dependencies

* Foundation `00` — OTEL Collector + Loki + Tempo + Prometheus + Grafana
* Epic `01-runtime-contract` — structured-log conventions
* No prior steps in this epic

## Scope

* Go HTTP service (`/health/live`, `/health/ready`, identity `/`, structured logs, env config, SIGTERM)
* Backend clients: read-only clients for Loki (logs), Tempo (traces), Prometheus (metrics) with health checks
* Correlation model document + shared definitions: `traceparent` propagation, `X-Forge-Request-ID`, resource attributes, log field names
* `GET /v1/health/backends` reporting Loki/Tempo/Prometheus reachability
* A published correlation contract under `docs/contracts/` + a small shared constants module other services import
* Dockerfile + Compose service `forge-observe` (4106); skeleton OpenAPI

## Explicitly out of scope

* Applying instrumentation to Control/Runtime/Gateway/Build (12.02)
* Dashboards (12.03)
* Log query API (12.04)
* CLI tail (12.05), alerts (12.06)

```text
services/forge-observe/
├── cmd/forge-observe/main.go
├── internal/config/config.go

docs/contracts/observability-correlation.md         # the correlation contract
```

Correlation contract (normative):

```text
Headers:  traceparent (W3C), X-Forge-Request-ID
Resource attributes / log fields:
  forge.project, forge.deployment, forge.service, forge.node,
  trace_id, span_id, request_id
```

## Contracts

* `GET /health/live` / `GET /health/ready` (ready requires backends reachable)
* `GET /` identity `{ "service":"forge-observe","language":"go","status":"running" }`
* `GET /v1/health/backends` → `{ loki:ok, tempo:ok, prometheus:ok }`
* `docs/contracts/observability-correlation.md` — the correlation contract
* `contracts/openapi/forge-observe.openapi.yaml` — skeleton

## Data model

No domain DB. Telemetry lives in the backends. The service is a query/correlation layer.

## Configuration

```text
PORT                    # default 4106

FORGE_LOKI_URL          # http://loki:3100

FORGE_OBSERVE_READY_REQUIRE_BACKENDS  # loki,tempo,prometheus (default all)
```

## Security considerations

* Read-only access to backends; no write/delete of telemetry from Observe
* Correlation attributes must not include secrets/PII (documented; e.g. no raw payloads, no tokens)
* Health/identity endpoints unauthenticated per contract; query APIs authorized later (12.04)

## Observability

* Observe instruments itself using the same correlation model (dogfooding)
* Metric: `forge_observe_backend_up{backend}`; log backend connectivity changes
* Trace span `observe.backend.health`

## Failure handling

* A required backend unreachable → `/health/ready` non-200; degraded mode documented (which features still work)
* Backend timeout → typed error from the client; never hangs a request indefinitely (context deadlines)
* SIGTERM → drain + exit 0
