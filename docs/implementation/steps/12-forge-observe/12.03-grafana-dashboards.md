## Goal

Provision, as code, four Grafana dashboards that load automatically: a **platform** overview (all services health/traffic/errors), a **service** dashboard (per-service latency/errors/throughput), a **deployment** dashboard (rollout status, replicas, per-deployment metrics), and a **runtime** dashboard (node capacity/allocation/health). They read the metrics emitted in 12.02.

## Why this step exists

`specs.md` Step 12 requires platform/service/deployment/runtime dashboards and that "predefined dashboards load automatically." Dashboards turn raw metrics into the operational views developers actually use and are a named acceptance criterion.

## Dependencies

* Step `12.02` — standard metrics + correlation labels emitted by services
* Foundation `00` — Grafana + Prometheus with provisioning support
* Epic `07`/`08` — deployment/node metrics for the deployment/runtime dashboards

## Scope

* Grafana provisioning: dashboards-as-code (JSON) + a provider config so they load on startup
* Four dashboards:
  * Platform: service up/down, aggregate request rate, error rate, p95 latency
  * Service: per-service (template variable) latency histogram, error %, throughput, log volume
  * Deployment: rollout phase, desired vs actual replicas, per-deployment error rate (uses `forge.deployment`)
  * Runtime: node count, per-node free/allocated slots, heartbeat age, offline nodes (uses `forge.node`)
* Datasource provisioning references (Prometheus/Loki/Tempo) if not already provided by `00`
* A provisioning test asserting dashboards load without errors

## Explicitly out of scope

* Log query API (12.04)
* Alert rules (12.06) — dashboards may show alert state but rules are next
* Custom per-user dashboards

## Architecture

```text
deploy/observability/grafana/
├── provisioning/dashboards/forge.yaml     # provider → load *.json
├── dashboards/platform.json
├── dashboards/service.json
├── dashboards/deployment.json
└── dashboards/runtime.json
```

Grafana auto-loads from the provisioning path at startup.

## Components

| Component | Responsibility |
|---|---|
| dashboard provider config | Tell Grafana where to load dashboards |
| platform.json | Fleet-wide health/traffic/errors |
| service.json | Per-service drill-down (templated) |
| deployment.json | Rollout + replica + per-deployment metrics |
| runtime.json | Node capacity + liveness |

## Contracts

* Dashboards query metric names defined in `12.02` (`forge_service_up`, `forge_http_requests_total`, `forge_http_request_duration_seconds`, `forge_replicas_ready`, `forge_node_free_slots`, etc.)
* No HTTP API; artifacts are provisioning files

## Data model

None. Panels are backed by Prometheus/Loki queries.

## Configuration

```text
GF_PATHS_PROVISIONING          # Grafana provisioning dir (from 00)
FORGE_GRAFANA_DASHBOARD_DIR    # where forge dashboards mount
```

## Security considerations

* Dashboards are read-only views; no secrets embedded in panel queries or titles
* Single local Grafana org; access control is deployment-environment concern (out of scope)

## Observability

* Dashboards are the observability surface; a smoke test verifies each renders with data
* Provisioning errors surfaced in Grafana logs (checked by the test)

## Failure handling

* Missing metric → panel shows "no data" rather than erroring the dashboard
* Provisioning file JSON error → Grafana logs it; the provisioning test fails the build
* Datasource unavailable → panels degrade to no-data (dashboards still load)

## Implementation tasks

1. Add Grafana dashboard provider config pointing at the forge dashboards dir.
2. Author `platform.json` (up/down, request rate, error rate, p95).
3. Author `service.json` with a `service` template variable (latency/error/throughput/log volume).
4. Author `deployment.json` using `forge.deployment` (rollout phase, replicas desired vs actual).
5. Author `runtime.json` using `forge.node` (slots free/allocated, heartbeat age, offline count).
6. Add a provisioning smoke test (dashboards load, key panels return data in the demo stack).
