## Goal

Apply a shared instrumentation checklist to the four core services (Control, Runtime, Gateway, Build) so they propagate trace context, emit structured logs carrying the correlation attributes, and export health/latency/error metrics — all to the foundation OTEL collector. After this step, a request that crosses these services shares one trace ID.

## Why this step exists

Observe (12.01) defined the correlation model but the services don't emit it yet. `specs.md` Step 12 requires "every completed service must be instrumented" and "one request can be followed across services." This step is the actual instrumentation work that makes distributed tracing and correlated logs real.

## Dependencies

* Step `12.01` — correlation contract + shared constants + backends
* Epics `02`/`04`/`05`/`06` — the services being instrumented
* Foundation `00` — OTEL collector endpoint

## Scope

* A written instrumentation checklist (in `docs/contracts/` / `docs/testing/`) every service must satisfy
* For each of Control, Runtime, Gateway, Build:
  * Initialize the OTEL SDK exporting to the collector (traces + metrics)
  * Extract inbound `traceparent`; create a server span; inject `traceparent` on outbound calls (context propagation)
  * Attach resource attributes `forge.project`/`forge.deployment`/`forge.service`/`forge.node` where known
  * Add `trace_id`/`span_id`/`request_id` to structured log lines
  * Emit standard metrics: request count, latency histogram, error count, health up gauge
* Gateway generates/propagates `X-Forge-Request-ID` at the edge if absent
* Telemetry export must be fail-open (never crash the service)

## Explicitly out of scope

* Dashboards (12.03)
* Log query API (12.04)
* CLI tail (12.05), alerts (12.06)
* Instrumenting services beyond these four (they follow the checklist when built)

## Architecture

```text
Gateway (edge): ensure traceparent + X-Forge-Request-ID → span → propagate
   ↓ (headers)
Control → Build / Runtime → ... each: extract → span → propagate → log with trace_id
All export spans+metrics → OTEL Collector → Tempo/Prometheus; logs → Loki (with trace_id)
```

## Components

| Component | Responsibility |
|---|---|
| Instrumentation checklist | Normative per-service requirements |
| OTEL bootstrap (per service) | SDK init + exporter to collector |
| Propagation middleware | Extract/inject `traceparent` + request id |
| Log enricher | Add correlation fields to structured logs |
| Standard metrics | request/latency/error/up per service |

## Contracts

* Consumes the correlation contract from `12.01`
* Outbound calls carry `traceparent` + `X-Forge-Request-ID`
* Logs include `trace_id`, `span_id`, `request_id`, `forge.*` fields
* No new HTTP API on Observe here

## Data model

None (telemetry export). Resource attributes attached at SDK init + per-request.

## Configuration

```text
# per service
FORGE_OTEL_EXPORTER_ENDPOINT   # otel collector (e.g. http://otel-collector:4317)
FORGE_OTEL_ENABLED             # default true (fail-open if endpoint down)
FORGE_SERVICE_NAME             # resource attribute service.name
FORGE_ENV                      # deployment.environment attribute
```

## Security considerations

* Never attach secrets/tokens/PII to spans, metrics labels, or logs (checklist rule; enforced by review + a lint/test)
* Bounded metric label cardinality (no request id in metric labels)
* Request IDs are opaque and safe to log

## Observability

* This step *is* observability: after it, traces span services and logs carry trace ids
* Add a self-test span from Gateway→Control to verify propagation
* Metric: `forge_service_up`, `forge_http_requests_total`, `forge_http_request_duration_seconds`

## Failure handling

* Collector unreachable → SDK buffers/drops without blocking; service stays healthy (fail-open) — verified by a test that kills the collector
* Missing inbound `traceparent` → generate a new root trace (edge) so every request is traced
* Malformed `traceparent` → start a new trace, log a debug note

## Manual verification

```bash
make dev
# make a request through the gateway that reaches control
RID=$(curl -s -D - -H 'Host: demo.localhost' localhost:4000/ -o /dev/null | awk -F': ' '/X-Forge-Request-ID/{print $2}' | tr -d '\r')
echo "request id: $RID"
# find the trace in tempo via observe backend health path or grafana; confirm multiple services in one trace
curl -s "localhost:3200/api/search?tags=forge.service" 2>/dev/null | jq '.traces | length' # traces present
# collector down → service still healthy (fail-open)
docker stop otel-collector; sleep 3
curl -s -o /dev/null -w '%{http_code}\n' -H 'Host: demo.localhost' localhost:4000/  # still 200
docker start otel-collector
```
